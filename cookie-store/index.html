1. 介绍

本节为非规范性内容。

这是一个提议，旨在为在HTML文档和服务工作线程中运行的脚本引入异步Cookie API。

HTTP cookies 自其在Netscape诞生以来（由archive.org保存的文档），为网络提供了宝贵的状态管理机制。

同步的单线程脚本级document.cookie 接口已成为复杂性和性能问题的来源，随着许多浏览器从以下模式转变：

  • 单个浏览器进程，

  • 单线程事件循环模型，和

  • 在处理cookie操作时没有对脚本事件处理响应的普遍期望

……到现代网络，追求平滑的响应和高性能：

  • 在多个浏览器进程中，

  • 使用多线程、多个事件循环模型，和

  • 期望在人体反射时间尺度内的响应。

在现代网络中，一个部分的cookie操作不能阻塞：

  • 网络应用的其余部分，

  • 同一网络源的其他部分，或

  • 整个浏览器。

新构建在服务工作线程中的网络部分也需要访问cookie，但无法使用同步的、阻塞的document.cookie 接口，因为它们没有document，且不能阻塞事件循环，否则会干扰不相关事件的处理。

1.1. 建议更改的概述

虽然重新考虑cookie的想法很诱人，但当今的网站仍然高度依赖cookie，并且使用它们的脚本API在几十年的使用中几乎没有改变。

如今，写入cookie意味着在等待浏览器同步更新cookie存储时阻塞事件循环，使用精心构造的cookie字符串以Set-Cookie格式：

document.cookie =
  '__Secure-COOKIENAME=cookie-value' +
  '; Path=/' +
  '; expires=Fri, 12 Aug 2016 23:05:17 GMT' +
  '; Secure' +
  '; Domain=example.org';
// 我们可以假设写入成功了，但由于失败时没有任何提示，因此很难判断是否成功 
// 因此我们读取来确认写入是否成功
var successRegExp =
  /(^|; ?)__Secure-COOKIENAME=cookie-value(;|$)/; 
if (String(document.cookie).match(successRegExp)) {
  console.log('成功了！'); 
} else { 
  console.error('失败了，我们不知道原因'); 
}

如果你能改写成：

const one_day_ms = 24 * 60 * 60 * 1000; 
cookieStore.set( 
  { 
    name: '__Secure-COOKIENAME', 
    value: 'cookie-value', 
    expires: Date.now() + one_day_ms, 
    domain: 'example.org' 
  }).then(function() { 
    console.log('成功了！'); 
  }, function(reason) { 
    console.error( 
      '失败了，原因是：', 
      reason); 
  }); 
// 同时，我们可以在等待cookie存储处理写入时做其他事情...

这种方式还具有不依赖document且不会阻塞的优点，这使得它可以在服务工作线程中使用，而后者的脚本无法访问cookie。

该提案还包括一个节能的监控API，用于替换基于setTimeout轮询的cookie监控器，采用cookie更改观察器。

1.2. 摘要

本提案概述了使用Promises/async函数的异步API，用于以下cookie操作：

  • 写入（或“设置”）和删除（或“过期”）cookie

  • 读取（或“获取”）脚本可见的cookie

    • ...包括在服务工作线程上下文中指定范围请求路径的cookie

  • 监控脚本可见的cookie更改，使用CookieChangeEvent

    • ...在长时间运行的脚本上下文（例如document）中

    • ...在服务工作线程上下文中，针对脚本提供的指定范围请求路径

1.3. 查询Cookie

文档和服务工作线程都通过全局对象上的cookieStore属性访问相同的查询API。

get() 和getAll()方法用于查询cookie。 这两个方法都返回Promise。 两个方法都接受相同的参数，可以是：

  • 一个名称，或

  • 一个选项字典（对于getAll()是可选的）

get()方法本质上是getAll()的一个形式，它只返回第一个结果。

读取cookie： 
try { 
  const cookie = await cookieStore.get('session_id'); 
  if (cookie) { 
    console.log(`找到 ${cookie.name} cookie: ${cookie.value}`); 
  } else { 
    console.log('未找到cookie'); 
  } 
} catch (e) { 
  console.error(`Cookie存储错误: ${e}`); 
}
读取多个cookie： 
try { 
  const cookies = await cookieStore.getAll('session_id'}); 
  for (const cookie of cookies) 
    console.log(`结果: ${cookie.name} = ${cookie.value}`); 
} catch (e) { 
  console.error(`Cookie存储错误: ${e}`); 
}

服务工作线程可以获取将在fetch到其范围下的任意URL时发送的cookie列表。

读取特定URL的cookie（在服务工作线程中）： 
await cookieStore.getAll({url: '/admin'});

文档只能获取当前URL下的cookie。换句话说，文档上下文中唯一有效的url值就是文档的URL。

get() 和getAll()方法返回的对象包含cookie存储中的所有相关信息，而不仅仅是旧API中document.cookie的名称和值。

访问所有cookie数据： 
await cookie = cookieStore.get('session_id'); 
console.log(`Cookie范围 - 域名: ${cookie.domain} 路径: ${cookie.path}`); 
if (cookie.expires === null) { 
  console.log('cookie将在会话结束时过期'); 
} else { 
  console.log(`cookie过期时间: ${cookie.expires}`); 
} 
if (cookie.secure) 
  console.log('此cookie仅限于安全源');

1.4. 修改Cookie

文档和服务工作线程都通过全局对象上的cookieStore属性访问相同的修改API。

使用set()方法来创建或修改（写入）cookie。

写入一个cookie： 
try { 
  await cookieStore.set('opted_out', '1'); 
} catch (e) { 
  console.error(`设置cookie失败: ${e}`); 
}

上面的set()调用是使用选项字典的简写形式，如下所示： 

await cookieStore.set({ 
  name: 'opted_out', 
  value: '1', 
  expires: null,  // 会话cookie 

  // 默认情况下，domain设置为null，这意味着作用域仅限于当前域。 
  domain: null, 
  path: '/' 
});

使用delete()方法删除（过期）cookie。

删除一个cookie： 
try { 
  await cookieStore.delete('session_id'); 
} catch (e) { 
  console.error(`删除cookie失败: ${e}`); 
}

实际上，删除cookie是通过将cookie的过期日期改为过去的日期来完成的，这种方式仍然有效。

通过更改过期日期删除cookie： 
try { 
  const one_day_ms = 24 * 60 * 60 * 1000; 
  await cookieStore.set({ 
    name: 'session_id', 
    value: '此值将被忽略', 
    expires: Date.now() - one_day_ms }); 
} catch (e) { 
  console.error(`删除cookie失败: ${e}`); 
}

1.5. 监控Cookie

为了避免轮询，可以观察cookie的变化。

在文档中，change事件将在所有相关的cookie更改时触发。

在文档中注册change事件： 
cookieStore.addEventListener('change', event => { 
  console.log(`${event.changed.length} 个cookie已更改`); 
  for (const cookie in event.changed) 
    console.log(`Cookie ${cookie.name} 已更改为 ${cookie.value}`); 

  console.log(`${event.deleted.length} 个cookie已删除`); 
  for (const cookie in event.deleted) 
    console.log(`Cookie ${cookie.name} 已删除`); 
});

在服务工作线程中，cookiechange事件会在全局范围内触发，但需要与服务工作线程的注册相关联的显式订阅。

在服务工作线程中注册cookiechange事件： 
self.addEventListener('activate', (event) => { 
  event.waitUntil(async () => { 
    // 快照当前的订阅状态。 
    const subscriptions = await self.registration.cookies.getSubscriptions(); 

    // 清除任何现有订阅。 
    await self.registration.cookies.unsubscribe(subscriptions); 

    await self.registration.cookies.subscribe([ 
      { 
        name: 'session_id',  // 获取名为session_id的cookie更改事件。 
      } 
    ]); 
  }); 
}); 

self.addEventListener('cookiechange', event => { 
  // 事件具有 |changed| 和 |deleted| 属性，其语义与文档事件相同。 
  console.log(`${event.changed.length} 个cookie已更改`); 
  console.log(`${event.deleted.length} 个cookie已删除`); 
});

调用subscribe()是累积的，因此独立维护的模块或库可以设置它们自己的订阅。正如预期的那样，服务工作线程的订阅会随服务工作线程注册一起保存。

订阅可以使用与get()和getAll()相同的选项。细粒度订阅的复杂性是合理的，因为向服务工作线程分发无关cookie更改事件的成本远高于向窗口分发等效事件的成本。 具体来说，向服务工作线程分发事件可能需要唤醒工作线程，这对电池寿命有显著影响。

getSubscriptions()允许服务工作线程检查已进行的订阅。

检查更改订阅： 
const subscriptions = await self.registration.cookies.getSubscriptions(); 
for (const sub of subscriptions) { 
  console.log(sub.name, sub.url); 
}

2. 概念

2.1. Cookie

cookie 在用户代理中规范定义于Cookies § 用户代理要求。

根据Cookies § 存储模型，cookie具有以下字段：名称、值、过期时间、域名、路径、 创建时间、最后访问时间、持久化标志、仅限主机标志、仅限安全标志、HttpOnly标志、SameSite标志。

当cookie在作用域内且未设置HttpOnly标志时，它是脚本可见的。这在处理模型中更为正式地执行，适时参考Cookies § 检索模型。

cookie还受某些大小限制。根据Cookies § 存储模型：

  • 名称和值字段的总长度不得超过4096字节（最大名称/值对大小）。

  • 除名称和值字段外，每个字段的长度不得超过1024字节（最大属性值大小）。

注意： cookie属性值存储为字节序列，而非字符串。

2.2. Cookie 存储

cookie存储 在用户代理中规范定义于Cookies § 用户代理要求。

当cookie存储发生以下任一情况时，执行处理cookie更改的步骤：

  • 将新创建的cookie插入cookie存储。

  • 用户代理从cookie存储中驱逐过期的cookie。

  • 用户代理从cookie存储中移除多余的cookie。

2.3. 对服务工作线程的扩展

[Service-Workers] 定义了服务工作线程注册，本规范对其进行了扩展。

每个服务工作线程注册都关联了一个cookie更改订阅列表，该列表是列表；每个成员都是一个cookie更改订阅。cookie更改订阅是一个元组，包含名称和URL。

3. CookieStore 接口

[Exposed=(ServiceWorker,Window),
 SecureContext]
interface CookieStore : EventTarget {
  Promise<CookieListItem?> get(USVString name);
  Promise<CookieListItem?> get(optional CookieStoreGetOptions options = {});

  Promise<CookieList> getAll(USVString name);
  Promise<CookieList> getAll(optional CookieStoreGetOptions options = {});

  Promise<undefined> set(USVString name, USVString value);
  Promise<undefined> set(CookieInit options);

  Promise<undefined> delete(USVString name);
  Promise<undefined> delete(CookieStoreDeleteOptions options);

  [Exposed=Window]
  attribute EventHandler onchange;
};

dictionary CookieStoreGetOptions {
  USVString name;
  USVString url;
};

enum CookieSameSite {
  "strict",
  "lax",
  "none"
};

dictionary CookieInit {
  required USVString name;
  required USVString value;
  DOMHighResTimeStamp? expires = null;
  USVString? domain = null;
  USVString path = "/";
  CookieSameSite sameSite = "strict";
  boolean partitioned = false;
};

dictionary CookieStoreDeleteOptions {
  required USVString name;
  USVString? domain = null;
  USVString path = "/";
  boolean partitioned = false;
};

dictionary CookieListItem {
  USVString name;
  USVString value;
  USVString? domain;
  USVString path;
  DOMHighResTimeStamp? expires;
  boolean secure;
  CookieSameSite sameSite;
  boolean partitioned;
};

typedef sequence<CookieListItem> CookieList;

3.1. get() 方法

cookie = await cookieStore . get(name)
cookie = await cookieStore . get(options)

返回一个promise，解析为给定cookie名称（或其他选项）的第一个在范围内的脚本可见值。 在服务工作线程的上下文中，默认路径为服务工作线程注册的作用域路径。 在文档中，它默认使用当前文档的路径，并且不会遵循replaceState() 或document.domain的更改。

get(name)方法的步骤如下：

  1. 令settings为this的相关设置对象。

  2. 令origin为settings的源。

  3. 如果origin是不透明源，则返回被拒绝的promise，并带有一个"SecurityError" DOMException。

  4. 令url为settings的创建URL。

  5. 令p为一个新的promise。

  6. 并行运行以下步骤：

    1. 令list为使用url和name运行查询cookie的结果。

    2. 如果list失败，则拒绝 p，并返回一个TypeError，并中止这些步骤。

    3. 如果list为空，则解析 p为null。

    4. 否则，解析p为list的第一个项目。

  7. 返回p。

get(options) 方法的步骤如下：

  1. 令settings为this的相关设置对象。

  2. 令origin为settings的源。

  3. 如果origin是不透明源，则返回被拒绝的promise，并带有一个"SecurityError" DOMException。

  4. 令url为settings的创建URL。

  5. 如果options为空，则返回被拒绝的promise，并带有一个TypeError。

  6. 如果options中存在"url"，则执行以下步骤：

    1. 令parsed为使用settings的基本URL解析器解析options["url"]的结果。

    2. 如果this的相关全局对象是一个窗口对象，并且parsed不等于url，则返回被拒绝的promise，并带有一个TypeError。

    3. 如果parsed的源和url的源不是同源，则返回被拒绝的promise，并带有一个TypeError。

    4. 将url设置为parsed。

  7. 令p为一个新的promise。

  8. 并行运行以下步骤：

    1. 令list为使用url和options["name"]（如果存在）运行查询cookie的结果。

    2. 如果list失败，则拒绝p，并返回一个TypeError，并中止这些步骤。

    3. 如果list为空，则解析p为null。

    4. 否则，解析p为list的第一个项目。

  9. 返回p。

3.3. set() 方法

await cookieStore . set(options)

写入（创建或修改）一个cookie。

选项的默认值为：

  • 路径：/

  • 域名：与当前文档或服务工作线程位置的域相同

  • 没有过期日期

  • SameSite：strict

set(name, value)方法的步骤如下：

  1. 令settings为this的相关设置对象。

  2. 令origin为settings的源。

  3. 如果origin是不透明源，则返回被拒绝的promise，并带有一个"SecurityError" DOMException。

  4. 令url为settings的创建URL。

  5. 令domain为null。

  6. 令path为"/"。

  7. 令sameSite为strict。

  8. 令partitioned为false。

  9. 令p为一个新的promise。

  10. 并行运行以下步骤并行：

    1. 令r为运行设置cookie的结果，参数为url、name、 value、domain、path、sameSite和 partitioned。

    2. 如果r失败，则拒绝 p，并返回一个TypeError， 并中止这些步骤。

    3. 解析p为undefined。

  11. 返回p。

  1. 令settings为this的相关设置对象。

  2. 令origin为settings的源。

  3. 如果origin是不透明源，则返回被拒绝的promise，并带有一个"SecurityError" DOMException。

  4. 令url为settings的创建URL。

  5. 令p为一个新的promise。

  6. 并行运行以下步骤并行：

    1. 令r为运行设置cookie的结果，参数为url、 options["name"]、 options["value"]、 options["expires"]、 options["domain"]、 options["path"]、 options["sameSite"]、 和options["partitioned"]。

    2. 如果r失败，则拒绝 p，并返回一个TypeError， 并中止这些步骤。

    3. 解析p为undefined。

  7. 返回p。

3.4. delete() 方法

await cookieStore . delete(name)
await cookieStore . delete(options)

删除（过期）具有给定名称或名称、可选域名和路径的cookie。

delete(name)方法的步骤如下：

  1. 令settings为this的相关设置对象。

  2. 令origin为settings的源。

  3. 如果origin是不透明源，则返回被拒绝的promise，并带有一个"SecurityError" DOMException。

  4. 令url为settings的创建URL。

  5. 令p为一个新的promise。

  6. 并行运行以下步骤并行：

    1. 令r为运行删除cookie的结果，参数为url、 name、 null、 "/"、 true 和 "strict"。

    2. 如果r失败，则拒绝 p，并返回一个TypeError， 并中止这些步骤。

    3. 解析p为undefined。

  7. 返回p。

delete(options)方法的步骤如下：

  1. 令settings为this的相关设置对象。

  2. 令origin为settings的源。

  3. 如果origin是不透明源，则返回被拒绝的promise，并带有一个"SecurityError" DOMException。

  4. 令url为settings的创建URL。

  5. 令p为一个新的promise。

  6. 并行运行以下步骤并行：

  7. 返回p。

7.1. 查询 Cookie

要使用 url 和可选的 name 查询 Cookie， 请按以下步骤进行：

  1. 执行 Cookies § 检索模型 中定义的步骤，以计算来自指定 cookie 存储的 "cookie-string"， 将 url 作为 request-uri。 cookie-string 本身被忽略，但中间的 cookie-list 会在后续步骤中使用。

    对于这些步骤，cookie-string 是为 "非 HTTP" API 生成的。

  2. 让 list 成为一个新的 列表。

  3. 对每一个 cookie 在 cookie-list 中，运行这些步骤：

    1. 断言：cookie 的 http-only-flag 为 false。

    2. 如果给定了 name，则运行以下步骤：

      1. 让 cookieName 成为在 cookie 的 name 上运行 UTF-8 无 BOM 解码 的结果。

      2. 如果 cookieName 不等于 name， 则 继续。

    3. 让 item 成为从 cookie 创建 CookieListItem 的结果。

    4. 将 item 添加到 list 中。

  4. 返回 list。

要从 cookie 创建 CookieListItem，运行以下步骤。

  1. 让 name 成为在 cookie 的 UTF-8 无 BOM 解码 的 name 的结果。

  2. 让 value 成为在 cookie 的 UTF-8 无 BOM 解码 的 value 的结果。

  3. 让 domain 成为在 cookie 的 UTF-8 无 BOM 解码 的 domain 的结果。

  4. 让 path 成为在 cookie 的 UTF-8 无 BOM 解码 的 path 的结果。

  5. 让 expires 成为 cookie 的 到期时间（作为时间戳）。

  6. 让 secure 成为 cookie 的 secure-only-flag。

  7. 根据 cookie 的 same-site-flag 切换：

    `None`

    让 sameSite 为 "none"。

    `Strict`

    让 sameSite 为 "strict"。

    `Lax`

    让 sameSite 为 "lax"。

  8. 让 partitioned 为一个布尔值，表示用户代理是否支持 cookie 分区，以及该 cookie 是否具有分区键。

  9. 返回 «[ "name" → name, "value" → value, "domain" → domain, "path" → path, "expires" → expires, "secure" → secure, "sameSite" → sameSite, "partitioned" → partitioned ]»

注意： cookie 的 创建时间、上次访问时间、持久标志、主机专用标志 和 http-only-flag 属性不向脚本公开。

7.2. 设置 Cookie

要使用 url、name、value，可选的 expires、domain、path、sameSite 和 partitioned 来设置一个 cookie，请执行以下步骤：

  1. 如果 name 或 value 包含 U+003B (;)、任何 C0 控制字符 除了 U+0009（水平制表符字符）或 U+007F，则返回失败。

    注意，这是否也应该适用于 expires、domain、path 和 sameSite 的字符限制尚待讨论。[Issue #httpwg/http-extensions#1593]

  2. 如果 name 的 长度 为 0 且 value 包含 U+003D (=)，则返回失败。

  3. 如果 name 的 长度 为 0 且 value 的 长度 为 0，则返回失败。

  4. 让 encodedName 成为 UTF-8 编码 name 的结果。

  5. 让 encodedValue 成为 UTF-8 编码 value 的结果。

  6. 如果 encodedName 的 字节序列 和 encodedValue 的字节序列的长度之和大于 最大名称/值对大小，则返回失败。

  7. 让 host 成为 url 的 主机。

  8. 让 attributes 成为一个新的 列表。

  9. 如果 domain 不为 null，则运行以下步骤：

    1. 如果 domain 以 U+002E (.) 开头，则返回失败。

    2. 如果 host 不等于 domain 且 host 不是以 U+002E (.) 和 domain 结尾，则返回失败。

    3. 让 encodedDomain 成为 UTF-8 编码 domain 的结果。

    4. 如果 encodedDomain 的字节序列长度大于 最大属性值大小，则返回失败。

    5. 将 `Domain`/encodedDomain 添加到 attributes 中。

  10. 如果给定了 expires，则 将 `Expires`/expires（日期序列化）添加到 attributes。

  11. 如果 path 不以 U+002F (/) 开头，则返回失败。

  12. 如果 path 不以 U+002F (/) 结尾，则将 U+002F (/) 添加到 path。

  13. 让 encodedPath 成为 UTF-8 编码 path 的结果。

  14. 如果 encodedPath 的字节序列长度大于 最大属性值大小，则返回失败。

  15. 将 `Path`/encodedPath 添加到 attributes 中。

  16. 将 `Secure`/`` 添加到 attributes 中。

  17. 根据 sameSite 切换：

    "none"

    将 `SameSite`/`None` 添加到 attributes。

    "strict"

    将 `SameSite`/`Strict` 添加到 attributes。

    "lax"

    将 `SameSite`/`Lax` 添加到 attributes。

  18. 如果 partitioned 为 true，将 `Partitioned`/`` 添加到 attributes。

  19. 执行在 Cookies § 存储模型 中定义的步骤，用户代理在接收到 url 作为 request-uri，encodedName 作为 cookie-name，encodedValue 作为 cookie-value，attributes 作为 cookie-attribute-list 时。

    为了这些步骤的目的，新创建的 cookie 是从 "非 HTTP" API 接收到的。

  20. 返回成功。

    注意： 由于 [RFC6265BIS-14] 中的要求，存储 cookie 可能仍然会失败，但这些步骤将被视为成功。

7.3. 删除 Cookie

要使用 url、name、domain、path 和 partitioned 删除一个 cookie，请执行以下步骤：

  1. 如果 path 不为 null，则运行以下步骤：

    1. 如果 path 不以 U+002F (/) 开头，则返回失败。

    2. 如果 path 不以 U+002F (/) 结尾，则将 U+002F (/) 添加到 path。

  2. 让 expires 成为最早可表示的日期，并表示为 时间戳。

    注意： expires 的确切值对这个算法的目的不重要，只要它在过去即可。

  3. 让 value 成为空字符串。

  4. 让 sameSite 成为 "strict"。

    注意： value 和 sameSite 的值不会被这个算法持久化。

  5. 返回运行 设置 cookie 的结果，使用 url、name、 value、expires、domain、path、sameSite 和 partitioned。

7.4. 处理更改

要处理 cookie 更改，请执行以下步骤：

  1. 对于每个 Window window，执行以下步骤：

    1. 令 url 为 window 的 相关设置对象的 创建 URL。

    2. 令 changes 为 url 的 可观察更改。

    3. 如果 changes 为空，则继续。

    4. 在全局任务队列上对 window 执行 触发一个名为“change”的更改事件，并在 window 的 CookieStore 处使用 changes。

  2. 对于每个 service worker 注册 registration，执行以下步骤：

    1. 令 changes 为一个新的 集合。

    2. 对于 registration 的 可观察更改中的每个 change，执行以下步骤：

      1. 令 cookie 为 change 的 cookie。

      2. 对于 registration 的 cookie 更改订阅列表中的每个 subscription，执行以下步骤：

        1. 如果 change 不在 subscription 的 url 中的 可观察更改内，则 继续。

        2. 令 cookieName 为对 cookie 的 无 BOM UTF-8 解码的结果，并取 cookie 的 名称。

        3. 如果 cookieName 等于 subscription 的 名称，则将 change 添加到 changes，并终止。

    3. 如果 changes 为空，则 继续。

    4. 令 changedList 和 deletedList 为从 changes 运行 准备列表 的结果。

    5. 触发一个名为“cookiechange”的功能事件，使用 ExtendableCookieChangeEvent 在 registration 上，带有以下属性：

      changed

      changedList

      deleted

      deletedList

可观察的更改 是与 url 相关的 集合，包括符合步骤 1 中的 cookie 更改，具体请参见 Cookies § 检索算法。

cookie 更改 是一个 cookie 和一个类型（更改 或 删除）：

  • 由于插入了另一个具有相同 名称、域名 和 路径 的 cookie，而被移除的 cookie 会被忽略。

  • 新创建的且未被立即逐出的 cookie 被视为 更改。

  • 新创建的且立即被逐出的 cookie 被视为 删除。

  • 被其他方式逐出或删除的 cookie 被视为 删除。

要触发一个更改事件，名称为 type，包含 changes 并作用于 target，请执行以下步骤：

  1. 令 event 为通过使用 CookieChangeEvent 创建事件的结果。

  2. 将 event 的 type 属性设置为 type。

  3. 将 event 的 bubbles 和 cancelable 属性设置为 false。

  4. 令 changedList 和 deletedList 为从 changes 运行 准备列表 的结果。

  5. 将 event 的 changed 属性设置为 changedList。

  6. 将 event 的 deleted 属性设置为 deletedList。

  7. 在目标处调度 event。

要准备列表，从 changes 执行以下步骤：

  1. 令 changedList 为一个新的 列表。

  2. 令 deletedList 为一个新的 列表。

  3. 对每个 change 运行以下步骤：

    1. 令 item 为通过 创建 CookieListItem 从 change 的 cookie 得到的结果。

    2. 如果 change 的类型是 更改，则将 item 添加到 changedList。

    3. 否则，运行以下步骤：

      1. 将 item["value"] 设置为 undefined。

      2. 将 item 添加到 deletedList。

  4. 返回 changedList 和 deletedList。

8. 安全注意事项

除了从 service worker 上下文访问 cookie 之外，此 API 无意向网络公开任何新功能。

8.1. 注意事项！

虽然浏览器的 cookie 实现现在正朝着更好的安全性和减少令人惊讶且容易出错的默认设置的方向发展，但目前关于 cookie 数据安全的保证很少。

  • 非安全源通常可以覆盖安全源使用的 cookie

  • 超级域通常可以覆盖子域看到的 cookie

  • 跨站脚本攻击和其他脚本与头部注入攻击也可以用来伪造 cookie

  • cookie 读取操作（无论是脚本还是网络服务器上的操作）都不会指示 cookie 的来源

  • 浏览器有时会以令人惊讶和违反直觉的方式截断、转换或驱逐 cookie 数据

    • ... 因为达到存储限制

    • ... 因为字符编码差异

    • ... 因为 cookie 的语法和语义规则不同

因此，最好在解释任何 cookie 的值时谨慎行事，切勿将 cookie 的值作为脚本、HTML、CSS、XML、PDF 或任何其他可执行格式来执行。

8.2. 限制？

此 API 可能会产生使 cookie 更易于使用的意外副作用，进而鼓励更多使用。如果它导致 非安全上下文 中的进一步使用，可能会导致对用户而言不太安全的网络。因此，此 API 仅限于 安全上下文。

8.3. 安全 Cookie

本节非规范性。

此 API 仅允许为 Secure cookie 进行写操作，以鼓励围绕安全性的更好决策。然而，该 API 仍将允许读取非 Secure cookie，以便于迁移到 Secure cookie。副作用是，当使用此 API 获取和修改非 Secure cookie 时，非 Secure cookie 将自动修改为 Secure。

8.4. 意外

某些现有的 cookie 行为（尤其是以域为基础而非以源为基础，非安全上下文 能够设置在 安全上下文 中可读取的 cookie，并且脚本能够设置在脚本上下文中不可读取的 cookie）可能从网络安全的角度来看非常令人意外。

其他意外在 Cookies § Introduction 中有记载——例如，一个 cookie 可以为超级域设置（例如 app.example.com 可能为整个 example.com 域设置 cookie），并且 cookie 可以在给定域名的所有端口号上被读取。

历史上，不同主要浏览器之间的 cookie 处理差异进一步加剧了这种复杂性，尽管其中一些差异（例如端口号处理）现在比以前处理得更一致了。

8.5. 前缀

在可行的情况下，示例使用了 __Host- 和 __Secure- 名称前缀，这使得某些当前的浏览器不允许从 非安全上下文 覆盖，不允许使用没有 Secure 标志进行覆盖，并且——对于 __Host-——不允许使用显式 Domain 或非 '/' Path 属性进行覆盖（有效地执行了同源语义）。这些前缀在那些实现安全 Cookie 的浏览器中提供了重要的安全好处，并且在其他浏览器中逐步降级（即特殊语义可能不会在其他 cookie API 中执行，但 cookie 正常工作，异步 cookie API 强制执行写操作的安全语义）。不过，此 API 的一个主要目标是与现有 cookie 的互操作性，因此还提供了一些没有这些前缀的 cookie 名称示例。

此 API 在写操作中也强制执行前缀规则，但在其他 API 中同一浏览器可能不会强制执行这些规则。因此，除非它们被更广泛地采用，否则不建议过度依赖它们的执行。

8.6. URL 作用域

虽然 service worker 脚本今天不能直接访问 cookie，但它已经可以通过受控呈现范围内的 HTML 和脚本资源来注入在 service worker 脚本远程控制下的 cookie 监控代码。这意味着在 service worker 范围内的 cookie 访问技术上已经是可能的，只是并不太方便。

当 service worker 的作用域比 / 更窄时，它仍然可以通过成功猜测/构造允许嵌入 IFRAME 的 404 页面 URL 并在其中运行脚本来读取超出其作用域路径空间的路径作用域 cookie。同样的技术可以扩展到整个源，但一个精心构建的网站（没有超出范围的页面可以嵌入 IFRAME）实际上可以在今天拒绝路径作用域的 service worker 的这种能力，而我不愿意在没有进一步讨论影响的情况下删除该限制。

8.7. Cookie 厌恶

为了减少开发人员的复杂性并消除对临时测试 cookie 的需求，此异步 cookie API 将明确拒绝在操作将被忽略时写入或删除 cookie 的尝试。同样，当操作将忽略实际 cookie 数据并模拟一个空 cookie jar 时，它将明确拒绝读取 cookie 的尝试。在这些上下文中尝试观察 cookie 更改仍然“有效”，但直到读取访问被允许（例如由于站点权限的变化）之前都不会调用回调。

今天，在禁止脚本启动 cookie 写入的上下文中，写入 document.cookie 通常是无效操作。然而，许多 cookie 写入脚本和框架总是写入一个测试 cookie 然后检查它是否存在以确定是否允许脚本启动的 cookie 写入。

同样，在禁止脚本启动 cookie 读取的上下文中，读取 document.cookie 通常返回一个空字符串。然而，合作的网络服务器可以验证服务器启动的 cookie 写入和读取是否有效并将此信息报告给脚本（脚本仍会看到空字符串），并且脚本可以使用此信息推断出脚本启动的 cookie 读取被禁止。

9. 隐私注意事项

9.1. 清除 cookies

本节非规范性。

当用户为某个源清除 cookies 时，用户代理需要清除该源的所有存储；包括该源的 service workers 和 DOM 可访问的存储。这是为了防止网站在用户发起该操作后恢复任何持久存储中的用户标识符。
