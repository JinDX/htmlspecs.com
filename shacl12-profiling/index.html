摘要

本文档定义了 SHACL 配置文件化。

SHACL，即形状约束语言，是一种用于描述 RDF 图结构的语言。 SHACL 可用于多种用途，例如验证、推理、 领域建模、生成本体以告知其他代理、构建用户 界面、生成代码，以及集成数据。

SHACL 配置文件化定义了 SHACL 形状约束语言中的元素，这些元素用于支持 SHACL 的 配置文件 以及使用 SHACL 进行配置文件化。本文档的范围限于 RDF 图的配置文件化，包括 包含 SHACL 形状的图。

本规范由 数据形状工作组发布。

1. 引言

1.1 什么是配置文件化？

配置文件化是为某物创建“配置文件”的行为。

一般而言，在英语中，某物的“profile”如下：

物理对象或特征的轮廓，或对此的表示

- 《牛津英语词典》，自 17 世纪以来对 “profile” 一词的用法

在数据领域中，与上述含义一致的“profile”派生定义为：

摘要或提取

在这个定义中，保留了该英语词的核心含义，因为对一个数据 对象的摘要或提取可能是它的轮廓；例如，三维空间对象的二维表示，或 对具有许多部分的数据集进行的统计 摘要。

警告：引言中截至此处的配置文件化背景内容，可能会被上游纳入 The Profiles Vocabulary 的更新 版本。

根据定义，SHACL 约束 RDF 数据。因此，任何根据 形状图 有效的数据，都将是该 数据 图的一个配置文件。如果某个 形状图 验证了 数据 图的所有元素，则得到的有效数据将是该 数据 图的“空”配置文件，这意味着它与原始数据 图相同。

The Profiles Vocabulary 在 规范的上下文中，将 配置文件定义为：

一种规范，它约束、扩展、组合或提供有关 使用 其他规范的指导或说明

如果一个 形状图被视为 “规范”，那么不仅根据该 形状图有效的数据是 被验证的 数据图的配置文件，而且 该 形状图本身也 充当用于该 数据 图的数据模型的配置文件。

在本文档中，我们描述如何打包 SHACL 信息以实现最佳配置文件化，并通过对 每个 SHACL 规范所创建的 SHACL 形状进行打包来举例说明。

SHACL 的打包是将 SHACL 元素相对于容器 对象进行呈现。

2.1 动机

人们可能出于多种原因，希望标识 SHACL 元素组，并 了解组与单个 SHACL 元素之间的关系。 这些原因包括想要理解一个组的目的、一个组中的变更如何影响 另一个组，以及如何创建新的组。 对此已经阐明了三个用例：

  2. 找出某个组所依赖的所有 SHACL 元素

2.2 建议摘要

对上述用例的解决方案产生了以下建议，这些建议建立在 已确立的实践之上，例如使用 sh:ShapesGraph 的实例来分组 SHACL 元素：

  1. sh:ShapesGraph 的实例SHOULD用于分组 SHACL 元素的实例
  2. SHACL 元素的实例SHOULD使用 rdfs:isDefinedBy 谓词指示定义它的 sh:ShapesGraph 实例
  3. 使用某个 SHACL 元素、 但并不定义它的 sh:ShapesGraph 实例，SHOULD使用 rdfs:member 谓词将该 SHACL 元素指示为成员
  4. 用于分组 SHACL 元素的 sh:ShapesGraph 实例，SHOULD使用 owl:imports 谓词指示其对其他 SHACL 元素组的依赖，这些组使用 sh:ShapesGraph 的实例标识

以下各小节详细说明这些建议。

2.3 标识 SHACL 元素组

sh:ShapesGraph 的实例SHOULD用于分组 SHACL 元素的实例。

SHACL 元素与许多其他 RDF 定义性数据一样，过去 通常被分组在 OWL 2 Web Ontology Language RDF-Based Semantics (Second Edition) 的 owl:Ontology 类实例中。

例如，开放地理空间联盟的 GeoSPARQL 1.1 空间 RDF 数据标准 提供了一个 SHACL 验证器， 其中在一个 Turtle 文件内包含 27 个 sh:NodeShape 实例以及数量相近的 sh:PropertyShape 实例。该文件还包含一个 owl:Ontology 实例，用于呈现该 验证资源的元数据，例如其 修改日期、使用许可证等等。

本建议取代该实践，并建议使用 sh:ShapesGraph 这一 owl:Ontology 的专门子类来分组 SHACL 元素。这是为了避免 有歧义的分组对象。

2.4 指示元素定义

SHACL 元素的实例SHOULD使用 rdfs:isDefinedBy 谓词指示定义它的 sh:ShapesGraph 实例。

将 SHACL 元素放入由 sh:ShapesGraph 实例标识的组中时，常见实践 只是将这些元素和本体实例包含在一个 RDF 文件中。对于 数据交换而言，这是合理的； 但是，当交换的文件被存储到 RDF 数据库中时， sh:ShapesGraph 实例 与其中定义的 SHACL 元素之间的关联可能会丢失。如果加载到 命名图中， 该关联可能会通过命名图保留下来，该命名图对本体和 SHACL 元素进行分组，但这是一种非语义的 关联，即不能直接在图中看到的关联。

本建议提出，SHACL 元素的创建者SHOULD指示 定义该 SHACL 元素的组。此类 RDF 断言的示例如下：

ex:ont-M a sh:ShapesGraph ; . ex:shape-1 a sh:NodeShape ; rdfs:isDefinedBy ex:ont-M ; .
注

在 SHACL 1.2 Core 的 2.5 形状和类的语法 变体一节中， 有一条警告指出，虽然 属性形状可以声明为 空白节点，但最好将它们声明为 IRI，因为 这能在图之间实现更好的复用。如果按本节加入有关 形状的信息，则SHOULD使用 IRI。

2.5 指示元素使用

使用某个 SHACL 元素、 但并不定义它的 sh:ShapesGraph 实例，SHOULD使用 rdfs:member 谓词将该 SHACL 元素指示为 成员

当一个 SHACL 元素在某个元素组内被使用但并非由该组 定义时，SHOULD NOT将其指示为由该组 定义，因此上一节中的建议并不适用。根据链接数据原则 [LDP]， 在不定义 SHACL 元素的组内复用这些元素是预期且受鼓励的。

以下示例展示了如何使用 rdfs:member 谓词来指示某个 SHACL 元素被包含在一个 并不定义它的组中，同时也展示了如上例所示的 rdfs:isDefinedBy 的使用：

# “Ontology M”使用由“Ontology Other”定义的 Shape ex:ont-M a sh:ShapesGraph ; rdfs:member ex:shape-1 ; . ex:ont-Other a sh:ShapesGraph ; . ex:shape-1 a sh:NodeShape ; rdfs:isDefinedBy ex:ont-Other ; .

2.6 指示组依赖

用于分组 SHACL 元素的 sh:ShapesGraph 实例，SHOULD使用 owl:imports 谓词指示其对其他 SHACL 元素组的依赖， 这些组使用 sh:ShapesGraph 的实例标识。

在 SHACL 验证工具中，从一个集合（例如包含 RDF 数据的 文件）通过 owl:imports 指令将 SHACL 元素导入 另一个集合，是已经确立的实践。许多 SHACL 验证引擎 都这样做，例如 pySHACL [PYSH]；推荐采用这种实践来指示从一个集合/组 到另一个集合/组的依赖，即使并不预期某个特定验证引擎会因该断言而采取行动： 对 SHACL 数据管理而言，这也很可能具有有用的信息价值。

owl:imports 的示例如下。

# “Ontology N”导入“Ontology M” ex:ont-M a owl:Ontology ; . ex:ont-N a owl:Ontology ; owl:imports ex:ont-M ; .

3.1 定义 SHACL 的配置文件

SHACL 的配置文件应使用 The Profiles Vocabulary 来声明。SHACL 1.2 Core 配置文件的声明如下：

# Core 配置文件的声明 <http://www.w3.org/ns/shacl/profile/core> a prof:Profile ; prof:isProfileOf <https://www.w3.org/TR/shacl/> ; prof:hasResource <https://www.w3.org/TR/shacl12-core/> , <http://www.w3.org/ns/shacl-shacl/core> ; . # Core 规范 <https://www.w3.org/TR/shacl12-core/> a prof:ResourceDescriptor ; prof:hasRole role:Specification ; dcterms:format "text/html" ; dcterms:conformsTo <http://www.w3.org/TR/html5/> ; prof:hasArtifact <https://www.w3.org/TR/shacl12-core/> ; . # Core 验证器 <http://www.w3.org/ns/shacl-shacl/core> a prof:ResourceDescriptor ; prof:hasRole role:Validation ; dcterms:format "text/turtle" ; dcterms:conformsTo <https://www.w3.org/TR/shacl/> ; prof:hasArtifact <http://www.w3.org/ns/shacl-shacl/core> ; .

此声明使用 IRI <http://www.w3.org/ns/shacl/profile/core> 标识 Core 配置文件，并 指示它是 prof:Profile 的一个实例，是 2015 年原始 SHACL 发布物的一个配置文件，并且具有两个配置文件资源：一个规范和一个验证器。

使用 The Profiles Vocabulary 中的术语描述时， 上面链接的每个规范的在线文档形式，只是共同构成 SHACL 1.2 配置文件的众多 资源之一。该文档 扮演 Specification 这一资源角色——“以人类可读的形式定义配置文件”——而其他资源 扮演其他角色。SHACL 用户尤其感兴趣的是每个 配置文件的 SHACL 形状图，它扮演 Validation 角色—— “提供关于如何验证数据是否符合该 配置文件的说明”。SHACL 1.2 的各个配置文件 可能会提供具有其他配置文件资源角色的附加资源。

形状图验证资源在 SHACL 1.2 Overview 的 SHACL-SHACL 附录中给出。

注

要根据这些规范配置文件或下面定义的联合配置文件中的任何一个来验证 RDF 数据，请使用 SHACL 1.2 Overview 的 SHACL-SHACL 附录中列出的 SHACL-SHACL 形状图。

描述如何使用 SHACL 的打包来挑选要在你的配置文件中支持的 SHACL 元素。

4.1 配置文件化规范

注

4.1.1 刻画 配置文件化情形

议题 617：[PROF] 用例：SKOS 配置文件 ProfilesUCR
此情形刻画部分是否在此回答了该用例？
议题 764：SHACL 数据配置文件化的用例： 多模型互操作性 ProfilesUCR
此用例是否在此得到回答？

SHACL 经常用于测试数据是否匹配某个已知的 Specification。 如果用于此类测试的 SHACL 除规范中的规则之外没有实现其他规则，则该 SHACL 形状图会创建该 规范的 空配置文件。

如果某个形状图实现的规则多于/不同于规范中的规则，那么它就是在测试该规范的一个 （非空）配置文件。

如果一个形状图 SG1 正在配置文件化一个规范 S1， 那么，按照 The Profiles Vocabulary 的刻画方式：

<SG1> a prof:Profile ; prof:isProfileOf <S1> ; .

然而，SG1 也在扮演 Validation 角色， 因此，对这种情形更完整的 The Profiles Vocabulary 刻画将是：

<SG1> a prof:Profile ; prof:isProfileOf <S1> ; prof:hasResource [ prof:hasArtifact <SG1> ; prof:hasRole role:validation ; ] ; .

这是对 The Profiles Vocabulary 的一种非预期用法， 其中配置文件本身也在扮演通常由该配置文件的一部分、 或某个资源所承担的角色，而不是让单个配置文件资源扮演所有 角色；但这是合法的， 并且仍将实现 PROF 的目标：允许概念性配置文件指示一个执行常见 配置文件资源角色——验证——的资源。

要指示某个配置文件资源正在使用 SHACL 1.2 Core 且以 Turtle 格式可用，也可以将以下内容添加到 该刻画中：

<SG1> a prof:Profile ; prof:isProfileOf <S1> ; prof:hasResource [ prof:hasArtifact <S1> ; prof:hasRole role:validation ; dcterms:conformsTo <https://www.w3.org/TR/shacl12-core/> ; dcterms:format "text/turtle" ; ] ; .

如果 SG1 还被视为该配置文件要求的规范性表达， 而不是例如由另一个包含这些要求的人类语言文档来表达，那么 prof:hasRole 的值 role:constraints 也可以应用于该资源。

如果没有关于该配置文件的单独人类语言文档，那么 prof:hasRole 的值 role:specification 也可以应用于该资源。不过，这并非最佳做法，因为 role:specification 被规定为“[以人类可读的形式]定义配置文件”， 虽然 SHACL 是 人类可读的，但这里预期的是自然语言散文。

<SG1> a prof:Profile ; prof:isProfileOf <S1> ; prof:hasResource [ prof:hasArtifact <S1> ; prof:hasRole role:constraints , role:specification , role:validation ; dcterms:conformsTo <https://www.w3.org/TR/shacl12-core/> ; dcterms:format "text/turtle" ; ] ; .

这种推理解决了哪些用例？

4.1.2 对数据的影响

按照 2.8 指示 有效性，我们可以用谓词 sh:conformsTo 指示某个 数据 图 相对于某个 形状图的有效性。 该谓词是 [DCTERMS] 谓词 dcterms:conformsTo 的子属性，后者用于 The Profiles Vocabulary。

按照 图 2，如果我们有 <DG1> sh:conformsTo <SG1>， 这可能是在通过 验证测试数据图对形状图的一致性之后 断言的， 则我们可以推断：

<DG1> dcterms:conformsTo <SG1> . # 通过 RDFS 或 OWL-RL 推理

按照 The Profiles Vocabulary 的公理 dcterms:conformsTo owl:propertyChainAxiom ( dcterms:conformsTo prof:isProfileOf )，那么， 如果给出

<DG1> dcterms:conformsTo <SG1> .

以及

<SG1> prof:isProfileOf <S1> .

我们可以得出

<DG1> dcterms:conformsTo <S1> .

这种推理允许对数据与规范配置文件以及规范本身的一致性作出静态声明， 这对那些必须知道其是否符合这些事物、但又 无法持续进行验证的数据具有实用价值。

4.3 配置文件部分角色

SHACL 是一种可为 RDF 数据实现约束和推理规则的语言。这两件事 并不是配置文件设计者可能希望做的事情的 完整集合；例如，他们可能希望包含 关于该配置文件为何以及如何创建的文档，或者在 扩展模式中提供新的模型元素（类、谓词等）。

由于 SHACL 不能用于所有可能的配置文件部分，配置文件设计者需要参考 SHACL 规范系列之外的规范，以获得关于配置文件 全部组成部分以及如何将它们彼此关联的指导。预计将使用 The Profiles Vocabulary 来完成这一点。

参考 The Profiles Vocabulary 的 资源 角色实例词汇表，似乎很明显， SHACL 图可以在配置文件中承担 Validation 角色，或许 也可以承担 Specification 角色，因为 SHACL 可用于 声明节点形状和属性形状，正如 OWL 可用于声明类和属性一样。

SHACL 形状天然约束模型和数据，因此如果用来指示某个配置文件如何约束 某个事物 ——标准或另一个配置文件——它们就可以承担 Constraints 角色。

如果实现了 SHACL 规则以将数据从一种模型转换为另一种模型， 它们也可以承担 Mapping 角色。

4.4 配置文件层级

配置文件层级是一种安排，其中一个配置文件是另一个配置文件的专门化，形成一条链， 更具体的配置文件从更宽泛的配置文件继承约束。按照 The Profiles Vocabulary 的规则，如果 C 是 B 的配置文件，而 B 是 A 的配置文件，则 希望符合 C 的数据，只有在也符合 B 和 A 时才可能有效。

检查数据对配置文件的一致性，意味着使用具有 Validation 角色的配置文件资源来测试该数据。

4.5 创建空配置文件

使用 SHACL 创建的某个规范的 空配置文件，是该规范的一个 配置文件，其中 SHACL 被用于实现该规范的一些规则，并且很可能实现全部规则，但不在 SHACL 中实现任何其他规则。

使用 SHACL 创建规范空配置文件的主要目的，是能够通过 SHACL 验证来测试 对该规范的一致性。这个目的之所以存在，是因为许多 RDF 数据模型 具有 模型规范，或许是 OWL 模型，或者只是模型的自然语言文档，但并不 提供 数据验证机制，例如 SHACL 验证器。此类模型的示例包括：Data Catalog Vocabulary (DCAT) - Version 3、Schema.org、 The RDF Data Cube Vocabulary，以及许多 W3C 标准。

注

虽然规范的空配置文件可以使用 SHACL 以外的机制创建，但这里我们 只关注 SHACL 的使用。

一个用于 RDF 数据、充当空配置文件的非 SHACL 验证器示例，是 W3C 的 Constraints of the PROV Data Model， 它提供了一组约束，适用于根据 PROV-O: The PROV Ontology 制定的溯源数据。 Constraints of the PROV Data Model 并未实现超出 PROV-O: The PROV Ontology 和概念性 PROV-DM: The PROV Data Model 中所述或所暗示的约束之外的 任何约束；不过 它确实包含针对模型没有明确建模但 正确使用时需要的事项的测试，例如时间实体的排序。那些 约束的实现已经制作出来，形式为执行 SPARQL 查询的 Python 脚本，从而允许对 RDF 数据进行验证。

无法给出创建规范空配置文件的通用且精确的方法， 因为就该任务的本质而言，规范的规则可能定义得并不精确，因此 用 SHACL 对它们进行刻画的方法——而 SHACL 是精确的——也会像不精确的规则 定义本身一样有所不同。

有关其他规范的空配置文件示例，请参见 B. 空配置文件示例。

5.5 规则：conforms to

“conforms to”规则是一条推理规则：如果存在与 Data Graph关联的 Validation Report， 该报告指示 Shapes Graph 被用于验证，并且 sh:conforms 属性指示为 true，则允许指示该 Data Graph 符合 该 Shapes Graph。

由于需要测试 sh:conforms 属性值， 这条推理规则不能表述为 OWL 属性链公理，因此这里同时以 SPARQL 1.2 Query Language 规则和 SHACL 1.2 Rules 规则的形式给出。

CONSTRUCT { ?x sh:conformsTo ?z } WHERE { ?x sh:validationReport ?y . ?y sh:usedShapesGraph ?z ; sh:conforms true ; . }
RULE { ?x sh:conformsTo ?z } WHERE { ?x sh:validationReport ?y . ?y sh:usedShapesGraph ?z ; sh:conforms true ; . }

C. 安全与隐私 考量

本节为非规范性内容。

与大多数基于 RDF 的技术一样，SHACL 处理器可能会操作由 各种来源组装而成的图。一些应用可能具有开放的“链接数据”（“LD”）架构，并动态 组装来自组织信任网络之外来源的 RDF 三元组。 由于 RDF 允许任何人添加关于任何资源的语句，三元组可能会修改形状定义或 数据 图中节点原本 预期的语义，从而导致误导性结果。 通过只使用受信任且已验证的 RDF 来源，并消除通过 owl:imports 和 sh:shapesGraph 动态添加图的可能性，可以防范这种 （以及下面的）场景。

在创建其他规范的配置文件时，配置文件创建者需要确保 其约束不会违反那些规范的规则。如果任何约束确实违反了规则，并且如果只使用 配置文件的规则、而不使用规范的规则来检查数据有效性，无论是偶然还是有意， 数据都可能被错误地计算为有效。这可能导致意外的数据发布或使用，从而可能 引入安全问题。
