摘要

本规范定义了一种基于JSON的文件格式，提供开发人员一个集中位置来放置与网页应用程序相关的元数据。这些元数据包括但不限于网页应用程序的名称、图标链接，以及用户启动网页应用程序时的首选打开URL。清单还允许开发人员声明网页应用程序的默认屏幕方向，并提供设置应用程序显示模式（例如全屏）的能力。此外，清单允许开发人员将网页应用程序“限定”到一个URL。这限制了清单应用的URL范围，并提供了从其他应用程序“深度链接”到网页应用程序的方法。

使用这些元数据，用户代理可以为开发人员提供手段，创建更接近原生应用程序的用户体验。

1. 网页应用程序清单

应用程序清单是一个 [JSON] 文档，包含网页应用程序启动时的启动参数和应用程序默认设置。

清单有一个相关的清单URL， 即[URL]，从该清单获取。

清单可以在其根目录包含以下任何成员，所有这些成员都是可选的。成员可以按任意顺序出现。

  • background_color
  • dir
  • display
  • icons
  • identity
  • lang
  • name
  • orientation
  • prefer_related_applications
  • related_applications
  • scope
  • short_name
  • shortcuts
  • start_url
  • theme_color
注意

尽管在清单中任何成员都是可选的，但某些用户代理可能要求某个或多个成员存在，才能充分利用本规范提供的功能。

    }

1.1.2 使用link元素链接到清单

本节为非规范性内容。

本示例还展示了如何使用类型为 "manifest" 的链接，以及如何使用其他 meta 和 link 元素为网页应用程序提供备用名称和一组图标。

示例 2：链接到清单 
<!doctype>
    <html>
    <title>Racer 3K</title>
    
    <!-- 启动配置 -->
    <link rel="manifest" href="manifest.webmanifest">
    
    <!-- 旧版浏览器的备用应用程序元数据 -->
    <meta name="application-name" content="Racer3K">
    <link rel="icon" sizes="16x16 32x32 48x48" href="lo_def.ico">
    <link rel="icon" sizes="512x512" href="hi_def.png">
注意：文件扩展名：.webmanifest还是.json？
IANA 已注册的清单文件扩展名是.webmanifest。 一些网络服务器识别此扩展名，并使用标准的应用程序清单媒体类型 (application/manifest+json) 传输该文件。 开发人员也可以选择不同的扩展名（例如.json）或完全不使用扩展名（例如/api/GetManifest），但建议使用 application/manifest+jsonMIME 类型传输清单，尽管使用任何JSON MIME类型也是可以的。

1.1.3 声明多个图标

本节为非规范性内容。

在以下示例中，开发人员对与网页应用程序相关的图标做出了如下选择：

  • 开发人员包含了两个相同尺寸的图标，但使用了两种不同的格式。其中一个通过 type成员明确标记为WebP。如果用户代理不支持WebP，它将回退到相同尺寸的第二个图标。然后该图标的 MIME类型 可以通过HTTP头确定，或者用户代理在接收到图标的前几字节后可以 嗅探出MIME类型。
  • 开发人员希望使用大于或等于257x257px的SVG图标。他们发现SVG文件在小尺寸下看起来太模糊，即使在高密度屏幕上也是如此。为了解决这个问题，开发人员包含了一个只在尺寸至少为257px时才使用的SVG图标。否则，用户代理将使用ICO文件（hd_hi.ico），其中包含一系列为小显示尺寸量身定制的光栅图标。
示例 3：多个图标 
{
      "icons": [
        {
          "src": "icon/lowres.webp",
          "sizes": "48x48",
          "type": "image/webp"
        },{
          "src": "icon/lowres",
          "sizes": "48x48"
        },{
          "src": "icon/hd_hi.ico",
          "sizes": "72x72 96x96 128x128 256x256"
        },{
          "src": "icon/hd_hi.svg",
          "sizes": "257x257"
        }]
    }

1.1.4 创建快捷方式

本节为非规范性内容。

在以下示例中，开发人员包含了两个快捷方式。假设清单的URL是 https://example.com/manifest.webmanifest：

  • 第一个快捷方式将显示为“稍后播放”。如果操作系统支持上下文菜单项的图标并且也支持SVG图像，用户代理将在文本旁边显示 https://example.com/icons/play-later.svg。当启动时，用户代理将实例化一个新的 顶级浏览上下文，并导航到 https://example.com/play-later。
  • 第二个快捷方式将显示为“订阅”。当启动时，用户代理将实例化一个新的 顶级浏览上下文，并导航到 https://example.com/subscriptions?sort=desc。
示例 4：添加快捷方式 
{
      "shortcuts": [
        {
          "name": "Play Later",
          "description": "查看稍后保存的播客列表",
          "url": "/play-later",
          "icons": [
            {
              "src": "/icons/play-later.svg",
              "type": "image/svg+xml"
            }
          ]
        },
        {
          "name": "Subscriptions",
          "description": "查看你订阅的播客列表",
          "url": "/subscriptions?sort=desc"
        }
      ]
    }

1.1.5 理解“scope”

本节为非规范性内容。

scope成员告诉浏览器哪些文档是网页应用程序的一部分，哪些不是——因此当用户在网站上导航时，清单会应用于哪些网页。

例如，{"scope": "/"}意味着清单适用于该源中的每个文档。另一方面，{"scope": "/racer/"}意味着只有在路径“/racer/”中的文档才 在范围内：因此“/racer/race1.html”、“/racer/race2.html”等都 在范围内，但“/elsewhere/”及根目录下的任何内容“/”则超出范围，清单不会应用于这些路径中的文档。仅支持一个范围路径。有关技术细节，请参见6. 导航范围。

应用清单意味着清单中影响显示的任何成员将生效，例如显示为“全屏”，或应用特定的屏幕方向。只要网页应用程序导航到 范围内的URL，浏览器将继续应用清单。然而，导航到超出范围的网页应用程序将导致清单不再应用，浏览器将使用其默认设置。例如，应用程序将不再以全屏显示，而是在浏览器标签页中显示为普通网页。如何处理网页在范围内和范围外的导航由实现者决定。有关技术细节，请参见1.18.5 应用清单。

最后，由于用户可以从源内的任何文档安装网页应用程序，因此最好在清单中始终声明scope成员。如果清单中缺少scope成员，则将使用start_url成员的路径作为回退。 如果start_url成员也缺失，则使用安装网页应用程序的文档URL作为范围。为了确保没有意外的导航行为，始终包括一个scope成员，最好设置为"/"。

1.2 dir 成员

清单的 dir 成员指定可本地化成员的 默认方向。 dir成员的值可以设置为 文本方向之一。

文本方向如下，表示可本地化成员的值默认是：

"ltr"
从左到右的文本。
"rtl"
从右到左的文本。
"auto" (默认)
文本方向未知。用户代理应使用启发式算法来估计文本的显示，例如 [UAX9] 中描述的首强算法。

文本方向列表为« "ltr", "rtl", "auto" »。

要处理 dir 成员，给定json和manifest两个有序映射：

  1. 将manifest["dir"]设置为"auto"。
  2. 如果json["dir"]不存在或不是字符串，则返回。
  3. 去除 json["dir"]的前后ASCII空白字符。
  4. 如果文本方向列表不包含 json["dir"]，则返回。
  5. 将manifest["dir"]设置为json["dir"]。

1.3 lang 成员

清单的 lang 成员是一个字符串，形式为 语言标签，用于指定清单中可本地化成员的语言。如果未指定lang成员，则该语言被视为未知。

注意

指定语言可以通过帮助用户代理选择最合适的处理方式或资源（如字体、样式、连字符或用于辅助功能的文本到语音语音）来改善用户体验。

语言标签 是与[BCP47]中定义的Language-Tag格式匹配的 字符串。

注意

语言标签不区分大小写。语言标签示例包括'fr'（法语）、'en-AU'（澳大利亚英语）或 'zh-Hans-CN'（简体中文）。

要处理 lang 成员，给定json和manifest两个有序映射：

  1. 如果json["lang"]不存在或不是字符串，则返回。
  2. 去除 json["lang"]的前后ASCII空白字符。
  3. 如果调用IsStructurallyValidLanguageTag并传入json["lang"]返回false，则返回。
  4. 将 manifest["lang"]设置为调用CanonicalizeUnicodeLocaleId抽象操作并传入json["lang"]的结果。

1.6 scope 成员

清单的 scope 成员是一个字符串，表示此网页应用程序的导航范围。

注意：默认范围

“默认范围”（当scope成员缺失、为空或失败时）是起始URL，但会删除其文件名、查询和片段。

为了处理scope成员，给定有序映射 json 和 有序映射 manifest：

  1. 将manifest["scope"]设置为解析manifest["start_url"]的结果。
  2. 如果json["scope"]为空字符串，则返回。
  3. 将scope设置为解析json["scope"]的结果，并以manifest URL作为基URL。
  4. 如果scope解析失败，则返回。
  5. 将scope的查询和片段设置为null。
  6. 如果manifest["start_url"]不在scope的范围内，则返回。
  7. 否则，将manifest["scope"]设置为scope。

1.9 orientation 成员

manifest的 orientation 成员是一个 字符串，表示该网页应用程序所有顶级浏览上下文的默认屏幕方向。

如果用户代理支持 orientation 成员作为默认屏幕方向，那么该值会成为网页应用程序生命周期内的默认屏幕方向，除非在运行时通过其他方式覆盖。

1.11 id 成员

manifest的 id 成员是一个字符串，表示应用程序的身份。身份的形式是一个与start URL同源的URL。

身份由用户代理用于唯一标识应用程序。当用户代理看到一个带有与已安装的应用程序不对应的身份的manifest时，应该将该manifest视为不同的应用程序的描述，即使它与另一个应用程序来自相同的URL。当用户代理看到一个manifest的manifest["id"]与已安装应用程序的身份相等时，应该视为这是已安装应用程序的manifest的替代，而不是一个不同的应用程序，即使它来自不同的URL。

注意: 排除片段是最佳实践
由于处理算法移除了id成员中的片段，因此在检查匹配应用程序时排除片段并非严格必要。然而，由于旧版规范（可能还有旧版用户代理）没有在解析时移除片段，并且仅在比较时依赖排除片段，历史应用数据可能会在id中包含片段。因此，最佳实践是用户代理在比较两个URL时排除片段。
注意

身份可以被用于收集网页应用程序列表的服务来唯一标识应用程序。

注意

身份像URL一样处理，但它并不指向一个可以导航到的资源，因此它不必在范围内。

  1. 将manifest["id"]设置为manifest["start_url"]。

  3. 如果json["id"]为空字符串，返回。

  6. 如果id解析失败，返回。
  7. 如果id与manifest["start_url"]不是同源的，返回。
  8. 将id的片段设置为null。
  9. 将manifest["id"]设置为id。

1.18 清单生命周期

本节定义了处理清单和 应用 清单的算法。

用户代理必须支持"manifest"链接类型以及如何获取和处理关联资源的相关步骤。

1.18.1 处理清单

当被指示忽略时，用户代理必须表现得好像导致该情况的清单、成员或值不存在。

以下算法提供了一个处理扩展点：建议其他增加新成员到清单的规范在算法的这一点挂钩。 它们不应修改已经存在于manifest对象中的值。

注意

处理扩展点 旨在帮助避免与猴子补丁相关的问题。

处理清单的步骤由以下算法给出。该算法接收一个URL 文档URL，一个URL 清单URL，以及一个字节序列 bodyBytes。

  1. 让json成为解析JSON字节为Infra值的结果，传入bodyBytes。
  2. 如果json是解析异常，或者json不是一个 有序映射：
    1. 将json设置为空的有序映射。
  3. 让manifest成为一个空的有序映射。
  4. 处理dir成员，传入 json和manifest。
  5. 处理lang成员，传入 json和manifest。
  6. 处理文本成员，传入 json，manifest，和 "name"。
  7. 处理 *_localized文本成员，传入json， manifest，"name_localized"，和manifest["dir"]。
  8. 处理文本成员，传入 json，manifest，和 "short_name"。
  9. 处理 *_localized文本成员，传入json， manifest，"short_name_localized"，和manifest["dir"]。
  10. 处理start_url 成员，传入json，manifest， manifest URL，和document URL。
  11. 处理id成员，传入json和manifest。
  12. 如果文档的处理过的清单不是 null，且文档的处理过的清单的id与manifest["id"]不相等，则返回。
  13. 处理scope成员， 传入json，manifest， 和manifest URL。
  14. 处理颜色成员，传入 json，manifest，和 "theme_color"。
  15. 处理颜色成员，传入 json，manifest，和 "background_color"。
  16. 处理display成员， 传入json和manifest。
  17. 处理图像资源，传入 json["icons"]， manifest，manifest URL，和 "icons"。
  18. 处理 *_localized图像资源成员，传入 json，manifest， "icons_localized"，和manifest URL。
  19. 处理 orientation成员，传入json， manifest。
  20. 处理 related_applications成员，传入json 和manifest。
  21. 处理shortcuts 成员，传入json，manifest， 和manifest URL。
  22. 处理扩展点：在此算法点处理任何专有或其他支持的成员。
  23. 让文档的已处理清单为manifest。

1.18.2 处理颜色成员

注意: 支持的颜色

仅支持sRGB 颜色，以及用户代理能够在没有外部信息的情况下转换为 sRGB 的颜色（例如，"AliceBlue"）。例如，lab(…)或color(display-p3, …)可以 在没有外部信息的情况下转换为sRGB， 但color(--custom-profile, …)需要查找匹配的 "@color-profile" 规则，而这不能在清单中指定。

要处理颜色成员，使用有序映射 json，有序 映射 map，和 字符串 member：

  1. 如果json[member]不存在 或json[member]不是一个 字符串，则返回。
  2. 去除前后ASCII空格从 json[member]。
  3. 让color成为通过根据CSS语法解析 json[member]的值为CSS颜色的结果。
  4. 如果color解析失败，则返回。
  5. 如果color可以使用用户代理固有的信息转换为 sRGB， 则将color转换为sRGB。
  6. 如果color不是 sRGB 颜色，则返回。
  7. 将map[member]设置为color。

1.18.3 处理文本成员

要处理文本成员，给定有序映射 json，有序 映射 map，和 字符串 member：

  1. 如果json[member]不存在或 json[member] 不是一个字符串，则返回。
  2. 去除前后ASCII空格从 json[member]。
  3. 将map[member]设置为 json[member]的值。

1.18.4 处理没有文档的 manifest

处理 manifest 的步骤由 [HTML] 的 link 元素的处理步骤触发，但用户代理也可以 在没有关联的 文档时调用这些步骤处理 manifest。

在这种情况下，为了符合 [HTML] 相应步骤提供的保证，用户代理应当 确保至少在某个时间点：

  • 至少有一个与文档 URL同源的 文档，并且该文档包含一个具有 rel 为 manifest 且 href 解析到 manifest URL 的 linkElement 元素。
  • 如果 manifest URL 与 文档 URL 不同源， 则 bodyBytes 数据通过 获取，并使用文档 URL的来源发起 CORS 请求，并根据 linkElement的 crossorigin 属性设置凭证模式。
注意：这些检查的理由

该条款允许用户代理在不首先加载链接到 manifest 的文档的情况下执行应用安装 （例如通过同步服务将应用安装到用户设备上）。但上述建议要求用户代理验证当应用程序范围位于源 A 而 manifest 位于不同源 B 时，两者之间存在双向关系。

第一个检查确保源 A 上至少有一个页面链接到源 B 上的 manifest （否则，源 B 上的 manifest 可能会控制源 A 上非关联应用程序的元数据）。

第二个检查确保源 B 允许（通过 CORS 协议）其 manifest 被源 A 应用，并且 manifest 是按两者约定的方式（带或不带凭证）获取的，就像通过文档直接获取一样。

1.18.5 应用 manifest

已处理的 manifest 被应用到顶级浏览上下文，这意味着 已处理的 manifest 的成员会影响浏览上下文的呈现或行为。

被应用 manifest 的顶级浏览上下文 被称为 应用程序上下文。

如果由于用户代理被要求导航到深层链接而创建了 应用程序上下文，用户代理必须立即导航到该深层链接，并将 historyHandling 设置为 "replace"。否则，当创建 应用程序上下文时，用户代理必须 立即导航到启动 URL，并将 historyHandling 设置为 "replace"。

注意

启动 URL 不一定是 start_url 成员的值：用户或用户代理 可以在应用程序安装时更改它。

适当的时间来应用 manifest 是在 应用程序上下文创建时，并且在导航到 启动 URL 开始之前。

1.18.6 更新 manifest

如 manifest link 关系所规定，manifest 在每次页面加载时都会被获取并处理。当 manifest 处理成功时， 用户代理可以将更新的 manifest 应用于与应用程序相关的任何当前和未来的 应用程序上下文。

为了更新，以下成员被认为是安全敏感成员， 因为它们在安装和启动界面时展示：

  1. short_name 及其在 short_name_localized中的本地化表示，
  2. icons 及其在 icons_localized中的本地化表示，
  3. name 及其在 name_localized中的本地化表示。

安全敏感成员 应当 按照 [UTS55] 所描述的双向隔离方式显示，无论其方向如何。

用户代理不应在未经用户 明确许可的情况下 自动应用对 安全敏感成员的更改。

相反，用户代理应当通过适当的管理选项展示对 安全敏感成员的更改， 以便用户可以对 web 应用的更新做出明智的决定。

如果更新不包含对 安全敏感成员的更改，用户代理可以自动应用这些更改。

如果用户更改了本地化设置，用户代理可以 自动调整在启动界面上可见的安全敏感成员为其在 *_localized成员中指定的本地化表示。 这些更改应当在用户下次打开 web 应用时向他们展示。

注意: 用户代理不会应用部分更新

当更新同时包含对 安全敏感成员和其他成员的更改时， 用户代理不会自动进行部分更新。例如，用户代理可能会向用户提供选项：

  1. 接受更新
  2. 卸载 web 应用

2. Manifest 图像资源

每个 manifest 图像资源 是一个 图像资源， 它在概念上是 web 应用程序的一部分，适合在各种上下文中使用，具体取决于使用该对象的成员的语义 （例如，作为应用程序菜单的一部分的图标等）。

一个 manifest 图像资源 与 图像资源 不同之处在于它可以有一个额外的 purpose 成员。

用户代理可以 修改与 manifest 图像资源相关联的图像，以更好地匹配平台的视觉风格， 在显示给用户之前。例如，通过将图像的角变圆或以特定颜色绘制。建议开发人员准备好他们的图像资源以适应这些情况， 以避免通过例如颜色变化或角被裁剪等方式丢失重要信息。

2.1 purpose 成员

purpose 成员是一个 由唯一空格分隔的标记组成的无序集合。允许的值是 图标用途。

当 manifest 图像资源 作为一个 图标 使用时， 开发者可以提示该图像在主机 OS 的上下文中具有某种特殊用途 （即为了更好的集成）。用户代理 不应 使用与声明的 图标用途 不符的图标。

注

例如，具有用途 "monochrome" 的图标可以用作徽章或带有实色填充的固定图标， 其视觉效果与应用程序的全彩启动图标不同。用户代理使用 purpose 成员的值作为提示，以决定图标显示的位置和方式。 除非开发者另行声明，否则用户代理可以将图标用于 任何用途。

图标用途 如下：

monochrome
用户代理可以在需要 带有实色填充的单色图标 的地方显示此图标。图标中的颜色信息会被丢弃，仅使用 alpha 数据。 用户代理可以将该图标作为任何实色的遮罩来使用。
maskable
该图像是以 图标遮罩和安全区 为设计目标的，因此图像中超出 安全区 的任何部分都可以被用户代理安全地忽略和遮罩掉。
any（默认）
用户代理可以自由地在不需要 purpose 的地方显示此图标。例如，具有 "any" 用途的 manifest 图像资源 不会用于需要 "monochrome" 的上下文中。

图标用途列表 为 列表 « "monochrome", "maskable", "any" »。

注

如果一个图标包含多个用途，它可以用于这些用途中的任何一个。如果没有识别出的用途，图标将被完全忽略。例如，具有用途 "monochrome fizzbuzz" 的图标可以作为单色图标使用，因为 "monochrome" 是一个有效的用途。 然而，如果一个图标只有 "fizzbuzz" 这个用途，那么它将被忽略。

要 确定图像的用途，给定 有序映射 json：

  1. 如果 json["purpose"] 不 存在 或 json["purpose"] 不是 字符串：
    1. 返回 集合 « "any" »。
  2. 令 keywords 为 按 ASCII 空白符拆分 json["purpose"] 的结果。
  3. 令 purposes 为新的 集合。
  4. 对于每个 keyword 进行迭代：
    1. 如果 图标用途列表 不 包含 keyword，则 继续。
    2. 否则，追加 keyword 到 purposes。
  5. 如果 purposes 为空，则返回失败。
  6. 返回 purposes。

2.3 图标遮罩和安全区域

一些平台有其首选的图标形状，但由于网络应用程序应在多个平台上工作，因此可以通过添加 maskable 目的，指示图标可以应用用户代理指定的遮罩。这样，平台可以确保图标与平台良好集成，甚至可以在平台的不同位置应用不同的遮罩和背景颜色。

安全区域 是 maskable 图标中始终可见的区域，无论用户代理的偏好如何。它被定义为一个圆，中心点位于图标的中心，半径为图标大小的 2/5（40%），即使图标不是正方形，也意味着图标宽度和高度中的较小值。

注意

maskable 图标的设计者需要确保所有重要部分都在 安全区域 内。

此区域内的所有像素在所有遮罩中都保证可见。安全区域外的像素无法保证始终可见，取决于所应用的遮罩。

用户代理 可以 应用任意大小的遮罩，使得距离图像中心（安全区域）超过图像大小 2/5 的像素透明。

用户代理 不得 将安全区域内的任何像素变为透明。

用户代理 可以 通过添加额外的填充来放大图标。

如果图标包含透明像素，用户代理 必须 将图标合成到用户代理选择的实色填充（例如白色）上。

注意

建议设计人员在 maskable 图标中避免使用透明像素。

3. 快捷方式项目

每个 快捷方式项目 是一个 有序映射，表示网络应用中的某个关键任务或页面的链接。它具有以下成员：

  • name
  • short_name
  • description
  • url
  • icons

用户代理可以使用这些成员来组装一个上下文菜单，供操作系统在用户与网络应用图标交互时显示。当用户从操作系统菜单中调用快捷方式时，用户代理 应当 运行 启动快捷方式。

3.1 name 成员

快捷方式项目的 name 成员是一个 字符串，表示快捷方式的名称，通常在上下文菜单中显示给用户。

name 成员是一个 可本地化成员。

3.2 short_name 成员

快捷方式项目的 short_name 成员是一个 字符串，表示快捷方式名称的简短版本。它旨在用于显示空间不足以显示完整快捷方式名称的情况。

short_name 成员是一个 可本地化成员。

3.3 description 成员

快捷方式项目的 description 成员是一个 字符串，允许开发者描述该快捷方式的用途。用户代理 可以 将此信息暴露给辅助技术。

description 成员是一个 可本地化成员。

3.4 url 成员

快捷方式项目的 url 成员是一个 在范围内 的 已处理的 manifest 的 URL，当激活关联的快捷方式时打开该 URL。

3.5 icons 成员

快捷方式项目的 icons 成员列出作为快捷方式在各种上下文中的标志性表示的图像。

icons 成员是一个 可本地化成员。

3.6 启动快捷方式

当 快捷方式项目 shortcut 含有 manifest 被调用时，运行步骤 启动网络应用程序，并传递 manifest 和 shortcut.url。

3.7 处理快捷方式项目

要 处理快捷方式，给定 有序映射 item，URL manifest URL，URL scope 和 文本方向 defaultDirection：

  1. 如果出现以下情况则返回失败：
    • item 不是 有序映射。
    • item["name"] 不 存在。
    • item["name"] 是空字符串。
    • item["url"] 不 存在。
    • item["url"] 不是 字符串。
  2. 将 url 设为 解析 item["url"]，以 manifest URL 为基础 URL。
  3. 如果 url 解析失败，则返回失败。
  4. 如果 url 不在 scope 的 范围内，则返回失败。
  5. 将 shortcut 设为 «[ "url" → url，"name" → item["name"] ]»。
  6. 处理一个 *_localized 文本成员，传递 item、shortcut、"name_localized" 和 defaultDirection。
  7. 如果 "short_name" 存在于 item 中，并且 item["short_name"] 是一个字符串，则 将 shortcut["short_name"] 设置为 item["short_name"]。
  8. 处理一个 *_localized 文本成员，传递 item、shortcut、"short_name_localized" 和 defaultDirection。
  9. 如果 "description" 存在于 item 中，并且 item["description"] 是一个字符串，则 将 shortcut["description"] 设置为 item["description"]。
  10. 处理一个 *_localized 文本成员，传递 item、shortcut、"description_localized" 和 defaultDirection。
  11. 处理图像资源，传递 item["icons"]、shortcut、manifest URL 和 "icons"。
  12. 处理一个 *_localized 图像资源成员，传递 item、shortcut、"icons_localized" 和 manifest URL。
  13. 返回 shortcut。

4. 外部应用程序资源

(功能存在风险) 问题 956: related_applications 是单引擎功能

此规范中的 related_applications 成员当前仅有一个实现。因此，按照 W3C 进程，它已被标记为 "存在风险"，这意味着：

[该功能] 在推进到建议推荐标准之前，可能会被删除，而无需发布新的候选推荐标准。

Web Apps 工作组正在寻求第二个实现，以便保留此功能在规范中。

每个 外部应用程序资源 代表与该网络应用程序相关的应用程序。

一个 外部应用程序资源 可以具有以下成员，其中某些成员需要成为 有效的外部应用程序资源：

  • fingerprints 成员
  • id 成员
  • min_version 成员
  • platform 成员
  • url 成员

一个 有效的外部应用程序资源 必须 具有 platform 成员，并且需要 url 或 id 成员（或者两者都有）。

在以下示例中，网络应用程序列出了两个不同的相关应用程序，一个在 Google Play 商店，另一个在 iTunes 商店。Google Play 商店中的应用程序有一个 Android 包名、一个最低版本说明符和用于验证的加密指纹，以特定于 Play 商店的方式使用。

{
      "related_applications": [
        {
          "platform": "play",
          "url": "https://play.google.com/store/apps/details?id=com.example.app1",
          "id": "com.example.app1",
          "min_version": "2",
          "fingerprints": [
            {
              "type": "sha256_cert",
              "value": "92:5A:39:05:C5:B9:EA:BC:71:48:5F:F2"

        },
        {
          "platform": "itunes",
          "url": "https://itunes.apple.com/app/example-app1/id123456789"
        }

4.1 platform 成员

platform 成员表示此 外部应用程序资源 相关联的平台。 platform 代表一个软件分发生态系统或可能是一个操作系统。此规范未定义 platform 成员的特定值。

工作组维护了已知平台值的 注册表。

4.2 url 成员

外部应用程序资源 的 url 成员是可以找到该应用程序的 URL。

处理应用程序的 url 成员 的步骤：

  1. 如果 application URL 缺失，则返回 null。
  2. 否则，解析 application URL，如果结果不是失败，则返回该结果。否则返回 null。

4.3 id 成员

外部应用程序资源 的 id 成员表示用于在平台上表示该应用程序的 id。

4.4 min_version 成员

外部应用资源的 min_version 成员表示与此 Web 应用相关的应用的最低版本。此版本是具有平台特定语法和语义的字符串。

4.5 fingerprints 成员

外部应用资源的 fingerprints 成员表示用于验证应用程序的一组加密指纹。

指纹 代表用于验证应用程序的加密指纹。一个指纹有以下两个成员：type 和 value。 它们都是字符串，但其语法和语义由平台定义。

5. 可安装的 Web 应用程序

manifest 的一个常见用例是用户代理 安装 Web 应用程序；用户代理为最终用户提供了一个创建新顶层浏览上下文的方式，并应用 manifest 的成员 应用到该上下文。 已安装的 Web 应用程序称为 已安装的 Web 应用程序。 这意味着 manifest 的成员或其默认值在顶层浏览上下文中有效。这使已安装的 Web 应用程序与传统书签不同，因为从传统书签打开的网页不会应用 manifest 的属性。

注意

例如，在支持安装的用户代理上，Web 应用程序可以以最终用户无法区分的方式展示和启动，与原生应用程序类似：例如，作为带标签的图标出现在主屏幕、启动器或开始菜单上。当 启动 Web 应用程序时，manifest 被用户代理 应用到 顶层浏览上下文，在加载 start URL 之前。这为用户代理提供了应用 manifest 的相关值的机会，可能会更改 Web 应用程序的显示模式和屏幕方向。或者，用户代理还可以将 Web 应用程序安装到用户代理本身的书签列表中。

5.1 应用程序名称

应用程序名称 来源于 name 成员或 short_name 成员。 用户代理 应当 优先解析对应的 *_localized 成员中的本地化值。

当 name 或 short_name 成员缺失、为空或类型错误时，用户代理 可以 使用 name 成员作为 short_name 的替代，或者使用 short_name 作为 name 的替代。

如果 name 和 short_name 成员缺失、为空或类型错误，用户代理 可以 回退到 Document， 寻找缺失 manifest 成员的合适替代（例如，使用 application-name 代替 name 或 short_name）。另一种选择是，用户代理 应当 分配一个符合平台约定的默认名称（例如 "未命名"）。用户代理 可以 允许终端用户输入可用作 应用程序名称 的文本。

当 name 和 short_name 成员都存在时，由实现决定哪个成员最适合可用的空间（例如，short_name 成员可能更适合图标下方的可用空间）。

5.2 启动 Web 应用程序

由操作系统或用户代理自行决定，执行启动 Web 应用程序的步骤，使用处理过的清单 processed manifest。

注意

这通常发生在用户从应用启动界面（例如，主屏幕、启动器或开始菜单）选择已安装的 Web 应用时。

启动 Web 应用程序的步骤由以下算法给出。算法接受处理过的清单 manifest，可选的 URL target URL，可选的 POST 资源 POST resource，并返回应用程序上下文 application context。

如果给定，target URL 必须 在清单的范围内 within scope。

其他规范 可以 使用其自己的步骤替换此算法的步骤。此替换将适用于所有启动 Web 应用程序的调用。

注意

该算法是可替换的，以允许实验性的 launch_handler 清单字段配置所有 Web 应用程序启动的行为。 替换算法默认调用创建新的应用程序上下文，但在某些条件下表现不同。

  1. 返回执行创建新应用程序上下文的步骤的结果，传递 manifest、target URL 和 POST resource。

创建新应用程序上下文的步骤由以下算法给出。算法接受处理过的清单 manifest，可选的 URL target URL，可选的 POST 资源 POST resource，并返回应用程序上下文 application context。

  1. 如果没有提供 target URL，则将其设置为 start URL。
  2. 执行创建新的顶级遍历项的步骤，使用 target URL 和 POST resource。
  3. 将 browsing context 设置为 traversable 的活动浏览上下文。
  4. 将处理过的清单 manifest 应用于 browsing context。
  5. 返回 browsing context。

5.3 隐私和安全考虑

建议用户界面提供用户安装 Web 应用程序的能力时，还应允许用户检查与该 Web 应用程序相关的图标、名称、启动 URL、来源等信息。 这样可以使终端用户有机会在安装前有意识地做出批准决定，甚至可能修改与 Web 应用程序相关的信息。这也为终端用户提供了辨别 Web 应用程序是否在冒充其他 Web 应用程序的机会，例如使用意外的图标或名称。

建议用户代理阻止其他应用程序确定系统上已安装的应用程序（例如，通过对用户代理缓存的计时攻击）。这可以通过例如在 Web 应用程序安装后，使清单中链接的资源（例如图标）失效，或使用与常规 Web 浏览不同的缓存来实现。

5.4 卸载

用户代理 应该 提供一种机制，允许用户删除已安装的 Web 应用程序。

建议在删除时，用户代理还应为用户提供撤销与应用程序相关的其他持久数据和设置的机会，例如权限和持久存储。

6. 导航范围

manifest 的导航范围 是 处理后的 manifest 的 "scope" 项。导航范围限制了当 manifest 被应用时， 应用上下文 可以导航的 URL 集合。

注意

如果 scope 成员未在 manifest 中出现， 它默认会使用 start_url 成员的父路径。 例如，如果 start_url 是 /pages/welcome.html，且 scope 缺失， 导航范围将是 /pages/，且在相同的源上。如果 start_url 是 /pages/ （末尾的斜线很重要！），导航范围将是 /pages/。

开发者在依赖默认行为时应谨慎，确保应用程序的所有页面 URL 都以 start_url 的父路径开头。为安全起见，建议明确指定 scope。

如果 URL target 符合以下算法，则被认为是在 URL scope 的 范围内：

  1. 如果 target 和 scope 不在同源，返回false。
  2. 让 scopePath 是 scope 的路径，使用 U+002F (/) 作为分隔符进行连接。
  3. 让 targetPath 是 target 的路径，使用 U+002F (/) 作为分隔符进行连接。
  4. 返回一个 布尔值， 指示 targetPath 是否以 scopePath 开头。

如果 URL target 位于 manifest 的 导航范围内，则该 target 被认为是 manifest 的 范围内（即位于 scope 成员的范围内）。

注意

没有什么可以阻止 应用程序上下文 导航到超出 manifest 的 导航范围 的 URL，同时仍然将该 manifest 应用于其上。

与之前版本的规范不同，用户代理不再需要或允许阻止超出范围的导航，或将其打开在新的 顶级浏览上下文中。此实践破坏了一些站点，例如导航到超出范围的 URL（如执行第三方身份验证）。参见 Issue 646。

6.1 安全性考量

本节为非规范性内容。

上述建议（当 应用程序上下文 导航到超出范围的 URL 时显示一些 UI）是出于安全考虑。 它确保用户始终了解与其交互的来源。

尽管如此，仍然存在潜在的欺骗风险，如果已安装的应用程序假装导航到另一个来源的超出范围的站点。该站点显示用户代理的虚假超出范围的 UI，向用户表明它在另一个来源上。 然而实际上，用户从未离开已安装应用程序的来源，用户代理未显示任何超出范围的 UI。用户代理可以尝试以一种已安装应用无法伪造的方式显示超出范围的 UI。 但是，由于用户代理的 UI 对于已安装的应用程序来说是最小化的，甚至是不存在的，这可能无法实现。

6.2 深度链接

深度链接 是 范围内的 已安装的网络应用程序 的 处理过的 manifest 的 URL。

可以通过 深度链接 启动 应用程序上下文， 在这种情况下，将应用 manifest，并在网络应用程序的上下文中加载深度链接。

注意

深度链接 的概念 非常有用，因为它允许从一个 已安装的网络应用程序 超链接到另一个。 这可以从本机应用程序到 已安装的网络应用程序，反之亦然。理论上，这可以通过标准超链接提供 原生和网络应用程序之间的无缝上下文切换。如果某个特定的网络应用程序尚未安装，OS 可以直接在用户首选的网络浏览器中打开链接。

鼓励实现者使这种上下文切换对用户明显，例如通过遵守底层平台的应用切换相关的人机界面指南。

7. 显示模式

显示模式 表示在 OS 环境下 如何呈现网络应用程序（例如，全屏显示等）。显示模式对应于给定平台上使用的用户界面 (UI) 隐喻和功能。 显示模式的 UI 规范仅为建议性质，开发者可以根据需要自由解释和实现。

本规范定义了以下 显示模式：

全屏
在隐藏浏览器 UI 元素的情况下打开网络应用程序，占据可用显示区域的全部空间。
独立模式
以类似于独立原生应用程序的方式打开网络应用程序。这可能包括应用程序有一个独立的窗口、自己的图标等。在此模式下，用户代理将排除标准的浏览器 UI 元素（如 URL 栏），但可以包含其他系统 UI 元素，如状态栏和/或系统返回按钮。
最小化 UI
此模式类似于 独立模式，但为用户提供了一些用于控制导航的基本 UI 元素（例如，返回、前进、重新加载，或查看文档地址的方式）。用户代理还可以包括平台特定的 UI 元素，如“分享”和“打印”按钮，或其他常用的功能。
浏览器模式（默认）
使用用户代理的特定平台约定（例如，在浏览器标签或新窗口中）打开网络应用程序。
注意

全屏 显示模式 与 全屏 API 标准 是独立的，彼此没有依赖关系。全屏 显示模式 会影响浏览器窗口的全屏状态，而 [FULLSCREEN] API 仅作用于视口内的某个元素。因此，网络应用程序的 显示模式 可以设置为 全屏， 同时 document.fullScreenElement 返回 null，且 fullscreenEnabled 返回 false。

当用户代理 应用 某个 显示模式 到 应用程序上下文后， 它将成为 默认显示模式， 即在 顶级浏览上下文 中， 当窗口被 导航时会使用的显示模式。 出于安全原因，用户代理 可以 覆盖默认显示模式（例如，顶级浏览上下文被导航到另一个来源时）， 或者用户代理 可以 为用户提供切换到其他显示模式的方法。

当 display 成员缺失，或者没有有效的 display 成员时，用户代理将使用 浏览器 显示模式 作为 默认显示模式。 因此，用户代理 必须 支持 浏览器 显示模式。

每个 显示模式 都有一个 回退链， 它是一个 显示模式 的列表。以下是各显示模式的回退链：

  1. 浏览器 的回退链为 «»。
  2. 最小化 UI 的回退链为 « "浏览器" »。
  3. 独立模式 的回退链为 « "最小化 UI", "浏览器" »。
  4. 全屏 的回退链为 « "独立模式", "最小化 UI", "浏览器" »。

确定 Web 应用程序选择的 显示模式步骤 的算法如下。该算法接受 处理后的清单 manifest 并返回一个 显示模式。

  1. 处理扩展点：在此算法点处理任何专有的和/或支持的显示模式。
  2. 如果用户代理支持 manifest["display"]，则返回 manifest["display"]。
  3. 对每个 fallback_mode 在 回退链 中的 manifest["display"] 执行：
    1. 如果用户代理支持 fallback_mode，则返回 fallback_mode。
注意

上述循环保证在断言之前返回一个值，因为 浏览器 存在于每个模式的 回退链 中，并且所有用户代理都必须支持 浏览器 显示模式。

示例 13

SuperSecure 浏览器（一个虚构的浏览器）仅支持 minimal-ui 和 browser 显示模式，但开发人员通过使用 display 属性在清单中声明他们希望使用 fullscreen。在这种情况下，用户代理会首先检查它是否支持 fullscreen（不支持），因此它回退到 standalone（也不支持），最终回退到 minimal-ui。

显示模式列表 是 列表 « "全屏", "独立模式", "最小化 UI", "浏览器" »。

用户代理 必须 在 显示模式 的 display-mode 媒体功能中反映应用的显示模式 [MEDIAQUERIES-5]。

注意

用户代理将展示实际应用的显示模式——而不一定是清单中声明的模式——通过 display-mode 媒体功能，通过 CSS 或 JavaScript 可访问。请注意，当没有应用清单时，该媒体功能也会反映网页的其他显示模式。例如，如果最终用户将页面设置为全屏模式， 用户代理将通过 display-mode 媒体功能向 CSS 和脚本反映此更改。

8. 隐私和安全考虑

本规范并未直接涉及高价值数据。然而，已安装的 Web 应用程序及其数据可以被视为“高价值”（特别是从隐私角度）。

由于清单格式为 JSON 并将使用 [UNICODE] 编码，安全考虑中描述的内容 [JSON] 和 [UNICODE-SECURITY] 适用。此外，由于无法阻止开发人员在 清单 中包含自定义/不受约束的数据，实施者需要在值的实现上施加特定的限制，以防止拒绝服务攻击，防止内存耗尽，或解决平台特定的限制。

Web 应用程序通常包含 ECMAScript、HTML、CSS 文件及其他媒体，这些文件在沙盒环境中执行。因此，实施者需要意识到所支持类型的安全隐患。特别是， 实施者需要考虑至少在以下规范中概述的安全隐患： [CSS-MIME]， [ECMAScript-MIME]， [HTML]。

由于 Web 应用程序能够同时与本地设备和远程主机交互，实施者需要考虑暴露私人信息给远程主机可能带来的隐私隐患。减轻和深入的防御措施是实施者的责任， 而非本规范规定的。然而，在设计这些措施时，建议实施者使用户了解信息共享，并提供方便的接口以撤销权限。

由于本规范允许在清单中的某些成员声明 URL，实施者需要考虑 [URL] 规范中讨论的安全考虑。打算显示 IRIs 和 IDNA 地址的实现者强烈建议遵循 [UNICODE-SECURITY] 中给出的安全建议。

开发人员需要注意 [CSP3] 规范中讨论的安全考虑，特别是关于将 data: 作为 内嵌 清单的有效来源的目的。这么做可能允许清单直接包含在文档本身中，从而启用 XSS 攻击； 应尽量完全避免这种情况。

推荐 提供用户界面以允许最终用户安装 Web 应用程序时，可以检查图标、名称、启动 URL、来源等。这使得最终用户能够有意识地决定是否批准，甚至修改与 Web 应用程序相关的信息。 这也使得最终用户有机会判断 Web 应用程序是否冒充其他 Web 应用程序，例如，通过使用意外的图标或名称。

推荐 用户代理防止其他应用程序确定系统上安装了哪些应用程序（例如，通过对用户代理缓存进行计时攻击）。这可以通过，例如，在 Web 应用程序 安装后，从用户代理缓存中使与清单关联的资源（例如，图标）无效，或者使用与常规 Web 浏览不同的缓存。

可以想象，快捷方式 url 可能被设计为指示应用程序从浏览器外部启动 （例如，"url": "/task/?from=homescreen"）。也可以想象，开发人员可能会将唯一识别用户的字符串编码到 url 中（例如，服务器分配的 UUID）。这属于指纹识别/隐私敏感信息，用户可能不会意识到这一点。

当 Web 应用程序正在运行时，推荐 用户代理向最终用户提供访问 Web 应用程序常见信息的方式，例如来源、启动和/或当前 URL、已授予的权限和相关图标。如何向最终用户显示此信息由实施者决定。

此外，当应用清单设置的 显示模式 为除 "浏览器" 之外的模式时， 推荐 用户代理明确向最终用户表明他们正在离开 Web 浏览器的正常浏览上下文。理想情况下，启动或切换到 Web 应用程序的操作与主机平台上启动或切换到其他应用程序的操作保持一致。例如，长时间且明显的动画过渡，或说出“正在启动应用程序 X”。

display 成员允许某个来源对用户代理的本地 UI 进行一定程度的控制。在全屏后，它可能会尝试模拟另一个应用程序的用户界面。这也可以通过 'display-mode' 媒体功能 [MEDIAQUERIES-5] 来实现，脚本可以知道 Web 应用程序的显示模式。

B.1.1 专有清单成员

本节为非规范性内容。

虽然专有扩展是不受欢迎的，但它们实际上是无法避免的。如果用户代理选择解释清单 JSON 中未在本文档中指定的成员，则它可以这样做，但需要谨慎。

我们鼓励添加专有扩展的实现者考虑它们是否可以成为标准（即使目前没有其他用户代理对此成员表达兴趣，但在其他平台上的第二个用户代理使用该成员是否合理）。 如果是这样，我们要求作者以中立的方式设计 API，并将其作为标准提议（见 C. 孵化）。如果新成员确实是专有的（即只在专有生态系统的上下文中才有意义），请使用此流程，并为其加上专有生态系统的简短名称前缀，以避免命名冲突。

不要使用你打算在其成为标准后删除的供应商前缀（这些前缀往往会永久存在）。只使用现在和未来都有意义的前缀。

我们鼓励实现者将专有扩展添加到我们的 扩展注册表。 这允许社区跟踪供应商和/或网络社区定义和记录的扩展。我们将定期考虑这些扩展是否应标准化。

      ...
      "kpl_fancy_feature": "some/url/img",
      "gmpc_awesome_thing": { ... },
      "blitzly_site_verification": "KEY_9864D0966935"

在此示例中，我们有意选择了（虚构的）名称，这些名称可能是外部站点或服务，而不是浏览器或浏览器供应商的名称。 这些不是属于发明它们的浏览器的供应商前缀；它们是专有服务的前缀。

E. 与 HTML 的 link 和 meta 元素的关系

本节为非规范性内容。

关于我们为何选择使用 JSON 而不是 HTML meta/link 标签的广泛讨论已在 GitHub 和 www-tag 列表中进行。 以下是这些讨论中提出的关键点的简短总结。

本规范中定义的文档格式提供了一种统一的方式来封装 Web 应用程序的元数据，我们希望这能避免现有的专有和 [HTML] 的 meta/link 标签所存在的问题。这些问题包括：

  • 开发者必须在网站的每个页面上重复应用图标和名称，导致页面间的显著冗余。如果这些信息从未被用户代理使用（例如，用户从未将 Web 应用程序添加到书签），这种冗余问题尤为严重。
  • 将元数据分散到多个文档中可能导致数据不同步。
  • 如果 Web 应用程序的元数据存放在 HTML 文档中，则这会显著增加用户代理（和用户）检查站点元数据更新的成本。由于 HTML 文件可能经常更改，这意味着用户代理将不得不经常下载整个 HTML 文件来检查是否有任何相关的 meta 标签已更改。如果该资源包含内联资源，如 JavaScript、图片或样式表，这可能是一个非小数目的下载。

尽管认为本规范不会带来其自身问题是不现实的，但将此数据外部化为清单的形式解决了上述问题。这些问题通过以下方式解决：

  • 使清单可以作为外部链接：外部清单文件可以作为外部资源进行缓存，从而节省标记中的字节和冗余。
  • 灵活的值类型：与 HTML 属性不同，清单的成员可以使用复杂类型（如对象和数组）来表示数据，而不仅仅是字符串。这解决了当前专有 meta 标签值格式不一致的问题，尤其是在标签的值包含多个子值时。

此外，将当前通过各种基于 meta 标签的解决方案提供的功能标准化到清单中，解决了拥有专有和标准的 [HTML] 标签同时实现相同目的的问题。当然，这依赖于该标准被浏览器实际实现，并且这些浏览器被广泛部署到用户中：如果这发生了，Web 社区可能能够淘汰许多撰写本文时困扰 Web 的专有 meta 标签。有关这些专有标签的更多信息，请参见 可安装 Web 应用的用例和需求。

最后，本规范不会使 [HTML] 中的标准化解决方案失效。当清单中缺少 name 或 icons 等成员时，用户代理可以在清单的所有者 [HTML] 文档中查找图标和应用名称（或者用户代理甚至可能回退到文档中存在的专有标签/元数据）。

G. 国际化

本节为非规范性内容。

预期作者将使用以下选项之一来本地化 manifest 的内容：

在 manifest 中提供本地化值：
作者可以为 manifest 的 本地化值 提供 可本地化的成员。 用户代理将所有本地化值传递给主操作系统。当用户在 OS 级别更改 语言环境 时，OS 可以显示 已安装 Web 应用程序 的更新本地化值。
动态设置语言：
例如，询问最终用户的首选语言并根据该语言偏好动态添加或替换文档中的 manifest 链接关系（例如，使用 "manifest.php?lang=fr" 这样的 URL）。
使用内容协商或地理定位等服务器端策略：
托管 Web 应用程序的服务器可以尝试通过 地理定位 或使用内容协商（例如，使用 HTTP "Accept-Language" 请求头 [RFC9110]，或甚至自定义 HTTP 请求头）来预判最终用户的语言。

鉴于上述选项，开发者需要谨慎处理与最终用户的语言偏好相关的隐私问题：当最终用户明确向 Web 应用程序表明其语言偏好时（即不仅仅使用用户代理的默认语言设置），以明文形式发送最终用户的语言偏好通常是不合适的。 这样做可能会泄露关于最终用户的个人信息。因此，开发者应被鼓励使用 [TLS] 来减少 Web 应用程序被广泛监控的可能性 [RFC7258]。
