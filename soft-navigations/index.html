1. 引言

本节是非规范性的。

现代 Web 应用经常响应用户交互而动态更新内容，而不执行完整的页面导航。 这些由交互发起的效果——例如结构性的 DOM 修改、内容绘制以及历史状态变更—— 过去一直难以度量，也难以归因到正确的用户操作。

考虑一个典型的单页应用模式：用户点击一个产品链接，这会触发一个 click 事件处理器。该处理器发起一次网络 fetch() 以获取产品详情。当 响应到达时，会执行一个回调，将新内容动态注入 DOM，并 使用 History 或 Navigation API 更新 URL。虽然这在用户看来像是一次 导航，但现有指标（例如 Largest Contentful Paint (LCP)（[LARGEST-CONTENTFUL-PAINT]））只 度量初始页面加载，而 Interaction to Next Paint (INP) 只度量点击本身的即时 视觉反馈，导致后续显著的渲染和“软” 导航未被捕获。

本规范利用 [EVENT-TIMING] API 来定义交互，并利用 [ASYNC-CONTEXT] 提案来跨异步任务边界跟踪因果关系。它定义了 浏览器如何识别并报告这些效果，包括“软导航”，方式是与 [PAINT-TIMING] 和 [LARGEST-CONTENTFUL-PAINT] 集成，以将渲染变化归因到 性能 时间线上。

此外，本规范还与 [CONTAINER-TIMING] 提案集成。当 用户 交互导致 DOM 修改时，这些被修改的节点会被指定为新的“容器根”。 这些子树中任何后续的内容绘制都会被归因到其相应的根， 然后再追溯回原始交互。这允许高效且准确地 度量在初始事件派发很久之后发生的丰富、动态页面更新。

2. 交互基础设施

2.1. 导航 ID

导航 ID 是分配给 全局对象生命周期内每次导航（包括硬导航和 软导航）的唯一标识符。

每个 全局对象都有一个 当前导航 ID，它是一个 unsigned long long，初始设置为与 初始化 Document 的 初始 interactionId 值以及 Navigation Timing 的初始 navigationId 相同的值。

2.2. 交互上下文简介

软导航检测依赖于跟踪任务因果关系，并观察 某些操作（例如附加一个 DOM 节点）是由特定用户交互触发的能力。

本规范利用 TC39 [ASYNC-CONTEXT] 提案来处理这种传播。每个 新的用户交互（由 Event Timing 定义）或相关导航事件都会创建一个新的 InteractionContext。此上下文存储在一个隐藏的、仅供内部使用的 AsyncContext.Variable （记作 [[ActiveInteractionContext]]）中。

Web 平台与 AsyncContext 的集成确保该变量会被自动 附加到异步延续（例如 setTimeout、fetch、await），从而允许 浏览器 将之后的效果归因回原始交互。

除了脚本传播之外，本规范还定义了修改 DOM 的用户交互如何建立 container roots（通过 [CONTAINER-TIMING]）。这些根 允许后续渲染 效果（例如内容绘制）追溯回发起该效果的交互上下文，即使 当前没有活动的异步脚本。

2.3. InteractionContext 结构体

InteractionContext 是一个 结构体，用于维护从单次交互中检测软 导航所需的数据。它具有以下 项：

  • id，初始未设置 - 与此上下文关联的交互 ID。

  • 上下文文档，一个 Document - 与此上下文关联的 Document。

  • 开始时间，初始未设置 - 表示交互的开始 时间。

  • 导航类型，初始未设置 - 表示 导航的类型（例如 "push"、 "replace"）。

  • 第一个 URL 值，初始未设置 - 表示 第一次 修改时 URL 的值。

  • 第一个 URL 更新时间戳， 初始未设置 - 表示此交互期间第一次 URL 修改的时间。

  • 第一个滚动时间戳，初始未设置 - 表示此交互期间第一次滚动 事件的时间。

  • 第一个输入时间戳，初始未设置 - 表示此交互期间第一次输入事件 的时间。

  • 首次内容绘制，初始为 null - 此交互的第一个 InteractionContentfulPaint 条目。

  • 最大内容绘制，初始为 null - 到目前为止此交互的最大 InteractionContentfulPaint 条目。

  • 最后一个 URL 值，初始未设置 - 表示 此交互期间最近一次 URL 修改的值。

  • 已发出标志，初始为 false - 表示一个软 导航条目已由该交互 发出。

注： 本规范的未来版本可能会跟踪 交互上下文期间发生的所有 URL 更新，以提供更完整的导航历史。最后一个 URL 值会在内部被 跟踪，以确保即使当前只有第一个 URL 值暴露在 PerformanceSoftNavigation 的 name 中，也能将效果准确归因回交互的最终状态。

2.4. 基础设施算法

要在给定 Document document 的情况下，获取 当前交互上下文：

  1. 令 context 为内部 AsyncContext.Variable [[ActiveInteractionContext]] 的值。

  2. 如果 context 不为 null，且 context 的上下文文档不等于 document，返回 null。

  3. 返回 context。

要在给定 Document document、一个 interaction id 和一个 Event event 的情况下，获取 交互的上下文：

  1. 如果 document 的交互 ID 到交互 上下文[interaction id] 存在， 返回 document 的交互 ID 到交互 上下文[interaction id]。

  2. 令 interaction context 为一个新的 InteractionContext。

  3. 将 interaction context 的 id 设置为 interaction id。

  4. 将 interaction context 的上下文文档设置为 document。

  5. 将 interaction context 的开始 时间设置为 event 的 startTime。

  6. 设置 document 的交互 ID 到交互 上下文[interaction id] 为 interaction context。

  7. 返回 interaction context。

要在给定 Document document 和一个 Event event 的情况下，更新事件的交互上下文：

  1. 令 timestamp 为在给定 document 的相关全局对象时得到的当前高精度时间。

  2. 如果 event 的 type 是 "scroll"，令 is scroll 为 true；否则为 false。

  3. 如果 event 的 type 是会触发 已派发输入事件的事件类型（如 [EVENT-TIMING] 中所定义），令 is input 为 true；否则为 false。

  4. 如果 is scroll 为 false 且 is input 为 false，返回。

  5. 对于 document 的交互 ID 到交互 上下文的 values 中的每个 interaction context：

    1. 如果 is scroll 为 true，且 interaction context 的第一个滚动时间戳 未设置：

      1. 将 interaction context 的第一个滚动 时间戳设置为 timestamp。

    2. 如果 is input 为 true，且 interaction context 的第一个输入时间戳 未设置：

      1. 将 interaction context 的第一个输入 时间戳设置为 timestamp。

要在给定 null 或一个 PerformanceEventTiming 对象 timingEntry 的情况下，设置事件派发的当前交互上下文：

  1. 如果 timingEntry 为 null，返回 null。

  2. 令 event 为 timingEntry 的关联事件。

  3. 令 interaction id 为 timingEntry 的 interactionId。

  4. 如果 interaction id 为 null 或 0，返回 null。

  5. 令 document 为 event 的相关全局对象的关联 Document。

  6. 在给定 document 和 event 的情况下，更新事件的交互上下文。

  7. 令 interaction context 为在给定 document、interaction id 和 event 的情况下，获取交互的上下文的结果。

  8. 将内部 AsyncContext.Variable [[ActiveInteractionContext]] 的值设置为 interaction context。

  9. 返回 interaction context。

要在事件派发后取消设置当前交互上下文：

  1. 将内部 AsyncContext.Variable [[ActiveInteractionContext]] 的值设置为 null。

3. 交互内容绘制

3.1. InteractionContentfulPaint 接口

[Exposed=Window]
interface InteractionContentfulPaint : PerformanceEntry {
    readonly attribute LargestContentfulPaint largestContentfulPaint;
    readonly attribute unsigned long long interactionId;

    object toJSON();
};

InteractionContentfulPaint includes PaintTimingMixin;

每个 InteractionContentfulPaint 具有：

  • 一个关联的 上下文，即一个 InteractionContext。

  • 一个关联的 最大内容绘制，即一个 LargestContentfulPaint 条目。

  • 一个关联的 绘制定时信息。

largestContentfulPaint 属性的 getter 必须返回 this 的 最大内容绘制。

interactionId 属性的 getter 必须返回 this 的 上下文的 id。

name 属性的 getter 必须返回空字符串。

entryType 属性的 getter 必须返回 "interaction-contentful-paint"。

startTime 属性的 getter 必须返回 this 的 上下文的 开始时间。

duration 属性的 getter 必须返回以下二者之间的差值： 默认绘制时间戳，其针对 this 的 绘制 定时信息；以及 this 的 上下文的 开始时间。

注： 虽然该条目会从其关联的 InteractionContext 动态查询静态 交互属性（例如 interactionId 和 startTime），但它会在创建时捕获 绘制的 绘制定时信息和候选 LargestContentfulPaint 条目的静态快照，以 确保这些指标反映该次视觉更新的精确状态。

3.2. 交互内容绘制算法

要在给定一个 全局对象 global、一个 InteractionContext interaction context、一个 LargestContentfulPaint lcpEntry，以及一个 绘制定时信息 paintTimingInfo 的情况下，创建交互内容绘制条目：

  1. 令 entry 为 global 的领域中的一个新的 InteractionContentfulPaint 对象。

  2. 将 entry 的上下文设置为 interaction context。

  3. 将 entry 的最大内容 绘制设置为 lcpEntry。

  4. 将 entry 的关联 绘制定时信息设置为 paintTimingInfo。

  5. 返回 entry。

要在给定一个 LargestContentfulPaint lcpEntry、一个 Document document，以及一个 InteractionContext interaction context 的情况下，处理新的已归因内容绘制 候选项：

  1. 如果 interaction context 的第一个滚动时间戳已设置， 且 lcpEntry 的 renderTime 大于 interaction context 的 第一个滚动时间戳，返回。

  2. 如果 interaction context 的第一个输入时间戳已设置，且 lcpEntry 的 renderTime 大于 interaction context 的 第一个输入时间戳，返回。

  3. 令 global 为 document 的相关全局对象。

  4. 令 paint timing info 为 lcpEntry 的关联 绘制定时信息。

  5. 令 icpEntry 为在给定 global、interaction context、lcpEntry 和 paint timing info 的情况下，创建交互内容 绘制条目的结果。

  6. 排队 icpEntry。

  7. 将 icpEntry 添加到 global 的性能条目缓冲区。

  8. 如果 interaction context 的首次内容绘制为 null：

    1. 将 interaction context 的首次内容绘制 设置为 icpEntry。

  9. 将 interaction context 的最大内容绘制设置为 icpEntry。

注： InteractionContentfulPaint 条目会在被 检测到时发出到性能时间线，而不取决于该交互最终是否产生软导航。这 允许开发者监视所有交互的渲染更新。

4. 软导航

4.1. PerformanceSoftNavigation 接口

[Exposed=Window]
interface PerformanceSoftNavigation : PerformanceEntry {
    readonly attribute DOMString navigationType;
    readonly attribute unsigned long long interactionId;
    InteractionContentfulPaint? getLargestInteractionContentfulPaint();
};

PerformanceSoftNavigation includes PaintTimingMixin;

每个 PerformanceSoftNavigation 具有：

  • 一个关联的 上下文，即一个 InteractionContext。

  • 一个关联的 绘制定时信息。

navigationType 属性的 getter 必须返回 this 的 上下文的 导航类型。

interactionId 属性的 getter 必须返回 this 的 上下文的 id。

getLargestInteractionContentfulPaint() 方法必须返回 this 的 上下文的 最大内容绘制。

name 属性的 getter 必须返回 this 的 上下文的 第一个 URL 值。

entryType 属性的 getter 必须返回 "soft-navigation"。

startTime 属性的 getter 必须返回 this 的 上下文的 开始时间。

duration 属性的 getter 必须返回 this 的 presentationTime 与 this 的 上下文的 开始时间在发出时的差值。

注： 实践中，getLargestInteractionContentfulPaint() 预计会返回一个非 null 条目，因为已确认的软导航至少需要检测到一次交互内容绘制才会 触发其发出。

然而，未来的迭代可以评估软导航是否可以在 URL 修改时立即被视为已提交， 即早于首次绘制。在这样的模型中，此方法可能会在发出时 返回 null。

这种时序的主要设计考量，是其他时间线条目（例如 LayoutShift、PerformanceResourceTiming、LongAnimationFrameTiming 等）的归因，这些条目发生在 URL 修改与首次绘制之间的区间内。例如，许多站点会在发起 fetch 请求时 立即提交新 URL，即便当前页面仍处于 上一个导航状态。为确保一致的时间线切片，本规范会将所有 此类条目归因到上一个导航标识符，直到首次绘制确认该转变。

4.2. 软导航算法

软导航 是一种 满足以下 条件的同文档导航：

  • 在一个 InteractionContext 处于活动状态时发生同文档 URL 变更。

  • 发生一次内容绘制，且该绘制被归因到同一个 InteractionContext。

要在给定一个 Document document 和一个 InteractionContext interaction context 的情况下，评估软导航发出：

  1. 如果 interaction context 的已发出为 true，返回。

  2. 如果 document 的活动软导航候选项 不是 interaction context，返回。

  3. 如果 interaction context 的第一个 URL 值未设置，返回。

  4. 如果 interaction context 的首次内容绘制为 null， 返回。

  5. 令 global 为 document 的相关全局对象。

  6. 令 entry 为在给定 global 和 interaction context 的情况下，创建 软导航条目的结果。

  7. 在给定 global 和 entry 的情况下，发出软导航条目。

  8. 将 interaction context 的已发出设置为 true。

要在给定一个 全局对象 global 和一个 InteractionContext interaction context 的情况下，创建 软导航条目：

  1. 令 entry 为 global 的领域中的一个新的 PerformanceSoftNavigation 对象。

  2. 将 entry 的上下文设置为 interaction context。

  3. 令 first paint 为 interaction context 的首次内容绘制。

  4. 如果 first paint 不为 null：

    1. 将 entry 的关联 绘制定时信息设置为 first paint 的 关联 绘制定时信息。

  5. 返回 entry。

注： navigationId 会在 下文的 排队 PerformanceEntry 中设置。

要在给定一个 全局对象 global 和一个 PerformanceSoftNavigation entry 的情况下，发出软 导航条目：

  1. 将 entry 的导航 ID设置为 entry 的 interactionId。

  2. 将 global 的当前导航 ID设置为 entry 的 interactionId。

  3. 排队 entry。

  4. 将 entry 添加到 global 的性能条目缓冲区。

要在给定一个 Document document、字符串 url 和 字符串 navigation type 的情况下，处理 同文档提交：

  1. 如果 url 等于 document 的 url，返回。

  2. 令 interaction context 为在给定 document 的情况下，获取当前交互上下文的结果。

  3. 如果 interaction context 为 null，返回。

  4. 将 interaction context 的最后一个 URL 值设置为 url。

  5. 如果 interaction context 的第一个 URL 更新时间戳 未设置：

    1. 将 interaction context 的第一个 URL 更新 时间戳设置为在给定 document 的相关全局对象时得到的 当前高精度时间。

    2. 将 interaction context 的第一个 URL 值设置为 url。

    3. 将 interaction context 的导航类型设置为 navigation type。

  6. 将 document 的活动软导航候选项 设置为 interaction context。

  7. 在给定 document 和 interaction context 的情况下，评估软导航发出。

[Exposed=(Window,Worker)]

    readonly attribute unsigned long long navigationId;
};

6. 规范集成

6.1. HTML 集成

6.1.1. Document

每个 文档都有一个 交互 ID 到 交互上下文，即一个 映射， 初始为空。

每个 文档都有一个 活动软导航 候选项，即一个 InteractionContext 或 null，初始为 null。

6.1.2. History

在 为历史步骤应用更新文档中，在 5.5.1 之前（如果 documentsEntryChanged 为 true，且如果 documentIsNew 为 false），在给定 Document、 目标条目的 url，以及 "traverse" 的情况下，处理同 文档提交。
在 共享 历史 push/replace 步骤 （如 [HTML] 中所定义）中，在 URL 更新后，在给定 Document、新的 URL，以及操作类型（"push" 或 "replace"）的情况下，处理同文档提交。

6.1.3. Node

每个 节点都有一个 关联的交互 上下文，初始为 null。

6.1.4. 硬导航

当一个新的 全局对象 global 被创建时（例如在一次“硬” 导航期间），它的 当前导航 ID 会按 § 2.1 导航 ID 中的规定进行初始化。

在 [NAVIGATION-TIMING] 中，当为初始 导航创建 PerformanceNavigationTiming 条目时，用户代理必须将其 navigationId 设置为全局对象的 当前导航 ID。

6.2. Event Timing 集成

本规范扩展了 [EVENT-TIMING] 对交互的定义，以包括与现代 Web 应用和单页应用相关的其他 事件类型。

以下事件类型被视为 交互 的一部分（如 [EVENT-TIMING] 中所定义）：

  • navigate

  • popstate

  • hashchange

当这些事件作为用户交互的结果被派发时，用户代理必须为其分配 一个唯一的 交互 ID，该 ID 通过为文档的 相关全局对象获取下一个 interactionId 得到。如果一个事件由已经具有 已分配交互 ID的先前交互触发，则用户代理应该复用同一个标识符。

注：这些事件只有在它们是来自 用户操作的发起事件时（例如点击浏览器 UI 的后退按钮），才会被报告为主要 交互。在大多数其他场景中——例如一个 click 处理器手动操作历史记录——后续的 popstate 或 navigate 事件并不 被视为独立的用户交互。

虽然这些以编程方式触发的事件可能未设置 isTrusted 标志，但它们会 通过 [ASYNC-CONTEXT] 被正确归因回原始交互，因为历史和 导航 API 会被视为发起任务的异步延续。

注： 事件 navigate、popstate 和 hashchange 被用作内部信号，用于 跟踪软导航并分配 interactionId。本规范 不要求这些 事件类型作为 PerformanceEventTiming 条目暴露到 性能时间线，而是将 该决定留给 [EVENT-TIMING] 规范。

注： [EVENT-TIMING] 规范预计会被 更新，以显式定义 processingStart 和 processingEnd 钩子，并 确保 interactionId 分配发生得足够早 以支持此集成。如今，interactionId 分配通常 延迟到事件处理 结束时。

本规范使用 unsigned long long 作为 interactionId， 以确保安全的全局 计数器，而 [EVENT-TIMING] 目前将其定义为 unsigned long。这种不匹配预计 会在两个规范的未来版本中解决。

要为 Window window 获取下一个 interactionId：

  1. 将 window 的交互计数设置为 window 的交互计数加 1。

  2. 返回 window 的初始 interactionId 值加上 （window 的交互计数 乘以 window 的interactionId 增量）。

在 [EVENT-TIMING] 中定义的对 DOM 规范事件 派发算法的修改中：

  1. 紧接在将 timingEntry 设置为 初始化并记录事件 定时处理开始的结果的步骤之后，运行以下步骤：

    1. 在给定 timingEntry 的情况下，设置事件派发的当前 交互上下文。

  2. 紧接在调用 记录事件定时处理结束的步骤之前，运行 以下步骤：

    1. 在事件派发后取消设置 当前交互上下文。

6.3. Largest Contentful Paint (LCP) 集成

本规范挂接到 [LARGEST-CONTENTFUL-PAINT] 算法中，以将绘制归因到 交互。

在 报告最大内容绘制中，令 candidates 为 已绘制图像列表 与 已绘制文本列表的并集：

  1. 令 contextToNewLargestCandidate 为一个新的 映射。

  2. 对于 candidates 中的每个 candidate：

    1. 令 element 为 candidate 的 element。

    2. 如果 element 为 null，继续。

    3. 令 interaction context 为 element 的关联交互上下文。

    4. 如果 interaction context 为 null，继续。

    5. 令 size 为 candidate 的 size。

    6. 如果 interaction context 的最大内容 绘制不为 null，且 size 小于或等于 interaction context 的最大内容 绘制的 largestContentfulPaint 的 size， 继续。

    7. 如果 contextToNewLargestCandidate[interaction context] 未设置，或 size 大于 contextToNewLargestCandidate[interaction context] 的 size：

      1. 设置 contextToNewLargestCandidate[interaction context] 为 candidate。

  3. 对于 contextToNewLargestCandidate 中的每个 interaction context → newLcpCandidate：

    1. 令 newLcpEntry 为表示 newLcpCandidate 的 LargestContentfulPaint 条目。

    2. 在给定 newLcpEntry、 document 和 interaction context 的情况下，处理新的 已归因内容绘制候选项。

    3. 在给定 document 和 interaction context 的情况下，评估软导航 发出。

注： § 6.4 Container Timing 集成一节定义了对 DOM 的 修改如何重置受影响元素的 先前已报告绘制，以确保它们由 [LARGEST-CONTENTFUL-PAINT] 算法重新评估，并正确归因到 当前交互。

注： [LARGEST-CONTENTFUL-PAINT] 算法预计会忽略任何具有 关联交互上下文的绘制，因为这些由本规范的 § 3.1 InteractionContentfulPaint 接口处理。

6.4. Container Timing 集成

[CONTAINER-TIMING] API 提供一种机制，可按共同的 DOM 祖先对渲染效果进行分组。本规范与该机制集成，以将渲染变化归因回 用户交互。

在 节点插入时，或当一个 节点以下列方式之一被 修改时：

  • 元素的 class 或 style 属性被修改。

  • 资源属性（例如 img 或 video 元素上的 src）被修改。

运行以下步骤：

  1. 令 document 为该节点的 节点文档。

  2. 令 interaction context 为在给定 document 的情况下，获取当前交互 上下文的结果。

  3. 如果 interaction context 不为 null：

    1. 将该节点的关联交互上下文 设置为 interaction context。

    2. 将该节点指定为 容器根（如 [CONTAINER-TIMING] 中所定义）。

    3. 对于该节点及其每个后代，将它们从 Document 的 先前已报告绘制中移除（如 [PAINT-TIMING] 中所定义）。

注： 通过从 先前已报告绘制集合中移除已修改的子树，用户 代理可确保这些元素的下一次内容绘制会被重新检测，并正确 归因到新的 InteractionContext。

注： 对“节点插入”的引用是一种高层级 描述，涵盖所有向文档结构中添加 新元素的操作（例如 appendChild、innerHTML 更新等）。具体如何将这些映射到跟踪文档结构 修改的内部实现钩子，由用户代理决定。

注： 鼓励用户代理维护一棵稀疏的 容器根树。如果被 修改的节点已经是与同一 交互上下文关联的现有容器根的后代，用户代理可以选择不创建新的冗余根。

此外，为避免在大型 DOM 修改期间产生昂贵的主线程工作，鼓励用户代理 优化从先前已报告绘制中移除后代的过程。用户代理可以不 立即进行穷尽式遍历，而是将已修改的根标记为“脏”，并在后续树遍历期间 （例如在布局或绘制期间）惰性地 传播此状态。已知不可见或被现有机制（例如 content-visibility 或 CSS containment）跳过的子树，也可以在此重置过程中跳过。

6.5. Performance Timeline 集成

在 排队 PerformanceEntry中，在步骤 1（初始化 条目）之后，添加以下步骤：

  1. 如果 newEntry 的导航 ID为 0：

    1. 将 newEntry 的导航 ID设置为 global 的当前导航 ID。

7. 安全与隐私考量

将软导航暴露到性能时间线本身并不会带来安全和隐私影响。 但是，作为检测到软 导航的结果而重置各种绘制定时条目可能会产生影响，尤其是在已访问链接被 分区之前。因此，在未对 :visited 缓存进行分区的情况下暴露 此类绘制操作，只能在仔细 分析相关绘制操作之后进行，以确保它们不会跨源暴露用户的历史记录。

8. 附录：重叠交互和竞态条件

本节是非规范性的。

Web 应用经常会快速连续处理多个用户交互。本规范 通过以下模型处理此类重叠交互：

  • 上下文独立性：每个交互都独立管理自己的 InteractionContext。 多个上下文可以同时处于“进行中”状态，每个上下文都跟踪自己的 URL 修改和 内容绘制。

  • 活动候选项单例：虽然许多交互可以处于活动状态，但 Document 在任意给定时刻只识别一个 活动软导航候选项。

  • 抢占：交互上下文一旦触发其第一次同文档 URL 修改， 就会成为 活动软导航候选项。 如果后续发生另一次交互 修改，它会抢占前一个候选项，并成为新的候选项。

  • 发出验证：只有当发出 条件（URL 变更 + 内容绘制）满足的那一刻，属于 活动软导航候选项 的交互 上下文，才会发出 PerformanceSoftNavigation。 这确保软导航报告 与文档当前的视觉和导航状态保持一致。

  • 持久归因：即使一个交互作为软导航 候选项被抢占， 只要它仍然 处于活动状态，它就会继续归因并报告自己的 InteractionContentfulPaint 条目。这允许对并发渲染更新进行准确度量，即便这些更新本身 并不 被视为“导航”。
