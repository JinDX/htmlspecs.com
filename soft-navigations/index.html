1. 引言

本节是非规范性的。

现代 Web 应用经常响应用户交互而动态更新内容，而不执行完整的页面导航。 这些由交互发起的效果——例如结构性的 DOM 修改、内容绘制以及历史状态变更—— 过去一直难以度量，也难以归因到正确的用户操作。

考虑一个典型的单页应用模式：用户点击一个产品链接，这会触发一个 click 事件处理器。该处理器发起一次网络 fetch() 以获取产品详情。当 响应到达时，会执行一个回调，将新内容动态注入 DOM，并 使用 History 或 Navigation API 更新 URL。虽然这在用户看来像是一次 导航，但现有指标（例如 Largest Contentful Paint (LCP)（[LARGEST-CONTENTFUL-PAINT]））只 度量初始页面加载，而 Interaction to Next Paint (INP) 只度量点击本身的即时 视觉反馈，导致后续显著的渲染和“软” 导航未被捕获。

本规范利用 [EVENT-TIMING] API 来定义交互，并利用 [ASYNC-CONTEXT] 提案来跨异步任务边界跟踪因果关系。它定义了 浏览器如何识别并报告这些效果，包括“软导航”，方式是与 [PAINT-TIMING] 和 [LARGEST-CONTENTFUL-PAINT] 集成，以将渲染变化归因到 性能 时间线上。

此外，本规范还与 [CONTAINER-TIMING] 提案集成。当 用户 交互导致 DOM 修改时，这些被修改的节点会被指定为新的“容器根”。 这些子树中任何后续的内容绘制都会被归因到其相应的根， 然后再追溯回原始交互。这允许高效且准确地 度量在初始事件派发很久之后发生的丰富、动态页面更新。

2. 性能时间线切片

本节为非规范性内容。

为了将性能指标（例如事件计时、布局偏移、资源加载或长动画帧）归因到触发它们的导航， 性能时间线会使用 navigationId 属性在 PerformanceEntry 上被切分为多个片段。

确定一个跨越时间的转换的确切切分点是复杂的。为确保归因一致， 本规范采用一种模型：时间线不会在每次 URL 修改时立即切分；相反，该转换会被推迟到 软导航 准备好被发送到性能时间线时。

有关设计权衡、归因规则和边界情况的详细讨论，参见 [PERFORMANCE-TIMELINE] issue comment。

2.1. 导航 ID

navigation id 是一个唯一 标识符，分配给 global object 生命周期内的每次导航（包括硬导航和 软导航）。

每个 Window 都具有：

  • 一个 current navigation id，一个 64-bit unsigned integer， 初始设置为该 Window 的 initial navigation id value。

  • 一个 navigation count，一个 64-bit unsigned integer，初始为 0。

  • 一个 initial navigation id value，一个 64-bit unsigned integer， 初始设置为 100 到 10000 之间的一个随机整数。

  • 一个 navigation id increment，一个 64-bit unsigned integer， 初始设置为由用户代理选择的一个小的正整数。

注： initial navigation id value 和 navigation id increment 用于计算 navigationId 值（参见 increment the current navigation id）。 这会阻止开发者依赖它来计算确切的导航次数，或 假定它从零开始。 用户代理应避免在不同的 Window 对象之间使用共享的全局导航值，以防止跨源泄露。

给定一个 Window window，要 increment the current navigation id：

  1. 将 window 的 navigation count 设置为 window 的 navigation count 加 1。

  2. 令 newId 为 window 的 initial navigation id value 加上 （window 的 navigation count 乘以 window 的 navigation id increment）。

  3. 将 window 的 current navigation id 设置为 newId。

2.2. PerformanceEntry 扩展

[Exposed=(Window,Worker)]
partial interface PerformanceEntry {
    readonly attribute unsigned long long navigationId;
};
每个 PerformanceEntry 都有关联的 navigation id， 一个 64-bit unsigned integer，初始为 0。

navigationId 属性的 getter 必须返回 this 的 navigation id。

2.3. 性能时间线集成

在 queue a PerformanceEntry 中，在步骤 1（初始化 entry）之后，添加以下步骤：

  1. 如果 newEntry 的 navigation id 为 0：

    1. 如果 global 是一个 Window 对象：

      1. 将 newEntry 的 navigation id 设置为 global 的 current navigation id。

3. 交互基础设施

3.1. 交互上下文简介

软导航检测依赖于跟踪任务因果关系的能力，并观察某些操作 （例如追加 DOM 节点）是由特定用户交互触发的。

本规范利用 TC39 [ASYNC-CONTEXT] 提案来处理这种传播。每个 新的用户交互（由 Event Timing 定义）或相关的导航事件都会创建一个新的 InteractionContext。该上下文存储在一个隐藏的、仅供内部使用的 AsyncContext.Variable （表示为 [[ActiveInteractionContext]]）中。

Web 平台与 AsyncContext 的集成确保该变量会自动 附加到异步延续（例如 setTimeout、fetch、await）上，使 浏览器 能够将后续影响归因回原始交互。

除了脚本传播之外，本规范还定义了修改 DOM 的用户交互如何建立 container roots（通过 [CONTAINER-TIMING]）。这些根 允许后续渲染 影响（如内容绘制）被追溯回触发它的交互上下文，即使当前没有 异步脚本处于活动状态。

3.2. InteractionContext 结构

InteractionContext 是一个 struct，用于维护从单个交互检测软 导航所需的数据。它具有以下 items：

  • id，初始未设置 - 与此上下文关联的 interaction id。

  • context document，一个 Document - 与此上下文关联的 Document。

  • start time，初始未设置 - 表示交互的开始 时间。

  • navigation type，初始未设置 - 表示 导航的类型（例如 "push"、 "replace"）。

  • first URL value，初始未设置 - 表示第一次 修改时的 URL 值。

  • first URL update timestamp， 初始未设置 - 表示此交互期间第一次 URL 修改的时间。

  • first scroll timestamp，初始未设置 - 表示此交互期间第一次滚动 事件的时间。

  • first input timestamp，初始未设置 - 表示此交互期间第一次输入事件 的时间。

  • first contentful paint，初始为 null - 此交互的第一个 InteractionContentfulPaint entry。

  • largest contentful paint，初始 为 null - 到目前为止此交互的最大 InteractionContentfulPaint entry。

  • last URL value，初始未设置 - 表示此交互期间 最近一次 URL 修改的值。

  • emitted 标志，初始为 false - 表示一个软 导航 entry 已由该交互发出。

注： 本规范的未来版本可能会跟踪 在一个交互上下文中发生的所有 URL 更新，以提供更完整的导航历史。last URL value 会被 内部跟踪，以确保即使当前仅在 PerformanceSoftNavigation 的 name 中公开 first URL value，仍能将影响准确归因回 交互的最终状态。

3.3. 基础设施算法

给定一个 Document document，要 get the current interaction context：

  1. 令 context 为内部 AsyncContext.Variable [[ActiveInteractionContext]] 的值。

  2. 如果 context 不为 null，并且 context 的 context document 不等于 document，返回 null。

  3. 返回 context。

给定一个 Document document、一个 interaction id，以及一个 Event event，要 get the context for an interaction：

  1. 如果 document 的 interaction id to interaction context[interaction id] exists， 返回 document 的 interaction id to interaction context[interaction id]。

  2. 令 interaction context 为一个新的 InteractionContext。

  3. 将 interaction context 的 id 设置为 interaction id。

  4. 将 interaction context 的 context document 设置为 document。

  5. 将 interaction context 的 start time 设置为 event 的 startTime。

  6. Set document 的 interaction id to interaction context[interaction id] 为 interaction context。

  7. 返回 interaction context。

给定一个 Document document 和一个 Event event，要 update the interaction contexts for an event：

  1. 令 timestamp 为给定 document 的 relevant global object 后得到的 current high resolution time。

  2. 如果 event 的 type 是 "scroll"，则令 is scroll 为 true；否则为 false。

  3. 如果 event 的 type 是会触发 has dispatched input event 的事件类型 （如 [EVENT-TIMING] 中定义），则令 is input 为 true；否则为 false。

  4. 如果 is scroll 为 false 且 is input 为 false，则返回。

  5. 对于 document 的 interaction id to interaction context 的 values 中的每个 interaction context：

    1. 如果 is scroll 为 true，并且 interaction context 的 first scroll timestamp 未设置：

      1. 将 interaction context 的 first scroll timestamp 设置为 timestamp。

    2. 如果 is input 为 true，并且 interaction context 的 first input timestamp 未设置：

      1. 将 interaction context 的 first input timestamp 设置为 timestamp。

给定 null 或一个 PerformanceEventTiming 对象 timingEntry，要 set the current interaction context for event dispatch：

  1. 如果 timingEntry 为 null，返回 null。

  2. 令 event 为 timingEntry 的 associated event。

  3. 令 interaction id 为 timingEntry 的 interactionId。

  4. 如果 interaction id 为 null 或 0，返回 null。

  5. 令 document 为 event 的 relevant global object 的 associated Document。

  6. 给定 document 和 event，执行 Update the interaction contexts for an event。

  7. 令 interaction context 为给定 document、interaction id 和 event 后执行 geting the context for an interaction 的结果。

  8. 将内部 AsyncContext.Variable [[ActiveInteractionContext]] 的值设置为 interaction context。

  9. 返回 interaction context。

要 unset the current interaction context after event dispatch：

  1. 将内部 AsyncContext.Variable [[ActiveInteractionContext]] 的值设置为 null。

4. 交互内容绘制

4.1. InteractionContentfulPaint 接口

[Exposed=Window]
interface InteractionContentfulPaint : PerformanceEntry {
    readonly attribute LargestContentfulPaint largestContentfulPaint;
    readonly attribute unsigned long long interactionId;

    object toJSON();
};

InteractionContentfulPaint includes PaintTimingMixin;

每个 InteractionContentfulPaint 都具有：

  • 一个关联的 context，一个 InteractionContext。

  • 一个关联的 largest contentful paint，一个 LargestContentfulPaint entry。

  • 一个关联的 paint timing info。

largestContentfulPaint 属性的 getter 必须返回 this 的 largest contentful paint。

interactionId 属性的 getter 必须返回 this 的 context 的 id。

name 属性的 getter 必须返回空字符串。

entryType 属性的 getter 必须返回 "interaction-contentful-paint"。

startTime 属性的 getter 必须返回 this 的 context 的 start time。

duration 属性的 getter 必须返回 default paint timestamp（针对 this 的 paint timing info）与 this 的 context 的 start time 之间的差值。

注： 虽然该 entry 会从其关联的 InteractionContext 动态查询静态的 交互属性（如 interactionId 和 startTime），但它会在创建时捕获该 paint 的 paint timing info 和候选 LargestContentfulPaint entry 的静态快照，以 确保这些指标反映该视觉更新的确切状态。

4.2. 交互内容绘制算法

给定一个 Window window、一个 InteractionContext interaction context、一个 LargestContentfulPaint lcpEntry，以及一个 paint timing info paintTimingInfo，要 create an interaction contentful paint entry：

  1. 令 entry 为 window 的 realm 中的一个新的 InteractionContentfulPaint 对象。

  2. 将 entry 的 context 设置为 interaction context。

  3. 将 entry 的 largest contentful paint 设置为 lcpEntry。

  4. 将 entry 关联的 paint timing info 设置为 paintTimingInfo。

  5. 返回 entry。

给定一个 LargestContentfulPaint lcpEntry、一个 Document document，以及一个 InteractionContext interaction context，要 process a new attributed contentful paint candidate：

  1. 如果 interaction context 的 first scroll timestamp 已设置 且 lcpEntry 的 renderTime 大于 interaction context 的 first scroll timestamp，则返回。

  2. 如果 interaction context 的 first input timestamp 已设置，且 lcpEntry 的 renderTime 大于 interaction context 的 first input timestamp，则返回。

  3. 令 global 为 document 的 relevant global object。

  4. 令 paint timing info 为 lcpEntry 关联的 paint timing info。

  5. 令 icpEntry 为给定 global、interaction context、lcpEntry 和 paint timing info 后执行 create an interaction contentful paint entry 的结果。

  6. Queue icpEntry。

  7. 将 icpEntry 添加到 global 的 performance entry buffer。

  8. 如果 interaction context 的 first contentful paint 为 null：

    1. 将 interaction context 的 first contentful paint 设置为 icpEntry。

  9. 将 interaction context 的 largest contentful paint 设置为 icpEntry。

注： InteractionContentfulPaint entry 会在被 检测到时被发送到性能时间线，而不取决于该交互最终是否会产生软导航。这 允许开发者监控所有交互的渲染更新。

5. 软导航

5.1. PerformanceSoftNavigation 接口

[Exposed=Window]
interface PerformanceSoftNavigation : PerformanceEntry {
    readonly attribute DOMString navigationType;
    readonly attribute unsigned long long interactionId;
    InteractionContentfulPaint? getLargestInteractionContentfulPaint();
};

PerformanceSoftNavigation includes PaintTimingMixin;

每个 PerformanceSoftNavigation 都具有：

  • 一个关联的 context，一个 InteractionContext。

  • 一个关联的 paint timing info。

navigationType 属性的 getter 必须返回 this 的 context 的 navigation type。

interactionId 属性的 getter 必须返回 this 的 context 的 id。

getLargestInteractionContentfulPaint() 方法必须返回 this 的 context 的 largest contentful paint。

name 属性的 getter 必须返回 this 的 context 的 first URL value。

entryType 属性的 getter 必须返回 "soft-navigation"。

startTime 属性的 getter 必须返回 this 的 context 的 start time。

duration 属性的 getter 必须返回 this 的 presentationTime 与 this 的 context 的 start time 在发出时的差值。

注： 实践中，getLargestInteractionContentfulPaint() 预期会返回一个非 null 的 entry，因为一个已确认的软导航至少需要检测到一个交互内容绘制来 触发其发出。

不过，未来的迭代可以评估：软导航是否可以在 URL 修改时立即视为已提交， 而早于第一次绘制。在这样的模型中，该方法在发出时可能会 返回 null。

这种计时方式的主要设计考量，是对其他时间线 entry（例如 LayoutShift、PerformanceResourceTiming、LongAnimationFrameTiming 等）的归因，这些 entry 发生在 URL 修改和第一次绘制之间的 间隔中。例如，许多站点会在发起 fetch 请求时立即提交新 URL，即使当前页面仍处于 先前的导航状态。为确保时间线切片一致，本规范会将所有 此类 entry 归因到先前的导航标识符，直到第一次绘制确认该转换。

5.2. 软导航算法

soft navigation 是一种 满足以下 条件的同文档导航：

  • 当一个 InteractionContext 处于活动状态时，发生同文档 URL 变更。

  • 发生一次内容绘制，并且该内容绘制被归因到同一个 InteractionContext。

给定一个 Document document 和一个 InteractionContext interaction context，要 evaluate soft navigation emission：

  1. 如果 interaction context 的 emitted 为 true，则返回。

  2. 如果 document 的 active soft navigation candidate 不是 interaction context，则返回。

  3. 如果 interaction context 的 first URL value 未设置，则返回。

  4. 如果 interaction context 的 first contentful paint 为 null， 则返回。

  5. 令 window 为 document 的 relevant global object。

  6. 令 entry 为给定 window 和 interaction context 后执行 create a soft navigation entry 的结果。

  7. 给定 window 和 entry，执行 Emit a soft navigation entry。

  8. 将 interaction context 的 emitted 设置为 true。

给定一个 Window window 和一个 InteractionContext interaction context，要 create a soft navigation entry：

  1. 令 entry 为 window 的 realm 中的一个新的 PerformanceSoftNavigation 对象。

  2. 将 entry 的 context 设置为 interaction context。

  3. 令 first paint 为 interaction context 的 first contentful paint。

  4. 如果 first paint 不为 null：

    1. 将 entry 关联的 paint timing info 设置为 first paint 关联的 paint timing info。

  5. 返回 entry。

给定一个 Window window 和一个 PerformanceSoftNavigation entry，要 emit a soft navigation entry：

  1. 给定 window，执行 Increment the current navigation id。

  2. Queue entry。

  3. 将 entry 添加到 window 的 performance entry buffer。

注： 此 navigationId for this PerformanceSoftNavigation entry 会作为将其排入性能时间线的一部分而被自动设置，与其他 PerformanceEntry 类型的行为保持一致。

给定一个 Document document、string url， 以及 string navigation type，要 process a same document commit：

  1. 如果 url 等于 document 的 url，则返回。

  2. 令 interaction context 为给定 document 后执行 getting the current interaction context 的结果。

  3. 如果 interaction context 为 null，则返回。

  4. 将 interaction context 的 last URL value 设置为 url。

  5. 如果 interaction context 的 first URL update timestamp 未设置：

    1. 将 interaction context 的 first URL update timestamp 设置为给定 document 的 relevant global object 后得到的 current high resolution time。

    2. 将 interaction context 的 first URL value 设置为 url。

    3. 将 interaction context 的 navigation type 设置为 navigation type。

  6. 将 document 的 active soft navigation candidate 设置为 interaction context。

  7. 给定 document 和 interaction context，执行 Evaluate soft navigation emission。

6. 规范集成

6.1. HTML 集成

6.1.1. 文档

每个 document 都有一个 interaction id to interaction context，一个 map， 初始为空。

每个 document 都有一个 active soft navigation candidate，一个 InteractionContext 或 null，初始为 null。

6.1.2. 历史

在 update document for history step application 中，在 5.5.1（如果 documentsEntryChanged 为 true 且 documentIsNew 为 false）之前，给定该 Document、 目标 entry 的 url 以及 "traverse"，执行 process a same document commit。
在 shared history push/replace steps （如 [HTML] 中定义）中，在 URL 更新之后，给定 Document、新的 URL，以及操作类型（"push" 或 "replace"），执行 process a same document commit。

6.1.3. 节点

每个 node 都有一个 associated interaction context，初始为 null。

6.1.4. 硬导航

当创建一个新的 Window 时（例如，在一次“硬”导航期间），其 current navigation id 会按照 § 2.1 Navigation ID 中的规定进行初始化。

在 [NAVIGATION-TIMING] 中，当为 初始导航创建 PerformanceNavigationTiming entry 时，用户代理必须将其 navigationId 设置为该 Window 的 current navigation id。

6.2. Event Timing 集成

本规范扩展了 [EVENT-TIMING] 对交互的定义，以包含与现代 Web 应用和单页应用相关的 其他事件类型。

以下事件类型被视为 interaction 的一部分（如 [EVENT-TIMING] 中所定义）：

  • navigate

  • popstate

  • hashchange

当这些事件作为用户交互的结果被分发时，用户代理必须为它们分配 一个唯一的 interaction id，该 id 通过为文档的 relevant global object getting the next interactionId 获得。如果某个事件是由此前已经具有 已分配 interaction id 的交互触发的，则用户代理应复用该同一标识符。

注：这些事件只有在它们是由 用户操作触发的发起事件时（例如点击浏览器 UI 的后退按钮），才会被报告为主要 交互。在大多数其他场景中——例如 click 处理器手动操纵历史记录——后续的 popstate 或 navigate 事件不会被 视为独立的用户交互。

虽然这些以编程方式触发的事件可能没有设置 isTrusted 标志，但它们会 通过 [ASYNC-CONTEXT] 被正确归因回原始交互，因为历史和 导航 API 会被视为发起任务的异步延续。

注： navigate、popstate 和 hashchange 事件被用作内部信号，用于 跟踪软导航并分配 interactionId。本规范 不要求这些 事件类型作为 PerformanceEventTiming entry 暴露到 性能时间线，而是将 该决定留给 [EVENT-TIMING] 规范。

注： [EVENT-TIMING] 规范预期会被 更新，以显式定义 processingStart 和 processingEnd 钩子，并且 确保 interactionId 分配足够早 发生，以支持此集成。目前，interactionId 分配通常会 延迟到事件处理 结束时。

本规范对 interactionId 使用 unsigned long long，以确保一个安全的全局 计数器，而 [EVENT-TIMING] 目前将其定义为 unsigned long。这个不匹配预期 会在两个规范的未来版本中解决。

给定一个 Window window，要 get the next interactionId：

  1. 将 window 的 interaction count 设置为 window 的 interaction count 加 1。

  2. 返回 window 的 initial interactionId value 加上 （window 的 interaction count 乘以 window 的 interactionId increment）。

在 [EVENT-TIMING] 中定义的对 DOM 规范的 event dispatch 算法的修改中：

  1. 紧接在将 timingEntry 设置为 initialize and record event timing processing start 结果的步骤之后，运行以下步骤：

    1. 给定 timingEntry，执行 Set the current interaction context for event dispatch。

  2. 紧接在调用 record event timing processing end 的步骤 之前，运行以下步骤：

    1. 执行 Unset the current interaction context after event dispatch。

6.3. Largest Contentful Paint (LCP) 集成

本规范挂接到 [LARGEST-CONTENTFUL-PAINT] 算法中，以将绘制归因到 交互。

在 报告最大内容绘制中，令 candidates 为 已绘制图像列表 与 已绘制文本列表的并集：

  1. 令 contextToNewLargestCandidate 为一个新的 映射。

  2. 对于 candidates 中的每个 candidate：

    1. 令 element 为 candidate 的 element。

    2. 如果 element 为 null，继续。

    3. 令 interaction context 为 element 的关联交互上下文。

    4. 如果 interaction context 为 null，继续。

    5. 令 size 为 candidate 的 size。

    6. 如果 interaction context 的最大内容 绘制不为 null，且 size 小于或等于 interaction context 的最大内容 绘制的 largestContentfulPaint 的 size， 继续。

    7. 如果 contextToNewLargestCandidate[interaction context] 未设置，或 size 大于 contextToNewLargestCandidate[interaction context] 的 size：

      1. 设置 contextToNewLargestCandidate[interaction context] 为 candidate。

  3. 对于 contextToNewLargestCandidate 中的每个 interaction context → newLcpCandidate：

    1. 令 newLcpEntry 为表示 newLcpCandidate 的 LargestContentfulPaint 条目。

    2. 在给定 newLcpEntry、 document 和 interaction context 的情况下，处理新的 已归因内容绘制候选项。

    3. 在给定 document 和 interaction context 的情况下，评估软导航 发出。

注： § 6.4 Container Timing 集成一节定义了对 DOM 的 修改如何重置受影响元素的 先前已报告绘制，以确保它们由 [LARGEST-CONTENTFUL-PAINT] 算法重新评估，并正确归因到 当前交互。

注： [LARGEST-CONTENTFUL-PAINT] 算法预计会忽略任何具有 关联交互上下文的绘制，因为这些由本规范的 § 4.1 InteractionContentfulPaint 接口处理。

6.4. Container Timing 集成

[CONTAINER-TIMING] API 提供一种机制，可按共同的 DOM 祖先对渲染效果进行分组。本规范与该机制集成，以将渲染变化归因回 用户交互。

在 节点插入时，或当一个 节点以下列方式之一被 修改时：

  • 元素的 class 或 style 属性被修改。

  • 资源属性（例如 img 或 video 元素上的 src）被修改。

运行以下步骤：

  1. 令 document 为该节点的 节点文档。

  2. 令 interaction context 为在给定 document 的情况下，获取当前交互 上下文的结果。

  3. 如果 interaction context 不为 null：

    1. 将该节点的关联交互上下文 设置为 interaction context。

    2. 将该节点指定为 容器根（如 [CONTAINER-TIMING] 中所定义）。

    3. 对于该节点及其每个后代，将它们从 Document 的 先前已报告绘制中移除（如 [PAINT-TIMING] 中所定义）。

注： 通过从 先前已报告绘制集合中移除已修改的子树，用户 代理可确保这些元素的下一次内容绘制会被重新检测，并正确 归因到新的 InteractionContext。

注： 对“节点插入”的引用是一种高层级 描述，涵盖所有向文档结构中添加 新元素的操作（例如 appendChild、innerHTML 更新等）。具体如何将这些映射到跟踪文档结构 修改的内部实现钩子，由用户代理决定。

注： 鼓励用户代理维护一棵稀疏的 容器根树。如果被 修改的节点已经是与同一 交互上下文关联的现有容器根的后代，用户代理可以选择不创建新的冗余根。

此外，为避免在大型 DOM 修改期间产生昂贵的主线程工作，鼓励用户代理 优化从先前已报告绘制中移除后代的过程。用户代理可以不 立即进行穷尽式遍历，而是将已修改的根标记为“脏”，并在后续树遍历期间 （例如在布局或绘制期间）惰性地 传播此状态。已知不可见或被现有机制（例如 content-visibility 或 CSS containment）跳过的子树，也可以在此重置过程中跳过。

7. 安全与隐私考量

将软导航暴露到性能时间线本身并不会带来安全和隐私影响。 但是，作为检测到软 导航的结果而重置各种绘制定时条目可能会产生影响，尤其是在已访问链接被 分区之前。因此，在未对 :visited 缓存进行分区的情况下暴露 此类绘制操作，只能在仔细 分析相关绘制操作之后进行，以确保它们不会跨源暴露用户的历史记录。

8. 附录：重叠交互和竞态条件

本节为非规范性内容。

Web 应用通常会快速连续地处理多个用户交互。本规范 通过以下模型处理这种重叠交互：

  • 上下文独立性：每个交互都会独立管理自己的 InteractionContext。 多个上下文可以同时处于“进行中”状态，每个上下文都会跟踪自己的 URL 修改和 内容绘制。

  • 活动候选单例：虽然许多交互都可以处于活动状态，但 Document 在任意给定时间只识别一个 active soft navigation candidate。

  • 抢占：交互上下文在触发其第一次同文档 URL 修改的那一刻，会成为 active soft navigation candidate。 如果随后发生另一个交互修改，它会抢占前一个交互并成为新的候选。

  • 发出验证：只有在满足发出 条件（URL 变更 + 内容绘制）的那一刻，作为 active soft navigation candidate 的交互 上下文，才会发出 PerformanceSoftNavigation。 这可确保软导航报告 与文档当前的视觉和导航状态保持一致。

  • 持续归因：即使一个交互作为软导航 候选被抢占， 只要它仍然处于活动状态，它就会继续归因并报告自己的 InteractionContentfulPaint entry。这允许准确测量并发渲染更新，即使这些更新本身 并不被视为“导航”。
