1. 简介

本节不是规范性的。

Web 应用程序通常需要在客户端处理 HTML 字符串，可能是作为客户端模板解决方案的一部分，或者是为了渲染用户生成的内容等。以安全的方式进行处理是非常困难的。将字符串简单拼接并插入到 Element 的 innerHTML 中是一种简单但充满风险的方法，因为它可能会以多种意想不到的方式导致 JavaScript 执行。

像 [DOMPURIFY] 这样的库试图通过在插入之前仔细解析和清理字符串来解决这个问题，它们通过构建 DOM 并将其成员过滤通过一个允许列表来完成。然而，这种方法已被证明是脆弱的，因为暴露给 Web 的解析 API 并不总是合理地映射到浏览器在“真实” DOM 中实际渲染字符串时的行为。此外，这些库需要随着浏览器行为的变化不断更新；以前安全的东西可能会随着平台级新特性的出现变成“定时炸弹”。

浏览器在何时执行代码方面有相当好的判断能力。我们可以通过教浏览器如何以安全的方式从任意字符串渲染 HTML，并以与浏览器自身解析器实现的变化保持一致的方式进行维护和更新，来改进用户空间库。本文档概述了一个旨在实现这一目标的 API。

1.1. 目标

  • 通过为开发者提供处理用户控制的 HTML 的机制，防止直接脚本执行以降低基于 DOM 的跨站脚本攻击风险。

  • 确保 HTML 输出在当前用户代理中是安全的，考虑其对 HTML 的当前理解。

  • 允许开发者覆盖默认的元素和属性集。添加某些元素和属性可以防止 脚本小工具 攻击。

1.2. API 概要

Sanitizer API 提供了将包含 HTML 的字符串解析为 DOM 树，并根据用户提供的配置过滤结果树的功能。这些方法有两种类型：

  • 安全和不安全：“安全”方法不会生成任何执行脚本的标记，即它们应该对 XSS 安全。“不安全”方法将解析并过滤它们应处理的内容。

  • 上下文：方法定义在 Element 和 ShadowRoot 上，并将替换这些 Node 的子节点，它们大体上类似于 innerHTML。也有静态方法定义在 Document 上，它们解析整个文档并大体上类似于 DOMParser.parseFromString()。

2. 框架

2.1. Sanitizer API

Element 接口定义了两个方法，setHTML() 和 setHTMLUnsafe()。 两者都接收一个包含 HTML 标记的 DOMString 以及一个可选的配置。

partial interface Element {
[CEReactions] undefined setHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLOptions options = {});
[CEReactions] undefined setHTML(DOMString html, optional SetHTMLOptions options = {});
};
Element的 setHTMLUnsafe(html, options) 方法步骤如下：

  1. 调用 获取受信任类型合规字符串 算法，并将结果赋值给 compliantHTML，该算法接收 TrustedHTML、 this的 相关全局对象、html、"Element setHTMLUnsafe" 和 "script"。

  2. 如果 this 是一个 template 元素，将 target 设为其 template 内容；否则，将 target 设为 this。

  3. 根据 target、this、 compliantHTML、options 和 false 调用 设置并过滤 HTML。

Element的 setHTML(html, options) 方法步骤如下：

  1. 如果 this 是一个 template元素， 则将 target 设为其 template 内容；否则，设为 this。

  2. 根据 target、this、html、 options 和 true 调用 设置并过滤 HTML。 

partial interface ShadowRoot {
  [CEReactions] undefined setHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLOptions options = {});
  [CEReactions] undefined setHTML(DOMString html, optional SetHTMLOptions options = {});
};

这些方法在 ShadowRoot 上也有对应的实现：

ShadowRoot的 setHTMLUnsafe(html, options) 方法步骤如下：

  1. 调用 获取受信任类型合规字符串 算法，并将结果赋值给 compliantHTML，该算法接收 TrustedHTML、 this的 相关全局对象、html、"ShadowRoot setHTMLUnsafe" 和 "script"。

  2. 根据 设置并过滤 HTML，使用 this，将其 shadow host 作为上下文元素，compliantHTML、options 以及 false。

ShadowRoot的 setHTML(html, options) 方法步骤如下：

  1. 根据 设置并过滤 HTML，使用 this（作为目标），this（作为上下文元素）、html、options 以及 true。

Document 接口新增了两个方法，能够解析整个 Document： 

partial interface Document {
  static Document parseHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLOptions options = {});
  static Document parseHTML(DOMString html, optional SetHTMLOptions options = {});
};
parseHTMLUnsafe(html, options) 方法的步骤如下：

  1. 调用 获取受信任类型合规字符串 算法，将结果赋值给 compliantHTML，该算法接收 TrustedHTML、 this 的 相关全局对象、html、"Document parseHTMLUnsafe" 和 "script"。

  2. 创建一个新的 Document， 它的 内容类型 是 "text/html"。

    注意：由于 document 没有浏览上下文，因此脚本执行被禁用。

  3. 将 document 的 允许声明性影子根 设置为 true。

  4. 调用 从字符串解析 HTML，传入 document 和 compliantHTML。

  5. 调用 从选项获取清理器配置，传入 options 和 false，得到 config。

  6. 如果 config 不是 空的， 则对 document 的 根节点 调用 清理，传入 config。

  7. 返回 document。

parseHTML(html, options) 方法的步骤如下：

  1. 创建一个新的 Document， 它的 内容类型 是 "text/html"。

    注意：由于 document 没有浏览上下文，因此脚本执行被禁用。

  2. 将 document 的 允许声明性影子根 设置为 true。

  3. 调用 从字符串解析 HTML，传入 document 和 html。

  4. 调用 从选项获取清理器配置，传入 options 和 true，得到 config。

  5. 对 document 的 根节点 调用 清理，传入 config。

  6. 返回 document。

2.2. SetHTML 选项和配置对象。

setHTML()-系列方法都接受一个选项字典。目前，只有该字典中的一个成员被定义： 

dictionary SetHTMLOptions {
  (Sanitizer or SanitizerConfig) sanitizer = {};
};

Sanitizer 配置对象封装了过滤配置。相同的配置可以用于安全或不安全方法。意图是尽早在页面生命周期中构建一个（或多个）配置，然后在需要时使用。这允许实现对配置的预处理。

配置对象也是可查询的，可以返回 规范化的配置字典，包含安全和不安全变体。这使得页面可以查询并预测给定配置的效果，或基于现有配置构建新配置。

[Exposed=(Window,Worker)]
interface Sanitizer {
  constructor(optional SanitizerConfig config = {});
  SanitizerConfig get();
  SanitizerConfig getUnsafe();
};
构造函数(config) 方法的步骤如下：

  1. 将 config 存储在 this 的 内部槽 中。

get() 方法的步骤如下：

  1. 返回使用 规范化配置 的结果，传入 this 的 内部槽 和 true。

getUnsafe() 方法的步骤如下：

  1. 返回使用 规范化配置 的结果，传入 this 的 内部槽 和 false。

2.3. 配置字典

dictionary SanitizerElementNamespace {
  required DOMString name;
  DOMString? _namespace = "http://www.w3.org/1999/xhtml";
};

// 用于 "elements"
dictionary SanitizerElementNamespaceWithAttributes : SanitizerElementNamespace {
  sequence<SanitizerAttribute> attributes;
  sequence<SanitizerAttribute> removeAttributes;
};

typedef (DOMString or SanitizerElementNamespace) SanitizerElement;
typedef (DOMString or SanitizerElementNamespaceWithAttributes) SanitizerElementWithAttributes;

dictionary SanitizerAttributeNamespace {
  required DOMString name;
  DOMString? _namespace = null;
};
typedef (DOMString or SanitizerAttributeNamespace) SanitizerAttribute;

dictionary SanitizerConfig {
  sequence<SanitizerElementWithAttributes> elements;
  sequence<SanitizerElement> removeElements;
  sequence<SanitizerElement> replaceWithChildrenElements;

  sequence<SanitizerAttribute> attributes;
  sequence<SanitizerAttribute> removeAttributes;

  boolean comments;
  boolean dataAttributes;
};

3. 算法

要设置并过滤HTML，给定一个Element 或DocumentFragment target，一个Element contextElement，一个字符串 html，一个字典 options，以及一个布尔值 safe：

  1. 如果safe为真且contextElement的本地名称是"script"，且 contextElement的命名空间是HTML 命名空间或SVG 命名空间，则返回。

  2. 将config设为调用从选项中获取一个过滤器配置，参数为options和safe。

  3. 将newChildren设为HTML片段解析算法步骤的结果，参数为 contextElement、html和true。

  4. 将fragment设为一个新的DocumentFragment， 其节点文档是contextElement的节点文档。

  5. 对每个 node在newChildren中，追加 node到fragment。

  6. 如果config不为空， 则使用config运行sanitize。

  7. 用fragment替换 target中的所有内容。

要为选项字典options和布尔值safe获取消毒器配置，请执行以下操作：

  1. 断言：options是字典。

  2. 如果options["sanitizer"]不存在，则返回 undefined。

  3. 断言：options["sanitizer"]是Sanitizer实例或字典。

  4. 如果options["sanitizer"]是Sanitizer实例：

    1. 那么让config为options["sanitizer"]的内部槽的值。

    2. 否则，让config为options["sanitizer"]的值。

  5. 返回调用config和safe时的规范化配置的结果。

3.1. 清理算法

对于主要的清理 操作，使用一个ParentNode node，以及规范化的SanitizerConfig config，按以下步骤执行：

  1. 断言：config是规范化的。

  2. 将current设为node。

  3. 对每个child在current的子节点中执行：

    1. 断言：child实现了Text、 Comment、 或Element。

      注意：当前，该算法仅在HTML解析器的输出上调用，且应满足此断言。如果未来该算法在不同的上下文中使用，需要重新检查这一假设。

    2. 如果child实现了Text：

      1. 继续。

    3. 否则如果child实现了Comment：

      1. 如果config的comments不为true：

        1. 移除child。

    4. 否则：

      1. 将elementName设为child的SanitizerElementNamespace，并包括child的本地名称和命名空间。

      2. 如果config["elements"]存在且config["elements"]不包含[elementName]：

        1. 移除child。

      3. 否则如果config["removeElements"]存在且config["removeElements"]包含[elementName]：

        1. 移除child。

      4. 如果config["replaceWithChildrenElements"]存在且config["replaceWithChildrenElements"]包含elementName：

        1. 调用清理 child，并传入config。

        2. 调用替换所有，用child的子节点替换child的内容。

      5. 如果elementName等于«[ "name" →"template"，"namespace" →HTML命名空间 ]»：

        1. 调用清理 child的模板内容，并传入config。

      6. 如果child是影子宿主：

        1. 调用清理 child的影子根，并传入config。

      7. 对每个attr在current的属性列表中：

        1. 将attrName设为SanitizerAttributeNamespace，并包括attr的本地名称和命名空间。

        2. 如果config["attributes"]存在且config["attributes"]不包含attrName：

          1. 如果"data-"是代码单元前缀，并且如果命名空间为null，且config["dataAttributes"]存在且为false：

            1. 从child中移除attr。

        3. 否则如果config["removeAttributes"]存在且config["removeAttributes"]包含attrName：

          1. 从child中移除attr。

        4. 如果config["elements"][elementName]存在，且如果config["elements"][elementName]["attributes"]存在，且如果config["elements"][elementName]["attributes"]不包含attrName：

          1. 从child中移除attr。

        5. 如果config["elements"][elementName]存在，且如果config["elements"][elementName]["removeAttributes"]存在，且如果config["elements"][elementName]["removeAttributes"]包含attrName：

          1. 从child中移除attr。

        6. 如果«[elementName, attrName]»匹配导航URL属性列表中的某个条目，且attr的协议是"javascript:"：

          1. 则从child中移除attr。

        7. 调用清理 child的影子根，并传入config。

      8. 否则：

        1. 移除child。

3.2. 配置处理

一个config是有效的，如果满足以下所有条件：

  1. config是字典

  2. config的键集不同时包含 "elements" 和 "removeElements"

  3. config的键集不同时包含 "removeAttributes" 和 "attributes"。

  4. 对于任何 key «[ "elements", "removeElements", "replaceWithChildrenElements", "attributes", "removeAttributes" ]»，如果config[key] 存在：

    1. config[key]是有效的。

  5. 如果config["elements"] 存在，那么对于任何 element在 config[key]中是字典：

    1. element不同时包含 "attributes" 和 "removeAttributes"。

    2. 如果element["attributes"] 或者 element["removeAttributes"] 存在，那么它是有效的。

    3. 设tmp为字典，并对于任何key «[ "elements", "removeElements", "replaceWithChildrenElements", "attributes", "removeAttributes" ]»将tmp[key]设为规范化的清理器元素列表，并使用HTML命名空间作为元素列表的默认命名空间，使用null作为属性列表的默认命名空间。

      注意：这里的意图是断言列表中的元素，而不考虑是使用字符串快捷方式还是显式字典语法。例如，在elements中包含"img"，而在removeElements中包含{ name: "img" }。实现可能会在此时进行规范化处理。

      1. 给定这些规范化的名称列表，所有以下条件成立：

        1. tmp["elements"] 和tmp["removeElements"] 之间的交集是空的。

        2. tmp["removeElements"] 和tmp["replaceWithChildrenElements"] 之间的交集是空的。

        3. tmp["replaceWithChildrenElements"] 和tmp["elements"] 之间的交集是空的。

        4. tmp["attributes"] 和tmp["removeAttributes"] 之间的交集是空的。

      1. 设tmpattrs为tmp["attributes"] （如果存在）， 否则为内置默认配置["attributes"]。

      2. 对于任何 item在 tmp["elements"]中：

        1. 如果item["attributes"] 或者 item["removeAttributes"] 存在：

          1. 那么它与tmpattrs之间的差集是空的。

一个list的名称是有效的，如果满足以下条件：

  1. list是列表。

  2. 对于所有成员name：

    1. name是字符串 或者字典。

    2. 如果name是字典：

      1. name["name"] 存在且是一个字符串。

一个config是规范的，如果满足以下条件：

  1. config是有效的。

  2. config的键集是子集，包含以下： «[ "elements", "removeElements", "replaceWithChildrenElements", "attributes", "removeAttributes", "comments", "dataAttributes" ]»

  3. config的键集包含 其中之一：

    1. 既包含"elements" 又包含"attributes"， 但不包含 "removeElements" 或"removeAttributes"。

    2. 或者既包含 "removeElements" 又包含"removeAttributes"， 但不包含 "elements" 或"attributes"。

  4. 对于以下的任何key «[ "replaceWithChildrenElements", "removeElements", "attributes", "removeAttributes" ]»，如果config[key] 存在：

    1. config[key]是规范的。

  5. 如果config["elements"] 存在：

    1. config["elements"] 是规范的。

  6. 对于以下的任何key «[ "comments", "dataAttributes" ]»：

    1. 如果config[key] 存在，config[key] 是布尔值。

一个list的名称是规范的，如果满足以下条件：

  1. list[key]是列表。

  2. 对于所有其list[key]的成员 name：

    1. name是字典。

    2. name的键集等于«[ "name", "namespace" ]»

    3. name的值是字符串。

一个list的名称是规范的，如果满足以下条件：

  1. list[key]是列表。

  2. 对于所有其list[key]的成员 name：

    1. name是字典。

    2. name的键集 等于以下之一：

      1. «[ "name", "namespace" ]»

      2. «[ "name", "namespace", "attributes" ]»

      3. «[ "name", "namespace", "removeAttributes" ]»

    3. name["name"] 和name["namespace"] 是字符串。

    4. name["attributes"] 和name["removeAttributes"] 是规范的，如果它们存在。

要规范化配置 config 并使用一个布尔值 safe：

注意：初始的断言将断言内置常量的属性，比如默认值，以及已知的元素和属性列表。

  1. 断言：内置默认配置是规范的。

  2. 断言：内置默认配置["elements"]是已知元素的子集。

  3. 断言：内置默认配置["attributes"]是已知属性的子集。

  4. 断言：«[ "elements" → 已知元素, "attributes" → 已知属性, ]»是规范的。

  5. 如果config是空的并且不是safe，则返回«[]»

  6. 如果config不是有效的，则抛出TypeError。

  7. 将result设为一个新的字典。

  8. 对于以下每个key «[ "elements", "removeElements", "replaceWithChildrenElements" ]»：

    1. 如果config[key]存在，则将result[key]设置为在config[key]上运行规范化的元素列表，默认使用HTML命名空间。

  9. 对于以下每个key «[ "attributes", "removeAttributes" ]»：

    1. 如果config[key]存在，则将result[key]设置为在config[key]上运行规范化的元素列表，默认使用null作为命名空间。

  10. 将result["comments"] 设置为config["comments"]。

  11. 将default设置为规范化内置默认配置的结果。

  12. 如果safe：

    1. 如果config["elements"] 存在：

      1. 让elementBlockList为差集，其计算方式为已知元素与default["elements"]之间的差集。

        注意：强制使用默认元素列表的“自然”方式是与其相交。但是，这样做也会消除任何未知的元素（即，非 HTML 提供的元素，如 <foo>）。因此，我们构建了这个辅助工具，以便能够使用它来减去任何“不安全”的元素。

      2. 将result["elements"] 设置为result["elements"] 与elementBlockList之间的差异。

    2. 如果config["removeElements"] 存在：

      1. 将result["elements"] 设置为default["elements"] 与result["removeElements"] 之间的差异。

      2. 移除 "removeElements" 从result中。

    3. 如果config["elements"] 和config["removeElements"] 都不存在：

      1. 将result["elements"] 设置为default["elements"]。

    4. 如果config["attributes"] 存在：

      1. 让attributeBlockList为差集，其计算方式为已知属性与default["attributes"]之间的差集。

      2. 将result["attributes"] 设置为result["attributes"] 与attributeBlockList之间的差异。

    5. 如果config["removeAttributes"] 存在：

      1. 将result["attributes"] 设置为default["attributes"] 与result["removeAttributes"] 之间的差异。

      2. 移除 "removeAttributes" 从result中。

    6. 如果config["attributes"] 和config["removeAttributes"] 都不存在：

      1. 将result["attributes"] 设置为default["attributes"]。

  13. 否则（如果不是safe）：

    1. 如果config["elements"] 和config["removeElements"] 都不存在：

      1. 将result["elements"] 设置为default["elements"]。

    2. 如果config["attributes"] 和config["removeAttributes"] 都不存在：

      1. 将result["attributes"] 设置为default["attributes"]。

  14. 断言：result 是有效的。

  15. 断言：result 是规范的。

  16. 返回result。

为了规范化一个 sanitizer 元素列表list，并使用默认命名空间defaultNamespace，请执行以下步骤：

  1. 让result成为一个新的有序集合。

  2. 对于 list中的每个name，调用规范化一个sanitizer名称，使用defaultNamespace，并将其 附加到result。

  3. 返回result。

为了规范化一个 sanitizer名称name，并使用默认命名空间defaultNamespace，请执行以下步骤：

  1. 断言： name是DOMString或者dictionary。

  2. 如果name是DOMString， 则返回«[ "name" → name, "namespace" → defaultNamespace]»。

  3. 断言： name是dictionary并且name["name"]存在。

  4. 返回«[
    "name" → name["name"],
    "namespace" → name["namespace"]（如果它存在），否则 defaultNamespace
    ]»。

3.3. 支持算法

对于该规范中使用的规范化 元素 和属性名列表， 列表成员资格基于同时匹配 "name" 和 "namespace" 条目： Sanitizer 名称list包含一个item，如果存在list的entry是一个 有序映射，并且 item["name"] 等于 entry["name"]，且item["namespace"] 等于 entry["namespace"]。
集合差异（或集合减法）是集合A的克隆，但移除了集合B中出现的所有成员： 计算两个有序集合 A和B的差异：

  1. 让set成为一个新的有序集合。

  2. 对于A的每个item：

    1. 如果B没有包含item，则附加 item到set。

  3. 返回set。

对于有序集合的 相等性是成员的相等，但不考虑顺序：有序集合A和B是相等，如果A是 B的超集，且B是 A的超集。

3.4. 默认值

注意：默认值应遵循特定形式，且会在规范化配置开始时进行检查。

内置默认配置如下：

{
  elements: [....],
  attributes: [....],
  comments: true,
}

已知元素如下：

[
  { name: "div", namespace: "http://www.w3.org/1999/xhtml" },
  ...
]

已知属性如下：

[
  { name: "class", namespace: null },
  ...
]

注意：已知元素和已知属性应当从HTML5规范中派生，而不是明确列在这里。目前， 尚无相关机制实现此功能。

导航URL属性列表， 其中"javascript:"导航是不安全的，如下所示：

4. 安全性考虑

Sanitizer API 旨在通过遍历提供的 HTML 内容并根据配置移除元素和属性来防止基于 DOM 的跨站脚本攻击。 指定的 API 必须不支持构建保留可执行脚本标记的 Sanitizer 对象，若出现这种情况，将视为威胁模型中的漏洞。

尽管如此，Sanitizer API 的正确使用仍无法防范某些安全问题，接下来的章节将描述这些场景。

4.1. 服务器端反射与存储型 XSS

本节非规范性。

Sanitizer API 仅在 DOM 中运行，提供遍历和过滤现有 DocumentFragment 的能力。 Sanitizer 并不处理服务器端反射或存储型 XSS。

4.2. DOM 遮蔽攻击

本节非规范性。

DOM 遮蔽攻击是一种利用恶意 HTML 通过id或name属性命名元素，从而使 DOM 中的 HTML 元素属性如children被恶意内容覆盖的攻击。

Sanitizer API 默认状态下无法防御 DOM 遮蔽攻击，但可以配置为移除id和name属性。

4.3. 通过脚本工具的 XSS

本节非规范性。

脚本工具是一种攻击技术，攻击者利用现有应用程序代码，特别是流行的 JavaScript 库中的代码，来执行其自身的代码。 这通常是通过注入看似无害的代码或无害的 DOM 节点完成的，而这些节点仅在框架解析时被执行。

Sanitizer API 无法防御这些攻击，但要求页面作者明确允许未知元素，并且必须显式配置未知属性和元素， 以及像 data- 和 slot 属性以及 <slot> 和 <template> 之类的模板和框架特定的代码。 我们认为这些限制并不全面，建议页面作者检查其第三方库的此类行为。

4.4. 变异型 XSS

本节非规范性。

变异型 XSS 或 mXSS 描述了一种基于解析器上下文不匹配的攻击，即在没有正确上下文的情况下解析 HTML 片段。 尤其是，当已解析的 HTML 片段被序列化为字符串后，该字符串在插入不同父元素时不一定能保证被相同地解析和解释。 此类攻击的一个示例是依赖于外来内容或嵌套标签的解析行为变化。

Sanitizer API 仅提供将字符串转换为节点树的功能。 上下文由所有 sanitizer 函数隐式提供：Element.setHTML() 使用当前元素；Document.parseHTML() 创建一个新文档。 因此 Sanitizer API 直接不受变异型 XSS 的影响。

如果开发者检索已消毒的节点树作为字符串（例如通过 .innerHTML），然后再次解析它，则可能发生变异型 XSS。 我们不建议这样做。如果在处理或传递 HTML 字符串时确有必要，任何字符串在插入 DOM 时应被视为不可信并再次进行消毒。 换句话说，已消毒且随后被序列化的 HTML 树不能再被视为已消毒。

更完整的 mXSS 处理可以参考 [MXSS]。

5. 致谢

Cure53 的 [DOMPURIFY] 是本文档所描述 API 的一个重要启发来源，正如 Internet Explorer 的 window.toStaticHTML()。
