摘要

本规范定义了一个 API，允许已安装的 Web 应用 设置应用徽章，通常会显示在设备主屏幕或应用程序坞上的应用图标旁边。

1. 用法示例

本节为非规范性内容。

示例 1：在应用图标上显示未读数量 
async function updateMailBadge() {
  // 检查 API 是否支持。
  if (!navigator.setAppBadge) return;

  const unreadCount = await getUnreadMailCount();

  // 尝试设置应用徽章。
  try {
    await navigator.setAppBadge(unreadCount);
  } catch (e) {
    // 徽章不支持，或者用户阻止了应用设置徽章。
  }
}

徽章可能会显示在操作系统中的应用图标上。如果在同一个应用内多次调用 API 来设置 或清除徽章，以最新调用为准，并且即使应用关闭后也可能继续显示。

示例 2：在应用图标上显示就绪状态 
async function showPlayerTurn(playerTurnId) {
  if (playerTurnId === localPlayerId)
    await navigator.setAppBadge();
  else
    await navigator.clearAppBadge();
}

在某些操作系统上，设置徽章可能需要用户授权。在这种情况下，开发者需要先查询 "通知"权限状态，然后再设置徽章。如果未获得授权，开发者需要通过 Notification.requestPermission() 进行权限申请。

示例 3：检查权限 
async function checkPermission() {
  permission = await navigator.permissions.query({
    name: "notifications",
  });
  const button = document.getElementById("permission-button");
  if (permission.state === "prompt") {
    // 提示用户授权。
    button.hidden = false;
    button.addEventListener("click", async () => {
      await Notification.requestPermission();
      checkPermission();
    }, { once: true });
    return;
  }
  button.hidden = true;
}

2. 模型

徽章旨在为已安装的 Web 应用 提供一种机制，用于通知用户有新的活动可能需要关注，或者用于指示一些简要信息，如未读数量。

徽章可以有以下值之一：

特殊值 "nothing"：
表示当前没有设置徽章。
特殊值 "flag"：
表示徽章已设置，但不包含具体数值。
数字值：
表示徽章已设置为大于0的数值。

已安装的 Web 应用有一个关联的徽章，其初始化值为"nothing"。

用户代理可以（重新）设置应用的徽章为"nothing"，以其自行决定（例如遵循系统惯例）。

3. 徽章显示

当应用的徽章被设置时，用户代理或操作系统应当 在用户操作系统的应用主要图标旁显示徽章（例如，在设备主屏幕上的应用图标上叠加一个小标记）。

用户代理可以要求用户明确授权后才能设置徽章。当用户代理要求此类权限时，应当将授权与"通知"权限绑定。

当徽章被设置为"flag"时，用户代理 或操作系统应当显示一个非具体符号的指示器（例如，一个有颜色的圆点）。

当徽章的值被设置为"nothing"时，用户代理或操作系统 应当清除徽章，即不再显示徽章。

当徽章被设置为数字时，用户代理 或操作系统：

  • 应当根据用户字体和格式化偏好格式化并显示数字。例如，在设备主屏幕上的应用图标上用圆圈显示数字。
  • 可以根据平台惯例简化或降级徽章的数字值。例如，平台可能会将数值为'100'的徽章显示为'99+'。
  • 应当根据用户的语言环境本地化数字。例如，值'7'在'en'（英语）环境下应显示为'7'，在'ar'（阿拉伯语）环境下应显示为'٧'。
  • 可以将数字值视为"flag"，或者采用最适合平台的展示方式，如果平台不支持在徽章上显示数字。

5. 设置应用徽章

要将平台对象context的应用徽章设置为可选的unsigned long long contents值：

  1. 令global为context的相关全局对象。
  2. 如果global是Window 对象，则：
    1. 令document为global的关联Document。
    2. 如果document不是完全激活状态，则返回一个被拒绝的 promise，错误为"InvalidStateError" DOMException。
    3. 如果document的相关设置对象的origin与this的相关设置对象的顶级 origin不是同源域，则返回一个被拒绝的 promise，错误为"SecurityError" DOMException。
  3. 令promise为一个新的 promise。
  4. 并行执行：
    1. 如果用户代理要求明确授权才能设置应用徽章，则：
      1. 令permissionState为使用"notifications"获取当前权限状态的结果。
      2. 如果permissionState不是"granted"， 在global上队列一个全局任务，任务源为用户交互任务源， 以rejectpromise，错误为NotAllowedError 并终止此算法。
    2. 根据contents的情况分支：
      未传递contents：
      设置badge为"flag"。
      contents为0：
      设置badge为"nothing"。
      contents：
      设置badge为contents。
    3. 在global上队列一个全局任务，任务源为DOM 操作任务源 ，以resolvepromise，值为undefined。
  5. 返回promise。

6. 隐私注意事项

该 API 设计为只写。站点无法读取之前设置的徽章值，以确保应用徽章不能被用作存储或指纹识别机制。

7. 安全注意事项

用户代理或操作系统可以根据自身决定清除徽章 ，并遵循任何系统惯例（例如系统重置时）。

Issue 68：限制设置徽章的频率

由 @grorg 在 webkit-dev提出：

我也希望看到一些规范文本，描述浏览器应当忽略在极短时间内快速执行的多个 set/clear 操作（比如用来制造闪烁徽章的情况）——也许限制为每分钟只能有一次徽章操作？

我们应当讨论这个问题；这可能更适合作为建议而非强制要求，因为它与用户代理的 UI 有关。

我希望能有一种“最终一致性”保证，即你最终设置的徽章值会最终显示给用户。这可以防止出现你先设置“3”，10秒后设置“12”，但因为频率限制，“12”永远没被设置，用户一直看到“3”。建议规则为“如果 UI 超过 N 秒未更新，则更新为新值，否则设定一个定时器，在 N - 距离上次更新的秒数后更新为新值”。
