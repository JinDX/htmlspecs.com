No-Vary-Search HTTP 缓存扩展

摘要

本规范定义了 HTTP 缓存的一项扩展，用于改变 URI 查询 参数对缓存的影响。¶

1. 引言

HTTP 缓存 [HTTP-CACHING] 基于复用在若干缓存键上匹配的资源，其中最重要的是所呈现的目标 URI（[HTTP] 的第 7.1 节）。然而， 有时多个 URI 可以表示同一资源。这会导致缓存并不总是像本可做到的那样有用：如果缓存中包含某个 URI 下的响应， 但随后又在另一个 URI 下请求该响应，则缓存版本将被忽略。¶

`No-Vary-Search` 响应标头字段定义了一种缓存扩展，如 [HTTP-CACHING] 的第 4 节 所述， 它处理这一一般问题的一个特定子集，即仅在某些查询参数上不同的 URI 标识同一资源的情形。 它允许资源声明查询组件的某些或全部部分不会在语义上影响所提供的响应，因此可在缓存匹配目的中被忽略。 例如，如果查询参数的顺序不会影响所标识的资源，则可使用以下方式表示¶ 

No-Vary-Search: key-order
¶

如果特定查询参数（例如用于分析的参数）不会在语义上影响所提供的资源， 则可使用以下方式表示¶

No-Vary-Search: params=("utm_source" "utm_medium" "utm_campaign")
¶

如果资源反而想采用基于允许列表的方法，即只有 某些已知查询参数会在语义上影响所提供的响应，它们可以使用¶

No-Vary-Search: except=("productId")
¶

请注意，通过发送唯一查询参数以避免取回 缓存响应的“缓存清除”可能会被 No-Vary-Search 标头字段变得无效。¶

第 3 节 使用 [STRUCTURED-FIELDS] 框架定义新的 响应标头字段 No-Vary-Search。第 4 节 和 第 5 节 说明字段值如何在规范中表示的数据模型，以及 将结构化字段解析器的原始输出解析为该数据模型的过程。第 6 节 给出用于比较 两个 URL 在该标头字段影响下是否等价的关键算法；值得注意的是，它依赖 [WHATWG-URL] 中指定的 application/x-www-form-urlencoded 格式 对查询组件给出的键和值分解。（因此，此 标头字段不适用于查询组件不遵循该格式的 URL。）最后，第 7 节 解释如何扩展 [HTTP-CACHING] 的第 4 节，以考虑这种新的等价性。¶

3. HTTP 标头字段定义

No-Vary-Search HTTP 标头字段是一个结构化字段 [STRUCTURED-FIELDS]，其值 MUST 是一个字典（[STRUCTURED-FIELDS] 的 第 3.2 节）。¶

它有以下约束：¶

  • 如果存在，key-order 条目的值 MUST 是一个布尔值（[STRUCTURED-FIELDS] 的 第 3.3.6 节）。¶

  • 如果存在，params 条目的值 MUST 是一个字符串内部列表（[STRUCTURED-FIELDS] 的 第 3.1.1 节）。¶

  • 如果存在，except 条目的值 MUST 是一个字符串内部列表（[STRUCTURED-FIELDS] 的 第 3.1.1 节）。¶

  • 如果 params 条目也存在，则 except 条目 MUST NOT 存在。¶

该字典 MAY 包含键不是 key-order、params 和 except 之一的条目，但其含义不由 本规范定义。本规范的实现将忽略这些条目（但未来的文档可能会为这些条目赋予含义）。¶

解析算法在第 5.2 节中定义。¶

4. 数据模型

一个 URL 变化配置由以下内容组成：¶

no-vary params

特殊值 wildcard 或一个字符串列表¶

vary params

特殊值 wildcard 或一个字符串列表¶

vary on key order

一个布尔值¶

默认 URL 变化 配置是一个 URL 变化配置，其 no-vary params 是一个空列表，vary params 是 wildcard，且 vary on key order 为 true。¶

获取 URL 变化配置 算法（第 5.2 节）会确保 所有 URL 变化配置都遵守以下约束：¶

  • 当且仅当 no-vary params 是 wildcard 时，vary params 是一个列表；并且¶

  • 当且仅当 vary params 是 wildcard 时，no-vary params 是一个列表。¶

5. 解析

5.1. 解析 URL 变化 配置

要在给定 value 的情况下解析 URL 变化 配置：¶

  1. 如果 value 为 null，则返回默认 URL 变化 配置。¶

  2. 令 result 为一个新的 URL 变化配置。¶

  3. 将 result 的 vary on key order 设置为 true。¶

  4. 如果 value["key-order"] 存在：¶

    1. 如果 value["key-order"] 不是 布尔值，则返回默认 URL 变化配置。¶

    2. 将 result 的 vary on key order 设置为 value["key-order"] 的布尔 否定。¶

  5. 如果 value["params"] 和 value["except"] 都存在，或者二者都不存在，则返回默认 URL 变化 配置。¶

  6. 如果 value["params"] 存在：¶

    1. 如果 value["params"] 不是一个 数组，则返回默认 URL 变化配置。¶

    2. 如果 value["params"] 中的任何项 不是字符串，则返回默认 URL 变化配置。¶

    3. 将 result 的 no-vary params 设置为 将解析键（第 5.3 节）应用于 value["params"] 中每一项所得的结果。¶

    4. 如果 result 的 no-vary params 中的任何项是 错误，则返回默认 URL 变化配置。¶

    5. 将 result 的 vary params 设置为 wildcard。¶

  7. 否则，如果 value["except"] 存在：¶

    1. 如果 value["except"] 不是一个 数组，则返回默认 URL 变化配置。¶

    2. 如果 value["except"] 中的任何项 不是字符串，则返回默认 URL 变化配置。¶

    3. 将 result 的 vary params 设置为 将解析键（第 5.3 节）应用于 value["except"] 中每一项所得的结果。¶

    4. 如果 result 的 vary params 中的任何项是 错误，则返回默认 URL 变化配置。¶

    5. 将 result 的 no-vary params 设置为 wildcard。¶

  8. 返回 result。¶

一般来说，此算法是严格的，并且在看到无法识别的内容时往往会返回默认 URL 变化 配置。这是因为默认 URL 变化 配置的行为只会导致更少的缓存命中，这是一种可接受的回退 行为。¶

不过，顶层无法识别的键会被忽略，以便将来更容易扩展本 规范。为避免现有客户端软件出现错误行为，此类扩展很可能会扩大，而不是缩小， 缓存响应可以匹配的请求集合。¶

此算法的输入通常是通过将结构化字段（[STRUCTURED-FIELDS] 的 第 4.2 节）按 field_type "dictionary" 解析而获得的。¶

5.2. 获取 URL 变化 配置

要在给定 HTTP 响应（[HTTP] 的 第 3.4 节） response 的情况下获取 URL 变化 配置：¶

  1. 令 fieldValue 为从 response 中将 No-Vary-Search 字段作为 Dictionary（[STRUCTURED-FIELDS] 的 第 4.2 节）进行解析所得的结果。¶

  2. 返回在给定 fieldValue 的情况下解析 URL 变化配置（第 5.1 节） 所得的结果。¶

5.2.1. 示例

以下内容根据各种输入对最终 no-vary params 和 vary params 的影响， 说明它们会如何被解析：¶

5.3. 解析键

要在给定 ASCII 字符串 keyString 的情况下解析键：¶

  1. 令 keyBytes 为 keyString 的同构编码 [WHATWG-INFRA]。¶

  2. 将 keyBytes 中的任何 0x2B (+) 替换为 0x20 (SP)。¶

  3. 令 keyBytesDecoded 为对 keyBytes 进行百分号解码 [WHATWG-URL] 的结果。¶

  4. 令 keyStringDecoded 为对 keyBytesDecoded 进行无 BOM 的 UTF-8 解码 [WHATWG-ENCODING] 的结果。¶

  5. 返回 keyStringDecoded。¶

5.3.1. 示例

解析键算法允许在 ASCII 结构化标头字段格式中 编码非 ASCII 键字符串，类似于 application/x-www-form-urlencoded 格式 [WHATWG-URL] 允许 在 URI（它被限制为 ASCII 字符）中编码由键和值组成的整个条目列表。例如，¶

No-Vary-Search: params=("%C3%A9+%E6%B0%97")
¶

将会得到一个 URL 变化配置，其 vary params 为 « "é 気" »。正如后面的示例所解释的那样，等价性测试期间的规范化过程 意味着这会将如下 URI 视为等价：¶

  • https://example.com/?é 気=1¶

  • https://example.com/?é+気=2¶

  • https://example.com/?%C3%A9%20気=3¶

  • https://example.com/?%C3%A9+%E6%B0%97=4¶

等等，因为它们都会被解析 [WHATWG-URL] 为具有相同的 键 "é 気"。¶

6. 比较

给定 URL 变化配置 variationConfig 时，如果以下算法返回 true，则两个 URL [WHATWG-URL] urlA 和 urlB 在变化配置下等价：¶

  1. 如果 urlA 和 urlB 的方案、用户名、密码、主机、端口或路径不同，则返回 false。¶

  2. 如果 variationConfig 等价于默认 URL 变化配置， 则：¶

    1. 如果 urlA 的查询等于 urlB 的查询，则 返回 true。¶

    2. 返回 false。¶

    在这种情况下，即使 URL 对在对其查询运行 application/x-www-form-urlencoded 解析器 [WHATWG-URL] 后 可能看起来相同，例如 https://example.com/a 和 https://example.com/a?，或 https://example.com/foo?a=b&&&c 和 https://example.com/foo?a=b&c=，也会被视为不等价。¶

  3. 令 searchParamsA 和 searchParamsB 为空列表。¶

  4. 如果 urlA 的查询不是 null，则将 searchParamsA 设置为 在给定 urlA 的查询的 同构编码 [WHATWG-INFRA] 的情况下， 运行 application/x-www-form-urlencoded 解析器 [WHATWG-URL] 所得的结果。¶

  5. 如果 urlB 的查询不是 null，则将 searchParamsB 设置为 在给定 urlB 的查询的 同构编码 [WHATWG-INFRA] 的情况下， 运行 application/x-www-form-urlencoded 解析器 [WHATWG-URL] 所得的结果。¶

  6. 如果 variationConfig 的 no-vary params 是一个列表，则：¶

    1. 将 searchParamsA 设置为一个列表，其中包含 searchParamsA 中 variationConfig 的 no-vary params 不包含 pair[0] 的那些项 pair。¶

    2. 将 searchParamsB 设置为一个列表，其中包含 searchParamsB 中 variationConfig 的 no-vary params 不包含 pair[0] 的那些项 pair。¶

  7. 否则，如果 variationConfig 的 vary params 是一个列表，则：¶

    1. 将 searchParamsA 设置为一个列表，其中包含 searchParamsA 中 variationConfig 的 vary params 包含 pair[0] 的那些项 pair。¶

    2. 将 searchParamsB 设置为一个列表，其中包含 searchParamsB 中 variationConfig 的 vary params 包含 pair[0] 的那些项 pair。¶

  8. 如果 variationConfig 的 vary on key order 为 false，则：¶

    1. 令 keyLessThan 为一个算法，输入为两个 对（keyA、valueA）和（keyB、valueB），该算法 返回 keyA 是否按代码单元小于 [WHATWG-INFRA] keyB。¶

    2. 将 searchParamsA 设置为使用 keyLessThan 对 searchParamsA 进行升序排序 [WHATWG-INFRA] 所得的结果。¶

    3. 将 searchParamsB 设置为使用 keyLessThan 对 searchParamsB 进行升序排序 [WHATWG-INFRA] 所得的结果。¶

  9. 如果 searchParamsA 的大小不等于 searchParamsB 的 大小，则返回 false。¶

  10. 令 i 为 0。¶

  11. 当 i < searchParamsA 的大小时：¶

    1. 如果 searchParamsA[i][0] 不等于 searchParamsB[i][0]，则返回 false。¶

    2. 如果 searchParamsA[i][1] 不等于 searchParamsB[i][1]，则返回 false。¶

    3. 将 i 设置为 i + 1。¶

  12. 返回 true。¶

6.1. 示例

由于 application/x-www-form-urlencoded 解析器对查询 字符串进行规范化的方式，有些查询字符串虽然看起来并不明显等价，但在解析后最终会被视为等价。¶

因此，例如，给定 No-Vary-Search 的任何非默认值， 如 No-Vary-Search: key-order，我们将得到以下等价关系：¶

表 3
第一个查询 第二个查询 说明
null ? null 查询与空字符串解析结果相同
?a=x ?%61=%78 解析会执行百分号解码
?a=é ?a=%C3%A9 解析会执行百分号解码
?a=%f6 ?a=%ef%bf%bd 两个值都会被解析为 U+FFFD (�)

?a= ?a 二者都会解析为具有空字符串值的 a

7. 缓存

如果缓存 [HTTP-CACHING] 实现本规范，则 [HTTP-CACHING] 的第 4 节 中的呈现目标 URI 要求：¶

所呈现的目标 URI（[HTTP] 的第 7.1 节）与所存储响应的目标 URI 匹配，且¶

替换为：¶

所呈现的目标 URI（[HTTP] 的第 7.1 节）与所存储响应的目标 URI 匹配或 按 URL 变体配置模等价，且¶

缓存实现 MAY 不复用其目标 URI 仅按 URL 变体配置模匹配的已存储响应，如果该缓存最近已 存储了一个响应，该响应：¶

  • 具有一个与所呈现目标 URI 相等的目标 URI（排除 query），并且¶

  • 其 No-Vary-Search 字段具有非空值，并且¶

  • 其 No-Vary-Search 字段值不同于正被考虑复用的 已存储响应。¶

一般来说，并不要求缓存复用已存储响应。然而，上述内容 明确授权缓存在出于性能或其他原因有利时，搜索 更少数量的已存储响应。¶

也就是说，因为缓存可能为给定 pathname 存储多个响应， 它们需要一种无需访问所有缓存响应即可高效查找 No-Vary-Search 值的方法。 这样的缓存可能会采取如下步骤，以高性能方式识别一个已存储响应， 然后再检查 [HTTP-CACHING] 的第 4 节 中的其他条件：¶

  1. 令 exactMatch 为 cache[presentedTargetURI]。如果它是可复用的已存储 响应，则返回它。¶

  2. 令 targetPath 为移除查询参数后的 presentedTargetURI。¶

  3. 令 lastNVS 为 mostRecentNVS[targetPath]。如果它不存在，则返回 null。¶

  4. 令 simplifiedURL 为根据 lastNVS 简化 presentedTargetURI 的结果（通过移除不重要的查询参数，并在应忽略键顺序时 按键升序排序 [WHATWG-INFRA] 参数）。¶

  5. 令 nvsMatch 为 cache[simplifiedURL]。如果它不存在，则返回 null。（假定这是在写入缓存时，除了精确 URL 外也写入的。）¶

  6. 令 variationConfig 从 nvsMatch 中获取（第 5.2 节）。¶

  7. 如果 nvsMatch 的目标 URI 和 presentedTargetURI 在给定 variationConfig 时 不是按 URL 变体配置模等价（第 6 节）， 则返回 null。¶

  8. 如果 nvsMatch 是可复用的已存储响应，则返回它。 否则，返回 null。¶

为了帮助提高缓存实现效率，服务器 SHOULD NOT 对某个给定 pathname 的请求随时间发送不同的非空 No-Vary-Search 字段值， 除非需要更新它们处理 query 组件的方式。这样做 会导致使用上述策略的缓存实现错过一些本可被复用的已存储响应。¶

8. 安全考虑事项

需要注意的主要风险是 URL 不匹配的影响。具体而言，这可能 导致用户看到的响应最初取自一个不同于其悬停链接时所显示 URL 或 URL 栏中所显示 URL 的 URL。¶

然而，由于影响仅限于查询参数，因此这并没有跨越 相关的安全边界，也就是源（[ORIGIN]）。（或者从Web 浏览器 安全 UI 的角度来看，也许只是 host。 [WHATWG-URL]）事实上，我们 已经给予源对其如何呈现（URL, response body）对的完全控制权， 包括在客户端通过 history.replaceState() 或 Service Worker 等技术实现。¶

9. 隐私考虑事项

本提案邻近高度涉及隐私的导航跟踪领域， 该领域通常使用查询参数传递用户标识符。然而，我们认为本提案 本身没有隐私影响。它不会干扰现有的 导航跟踪缓解措施，也不会干扰任何已知正在考虑的未来措施。事实上，如果 页面在其 URI 中编码用户标识符，本提案提供的唯一能力是 通过阻止服务器处理此类用户 ID（因为服务器会被绕过而使用缓存） 来减少此类用户跟踪。[NAV-TRACKING-MITIGATIONS]¶
