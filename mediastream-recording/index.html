1. 概述

此 API 旨在使基本录制非常简单，同时仍允许更复杂的用例。 在最简单的情况下，应用实例化一个 MediaRecorder 对象，调用 start() 然后调用 stop() 或等待 MediaStreamTrack（s） [GETUSERMEDIA] 结束。录制内容将通过 ondataavailable 事件以平台默认编码的形式提供。可用函数可查询平台支持的一组编码，并在作者需要时选择想要的编码。应用还可以选择一次想要接收多少数据。默认情况下，录制结束时会返回包含整个录制内容的 Blob。然而，应用可以选择在固定间隔接收更小的数据缓冲区。

2. 媒体录制 API

[Exposed=Window]
interface MediaRecorder : EventTarget {
  constructor(MediaStream stream, optional MediaRecorderOptions options = {});
  readonly attribute MediaStream stream;
  readonly attribute DOMString mimeType;
  readonly attribute RecordingState state;
  attribute EventHandler onstart;
  attribute EventHandler onstop;
  attribute EventHandler ondataavailable;
  attribute EventHandler onpause;
  attribute EventHandler onresume;
  attribute EventHandler onerror;
  readonly attribute unsigned long videoBitsPerSecond;
  readonly attribute unsigned long audioBitsPerSecond;
  readonly attribute BitrateMode audioBitrateMode;

  undefined start(optional unsigned long timeslice);
  undefined stop();
  undefined pause();
  undefined resume();
  undefined requestData();

  static boolean isTypeSupported(DOMString type);
};

2.1. 构造函数

MediaRecorder(MediaStream stream, optional MediaRecorderOptions options = {})
当调用 MediaRecorder() 构造函数时，用户代理 MUST（必须）执行下列步骤：
  1. 令 stream 为构造函数的第一个参数。
  2. 令 options 为构造函数的第二个参数。
  3. 如果使用 options 的 mimeType 成员作为参数调用 is type supported 返回 false， 则抛出一个 NotSupportedError DOMException 并中止这些步骤。
  4. 令 recorder 为新构造的 MediaRecorder 对象。
  5. 令 recorder 具有一个内部槽位 [[ConstrainedMimeType]]，初始化为 options 的 mimeType 成员的值。
  6. 令 recorder 具有一个内部槽位 [[ConstrainedBitsPerSecond]]，初始化为 options 的 bitsPerSecond 成员（如果存在），否则为 null。
  7. 令 recorder 具有一个内部槽位 [[VideoKeyFrameIntervalDuration]]，初始化为 options 的 videoKeyFrameIntervalDuration 成员（如果存在），否则为 null。
  8. 令 recorder 具有一个内部槽位 [[VideoKeyFrameIntervalCount]]，初始化为 options 的 videoKeyFrameIntervalCount 成员（如果存在），否则为 null。
  9. 将 recorder 的 stream 属性初始化为 stream。
  10. 将 recorder 的 mimeType 属性初始化为 recorder 的 [[ConstrainedMimeType]] 槽位的值。
  11. 将 recorder 的 state 属性初始化为 inactive。
  12. 将 recorder 的 videoBitsPerSecond 属性初始化为 options 的 videoBitsPerSecond 成员的值（如果存在）。否则，为视频选择一个用户代理认为合理的目标值。
  13. 将 recorder 的 audioBitsPerSecond 属性初始化为 options 的 audioBitsPerSecond 成员的值（如果存在）。否则，为音频选择一个用户代理认为合理的目标值。
  14. 如果 recorder 的 [[ConstrainedBitsPerSecond]] 槽位不是 null，则将 recorder 的 videoBitsPerSecond 和 audioBitsPerSecond 属性设置为用户代理认为对各自媒体类型合理的值，使得 videoBitsPerSecond 与 audioBitsPerSecond 之和接近 recorder 的 [[ConstrainedBitsPerSecond]] 槽位的值。
  15. 如果 recorder 支持由 options 的 audioBitrateMode 成员指定的 BitrateMode， 则将 recorder 的 audioBitrateMode 属性初始化为 options 的 audioBitrateMode 成员的值，否则将 recorder 的 audioBitrateMode 属性初始化为 "variable"。
  16. 返回 recorder。

2.2. 属性

stream, 类型为 MediaStream, 只读
要录制的 MediaStream [GETUSERMEDIA]。
mimeType, 类型为 DOMString, 只读
MediaRecorder 对象使用的 MIME 类型（参见 [RFC2046]）。 用户代理应能够回放其支持的任一录制 MIME 类型。 例如，应能够在 HTML <video> 标签中显示视频录制。
mimeType 通过 type/subtype 组合指定录制的媒体类型和容器格式，并在可能产生歧义的情况下指定 codecs 和/或 profiles 参数 [RFC6381]。 各个 codec 可能还有进一步的可选特定参数。
state, 类型为 RecordingState, 只读
当前 MediaRecorder 对象的状态。
onstart, 类型为 EventHandler
用于处理 start 事件。
onstop, 类型为 EventHandler
用于处理 stop 事件。
ondataavailable, 类型为 EventHandler
用于处理 dataavailable 事件。录制数据的 Blob 包含在该事件中，可通过其 data 属性访问。
onpause, 类型为 EventHandler
用于处理 pause 事件。
onresume, 类型为 EventHandler
用于处理 resume 事件。
onerror, 类型为 EventHandler
用于处理 ErrorEvent。
videoBitsPerSecond, 类型为 unsigned long, 只读
用于对视频轨道进行编码的目标比特率。
audioBitsPerSecond, 类型为 unsigned long, 只读
用于对音频轨道进行编码的目标比特率。
audioBitrateMode, 类型为 BitrateMode, 只读
用于对音频轨道进行编码的 BitrateMode。

2.3. 方法

出于历史原因，以下方法同步更改 state 并异步触发事件。
start(optional unsigned long timeslice)
当调用 MediaRecorder 对象的 start() 方法时，UA 必须执行以下步骤：
  1. 令 recorder 为调用该方法的 MediaRecorder 对象。
  2. 令 timeslice 为方法的第一个参数（如果提供），否则为 undefined。
  3. 令 stream 为 recorder 的 stream 属性的值。
  4. 令 tracks 为 stream 的 track set 中的 live 轨道集合。
  5. 如果 recorder 的 state 属性的值不是 inactive，则抛出 InvalidStateError DOMException 并中止这些步骤。
  6. 如果 stream 的 隔离属性不允许 recorder 访问，则抛出 SecurityError DOMException 并中止这些步骤。
  7. 如果 stream 是 inactive，则抛出 NotSupportedError DOMException 并中止这些步骤。
  8. 如果 [[ConstrainedMimeType]] 槽位指定了媒体类型、容器或编解码，则将 recorder 的配置约束为该槽位中指定的类型、容器和编解码。
  9. 如果 recorder 的 [[ConstrainedBitsPerSecond]] 槽位不为 null，则将 recorder 的 videoBitsPerSecond 和 audioBitsPerSecond 属性设置为 UA 认为可用于 tracks 录制的合理值，使得两者之和接近 recorder 的 [[ConstrainedBitsPerSecond]] 槽值。
  10. 令 videoBitrate 为 recorder 的 videoBitsPerSecond，并将 recorder 配置约束为所有视频轨道的比特率聚合目标为 videoBitrate。videoBitrate 只是编码器的提示，最终可能超过、无法达到或仅长周期内能达到该值。
  11. 令 audioBitrate 为 recorder 的 audioBitsPerSecond，并将 recorder 配置约束为所有音频轨道的比特率聚合目标为 audioBitrate。audioBitrate 只是编码器的提示，最终可能超过、无法达到或仅长周期内能达到该值。
  12. 令 videoKeyFrameIntervalDuration 为 recorder.[[VideoKeyFrameIntervalDuration]]，videoKeyFrameIntervalCount 为 recorder.[[VideoKeyFrameIntervalCount]]。UA 应约束 recorder 配置使视频编码器遵循如下规则：
    • 若 videoKeyFrameIntervalDuration 非 null 且 videoKeyFrameIntervalCount 为 null，则视频编码器会在距上一个关键帧 videoKeyFrameIntervalDuration 毫秒时到达的第一个帧产生关键帧。
    • 若 videoKeyFrameIntervalCount 非 null 且 videoKeyFrameIntervalDuration 为 null，则视频编码器会在距上个关键帧满 videoKeyFrameIntervalCount 帧时到达的首帧产生关键帧。
    • 若二者均非 null，则抛出 NotSupportedError DOMException 并中止这些步骤。
    • 若二者均为 null，UA 可自行决定关键帧的发出时机。

    注意编码器有时会自行决定何时产生关键帧。

  13. 约束 recorder 的配置，使其对所有需要录制的音频轨道采用 BitrateMode，该枚举由 recorder 的 audioBitrateMode 属性决定。
  14. 对于 tracks 内的每个轨道，如果 UA 在当前配置下无法录制，抛出 NotSupportedError DOMException 并中止这些步骤。
  15. 将 recorder 的 state 设置为 recording，并 并行执行以下操作：
    1. 若录制要用的封装和编解码器尚未确定，UA 会在 recorder 当前配置中指定。UA 可根据 tracks 的轨道来源来决定采用哪种容器和编解码器。
      通过查看录制开始时 tracks 的来源，UA 可选择可避免转码轨内容的配置。例如，如果 MediaStreamTrack 来源于 RTCPeerConnection 的远端轨，则 UA 可选用与 RTP 流一致的 codec。若 RTP 流中的 codec 录制期间发生变更，UA 需准备转码以避免中断。
    2. 根据 recorder 当前配置录制 tracks 内的所有轨道，将数据收集到 Blob blob。使用 DOM 操作任务源排队一个任务，执行下列操作：
      1. 令 extendedMimeType 为 recorder 的 [[ConstrainedMimeType]] 槽值。
      2. 如尚未存在，根据 MediaRecorder 用于录制所有 tracks 的配置，向 extendedMimeType 增加媒体类型、子类型和 codecs 参数。该参数可能包括 [RFC6381] 的 profiles 或更多 codec 特定参数。
      3. 将 recorder 的 mimeType 属性设为 extendedMimeType。
      4. 触发事件，事件名为 start，对象为 recorder。
    3. 若 stream 的 隔离属性在任意时点变化，致使 MediaRecorder 不再被允许访问，应立刻停止收集并丢弃数据，并排队一个任务，任务内容如下：
      1. 停用 recorder，参数为 recorder。
      2. 触发错误事件，事件名为 SecurityError ，对象为 recorder。
      3. 触发 blob 事件，事件名为 dataavailable，对象为 recorder，携带 blob。
      4. 触发事件，事件名为 stop，对象为 recorder。
    4. 若 stream 的 track set 任意时刻有轨道被增加或移除，UA 必须停止采集数据，并通过 DOM 操作任务源排队以下任务：
      1. 停用 recorder，参数为 recorder。
      2. 触发错误事件，事件名为 InvalidModificationError ，对象为 recorder。
      3. 触发 blob 事件，事件名为 dataavailable，对象为 recorder，携带 blob。
      4. 触发事件，事件名为 stop，对象为 recorder。
    5. 如果 UA 因除 隔离属性 或 stream 的 track set 以外的原因无法继续采集，则也要停止采集，并排队以下 DOM 操作任务源任务：
      1. 停用 recorder，参数为 recorder。
      2. 触发错误事件，事件名为 UnknownError ，对象为 recorder。
      3. 触发 blob 事件，事件名为 dataavailable，对象为 recorder，携带 blob。
      4. 触发事件，事件名为 stop，对象为 recorder。
    6. 如果 timeslice 不为 undefined，则每采集到不少于 timeslice 毫秒的数据（或者 UA 的最小分片，两者取大），就开启新 Blob blob 进行收集，然后排队一个 DOM 操作任务源任务 触发 blob 事件，事件名为 dataavailable，对象为 recorder，携带 blob。

      注意 timeslice 为 undefined 视作最大 unsigned long。

    7. 如果录制轨道全部变成 ended，需停止收集数据，排队以下任务：
      1. 停用 recorder，参数为 recorder。
      2. 触发 blob 事件，事件名为 dataavailable，对象为 recorder，携带 blob。
      3. 触发事件，事件名为 stop，对象为 recorder。

注意，stop()、 requestData()、 pause() 也会影响录制行为。

UA 必须以便于回放时能恢复原始 Track 的方式录制 stream。当返回多个 Blob （由于 timeslice 或 requestData()）， 单个 Blob 可以不可播放，但全部组合结果必须可回放。

若 MediaStream 内的任何 Track 在任何时刻被 muted 或非 enabled，UA 只会录制黑帧或静音。

停用 recorder 算法参数 recorder 步骤如下：

  1. 将 recorder 的 mimeType 属性设成 [[ConstrainedMimeType]] 槽值。
  2. 将 recorder 的 state 属性设为 inactive。
  3. 若 recorder 的 [[ConstrainedBitsPerSecond]] 槽位不是 undefined，
    将 recorder 的 videoBitsPerSecond 和 audioBitsPerSecond 属性设为 UA 认为合理的值，使两者之和接近 [[ConstrainedBitsPerSecond]] 槽值。
stop()
当调用 MediaRecorder 对象的 stop() 方法时，UA 必须执行下列步骤：
  1. 令 recorder 为调用该方法的 MediaRecorder 对象。
  2. 如果 recorder 的 state 属性为 inactive， 则中止步骤。
  3. 停用 recorder，参数为 recorder。
  4. 排队 DOM 操作任务源的任务执行以下内容：
    1. 停止采集数据。
    2. 令 blob 是迄今为止收集的数据的 Blob，触发 blob 事件，事件名为 dataavailable，对象为 recorder，携带 blob。
    3. 触发事件，事件名为 stop，对象为 recorder。
  5. 返回 undefined。
pause()
当调用 MediaRecorder 对象的 pause() 方法时，UA 必须执行以下步骤：
  1. 若 state 为 inactive，则抛出 InvalidStateError DOMException 并中止步骤。
  2. 若 state 为 paused，则中止步骤。
  3. 将 state 设为 paused，排队 DOM 操作任务源任务执行：
    1. 停止向 blob 采集数据（但保留该 blob，以备将来恢复录制）。
    2. 令 target 为 MediaRecorder 上下文对象。触发事件，事件名为 pause，对象为 target。
  4. 返回 undefined。
resume()
当调用 MediaRecorder 对象的 resume() 方法时，UA 必须执行以下步骤：
  1. 若 state 为 inactive，则抛出 InvalidStateError DOMException 并中止步骤。
  2. 若 state 为 recording，则中止步骤。
  3. 将 state 设为 recording，排队 DOM 操作任务源任务：
    1. 恢复或继续向当前 blob 收集数据。
    2. 令 target 为 MediaRecorder 上下文对象。触发事件，事件名为 resume，对象为 target。
  4. 返回 undefined。
requestData()
当调用 MediaRecorder 对象的 requestData() 方法时，UA 必须执行以下步骤：
  1. 若 state 为 inactive，则抛出 InvalidStateError DOMException 并终止；否则 UA 必须队列一个 DOM 操作任务源任务，执行：
    1. 令 blob 是当前为止收集的数据 Blob，target 为 MediaRecorder 上下文对象，触发 blob 事件，事件名为 dataavailable，对象为 target，携带 blob。
      （注意：如果尚未收集到数据，blob 为空）
    2. 新建一个 Blob，后续数据收集到该 Blob。
  2. 返回 undefined。
isTypeSupported(DOMString type)
检查 MediaRecorder 是否能录制指定的 MIME 类型。 如果此方法返回 true，仅表示 MediaRecorder 实现可为指定类型生成 Blob 对象。若资源不足，录制仍可能失败。调用时 UA 必须返回 is type supported 的结果（参数为此方法第一个参数）。
is type supported 算法如下：
  1. 令 type 为该算法参数。
  2. 如果 type 是空字符串，则返回 true（相当于将容器与 codec 的选择交给 UA）。
  3. 如果 type 不包含有效 MIME 类型字符串，则返回 false。
  4. 如果 type 包含 MediaRecorder 不支持的媒体类型或子类型，则返回 false。
  5. 如果 type 包含 MediaRecorder 不支持的媒体容器，则返回 false。
  6. 如果 type 包含多于一个音频 codec 或多于一个视频 codec，则返回 false。
  7. 如果 type 包含 MediaRecorder 不支持的 codec，则返回 false。
  8. 如果 MediaRecorder 不支持指定媒体类型/子类型、codec 与容器组合，则返回 false。
  9. 返回 true。

2.4. 数据处理

对 fire a blob event 带有 Blob blob，表示在 target 上 触发一个事件，事件类型为 BlobEvent，其 data 属性初始化为 blob。

通常情况下，blob 为 UA 在最后一次进入 recording state 后收集的数据。

2.5. MediaRecorderOptions

dictionary MediaRecorderOptions {
  DOMString mimeType = "";
  unsigned long audioBitsPerSecond;
  unsigned long videoBitsPerSecond;
  unsigned long bitsPerSecond;
  BitrateMode audioBitrateMode = "variable";
  DOMHighResTimeStamp videoKeyFrameIntervalDuration;
  unsigned long videoKeyFrameIntervalCount;
};

2.5.1. 成员

mimeType, 类型为 DOMString，默认值为 ""
录制所用的容器和编码格式 [RFC2046]， 可以包含该格式定义的任意参数。
mimeType 通过类型/子类型组合指定录制的媒体类型和容器格式，如果存在歧义，则指定 codecs 和/或 profiles 参数 [RFC6381]。 某些编码器还可能有其它可选或必需的专用参数。
audioBitsPerSecond, 类型为 unsigned long
音频轨道的聚合目标编码比特率（如有）。
videoBitsPerSecond, 类型为 unsigned long
视频轨道的聚合目标编码比特率（如有）。
bitsPerSecond, 类型为 unsigned long
所有视频与音频轨道的聚合目标编码总比特率。如果同时存在 audioBitsPerSecond 或 videoBitsPerSecond，本成员会覆盖它们，具体分配由 UA 决定。
audioBitrateMode, 类型为 BitrateMode，默认值为 "variable"
指定音频轨道的编码 BitrateMode。
videoKeyFrameIntervalDuration, 类型为 DOMHighResTimeStamp
指定编码视频流的关键帧时间间隔。UA 会结合本成员和 videoKeyFrameIntervalCount 共同控制关键帧生成。
videoKeyFrameIntervalCount, 类型为 unsigned long
指定编码视频流时关键帧之间的帧数间隔。UA 会结合本成员和 videoKeyFrameIntervalDuration 共同控制关键帧生成。

2.6. BitrateMode

enum BitrateMode {
  "constant",
  "variable"
};

2.6.1. 取值

constant
以固定比特率编码。
variable
以可变比特率编码，使复杂信号得到更多空间，简单信号用更少空间。

2.7. RecordingState

enum RecordingState {
  "inactive",
  "recording",
  "paused"
};

2.7.1. 取值

inactive
未进行录制：要么尚未开始，要么已停止。
recording
录制已开始，UA 正在捕获数据。
paused
录制已开始，被暂停，但尚未停止或恢复。

BlobEvent(DOMString type, BlobEventInit eventInitDict)

3.2. 属性

data, 类型为 Blob，只读

timecode, 类型为 DOMHighResTimeStamp，只读

timecode, 类型为 DOMHighResTimeStamp
初始化 BlobEvent 时所用的 timecode。

4. 错误处理

4.1. 一般原则

本节为非规范性内容。

当错误可以在调用时检测到，UA 会抛出 DOMException。 其它所有情况下，UA 会 触发错误事件。 如果错误发生时录制已开始但尚未停止，让 blob 是到目前为止收集的数据 Blob； 抛出错误后，UA 会使用 blob 触发 dataavailable 事件； UA 紧接着会 触发名为 stop 的事件。 UA 可设置平台相关的限制，如其支持的最小与最大 Blob 大小， 或一次能录制的 MediaStreamTrack 数量。 如超出这些限制将发出致命错误信号。

4.2. 错误事件

对 fire an error event，意味着使用 ErrorEvent 作为 eventConstructor 触发事件。

4.3. 异常总结

本文件中定义的所有异常均为带特殊类型的 DOMException。

名称 描述
InvalidStateError 对不允许的对象或不允许的时机调用了操作，或请求了已被删除或移除的源对象。
NotSupportedError 操作因 MIME 类型不支持或轨道集合无法用该 MIME 类型录制而无法完成。用户代理应尽量在 message 属性中提供更多信息。
SecurityError MediaStream 的 隔离属性不允许 MediaRecorder 访问。
InvalidModificationError 所录 MediaStreamTrack 集合已变更，无法继续录制。

5. 事件汇总

下列附加事件会在 MediaRecorder 对象上触发：

事件名 接口 触发 条件
start Event UA 已开始录制 MediaStream 数据。
stop Event UA 已停止录制 MediaStream 数据。
dataavailable BlobEvent UA 触发该事件以将数据返回给应用。该事件的 data 属性包含录制数据的 Blob。
pause Event UA 已暂停录制 MediaStream 数据。
resume Event UA 已恢复录制 MediaStream 数据。
error ErrorEvent 发生错误，如内存不足、流发生更改（例如正在录制时有 Track 被添加或移除），导致无法继续录制。

6. 隐私与安全考虑

本节为非规范性内容。

鉴于 MediaRecorder 的数据源始终会是 MediaStream， 大部分安全性实际上交由 [GETUSERMEDIA] 及其“隐私与安全性考虑”部分处理。 特别地，默认为数据源 MediaStream 来自 安全上下文。

6.1. 资源耗尽

视频和音频编码可能消耗大量资源。恶意网站可能尝试通过配置巨大的负载阻塞或拖垮 UA，如编码大分辨率和/或高帧率。

MediaRecorder 可通过 start() 的 timeslice 参数 设置，在一段时间内保存编码后的数据。time slice 参数过大可能让 UA 缓存大量数据，导致卡顿甚至内存耗尽。

UA 应采取措施防止编码与缓存过程耗尽资源。

6.2. 指纹识别

MediaRecorder 通过 isTypeSupported() 方法，提供对支持的视频和音频 MIME 类型的信息。当未在 MediaRecorderOptions 中指定时，也会为编码器和带宽分配选择合适组合，并可通过事件 type 属性查询；会尝试遵循指定的 MediaRecorderOptions 配置。

恶意网站可用这些信息做主动指纹识别，例如：

  • 推测设备硬件特性或操作系统厂商/版本差异：如 UA 仅在特定平台（或系列）才支持特定 codec 或硬件编码加速，或分辨率/帧率有限制，易被指纹识别。

  • 通过性能统计推断，如 UA 默认带宽分配因硬件不同而异，或 UA 在编码不同分辨率输入向量时测量系统负载。

UA 应通过如广泛支持特定 codec/MIME 类型并不依赖于体系架构/硬件型号/操作系统版本等方式，缓解指纹表面扩大，防止通过浏览器推断硬件特征。UA 还应通过合理的默认值限制 UA 能力的暴露与可识别性。

7.2. 录制摄像头视频及音频

此示例使用 MediaStream 结合 getUserMedia() 捕获视频+音频， 并塞入一个 <video> 标签进行预览，尝试录制并通过 ondataavailable 事件获取录制片段。注意，录制会一直持续直到 MediaRecorder 被 stop() 或被录制 MediaStreamTrack 均 ended。

下述示例也可在例如 此 codepen 上找到，基本未改动。 
<html>
<body>
<video autoplay/>
<script>
  var recordedChunks = [];

  function gotMedia(stream) {
    // |video| shows a live view of the captured MediaStream.
    var video = document.querySelector('video');
    video.src = URL.createObjectURL(stream);

    var recorder = null;
    try {
      recorder = new MediaRecorder(stream, {mimeType: "video/webm"});
    } catch (e) {
      console.error('Exception while creating MediaRecorder: ' + e);
      return;
    }

    recorder.ondataavailable = (event) => {
      console.log(' Recorded chunk of size ' + event.data.size + "B");
      recordedChunks.push(event.data);
    };

    recorder.start(100);
  }

  navigator.mediaDevices.getUserMedia({video: true, audio: true})
      .then(gotMedia)
      .catch(e => { console.error('getUserMedia() failed: ' + e); });
</script>
</body>
</html>
可以使用例如 MediaRecorder Web Fundamentals 文章中的 download() 函数将 recordedChunks 保存成文件。
