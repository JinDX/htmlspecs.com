摘要

W3C 可访问性指南（WCAG）3.0 将为使网络内容对残障用户更易访问提供广泛建议。遵循这些指南，可以满足盲人、低视力及其他视觉障碍者；听力障碍及耳聋者；行动与灵活性受限者；语言障碍者；感官障碍、认知障碍与学习障碍者；以及上述多种障碍组合用户的需求。指南涵盖台式机、笔记本电脑、平板、移动设备、可穿戴设备及其他物联网设备上的网页内容可访问性。适用内容类型包括静态、动态、交互和流媒体内容；视听媒体；虚拟与增强现实；以及替代访问方式的呈现与控制。指南还涵盖相关网页工具，如用户代理（浏览器和辅助技术）、内容管理系统、创作工具和测试工具。

本标准中的每条 指南 都提供了有关可访问性实践的信息，用以满足已记录的残障人士的 用户需求。每条指南都由多个 要求 和 断言 作为支撑，用于判断该需求是否得到满足。每条指南还由特定技术的 方法 支持，以满足每项要求或断言。

为跟上技术变革步伐，预期本规范将定期更新，增补与更新方法、要求和指南，以应对技术演进带来的新需求。对于那些正式宣称 符合 本指南的实体，提供了多个符合等级，以适应数字内容的多样性以及采用的测试类型。

有关 WCAG 3 的概述以及指向 WCAG 技术和教育材料的链接，请参阅 WCAG 3 介绍。

1. 简介

本节（及其子节）仅提供建议，并未规定指南，因此其内容为资料性或非规范性。

简介的简明语言摘要
  • W3C 可访问性指南（WCAG）3.0 展示了如何让网页内容和应用对残障人士可用。WCAG 3 是比网页内容可访问性指南（WCAG）2 更新的标准。
  • WCAG 3 不会取代 WCAG 2。WCAG 2 在全球范围内使用，并且在很长一段时间内仍会是不同国家的要求。
  • 达到 WCAG 2 AA 级别意味着你已非常接近满足 WCAG 3，但两者之间可能仍有差异。
  • 本草案仅包含已达到“开发中”状态的要求。这意味着我们在该主题上已达成总体共识，但尚未敲定所有细节。
  • 我们希望获得对本草案的反馈。你可以提交一个 GitHub issue，或发送电子邮件至 public-agwg-comments@w3.org。

1.1.1 草案要求

本要求列表比 WCAG 2 中的成功准则列表更长，原因在于：

  • 当前阶段的意图是尽可能包容潜在要求，并且
  • WCAG 3 的要求比 WCAG 2 的成功准则粒度更细。

  • 占位：此内容为临时内容，用于展示此处将出现的内容类型。预计这些内容会全部被替换，因此无需对占位内容提供反馈。
  • 探索中：此内容尚未细化，可能缺少细节和定义。工作组正在探索本节应采纳的方向。反馈应围绕拟议的方向展开。
  • 开发中：就本节所需内容而言，工作组已大致达成共识，但并未解决所有高层次问题。已添加了一些细节，但仍需进一步完善。反馈应聚焦于从整体上确保本节内容可用且合理。
  • 完善中：此内容已准备好接受广泛的公开审阅和试验性采纳。工作组已就本节内容达成共识。反馈应侧重于实现的可行性。

1.2 关于 WCAG 3

本规范提出了一个新的模型和新的指南，以帮助使网页内容和应用对残障人士可访问。W3C 可访问性指南（WCAG）3.0 支持广泛的用户需求，采用新的测试方法，并允许对指南及相关内容进行更频繁的维护，以跟上快速变化的技术。WCAG 3 通过聚焦用户的功能性需求来支持这种演进。这些需求通过写成结果陈述的指南，以及相应的要求、断言和特定技术方法得到支持，以满足这些需求。

WCAG 3 是网页内容可访问性指南 2.2 [WCAG22] 及更早版本的后续者，但并不废弃 WCAG 2。它还将整合部分内容并在一定程度上扩展用户代理可访问性指南 2.0 [UAAG20] 和 创作工具可访问性指南 2.0 [ATAG20]。这些早期版本提供了一个灵活的模型，使其在 15 年以上的时间里保持了相关性。然而，技术的变化以及残障人士需求的变化，已经导致需要一个新模型以更全面、更灵活地解决内容可访问性问题。

WCAG 2 与 WCAG 3 之间存在许多差异。WCAG 3 指南涉及台式机、笔记本电脑、平板电脑、移动设备、可穿戴设备和其他物联网设备上的网页内容可访问性。该指南适用于多种类型的网页内容，包括静态、动态、交互式和流媒体内容；视觉和听觉媒体；虚拟与增强现实；以及替代访问呈现和控制方式。这些指南还涵盖相关的网页工具，如用户代理（浏览器和辅助技术）、内容管理系统、创作工具和测试工具。

本标准中的每条指南都提供了有关可访问性实践的信息，以满足已记录的残障人士用户需求。每条指南由多项要求支撑，用于判断这些需求是否得到满足。每条指南还由特定技术的方法支撑，以满足每项要求。

符合 WCAG 2.2 A 级和 AA 级的内容预计能够满足本新标准的大部分最低符合性级别，但由于 WCAG 3 包含额外测试与不同的计分机制，要完全满足新标准仍需要进行额外工作。由于新标准将采用不同的符合性模型，可访问性指南工作组预计，有些组织可能会选择继续使用 WCAG 2，而另一些则可能希望迁移到新标准。对于希望迁移到 WCAG 3 的组织，工作组将提供迁移支持材料，其中可能包括映射及其他方法，以便于完成迁移。

Editor's note

另见：结构，因为这些指南彼此密切相关。

Editor's note

为了确保本指南在不同语言中发挥良好作用， AGWG 成员、认知与学习障碍无障碍任务组（COGA）以及国际化（i18n）小组就一组初始语言达成一致，用于对这些指导进行压力测试。

这五种“护栏”语言是：

  • 阿拉伯语
  • 英语
  • 印地语
  • 普通话
  • 俄语

我们从联合国（UN）的六种官方语言开始。随后移除了法语和西班牙语，因为它们与英语相似。我们加入了印地语，因为它是最常用、但未在联合国列表中的语言。

这五种语言涵盖了丰富多样的语言特征，例如：

  • 从右到左的文本布局
  • 竖排文字布局
  • 影响含义的声调

该列表并未涵盖所有语言，但在保持工作量可控的同时，有助于使指南对更广泛的受众更有用。

我们将与 W3C 的全球包容性社区组、国际化（i18n）任务组及其他相关方合作，审查并完善针对这些要求的测试和技术。我们还计划在未来创建关于将这些指南翻译成更多语言的指导。

“解释规范性条款”的通俗语言总结

你可能希望声明你的内容或数字产品符合 WCAG 3 指南。当内容符合这些指南时，你可以做出“符合性声明”。

若要做出正式的符合性声明，你必须使用本文档中描述的流程。符合性声明并非强制要求。即使你不想做出声明，你的内容仍然可以符合 WCAG 3。

WCAG 3 包含两类内容：

  • 规范性（Normative）：为满足指南，你必须遵守的规则。
  • 信息性（Informative）：帮助你理解或应用这些规则的信息——也称为非规范性内容。

本指南通过三类条款来规定必须完成的事项：

  • 核心要求（Core requirements）：你必须完成的主要事项
  • 补充要求（Supplemental requirements）：你可以额外完成、以提升可访问性的内容
  • 声明（Assertions）：你可以就自己为改进内容或产品可访问性而采取或正在采取的实践做出的说明

此外，WCAG 3 还提供最佳实践（Best Practices）。这些内容提供了重要的指导，往往能节省时间并提升可访问性，但并不总是适用于所有场景。

你用于满足 WCAG 3 的方法必须是无障碍支持的。这意味着，仅当浏览器和常用辅助技术支持某种方法时，该方法才可以计入符合性。WCAG 3 使用已定义的无障碍支持集来标识将针对哪些技术进行测试，以检验所采用的方法是否满足某项要求。WCAG 3 中的初始无障碍支持集针对的是 HTML 方法。

在评估可访问性时，WCAG 3 会应用于明确界定的内容部分——例如特定的页面/视图，或流程——从而使你可以将符合性声明精确限定到实际测试过的范围。

每一项核心要求、补充要求和声明都会关联一个或多个功能限制条目。这些条目涵盖了广泛的残障类型，有助于说明满足这些要求将使哪些人群获益。

3.1 解释规范性条款

WCAG 2.2 的主要内容是规范性的，并定义了影响符合性声明的要求。引言性材料、附录、标明为“非规范性”的章节、图示、示例和注释均为信息性（非规范性）。非规范性材料为理解指南提供建议性信息，但不会产生影响符合性声明的强制性要求。

关键词 MAY、MUST、MUST NOT、NOT RECOMMENDED、RECOMMENDED、SHOULD 和 SHOULD NOT 应按 [RFC2119] 中的解释来理解。

3.1.1 条款类型

WCAG 3 由三种类型的条款构成。

核心要求（Core Requirements）——为了实现符合性必须满足的要求。这些要求包括确保：

  • 用户代理和辅助技术可以检测到内容；
  • 内容可被多种感官获取，并可通过多种输入方式使用；
  • 内容不会导致即时的身体伤害；以及
  • 必要的支持可用，例如字幕、文本替代内容和缩写的扩展形式。

补充要求（Supplemental Requirements）——在核心集基础上进一步扩展的要求，包括：

  • 质量检查；
  • 相较于对应核心要求更严格或更高水平的要求；
  • 当前实现成本较高或较为困难的要求；以及
  • 仅在某些特定情境下适用的要求。

声明（Assertions）——你可以就自己的组织为改进内容或产品的可访问性而遵循的特定无障碍实践做出的书面说明。

3.1.2 最佳实践

除要求和声明外，WCAG 3 还包括最佳实践。这些最佳实践在条款和方法层级中列出。它们提供了重要指导，往往可以节省时间并提升可访问性，但可能并非在所有情境下都适用。这些内容与要求、声明和方法一并呈现，并明确标注为最佳实践。

最佳实践不需要满足即可实现符合性。

  • 对于银级（Silver）符合性，符合性范围内的所有页面/视图和流程MUST满足所有核心要求，并在每个功能绩效声明中满足的补充要求和声明比例高于铜级。
  • 对于金级（Gold）符合性，符合性范围内的所有页面/视图和流程须满足所有核心要求，并在每个功能绩效声明中满足的补充要求和声明比例高于银级。

3.2.2 无障碍支持

许多方法（例如使用 ARIA 或 CSS 的方法）只有在真实用户所使用的用户代理（UA）和辅助技术（AT）支持的前提下才是有效的。

用户使用的 UA 和 AT 会因语言和地区而异，在防火墙后的封闭环境或自助终端等环境中也可能受到限制。因此，了解哪些 UA 和 AT 支持哪些方法，并在符合性声明中清晰说明所假定的 UA 和 AT，十分重要。要求方法必须同时得到用户辅助技术以及浏览器和其他用户代理无障碍功能支持这一概念，被称为“无障碍支持”（accessibility supported）。

3.2.2.1 无障碍支持集

WCAG 3 引入了“无障碍支持集（accessibility-support sets）”的概念。该集合由 AGWG 在确定某一方法是否支持某项要求时要测试的 UA、AT 和无障碍设置构成。

WCAG 3 将采用两类无障碍支持集：

  1. 默认无障碍支持集（Default accessibility-support set）——一组常见浏览器、辅助技术和无障碍设置，用于支持英文内容。默认无障碍支持集将针对 HTML 方法。未来还可以为移动或 VR 等技术创建额外的支持集。
  2. 替代无障碍支持集（Alternative accessibility support set）——个人、组织或区域监管机构也可以指定替代无障碍支持集。当默认支持集无法覆盖预期的语言、地区和技术时，可以使用另一组浏览器、辅助技术和无障碍设置来提供支持。

尽管 AGWG 将努力确保各项要求尽可能地获得广泛支持，但在所有 UA 与 AT 组合上满足每一项要求可能并不可行。AGWG 只会在核心要求中纳入那些在发布时至少具有一种适用于默认无障碍支持集的 HTML 方法的要求。

无障碍支持集必须包含在符合性声明中。如果你在封闭环境中工作、使用非 HTML 技术，或所在地区依赖默认无障碍支持集中未包含的 UA 和 AT，我们也建议在公开的无障碍声明中包含你的无障碍支持集。例如，如果你针对某个使用非标准方式提供文本转语音功能的自助终端声明符合性，则你所采用的无障碍支持集应包含在符合性声明中，并建议在关于该自助终端无障碍性的公开声明中说明。

Editor's note
  • 默认无障碍支持集尚未定义。
  • 需要补充性文档，说明人们如何创建自己的替代无障碍支持集。
  • 如何在时间维度上维护无障碍支持集仍需进一步讨论。
  • 针对辅助技术中长期存在缺陷的例外情况仍在讨论中。

3.2.3 明确的符合性范围

在评估内容的可访问性时，WCAG 3 要求将指南应用于特定范围。

WCAG 3 定义了两种内容范围界定方式：页面/视图和流程。评估在一个或多个完整的页面/视图或流程上进行，并据此确定符合性。

符合性仅针对被清晰界定的页面/视图和流程进行定义。然而，可以基于一个页面/视图和流程、一系列页面/视图和流程，或多组相关的页面/视图和流程来做出符合性声明。流程中的所有独特步骤MUST在视图集合中得到体现。流程以外的视图MAY也可纳入范围。

Editor's note

我们认识到代表性抽样是大型复杂网站评估可访问性的重要策略。尽管当前文档尚未涵盖该内容，但我们的目标是在指南进入候选推荐（Candidate Recommendation）阶段之前，在本文件或单独文件中进行说明。我们欢迎你就如何在 WCAG 3 中最佳纳入代表性抽样提供建议和反馈。

general flash and red flash thresholdsDeveloping

如果满足下列任一条件，则闪烁或快速变化的图像序列低于阈值（即内容通过）：

  • 在任一一秒时间段内，一般闪烁（general flashes）不超过三次和/或红色闪烁（red flashes）不超过三次；或
  • 在任何 10 度视野范围内，同时发生闪烁的组合区域总和，在典型观看距离下，占据屏幕上不超过 0.006 立体弧度（即屏幕上任意 10 度视野区域的 25%）

其中：

  • 一般闪烁（general flash）被定义为：一对相对亮度相反变化，且变化幅度达到最大相对亮度（1.0）的 10% 或以上，且较暗图像的相对亮度低于 0.80；“一对相反变化”指一次上升接一次下降，或一次下降接一次上升；并且
  • 红色闪烁（red flash）被定义为任意涉及饱和红色的一对相反转换

例外：精细且平衡的图案闪烁，例如白噪声，或由边长小于 0.1 度（在典型观看距离下对应的视野）的方块构成的棋盘格交替图案，不违反上述阈值。

Note 1

对于通用软件或 Web 内容，当在 1024 × 768 像素分辨率下查看内容时，在显示屏任意位置选取一个 341 × 256 像素矩形，可较好地估算标准屏幕尺寸和观看距离（例如 15–17 英寸屏幕、22–26 英寸观看距离）下的 10 度视野。该 75–85 ppi 的分辨率被认为低于（因此更保守于）CSS 规范中名义 CSS 像素分辨率 96 ppi。分辨率更高的显示器在呈现相同内容时会得到更小、更安全的图像，因此阈值是基于较低分辨率来定义的。

Note 2

“转换（transition）”指在将相对亮度（或针对红色闪烁时的相对亮度/颜色）对时间作图时，相邻峰值和谷值之间的变化。一个闪烁由两个相反方向的转换组成。

Note 3

目前在该领域中，对“涉及饱和红色的一对相反转换（pair of opposing transitions involving a saturated red）”的新工作定义（来自 WCAG 2.2）是：一对相反转换，其中一端状态的 R/(R + G + B) 值大于或等于 0.8，且这两个状态在 CIE 1976 UCS 色度图中的差异大于 0.2（无量纲）。[ISO_9241-391]

Note 4

有工具可以基于视频屏幕截图执行上述分析。然而，如果闪烁在任一一秒时间段内不超过 3 次，则无需使用工具即可进行评估，内容会自动通过（见上文 #1 和 #2）。
