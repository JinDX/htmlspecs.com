摘要

W3C 无障碍指南（WCAG）3.0将为提升残障用户的网页内容可访问性提供广泛建议。遵循这些指南可满足盲人、低视力及其他视力障碍者，聋人及听力受损者，行动及灵活性受限者，语言障碍者，感官障碍者，认知及学习障碍者，以及多种障碍组合者的需求。这些指南涵盖桌面、笔记本、平板电脑、移动设备、可穿戴设备及其他物联网设备上的网页内容无障碍问题。指南适用于多种类型的网页内容，包括静态、动态、交互式和流媒体内容；视听媒体；虚拟和增强现实；以及替代访问方式和控制。指南也涵盖相关网页工具，如用户代理（浏览器和辅助技术）、内容管理系统、创作工具和测试工具。

本标准中的每条指南都提供了解决残障人士已记录的用户需求的无障碍实践信息。每条指南配有多个要求和断言，用于判断需求是否得到满足。指南还配有针对技术的方法，以满足每项要求或断言。

为跟上技术变化，本规范预计将定期更新，新增方法、要求和指南以应对技术发展带来的新需求。对于正式声明符合性的相关方，提供多种符合性级别，以适应数字内容的多样性及测试类型。

关于WCAG 3的概览及相关技术和教育资料，请参阅WCAG 3.0简介。

1.1.1 草案要求

本要求列表比WCAG 2中的成功标准列表更长，原因包括：

  • 现阶段意在尽可能包容所有潜在要求，
  • WCAG 3.0的要求比WCAG 2的成功标准更细化。

WCAG 3.0最终的要求集会与本草案不同，可能会增删、合并要求，也会调整文本。只有部分要求会用于达到基础符合性级别。

1.1.2 章节状态级别

在WCAG 3.0起草过程中，本文件每个规范性章节都被赋予一个状态。该状态用于指示章节开发进度、实验采用的准备程度，以及无障碍指南工作组希望收到哪类反馈。

  • 占位：内容为临时，仅展示此处将出现的内容类型，预计全部替换。占位内容无需反馈。
  • 探索：内容尚未完善，可能缺少细节和定义。工作组正在探索方向，反馈建议聚焦于方向本身。
  • 开发：内容大致达成共识，但高层问题未完全解决。已补充细节但尚待完善。反馈应关注章节整体可用性与合理性。
  • 完善：内容已准备好广泛公开评审和实验采用，工作组就该章节达成一致。反馈重点为实施的可行性。
  • 成熟：工作组认为内容已准备好推荐发布。反馈应关注工作组未预见的边缘情况。

1.2 关于WCAG 3.0

本规范提出了新的模型和指南，以使网页内容和应用对残障人士可访问。W3C 无障碍指南（WCAG）3.0支持广泛的用户需求，采用新的测试方法，并允许指导内容频繁维护，以跟上技术快速变化。WCAG 3.0通过关注用户的功能需求来推动这一演进。这些需求由以结果为导向的指南、要求、断言和技术方法来支持。

WCAG 3.0是Web内容无障碍指南2.2[WCAG22]和更早版本的继任者，但不会废止WCAG 2。还将吸收部分内容并扩展用户代理无障碍指南2.0[UAAG20]和创作工具无障碍指南2.0[ATAG20]。前述版本为其保持15年以上的相关性提供了灵活模型，但技术变化和残障人士需求变化促使我们提出更全面、灵活的新模型以解决内容可访问性。

WCAG 2与WCAG 3.0有诸多不同。WCAG 3.0指南涵盖桌面、笔记本、平板、移动设备、可穿戴设备及其他物联网设备上的网页内容无障碍。指南适用于各类网页内容，包括静态、动态、交互式和流媒体内容；视觉和听觉媒体；虚拟与增强现实；以及替代访问呈现和控制方式。指南也涵盖相关网页工具，如用户代理（浏览器与辅助技术）、内容管理系统、创作工具和测试工具。

本标准中的每条指南都提供了关于解决残障人士已记录用户需求的无障碍实践信息。每条指南由多个要求支持，以判断需求是否满足。指南还配有针对技术的方法，以实现每项要求。

符合WCAG 2.2 Level A和Level AA的内容预计可满足新标准的大部分最低符合性要求，但因WCAG 3.0有额外测试和不同评分机制，要实现完全符合还需额外努力。由于新标准采用不同的符合性模型，无障碍指南工作组预计部分机构将继续使用WCAG 2，部分机构则会迁移至新标准。对于希望迁移到WCAG 3的机构，工作组会提供迁移支持材料，包括映射等方式以便迁移。

编辑注

使用WCAG的个人和组织类型多样，包括网页设计师与开发者、政策制定者、采购人员、教师和学生。为满足不同受众需求，将提供多层指导，包括以结果为导向的指南、可测试的要求、断言、丰富的方法集合、资源链接和代码示例。

以下列表是工作组将要探索的潜在指南和要求初始集合，旨在为下一阶段工作提供方向。这些都是草案内容，不应视为WCAG 3.0的最终内容。

通常，探索性内容会在每项下方附有编辑注，列出关切和问题。由于本指南部分仍处于WCAG 3.0早期阶段，这条编辑注涵盖了本节大部分内容。除非另有说明，列表中的所有条目当前均为探索性主题，仅供考虑。并非所有条目都会纳入WCAG 3.0最终版本。

下面列出的指南和要求来源于工作组分析、审查和研究的用户需求。这些内容尚未完善，也未包含必要的例外或方法。部分要求更适合由创作工具或平台层面实现。许多要求还需进一步完善范围，并确保能正确适用于多种语言、文化和书写体系。我们将在进一步探索每项要求时解决这些问题。

Guideline 2.1.1 图像替代方案

用户可获得与图像等效的替代方案。

哪些基础要求适用？

针对每个图像：

  1. 移除该图像会影响用户对页面的理解吗？
    • 否，装饰性图像需以编程方式隐藏。结束。
    • 是，继续。
  2. 该图像是否以可被用户代理 和辅助技术访问的方式呈现？
    • 是，图像必须满足 图像可由程序确定 并且 可访问性支持集 满足 图像有可用的等效文本替代，可用于传达内容. 停止.
    • 否，继续。
  3. 该图像是否有等效的文本替代？
    • 是，图像应满足为传递内容的图像提供等效文本替代。结束。
    • 否，未通过。

基础要求: 有意义的声音开发中

理解媒体所需的声音应在媒体替代中标明或描述。

编辑注

包括音效和其他非口语音频内容。

基础要求: 有意义的视觉信息开发中

理解媒体所需但未在音频内容中描述的视觉信息，应包含在媒体替代中。

编辑注

包括动作、图表或信息性视觉、场景变化及屏幕上的文本。

Guideline 2.1.4 字幕

基础要求: 预录字幕开发中

所有直播音频内容均应提供字幕。

基础要求: 字幕避免遮挡开发中

字幕应在屏幕上摆放，使其不会遮挡理解视频内容所需的视觉信息。

基础要求: 字幕一致性开发中

字幕在整个媒体及相关作品中应保持一致，除非有必要例外。包括字幕文本样式和位置一致，以及发言人、语言、声音的标识方法一致。

字幕及其相关视觉指示器的外观可调，包括字体大小、粗细、风格、颜色、背景色、背景透明度和位置；

基础要求: 360度字幕位置开发中

在360度数字环境中，字幕始终保持在用户正前方。

基础要求: 360度空间视觉指示器开发中

在360度数字环境中，当音频来自当前视角之外时，应指示声音或语音的方向。

  • 视觉指示：带描边的箭头指向发言人或声音方向

Guideline 2.1.5 音频描述

基础要求: 预录音频描述开发中

编辑注

WCAG 3需要明确如何处理无插入音频描述间隙的音频视频内容。两种可能方案是允许内容创作者使用描述性文字稿作为例外，或要求内容创作者提供扩展音频描述。

基础要求: 音频描述时序开发中

音频描述需与视频内容同步，且不能与对白及有意义的音频内容重叠。

补充要求: 直播音频描述开发中

为理解媒体所需视觉内容的直播视频应提供音频描述。

补充要求: 扩展音频描述开发中

若原有配乐中的停顿不够长，视频应暂停以延长音频轨道，并提供扩展音频描述，用于描述理解媒体所需的视觉信息。

补充要求: 音频描述音量开发中

应提供机制，允许用户独立调节音频描述音量，与视频的音频音量分开，并允许切换音频描述语言（若提供多种语言）。

补充要求: 音频描述语言开发中

应提供机制，允许用户切换音频描述语言（如有多种可选语言）。

Guideline 2.2.1 文本外观

用户可以阅读以视觉呈现的文本。

哪些基础要求适用？

针对每个词语：

  1. 文本是否纯属装饰，或任何人都无法阅读？
    • 是，通过
    • 否，继续
  2. 默认/作者设定的呈现是否满足最低要求？
    1. 是，默认/作者呈现符合可读文本块（基础）和 可读文本风格（基础），继续
    2. 否，不通过
  3. 用户能否调整文本外观？
    1. 是，文本必须是用户可操控文本，并且：
      1. 无障碍支持集应满足：
        1. 可调整文本块
        2. 可调整文本风格
      2. 文本调整后内容和功能不丢失
      3. 通过
    2. 是，通过产品提供的主题：
      1. 可调整文本块
      2. 可调整文本风格
      3. 文本调整后内容和功能不丢失
      4. 通过
    3. 否，且产品不提供其自有主题：
      1. 不通过。

编辑注

五种“护栏”语言为：

  • Arabic
  • English
  • Hindi
  • Mandarin
  • Russian

我们从联合国（UN）的六种官方语言开始，然后移除了法语和西班牙语，因为它们与英语相似。我们加入了印地语，因为它是不在联合国名单上、使用最广泛的语言。

  • 从右到左的文本布局
  • 竖排文本布局
  • 影响意义的声调

该列表不包含所有语言，但有助于在可控范围内开展工作，同时使指导对更广泛受众更有用。

我们将与W3C的全球包容社区组、国际化（i18n）工作组等合作，审查并完善这些要求的测试与技术。同时，我们计划在未来为将本指南翻译成更多语言提供指导。

    • 否，不通过。

2.4 输入 / 操作

Guideline 2.4.1 键盘接口输入

用户仅用键盘即可导航和操作内容。

基础要求： 所有元素可键盘操作开发中

所有可通过指针、音频（语音或其他）、手势、相机输入或其他方式控制或激活的元素，均可通过键盘接口进行控制或激活。

基础要求： 所有内容可通过键盘访问开发中

所有可通过其他输入方式访问的内容，也必须仅用键盘接口即可访问。

Note 1

“所有内容”包括通过悬停、右键点击等方式提供的内容。

Note 2

其他输入方式包括指点设备、语音与语音识别、手势、相机输入，以及任何其他输入或控制方式。

Note 3

所有元素可键盘操作要求允许你导航至所有可操作元素，但如果下一个元素在5屏之外，你仍需能够访问所有内容。并且，如果内容位于可展开的章节中，你不仅需要能够打开它们，还需要能访问所有内容，而不仅仅是可操作的元素。

基础要求： 双向导航开发中

在任意位置都应可通过键盘导航向前与向后移动。

编辑注

我们考虑要求导航必须是对称的（即：如果先前进再后退，总能回到原位置），对此欢迎反馈意见。

基础要求： 键盘命令不冲突开发中

作者定义的键盘命令不得与标准平台键盘命令冲突，或必须可以重新映射。

基础要求： 响应式下可键盘导航开发中

如果页面/视图使用了响应式设计，则页面/视图必须保持完全可用键盘导航。

基础要求： 无键盘陷阱开发中

在导航到、进入或激活某元素后，始终可以通过常见键盘导航技术离开该元素，或通过页面/视图上（或流程中更早的页面/视图上）描述的技术离开该元素。

基础要求： 用户对键盘焦点的控制开发中

当键盘焦点被移动时，以下之一为真：

  • 焦点是在用户直接控制下被移动；
  • 引入了新的视图（如对话框），并将焦点移至该视图；
  • 在焦点移动发生前告知用户该潜在移动，并给予用户避免移动的机会；
  • 在某些用户操作完成后，键盘焦点自动移动到键盘导航顺序中的下一个项。
示例

以下场景中，经用户允许将其移动到其他位置可能很有用：

  • 表单消息提示：“如果你选择‘否’，可以跳过第8到第15题；是否跳到第16题？”
  • 提交时检测到表单错误，消息提示：“页面上有一个错误；是否跳转到该位置？”
基础要求： 相关的Tab顺序键盘焦点开发中

除跳转链接和其他为辅助键盘导航而特意添加但默认隐藏的元素外，按Tab不会将键盘焦点移入Tab动作之前不可见的内容中。

Note

手风琴、下拉菜单和ARIA tab panels是可展开内容的示例。根据该要求，它们不会仅因内部包含在Tab顺序中的元素而展开。它们要么不展开，要么内部不包含任何处于Tab顺序中的元素。

示例

一个菜单在你Tab到它时展开，但随后使用方向键在其中导航，这样算通过。但若菜单展开后要求你通过Tab遍历所有新可见元素才能越过它，则不通过。

Guideline 2.4.2 使用键盘时的体力或认知负担

用户可在无需不必要体力或认知负担的情况下使用键盘。

基础要求： 逻辑的键盘焦点顺序开发中

键盘焦点按照保持意义与可操作性的顺序与方式在内容中移动。

基础要求： 保持键盘焦点开发中

当键盘焦点在同一页面/视图内从一个上下文移动到另一个上下文时，无论是自动还是用户请求，均应保持该焦点，以便当用户返回上一上下文时，键盘焦点恢复到先前位置（除非该位置已不存在）。

示例

当用户关闭模态对话框或其他弹出窗口时，键盘焦点返回到触发该对话框或弹窗打开的元素。

Note 1

当先前的焦点位置已不存在时，最佳做法是将焦点放在被移除位置之前的可聚焦位置。例如：文档中的一组主题标签，每个标签有删除按钮。用户点击位于中间的某个标签的删除按钮。删除后，焦点应放到被删除标签之前的那个标签上。

Note 2

在页面之间移动时这也很有用，但通常需要由浏览器完成，除非用户处于某个流程中，且该信息在页面之间通过cookie或服务器保存，这样当人返回到页面时仍能恢复旧位置。

断言： 可比的键盘操作负担开发中

内容作者遵循的用户界面设计原则应包括：尽量减小仅使用键盘接口所需的输入命令数量与使用其他输入方式所需命令数量之间的差异。

Note

其他输入方式包括指点设备、语音与语音识别、手势、相机输入，以及任何其他输入或控制方式。

Guideline 2.4.3 指针输入

指针输入应保持一致，且所有功能都可通过简单指针输入以对时间与压力不敏感的方式完成。

基础要求： 指针取消开发中

对于可通过简单指针输入激活的功能，至少满足以下之一：

无按下事件
指针的按下事件不会执行功能的任何部分
中止或撤销
在抬起事件完成功能，并提供机制以在完成前中止或在完成后撤销该功能
抬起反转
抬起事件会反转先前按下事件的任何结果
必要
在按下事件上完成功能属于必要例外
示例 1

“中止”的示例：拖拽操作在按下按钮时拾取，但可通过在非投放区域释放来取消。

示例 2

在按下事件上执行动作属于必要的情形示例包括荷兰式拍卖或游戏触发器。

基础要求： 简单指针输入开发中

任何使用指针输入但非简单指针输入的功能，也可以通过一次简单指针输入或无需计时的一系列简单指针输入来操作。

Note 1

非简单指针输入的示例包括：双击、滑动手势、多点手势（如捏合、分点点击或双指旋钮）、可变压力或时间要求、以及拖动动作。

Note 2

复杂指针输入并非被禁止，但它不能成为完成某动作的唯一方式。

Note 3

简单指针输入不同于单一指针输入，并且比仅使用单一指针更为严格。

基础要求： 一致的指针取消方式开发中

指针取消的方法在一组页面/视图内对每种交互类型保持一致，除非不同做法属于必要例外。

Note

在必须不同的情况下，提醒用户会有所帮助。

基础要求： 指针压力替代开发中

实现任一功能不应仅能通过特定指针压力完成，除非特定压力对该功能属于必要例外。

示例

画笔功能或高级签名验证需要特定压力，属于必要情形。

基础要求： 指针速度替代开发中

实现任一功能不应仅能通过特定指针速度完成，除非特定指针速度对该功能属于必要例外。

示例

画笔功能、高级签名验证或时间受限的游戏对特定指针速度是必要的。

Guideline 2.4.4 语音输入

为语音输入提供替代方式，并便于语音控制。

基础要求： 语音替代开发中

实现任一功能不应仅能通过语音输入完成，除非语音输入对该功能属于必要例外。

基础要求： 实时双向语音通信开发中

凡有实时双向语音通信的地方，应提供实时文本选项。

Guideline 2.4.5 输入操作

用户可以选择使用不同的输入技术及其组合，并在它们之间切换。

基础要求： 通过指针设备改变键盘焦点开发中

如果内容干扰了用户代理的指针或键盘焦点行为，则在视图中用指针选中任意内容，会将键盘焦点移动到该交互元素上，即使用户将指针拖离该元素（以避免激活它）。

示例

用户将文档向下滚动六屏，然后用指针点击某段落。接着按下Tab键，焦点移动到屏幕上所点击位置之后的第一个交互组件，而不是从文档上方六屏处的先前位置开始。

基础要求： 悬停或键盘聚焦时的内容开发中

当接收并移除指针悬停或键盘焦点时触发了额外内容的显示与隐藏，且该额外内容的视觉呈现由作者控制而非用户代理控制，则以下全部为真：

可关闭
提供机制在不移动指针悬停或键盘焦点的情况下关闭该额外内容，除非该额外内容未遮挡或替换其他内容
可悬停
如果指针悬停可触发额外内容，则指针应能移动到额外内容上而不导致其消失
可持续
该额外内容保持可见，直到移除悬停或键盘焦点触发、用户将其关闭，或其信息不再有效
Note 1

由用户代理控制的额外内容示例包括通过HTML title 属性创建的浏览器工具提示。

Note 2

这适用于除触发交互元素本身之外出现的内容。隐藏的交互元素在键盘聚焦时显示（如用于跳转页面其他部分的链接）并不属于“额外内容”，因此不在本要求范围内。

示例

在悬停与键盘聚焦时显示的自定义工具提示、子菜单及其他非模态弹窗。

基础要求： 手势替代开发中

手势不应成为实现任何功能的唯一方式，除非手势对该功能属于必要例外。

基础要求： 输入方式灵活性开发中

当某项功能（包括输入或导航）可通过不同输入方法实现时，用户可随时在这些输入方式之间切换。

Note

这并不意味着必须支持所有输入技术（指针、键盘、语音、手势），但若支持某种输入方式，则应在整个内容中均予支持，除非特定输入方式对功能属于必要例外。

基础要求： 无需身体移动即可使用开发中

实现任一功能不应仅能通过整体或大幅身体移动完成，除非整体或大幅身体移动对该功能属于必要例外。

Note

这既包括对身体移动的检测，也包括对设备的动作（如摇晃）等需要身体移动的操作。

Guideline 2.4.6 认证

应为用户提供可替代的认证方法。

基础要求： 生物识别身份验证开发中

生物识别不应成为识别或认证的唯一方式。

基础要求： 语音识别身份验证开发中

2.5 错误处理

Guideline 2.5.1 纠正错误

用户能知晓并纠正错误。

基础要求： 错误通知开发中

检测到错误时，应以视觉和编程方式通知用户已发生错误。

基础要求： 错误原因标识开发中

出错的内容用编程方式标识。

基础要求： 描述性错误开发中

错误消息应清楚描述问题。

编辑注

清晰语言指导中列出了可理解内容的写作要求。

补充要求： 通知中说明错误原因开发中

当错误因用户与交互元素互动而产生时，错误消息应包含出错元素的可读名称。如果该交互元素位于流程的其他位置，则错误消息中应包含该页面/视图或流程步骤。

基础要求： 错误修正建议开发中

错误消息应包含可自动判定的修正建议，除非这会危及内容的安全或目的。

基础要求： 错误可见性开发中

错误消息应至少通过以下两种方式在视觉上可识别：

  • 符号。
  • 颜色与周围内容区分。
  • 文本明确标明这是错误。
Note

不同文化背景下代表错误的符号和颜色可能不同，应据此调整。

基础要求： 错误持续显示开发中

错误消息应持续显示，直到用户关闭或错误被解决。

基础要求： 错误原因关联开发中

错误消息应与错误来源编程方式关联。

补充要求： 错误链接开发中

当错误通知与出错项不相邻时，应提供指向错误的链接。

补充要求： 错误位置开发中

错误消息应在视觉上与错误来源同位。

Guideline 2.5.2 预防错误

当用户提交信息时，至少满足以下之一：

  • 用户可在提交前审查、确认并纠正信息
  • 信息已校验，用户可纠正发现的错误
  • 用户可撤销提交
补充要求： 提交确认开发中

提交后，应通知用户已提交的信息及提交状态。

补充要求： 数据输入后校验开发中

数据录入过程中，应在用户录入数据后且表单提交前进行数据校验。

补充要求： 边录入边校验开发中

完成多步骤流程时，应在用户进入下一步骤前完成校验。

3.1 符合性

符合性的简明语言摘要

你可能希望声明你的内容或产品符合WCAG 3.0指南。如果确实符合这些指南，我们称之为“符合性”。

如果你想正式声明符合性，必须采用本文件描述的流程。符合性声明不是强制的，即使你不声明，内容也可以符合WCAG 3.0。

本文件包含两类内容：

  • 规范性：你必须做的，以满足指南要求。
  • 说明性：帮助你达成指南的建议，也称为非规范性。

我们正在探索WCAG 3.0的不同符合性方法。待足够多的指南开发完成后，我们会测试每种方法的效果。

符合性摘要结束

WCAG 3.0将采用不同于WCAG 2.2的符合性模型，以满足自身要求。开发和验证符合性模型是AG未来几年需完成的重要任务。

AG正在探索基于基础要求、补充要求和断言的模型。

最基本的符合性级别要求满足所有基础要求。这一集合大致等同于WCAG 2.2 AA级。

更高的符合性级别将通过补充要求和断言来定义和达成。AG正探索更高级别是否采用积分、百分比或预定义要求集合（模块）等方式更合适。

AG还在继续探索其他符合性概念，包括符合性级别、问题严重性、形容词评级和预评估检查。

更多信息请参见W3C无障碍指南（WCAG）3.0说明。

3.1.1 仅采用获得无障碍支持的技术方式

“无障碍支持”概念考虑了多种用户代理和使用场景。作者如何确定满足指南的某种技术手段在实际中能被真实用户使用的用户代理支持？

本意是不同符合性级别下，用户代理测试责任由作者承担的程度不同。

在基础符合性级别，作者可假定WCAG 3.0提供的方法和技术是可用的。更高符合性级别下，作者可能需要测试技术是否有效，或检查可用用户代理是否满足要求，或两者结合。

这种方法意味着工作组将确保纳入的方法和技术获得用户代理的广泛、国际化支持，并有足够的技术手段满足各项要求。

WCAG 3.0计划采用内容管理系统，为方法/技术标记支持信息。还应有流程让相关方提供信息。

“无障碍支持集”用于更高符合性级别，定义测试所用用户代理和辅助技术。它将包含在符合性声明中，并允许作者使用WCAG 3.0未提供的技术。

针对辅助技术长期存在的bug是否例外，仍在讨论中。

3.1.2 定义符合性范围探索中

评估内容可访问性时，WCAG 3.0要求指南适用于特定范围。范围可以是一个数字产品内的所有内容，但通常是全体内容的一个或多个子集。原因包括：

  • 内容量庞大时，除自动化评估外，全面评估几乎不可行；
  • 很多情况下内容更新频繁，导致评估仅在某一时刻有效；
  • 部分内容对多数用户更重要；
  • 内容大部分符合要求但部分有问题时，会妨碍用户完成流程。

因此WCAG 3.0定义了两种内容范围方式：视图和流程。评估以一个或多个完整视图或流程为单位，符合性也以此为依据。

符合性仅针对流程和视图定义。但符合性声明可覆盖一个流程和视图，一系列流程和视图，或多个相关流程和视图。流程中的所有唯一步骤MUST在视图集中体现。流程外的视图MAY也可纳入范围。

编辑注

我们认识到代表性抽样是大型复杂网站评估可访问性的重要策略。虽然本文件目前未涉及，但我们的目标是在本文件或单独文件中补充说明，确保在指南进入候选推荐阶段前涵盖该内容。欢迎你就如何纳入代表性抽样提出建议和反馈。

general flash and red flash thresholds开发中

若满足以下任一条件，则闪光或快速变化的图像序列低于阈值（即内容通过）：

  • 在任意一秒内，不超过三个通用闪光和/或不超过三个红色闪光；或
  • 同时发生的闪光组合区域，在典型观看距离下，于屏幕上任意 10 度视野内总计不超过 0.006 球面度（即屏幕上任意 10 度视野的 25%）。

其中：

  • 通用闪光定义为：当较暗图像的相对亮度低于 0.80 时，在最大相对亮度（1.0）的 10% 或更大幅度上的一对相对亮度相反变化；“一对相反变化”指先增加后减少，或先减少后增加；并且
  • 红色闪光定义为涉及饱和红色的任意一对相反过渡。

例外：细密、平衡的图案（如白噪声或方格小于 0.1 度（典型观看距离下的视角）的棋盘交替图案）的闪烁不违反上述阈值。

Note 1

对于通用软件或网页内容，在 1024 × 768 分辨率下观看时，于屏幕任意位置取一块 341 × 256 像素矩形区域，可很好估算标准屏幕与观看距离（如 15–17 英寸屏幕、22–26 英寸观看距离）下的 10 度视野。该 75–85 ppi 的分辨率低于 CSS 规范中的名义 96 ppi，因而更为保守。更高分辨率显示设备呈现相同内容时，图像更小更安全，因此用较低分辨率来定义阈值。

Note 2

“过渡”是指在相对亮度（或红色闪光的相对亮度/颜色）对时间曲线中，相邻峰谷之间的变化。一次闪光由两次相反的过渡组成。

Note 3

领域内关于“涉及饱和红色的一对相反过渡”（源自 WCAG 2.2）的新工作定义为：其中一次过渡指向或来自一个状态，该状态满足 R/(R + G + B) ≥ 0.8，且两状态在 CIE 1976 UCS 色度图中的差异大于 0.2（无量纲）。[ISO_9241-391]

Note 4

已有工具可基于视频屏幕捕获进行分析。然而，若任意一秒内闪光次数 ≤ 3，则无需任何工具即可判断内容自动通过（见上述第 1、2 条）。

relative luminance开发中

在某一颜色空间中任意点的相对亮度，归一化为最暗黑色为 0、最亮白色为 1

Note 1

在 sRGB 颜色空间中，颜色的相对亮度定义为 L = 0.2126 * R + 0.7152 * G + 0.0722 * B，其中 R、G、B 定义为：

  • 若 RsRGB <= 0.04045，则 R = RsRGB/12.92；否则 R = ((RsRGB+0.055)/1.055) ^ 2.4
  • 若 GsRGB <= 0.04045，则 G = GsRGB/12.92；否则 G = ((GsRGB+0.055)/1.055) ^ 2.4
  • 若 BsRGB <= 0.04045，则 B = BsRGB/12.92；否则 B = ((BsRGB+0.055)/1.055) ^ 2.4

且 RsRGB、GsRGB、BsRGB 定义为：

  • RsRGB = R8bit/255
  • GsRGB = G8bit/255
  • BsRGB = B8bit/255

“^” 表示乘方运算符。（公式来源于 [SRGB]。）

Note 2

2021 年 5 月之前，定义中的 0.04045 为 0.03928，源自更早版本的规范，现已更新。该调整对在本指南语境中的计算无实际影响。

Note 3

几乎所有当前用于查看网页内容的系统都假定采用 sRGB 编码。除非已知会用其他颜色空间处理与显示内容，否则作者应以 sRGB 颜色空间进行评估。

Note 4

若在分发后发生抖动（dithering），则使用源颜色值。若在源端发生抖动，则应使用抖动颜色的平均值（平均 R、平均 G、平均 B）。

Note 5

已有工具可在进行对比度与闪光测试时自动完成计算。

Editor's note

WCAG 2.2 单独提供了页面以 MathML 展示相对亮度的定义与公式。如何将其纳入 WCAG 3 仍需处理。
