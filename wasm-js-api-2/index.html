1. 简介

根据设计，WebAssembly 核心规范 范围 [WEBASSEMBLY] 并不包含 WebAssembly 程序如何与其外部执行环境交互的描述。 它定义了 WebAssembly 与其环境（称为 嵌入者）之间的抽象嵌入接口。 嵌入者仅通过该接口与 WebAssembly 语义交互，并实现其宿主环境与嵌入 API 之间的连接。 本文档描述了 WebAssembly 嵌入到 JavaScript [ECMASCRIPT] 环境的方式，包括如何构建和实例化 WebAssembly 模块，如何调用导入与导出函数，如何交换数据，以及如何处理错误。 当 JavaScript 环境本身被嵌入在 Web 浏览器中时，Web API 规范 [WASMWEB] 描述了与 Web 环境相关的额外行为。

2. API 用例示例

本节为非规范性内容。

假设有 demo.wat（编码为 demo.wasm）：

(module
    (import "js" "import1" (func $i1))
    (import "js" "import2" (func $i2))
    (func $main (call $i1))
    (start $main)
    (func (export "f") (call $i2))
)

以及在浏览器中运行的如下 JavaScript：

var importObj = {js: {
    import1: () => console.log("hello,"),
    import2: () => console.log("world!")
}};
fetch('demo.wasm').then(response =>
    response.arrayBuffer()
).then(buffer =>
    WebAssembly.instantiate(buffer, importObj)
).then(({module, instance}) =>
    instance.exports.f()
);

本规范依赖 Infra 标准。[INFRA]

WebAssembly 的 sequence 类型等价于该标准中定义的 list 类型；二者的值可直接转换，无需特殊处理。

4. 内部存储

4.1. WebAssembly Store 与 JavaScript 的交互

注： WebAssembly 语义是基于抽象的store定义的，store 表示 WebAssembly 抽象机的状态。 WebAssembly 操作接受 store 并返回更新后的 store。

每个 agent 都有一个 关联 store。当新 agent 创建时，其关联 store 初始化为 store_init() 的结果。

注： 本规范下，任何 WebAssembly 相关对象、内存或地址都不能在同一个agent cluster中的 agent 之间共享。未来版本的 WebAssembly 可能会改变这一点。

WebAssembly store 的各元素可通过 JavaScript 值标识。特别是，有对应 Memory 对象的 WebAssembly memory instance 会被标识为一个 JavaScript Data Block；对该 Data Block 的修改等价于将 agent 的 store 更新为反映这些变化的新 store，反之亦然。

4.2. WebAssembly JS 对象缓存

注： 某些 WebAssembly 对象可能有对应的 JavaScript 对象。这种对应关系保存在每个 agent 的 WebAssembly 地址到 JavaScript 对象的映射中。 该映射用于确保针对同一 agent，同一个 WebAssembly 地址至多对应一个 JavaScript 对象。但该属性对共享对象不成立。

每个 agent 都关联如下 有序映射：

  • 内存对象缓存：把memory address 映射为 Memory 对象。

  • 表对象缓存：把table address 映射为 Table 对象。

  • 导出函数缓存：把function address 映射为 导出函数对象。

  • 导出 GC 对象缓存：把struct address 和 array address 映射为 导出 GC 对象。

  • 全局对象缓存：把global address 映射为 Global 对象。

  • 标签对象缓存：把tag address 映射为 Tag 对象。

  • 异常对象缓存：把exception address 映射为 Exception 对象。

  • 宿主值缓存：把host address 映射为相应值。

5. WebAssembly 命名空间 

dictionary WebAssemblyInstantiatedSource {
    required Module module;
    required Instance instance;
};

dictionary WebAssemblyCompileOptions {
    USVString? importedStringConstants;
    sequence<USVString> builtins;
};

[Exposed=*]
namespace WebAssembly {
    boolean validate(BufferSource bytes, optional WebAssemblyCompileOptions options = {});
    Promise<Module> compile(BufferSource bytes, optional WebAssemblyCompileOptions options = {});

    Promise<WebAssemblyInstantiatedSource> instantiate(
        BufferSource bytes, optional object importObject, optional WebAssemblyCompileOptions options = {});

    Promise<Instance> instantiate(
        Module moduleObject, optional object importObject);

    readonly attribute Tag JSTag;
};
若要从源字节 bytes 编译 WebAssembly 模块，请执行以下步骤：

  1. 令 module 为 module_decode(bytes)。如 module 为 error，则返回 error。

  2. 若 module_validate(module) 为 error，则返回 error。

  3. 返回 module。

要对模块 module、已启用的内建 builtinSetNames 和 importedStringModule 验证 WebAssembly 模块的内建和导入字符串，请执行以下步骤：

  1. 若对 builtinSetNames 进行内建集合名验证 的结果为 false，则返回 false。

  2. 遍历 module_imports(module) 中每个 import，

    1. 若 importedStringModule 不为 null 且 import[0] 等于 importedStringModule，

      1. 令 importExternType 为 import[2]。

      2. 令 stringExternType 为 global const (ref extern)。

      3. 若 match_externtype(stringExternType, importExternType) 结果为 false，则返回 false

    2. 否则，

      1. 若用 import 和 builtinSetNames 验证 import 是否为内建导入结果为 false，则返回 false。

  3. 返回 true。

当调用 validate(bytes, options) 方法时， 执行如下步骤：

  1. 令 stableBytes 为 buffer 所持有字节的副本 bytes。

  2. 将 stableBytes 编译 为一个 WebAssembly 模块，并将结果存为 module。

  3. 如果 module 为 error，则返回 false。

  4. 令 builtinSetNames 为 options["builtins"]。

  5. 令 importedStringModule 为 options["importedStringConstants"]。

  6. 如果对 module 以 builtinSetNames 和 importedStringModule 验证内建与导入字符串 的结果为 false，则返回 false。

  7. 返回 true。

Module 对象表示单个 WebAssembly 模块。每个 Module 对象具有以下内部槽：

  • [[Module]]：一个 WebAssembly module

  • [[Bytes]]：[[Module]] 的源字节。

  • [[BuiltinSets]]：builtin sets 的名称有序集合。

  • [[ImportedStringModule]]：一个可选的模块指示符字符串，字符串常量可自其中导入。

要从模块 module、源字节 bytes、已启用的内建 builtinSetNames 与 importedStringModule 构造一个 WebAssembly 模块对象，执行以下步骤：

  1. 令 moduleObject 为一个新的 Module 对象。

  2. 将 moduleObject.[[Module]] 设为 module。

  3. 将 moduleObject.[[Bytes]] 设为 bytes。

  4. 将 moduleObject.[[BuiltinSets]] 设为 builtinSetNames。

  5. 将 moduleObject.[[ImportedStringModule]] 设为 importedStringModule。

  6. 返回 moduleObject。

要从源字节 bytes 和 WebAssemblyCompileOptions options 异步编译一个 WebAssembly 模块，并使用可选的 task source taskSource，执行以下步骤：

  1. 令 promise 为 一个新的 promise。

  2. 并行执行以下步骤：

    1. 编译 WebAssembly 模块 bytes 并将结果存为 module。

    2. 排队一个任务（Queue a task） 以执行以下步骤。若提供了 taskSource，则在该 task source 上排队该任务。

      1. 如果 module 为 error，则以 CompileError 异常拒绝 promise。

      2. 令 builtinSetNames 为 options["builtins"]。

      3. 令 importedStringModule 为 options["importedStringConstants"]。

      4. 如果对 module 以 builtinSetNames 和 importedStringModule 验证内建与导入字符串 的结果为 false，则以 CompileError 异常拒绝 promise。

      5. 否则，

        1. 从 module、bytes、builtinSetNames、 importedStringModule 构造一个 WebAssembly 模块对象，并令 moduleObject 为结果。

        2. Resolve promise，其值为 moduleObject。

  3. 返回 promise。

当调用 compile(bytes, options) 方法时， 执行如下步骤：

  1. 令 stableBytes 为 buffer 所持有字节的副本 bytes。

  2. 使用 options 从 stableBytes 异步编译一个 WebAssembly 模块 并返回结果。

要用模块 module 和 importedStringModule 实例化导入的字符串，执行以下步骤：

  1. 断言：importedStringModule 不为 null。

  2. 令 exportsObject 为 ! OrdinaryObjectCreate(null)。

  3. 对于每个 (moduleName, componentName, externtype) 属于 module_imports(module)，

    1. 如果 moduleName 不等于 importedStringModule，则继续。

    2. 令 stringConstant 为 componentName。

    3. 令 status 为 ! CreateDataProperty(exportsObject, stringConstant, stringConstant)。

    4. 断言：status 为 true。

  4. 返回 exportsObject。

要从 WebAssembly 模块 module、imports 对象 importObject、已启用内建 builtinSetNames 与 importedStringModule 读取导入（read the imports），执行以下步骤：

  1. 如果 module.imports 非空 且 importObject 为 undefined，则抛出 TypeError 异常。

  2. 令 builtinOrStringImports 为有序映射 « »。

  3. 对于每个 builtinSetName 属于 builtinSetNames，

    1. 断言：builtinOrStringImports 不包含 builtinSetName

    2. 如果 builtinSetName 不指代一个 builtin set，则继续。

    3. 令 exportsObject 为使用 builtinSetName 实例化一个 builtin set 的结果。

    4. 令 builtinSetQualifiedName 为以 "wasm:" 作为前缀的 builtinSetName。

    5. 将 builtinOrStringImports[builtinSetQualifiedName] 设为 exportsObject

  4. 如果 importedStringModule 不为 null，

    1. 令 exportsObject 为以 module 和 importedStringModule 实例化导入的字符串 的结果。

    2. 将 builtinOrStringImports[importedStringModule] 设为 exportsObject

  5. 令 imports 为 « »。

  6. 对于每个 (moduleName, componentName, externtype) 属于 module_imports(module)，

    1. 如果 builtinOrStringImports 包含 moduleName，

      1. 令 o 为 builtinOrStringImports[moduleName]。

      2. 如果 o 不是对象，或 o 不包含 componentName，

        1. 将 o 设为 ? Get(importObject, moduleName) 的结果。

    2. 否则，

      1. 令 o 为 ? Get(importObject, moduleName) 的结果。

    3. 如果 o 不是对象，抛出 TypeError 异常。

    4. 令 v 为 ? Get(o, componentName) 的结果。

    5. 若 externtype 形式为 func functype，

      1. 若 IsCallable(v) 为 false，则抛出 LinkError 异常。

      2. 若 v 具有 [[FunctionAddress]] 内部槽，因此是一个 导出函数（Exported Function），

        1. 令 funcaddr 为 v 的 [[FunctionAddress]] 内部槽的值。

      3. 否则，

        1. 从 v 与 functype 创建一个宿主函数，并令 funcaddr 为结果。

        2. 令 index 为 imports 中外部函数的数量。该值 index 即 宿主函数的索引 funcaddr。

      4. 令 externfunc 为 external value func funcaddr。

      5. 将 externfunc 追加到 imports。

    6. 若 externtype 形式为 global mut valtype，

      1. 如果 v 实现（implements）了 Global，

        1. 令 globaladdr 为 v.[[Global]]。

      2. 否则，

        1. 若 valtype 为 i64 且 v 不是 BigInt，

          1. 抛出 LinkError 异常。

        2. 若 valtype 属于 i32、f32 或 f64 且 v 不是 Number，

          1. 抛出 LinkError 异常。

        3. 若 valtype 为 v128，

          1. 抛出 LinkError 异常。

        4. 令 value 为 ToWebAssemblyValue(v, valtype)。若该操作抛出 TypeError， 捕获之，并抛出 LinkError 异常。

        5. 令 store 为 外围 agent 的 关联 store。

        6. 令 (store, globaladdr) 为 global_alloc(store， const valtype， value) 的结果。

        7. 将 外围 agent 的 关联 store 设为 store。

      3. 令 externglobal 为 global globaladdr。

      4. 将 externglobal 追加到 imports。

    7. 若 externtype 形式为 mem memtype，

      1. 若 v 未 实现 Memory， 则抛出 LinkError 异常。

      2. 令 externmem 为 external value mem v.[[Memory]]。

      3. 将 externmem 追加到 imports。

    8. 若 externtype 形式为 table tabletype，

      1. 若 v 未 实现 Table， 则抛出 LinkError 异常。

      2. 令 tableaddr 为 v.[[Table]]。

      3. 令 externtable 为 external value table tableaddr。

      4. 将 externtable 追加到 imports。

    9. 若 externtype 形式为 tag attribute functype，

      1. 断言：attribute 为 exception。

      2. 若 v 未 实现 Tag， 则抛出 LinkError 异常。

      3. 令 tagaddr 为 v.[[Address]]。

      4. 令 externtag 为 external value tag tagaddr。

      5. 将 externtag 追加到 imports。

  7. 返回 imports。

注： 此算法只验证传入的 JavaScript 值类型是否正确。 对 WebAssembly 类型需求的验证被推迟到 “实例化 WebAssembly 模块核心” 算法。

要从 WebAssembly 模块 module 和实例 instance 创建一个 exports 对象，执行以下步骤：

  1. 令 exportsObject 为 ! OrdinaryObjectCreate(null)。

  2. 对于每个 (name, externtype) 属于 module_exports(module)，

    1. 令 externval 为 instance_export(instance, name)。

    2. 断言：externval 不是 error。

    3. 若 externtype 形式为 func functype，

      1. 断言：externval 形式为 func funcaddr。

      2. 令 func funcaddr 为 externval。

      3. 令 func 为从 funcaddr 创建的新导出函数（Exported Function）。

      4. 令 value 为 func。

    4. 若 externtype 形式为 global mut globaltype，

      1. 断言：externval 形式为 global globaladdr。

      2. 令 global globaladdr 为 externval。

      3. 令 global 为从 globaladdr 创建的新 Global 对象。

      4. 令 value 为 global。

    5. 若 externtype 形式为 mem memtype，

      1. 断言：externval 形式为 mem memaddr。

      2. 令 mem memaddr 为 externval。

      3. 令 memory 为从 memaddr 创建的新 Memory 对象。

      4. 令 value 为 memory。

    6. 若 externtype 形式为 table tabletype，

      1. 断言：externval 形式为 table tableaddr。

      2. 令 table tableaddr 为 externval。

      3. 令 table 为从 tableaddr 创建的新 Table 对象。

      4. 令 value 为 table。

    7. 若 externtype 形式为 tag attribute functype，

      1. 断言：attribute 为 exception。

      2. 断言：externval 形式为 tag tagaddr。

      3. 令 tag tagaddr 为 externval。

      4. 令 tag 为从 tagaddr 创建的新 Tag 对象。

      5. 令 value 为 tag。

    8. 令 status 为 ! CreateDataProperty(exportsObject, name, value)。

    9. 断言：status 为 true。

    注： 在 WebAssembly 模块验证 过程中执行的有效性与唯一性检查可确保每个属性名有效，且无属性被定义两次。

  3. 执行 ! SetIntegrityLevel(exportsObject, "frozen")。

  4. 返回 exportsObject。

要从 WebAssembly 模块 module 和实例 instance 初始化一个实例对象 instanceObject，执行以下步骤：

  1. 从 module 与 instance 创建一个 exports 对象，并令 exportsObject 为结果。

  2. 将 instanceObject.[[Instance]] 设为 instance。

  3. 将 instanceObject.[[Exports]] 设为 exportsObject。

要从模块 module 与导入 imports 实例化 WebAssembly 模块的核心，执行以下步骤：

  1. 令 store 为 外围 agent 的 关联 store。

  2. 令 result 为 module_instantiate(store， module，imports)。

  3. 如果 result 为 error，则抛出适当的异常类型：

    • 大多数链接时发生的错误，抛出 LinkError 异常。

    • 若错误来自运行 start 函数，则对于多数由 WebAssembly 产生的错误抛出 RuntimeError， 或抛出从内部 ECMAScript 代码传播的错误对象。

    • 如适用，抛出其它错误类型，例如内存耗尽异常，详见 WebAssembly 错误映射。

  4. 令 (store, instance) 为 result。

  5. 将 外围 agent 的 关联 store 设为 store。

  6. 返回 instance。

要从 Module moduleObject 与导入 importObject 异步实例化一个 WebAssembly 模块，执行以下步骤：

  1. 令 promise 为 一个新的 promise。

  2. 令 module 为 moduleObject.[[Module]]。

  3. 令 builtinSetNames 为 moduleObject.[[BuiltinSets]]。

  4. 令 importedStringModule 为 moduleObject.[[ImportedStringModule]]。

  5. 读取导入（read the imports） module 的导入，使用 importObject、builtinSetNames 和 importedStringModule，并令 imports 为结果。 如果此操作抛出异常，则捕获之，以该异常拒绝 promise，并返回 promise。

  6. 并行执行以下步骤：

    1. 排队一个任务（Queue a task） 以执行以下步骤： 注：此处可执行与实现相关的工作。

      1. 用 imports 实例化 WebAssembly 模块的核心 module，令 instance 为结果。 若此步骤抛出异常，捕获之，以该异常拒绝 promise，并终止这些子步骤。

      2. 令 instanceObject 为一个 new 的 Instance。

      3. 初始化 instanceObject，来自 module 与 instance。 若此步骤抛出异常，捕获之，以该异常拒绝 promise，并终止这些子步骤。

      4. Resolve promise，其值为 instanceObject。

  7. 返回 promise。

要以导入 importObject 实例化一个模块的 promise promiseOfModule，执行以下步骤：

  1. 令 promise 为 一个新的 promise。

  2. 对 promiseOfModule 作出响应（React）：

    • 如果 promiseOfModule 以值 module 得到兑现：

      1. 实例化 WebAssembly 模块 module，导入 importObject，并令 innerPromise 为结果。

      2. 对 innerPromise 作出响应（React）：

        • 如果 innerPromise 以值 instance 得到兑现。

          1. 令 result 为 WebAssemblyInstantiatedSource 值 «[ "module" → module, "instance" → instance ]»。

          2. Resolve promise，其值为 result。

        • 如果 innerPromise 因原因 reason 被拒绝：

          1. 以 reason 拒绝 promise。

    • 如果 promiseOfModule 因原因 reason 被拒绝：

      1. 以 reason 拒绝 promise。

  3. 返回 promise。

当调用 instantiate(bytes, importObject, options) 方法时，执行如下步骤：

  1. 令 stableBytes 为 buffer 所持有字节的副本 bytes。

  2. 使用 options 从 stableBytes 异步编译一个 WebAssembly 模块，并令 promiseOfModule 为结果。

  3. 实例化 promiseOfModule，使用导入 importObject，并返回结果。

当调用 instantiate(moduleObject, importObject) 方法时，执行如下步骤：

  1. 异步实例化 WebAssembly 模块 moduleObject，导入 importObject，并返回结果。

注： 后续的流式 API 记录于 WebAssembly Web API 中。

当调用 WebAssembly 命名空间的 JSTag 属性的 getter 时，执行如下步骤：

  1. 令 JSTagAddr 为 获取 JavaScript 异常标签 的结果。

  2. 令 JSTagObject 为从 JSTagAddr 创建 Tag 对象 的结果。

  3. 返回 JSTagObject。

  1. 令 module 为 moduleObject.[[Module]]。

  2. 令 exports 为 « »。

  3. 对于每个 属于 module_exports(module) 的 (name, type)，

    1. 令 kind 为 extern 类型的字符串值 type。

    2. 令 obj 为 «[ "name" → name, "kind" → kind ]»。

    3. 将 obj 追加到 exports。

当调用 imports(moduleObject) 方法时，执行如下步骤：

  1. 令 module 为 moduleObject.[[Module]]。

  2. 令 builtinSetNames 为 moduleObject.[[BuiltinSets]]。

  3. 令 importedStringModule 为 moduleObject.[[ImportedStringModule]]。

  4. 令 imports 为 « »。

  5. 对于每个 属于 module_imports(module) 的 (moduleName, name, type)，

    1. 如果对 (moduleName, name, type) 和 builtinSetNames 查找内建（find a builtin） 的结果不为 null，则继续。

    2. 如果 importedStringModule 不为 null 且 moduleName 等于 importedStringModule，则继续。

    3. 令 kind 为 extern 类型的字符串值 type。

    4. 令 obj 为 «[ "module" → moduleName, "name" → name, "kind" → kind ]»。

    5. 将 obj 追加到 imports。

  6. 返回 imports。

当调用 customSections(moduleObject, sectionName) 方法时，执行如下步骤：

  1. 令 bytes 为 moduleObject.[[Bytes]]。

  2. 令 customSections 为 « »。

  3. 对于每个 custom section customSection 属于 bytes， 按照 module 语法 进行解释，

    1. 令 name 为 customSection 的 name，按 UTF-8 解码。

    2. 断言：name 非失败（moduleObject.[[Module]] 有效）。

    3. 若 name 作为字符串等于 sectionName，

      1. 将 一个新的 ArrayBuffer （其包含 bytes 中由该 customsec 产生式匹配范围的字节副本）追加到 customSections。

  4. 返回 customSections。

当调用 Module(bytes, options) 构造函数时，执行如下步骤：

  1. 令 stableBytes 为 buffer 所持有字节的副本 bytes。

  2. 编译 WebAssembly 模块 stableBytes 并将结果存为 module。

  3. 若 module 为 error，则抛出 CompileError 异常。

  4. 令 builtinSetNames 为 options["builtins"]。

  5. 令 importedStringModule 为 options["importedStringConstants"]。

  6. 如果对 module 以 builtinSetNames 和 importedStringModule 验证内建与导入字符串 的结果为 false，则抛出 CompileError 异常。

  7. 设置 this.[[Module]] 为 module。

  8. 设置 this.[[Bytes]] 为 stableBytes。

  9. 设置 this.[[BuiltinSets]] 为 builtinSetNames。

  10. 设置 this.[[ImportedStringModule]] 为 importedStringModule。

注： 某些实现会对 bytes 施加大小限制。不推荐使用此 API，建议使用异步 API。

Memory 对象表示一个 memory instance， 可被多个 Instance 对象同时引用。每个 Memory 对象具有以下内部槽：

  • [[Memory]]：一个 memory address

  • [[BufferObject]]：一个 ArrayBuffer， 其 Data Block 与 上述内存地址相对应

要从 memory address memaddr 创建定长内存缓冲区，执行如下步骤：

  1. 令 block 为一个 Data Block，其 与 memaddr 的底层内存相对应。

  2. 令 buffer 为一个新的 ArrayBuffer， 其内部槽包含 [[ArrayBufferData]]、[[ArrayBufferByteLength]] 和 [[ArrayBufferDetachKey]]。

  3. 设 buffer.[[ArrayBufferData]] 为 block。

  4. 设 buffer.[[ArrayBufferByteLength]] 为 block 的长度。

  5. 设 buffer.[[ArrayBufferDetachKey]] 为 "WebAssembly.Memory"。

  6. 返回 buffer。

要从 memory address memaddr 和 maxsize 创建可调整大小的内存缓冲区，执行如下步骤：

  1. 令 block 为一个 Data Block，其 与 memaddr 的底层内存相对应。

  2. 令 length 为 block 的长度。

  3. 令 buffer 为一个新的 ArrayBuffer， 其内部槽包含 [[ArrayBufferData]]、[[ArrayBufferByteLength]]、[[ArrayBufferMaxByteLength]] 和 [[ArrayBufferDetachKey]]。

  4. 设 buffer.[[ArrayBufferData]] 为 block。

  5. 设 buffer.[[ArrayBufferByteLength]] 为 length。

  6. 设 buffer.[[ArrayBufferMaxByteLength]] 为 maxsize。

  7. 设 buffer.[[ArrayBufferDetachKey]] 为 "WebAssembly.Memory"。

  8. 返回 buffer。

要从 memory address memaddr 初始化一个内存对象 memory，执行如下步骤：

  1. 令 map 为 外围 agent 关联的 Memory object cache。

  2. 断言：map[memaddr] 不存在。

  3. 令 buffer 为从 memaddr 创建定长内存缓冲区 的结果。

  4. 设 memory.[[Memory]] 为 memaddr。

  5. 设 memory.[[BufferObject]] 为 buffer。

  6. 设置 map[memaddr] 为 memory。

要从 memory address memaddr 创建一个内存对象，执行如下步骤：

  1. 令 map 为 外围 agent 关联的 Memory object cache。

  2. 如果 map[memaddr] 存在，

    1. 返回 map[memaddr]。

  3. 令 memory 为一个 new 的 Memory。

  4. 初始化 memory，来自 memaddr。

  5. 返回 memory。

当调用 Memory(descriptor) 构造函数时，执行如下步骤：

  1. 若 descriptor["address"] 存在，令 addrtype 为 descriptor["address"]；否则，令 addrtype 为 "i32"。

  2. 令 initial 为 ? AddressValueToU64(descriptor["initial"], addrtype)。

  3. 若 descriptor["maximum"] 存在，令 maximum 为 ? AddressValueToU64(descriptor["maximum"], addrtype)；否则，令 maximum 为空。

  4. 令 memtype 为 memory type addrtype { min initial, max maximum }。

  5. 若 memtype 非 有效，抛出 RangeError 异常。

  6. 令 store 为 外围 agent 的 关联 store。

  7. 令 (store, memaddr) 为 mem_alloc(store, memtype)。若分配失败，抛出 RangeError 异常。

  8. 将 外围 agent 的 关联 store 设为 store。

  9. 初始化 this，来自 memaddr。

要 刷新 Memory 缓冲区 memaddr，执行如下步骤：

  1. 令 map 为 外围 agent 关联的 Memory object cache。

  2. 断言：map[memaddr] 存在。

  3. 令 memory 为 map[memaddr]。

  4. 令 buffer 为 memory.[[BufferObject]]。

  5. 若 IsFixedLengthArrayBuffer(buffer) 为 true，

    1. 执行 ! DetachArrayBuffer(buffer, "WebAssembly.Memory")。

    2. 令 buffer 为从 memaddr 创建定长内存缓冲区 的结果。

    3. 设 memory.[[BufferObject]] 为 buffer。

  6. 否则，

    1. 令 block 为一个 Data Block，其 与 memaddr 的底层内存相对应。

    2. 设 buffer.[[ArrayBufferData]] 为 block。

    3. 设 buffer.[[ArrayBufferByteLength]] 为 block 的长度。

要将与 memory address memaddr 关联的 内存缓冲区扩展 delta，执行如下步骤：

  1. 令 store 为 外围 agent 的 关联 store。

  2. 令 ret 为 mem_size(store, memaddr)。

  3. 令 store 为 mem_grow(store, memaddr, delta)。

  4. 若 store 为 error，抛出 RangeError 异常。

  5. 将 外围 agent 的 关联 store 设为 store。

  6. 刷新 Memory 缓冲区 memaddr。

  7. 返回 ret。

当调用 grow(delta) 方法时，执行如下步骤：

  1. 令 memaddr 为 this.[[Memory]]。

  2. 令 store 为 外围 agent 的 关联 store。

  3. 令 addrtype 为 address type，来自 mem_type(store, memaddr)。

  4. 令 delta64 为 ? AddressValueToU64(delta, addrtype)。

  5. 令 ret 为 扩展内存缓冲区（与 memaddr 关联）delta64 的结果。

  6. 返回 U64ToAddressValue(ret, addrtype)。

在 WebAssembly memory.grow 指令执行后立即，执行如下步骤：

  1. 如果栈顶不是 i32.const (−1)，

    1. 令 frame 为 当前帧（current frame）。

    2. 断言：由于验证，frame.module.memaddrs[0] 存在。

    3. 令 memaddr 为内存地址 frame.module.memaddrs[0]。

    4. 刷新 Memory 缓冲区 memaddr。

当调用 toFixedLengthBuffer() 方法时，执行如下步骤：

  1. 令 buffer 为 this.[[BufferObject]]。

  2. 若 IsFixedLengthArrayBuffer(buffer) 为 true，返回 buffer。

  3. 令 memaddr 为 this.[[Memory]]。

  4. 令 fixedBuffer 为从 memaddr 创建定长内存缓冲区 的结果。

  5. 执行 ! DetachArrayBuffer(buffer, "WebAssembly.Memory")。

  6. 设 this.[[BufferObject]] 为 fixedBuffer。

  7. 返回 fixedBuffer。

当调用 toResizableBuffer() 方法时，执行如下步骤：

  1. 令 memaddr 为 this.[[Memory]]。

  2. 令 store 为 外围 agent 的 关联 store。

  3. 令 memtype 为 mem_type(store, memaddr)。

  4. 若 memtype 没有 max，

    1. 抛出 TypeError 异常。

  5. 令 buffer 为 this.[[BufferObject]]。

  6. 若 IsFixedLengthArrayBuffer(buffer) 为 false，返回 buffer。

  7. 断言：memtype 具有 max。

  8. 令 maxsize 为 memtype 中的 max 值 * 65536。

  9. 令 resizableBuffer 为从 memaddr 和 maxsize 创建可调整大小的内存缓冲区 的结果。

  10. 执行 ! DetachArrayBuffer(buffer, "WebAssembly.Memory")。

  11. 设 this.[[BufferObject]] 为 resizableBuffer。

  12. 返回 resizableBuffer。

由 ArrayBuffer 对象（由 Memory 对象返回）必须具有 WebAssembly page size（常量 65536）的整数倍大小。因此 HostResizeArrayBuffer 重新定义如下。

抽象操作 HostResizeArrayBuffer 接受参数 buffer（一个 ArrayBuffer） 和 newLength。调用时执行以下步骤。

  1. 若 buffer.[[ArrayBufferDetachKey]] 为 "WebAssembly.Memory"，

    1. 令 map 为 外围 agent 关联的 Memory object cache。

    2. 断言：buffer 是 map 中恰好一个值的 [[BufferObject]]。

    3. 对于每个 memaddr → mem 属于 map，

      1. 若 SameValue(mem.[[BufferObject]], buffer) 为 true，

        1. 断言：buffer.[[ArrayBufferByteLength]] 模 65536 等于 0。

        2. 令 lengthDelta 为 newLength - buffer.[[ArrayBufferByteLength]]。

        3. 若 lengthDelta < 0 或 lengthDelta 模 65536 不为 0，

          1. 抛出 RangeError 异常。

        4. 令 delta 为 lengthDelta ÷ 65536。

        5. 扩展内存缓冲区 与 memaddr 关联，扩展量为 delta。

    4. 返回 handled。

  2. 否则，返回 unhandled。

Memory 的 buffer 属性的 getter 返回 this.[[BufferObject]]。

  1. 令 elementtype 为 ToValueType(descriptor["element"])。

  2. 若 elementtype 不是一个 reftype，

    1. 抛出 TypeError 异常。

  3. 若 descriptor["address"] 存在，令 addrtype 为 descriptor["address"]；否则令 addrtype 为 "i32"。

  4. 令 initial 为 ? AddressValueToU64(descriptor["initial"], addrtype)。

  5. 若 descriptor["maximum"] 存在，令 maximum 为 ? AddressValueToU64(descriptor["maximum"], addrtype)；否则令 maximum 为空。

  6. 令 type 为 table type addrtype { min initial, max maximum } elementType。

  7. 若 type 非 有效，抛出 RangeError 异常。

  8. 若 value 缺失，

    1. 令 ref 为 DefaultValue(elementtype)。

    2. 断言：ref 非 error。

  9. 否则，

    1. 令 ref 为 ? ToWebAssemblyValue(value, elementType)。

  10. 令 store 为 外围 agent 的 关联 store。

  11. 令 (store, tableaddr) 为 table_alloc(store, type, ref)。若分配失败，抛出 RangeError 异常。

  12. 将 外围 agent 的 关联 store 设为 store。

  13. 初始化 this，来自 tableaddr。

  2. 令 store 为 外围 agent 的 关联 store。

  3. 令 initialSize 为 table_size(store, tableaddr)。

  4. 令 (addrtype, limits, elementtype) 为 table_type(store, tableaddr)。

  5. 令 delta64 为 ? AddressValueToU64(delta, addrtype)。

  6. 若 value 缺失，

    1. 令 ref 为 DefaultValue(elementtype)。

    2. 若 ref 为 error，抛出 TypeError 异常。

  7. 否则，

    1. 令 ref 为 ? ToWebAssemblyValue(value, elementtype)。

  8. 令 result 为 table_grow(store, tableaddr, delta64, ref)。

  9. 若 result 为 error，抛出 RangeError 异常。

    注： 上述异常可能由内存不足或无效的尺寸参数导致。

  10. 将 外围 agent 的 关联 store 设为 result。

  11. 返回 initialSize。

  5. 令 index64 为 ? AddressValueToU64(index, addrtype)。

  6. 令 result 为 table_read(store, tableaddr, index64)。

  7. 若 result 为 error，抛出 RangeError 异常。

  8. 返回 ToJSValue(result)。

当调用 set(index, value) 方法时，执行如下步骤：

  1. 令 tableaddr 为 this.[[Table]]。

  2. 令 store 为 外围 agent 的 关联 store。

  3. 令 (addrtype, limits, elementtype) 为 table_type(store, tableaddr)。

  4. 若 elementtype 匹配 exnref，

    1. 抛出 TypeError 异常。

  5. 令 index64 为 ? AddressValueToU64(index, addrtype)。

  6. 若 value 缺失，

    1. 令 ref 为 DefaultValue(elementtype)。

    2. 若 ref 为 error，抛出 TypeError 异常。

  7. 否则，

    1. 令 ref 为 ? ToWebAssemblyValue(value, elementtype)。

  8. 令 store 为 table_write(store, tableaddr, index64, ref)。

  10. 将 外围 agent 的 关联 store 设为 store。

  7. 令 store 为当前 agent 的 关联 store。

  8. 令 (store, globaladdr) 为 global_alloc(store, globaltype, value)。

  9. 将当前 agent 的 关联 store 设为 store。

  10. 初始化 this，来自 globaladdr。

算法 GetGlobalValue(Global global) 执行以下步骤：

  1. 令 store 为当前 agent 的 关联 store。

  2. 令 globaladdr 为 global.[[Global]]。

  3. 令 globaltype 为 global_type(store, globaladdr)。

  4. 若 globaltype 形式为 mut valuetype 且 valuetype 匹配 v128 或 exnref，抛出 TypeError。

  5. 令 value 为 global_read(store, globaladdr)。

  6. 返回 ToJSValue(value)。

当访问 Global 的 value 属性的 getter 时，执行如下步骤：

  1. 返回 GetGlobalValue(this)。

当设置 Global 的 value 属性时，执行如下步骤：

  1. 令 store 为当前 agent 的 关联 store。

  2. 令 globaladdr 为 this.[[Global]]。

  3. 令 mut valuetype 为 global_type(store, globaladdr)。

  4. 若 valuetype 匹配 v128 或 exnref，抛出 TypeError。

  5. 若 mut 为 const，抛出 TypeError。

  6. 令 value 为 ToWebAssemblyValue(the given value, valuetype)。

  7. 令 store 为 global_write(store, globaladdr, value)。

  8. 若 store 为 error，抛出 RangeError 异常。

  9. 将当前 agent 的 关联 store 设为 store。

当调用 valueOf() 方法时，执行如下步骤：

  1. 返回 GetGlobalValue(this)。

5.6. 导出函数

WebAssembly 函数以 导出函数（Exported Function） 的形式在 JavaScript 中可用。 导出函数是内建函数对象（Built-in Function Objects），不是构造函数，且具有 [[FunctionAddress]] 内部槽。 该内部槽保存一个相对于外围 agent的关联 store的函数地址。

WebAssembly 函数的名称 funcaddr 通过以下步骤获得：

  1. 令 store 为 外围 agent 的 关联 store。

  2. 令 funcinst 为 store.funcs[funcaddr]。

  3. 若 funcinst 形式为 {type functype, hostcode hostfunc}，

    1. 断言：hostfunc 是一个 JavaScript 对象且 IsCallable(hostfunc) 为 true。

    2. 令 index 为 宿主函数的索引 funcaddr。

  4. 否则，

    1. 令 moduleinst 为 funcinst.module。

    2. 断言：funcaddr 包含在 moduleinst.funcaddrs 中。

    3. 令 index 为在 moduleinst.funcaddrs 中找到 funcaddr 的索引。

  5. 返回 ! ToString(index)。

要从 WebAssembly 函数地址 funcaddr 创建一个新的导出函数（Exported Function），执行以下步骤：

  1. 令 map 为 外围 agent 关联的 导出函数缓存（Exported Function cache）。

  2. 若 map[funcaddr] 存在，

    1. 返回 map[funcaddr]。

  3. 令 steps 为 “调用导出函数 funcaddr 并传入参数”。

  4. 令 realm 为 当前 Realm。

  5. 令 store 为 外围 agent 的 关联 store。

  6. 令 functype 为 func_type(store, funcaddr)。

  7. 令 [paramTypes] → [resultTypes] 为 functype。

  8. 令 arity 为 paramTypes 的大小。

  9. 令 name 为 WebAssembly 函数的名称 funcaddr。

  10. 令 function 为 ! CreateBuiltinFunction(steps, arity, name, « [[FunctionAddress]] », realm)。

  11. 将 function.[[FunctionAddress]] 设为 funcaddr。

  12. 设置 map[funcaddr] 为 function。

  13. 返回 function。

要用函数地址 funcaddr 和一列 JavaScript 参数 argValues 调用导出函数，执行以下步骤：

  1. 令 store 为 外围 agent 的 关联 store。

  2. 令 functype 为 func_type(store, funcaddr)。

  3. 令 [parameters] → [results] 为 functype。

  4. 若 parameters 或 results 中的任一类型匹配 v128 或 exnref，抛出 TypeError。

    注： 上述错误会在每次调用 [[Call]] 方法时抛出。

  5. 令 args 为 « »。

  6. 令 i 为 0。

  7. 对于每个 t 属于 parameters，

    1. 若 argValues 的大小 > i，令 arg 为 argValues[i]。

    2. 否则，令 arg 为 undefined。

    3. 将 ToWebAssemblyValue(arg, t) 追加到 args。

    4. 设 i 为 i + 1。

  8. 令 (store, ret) 为 func_invoke(store, funcaddr, args) 的结果。

  9. 将 外围 agent 的 关联 store 设为 store。

  10. 若 ret 为 error，则抛出异常。除非 WebAssembly 错误映射 另有说明，否则该异常应为 WebAssembly RuntimeError。

  11. 若 ret 为 THROW ref.exn exnaddr，则

    1. 令 tagaddr 为 exn_tag(store, exnaddr)。

    2. 令 payload 为 exn_read(store, exnaddr)。

    3. 令 jsTagAddr 为获取 JavaScript 异常标签的结果。

    4. 若 tagaddr 等于 jsTagAddr，

      1. 抛出从 payload[0] 检索宿主值 的结果。

    5. 否则，

      1. 令 exception 为从 exnaddr 创建的新 Exception。

      2. 抛出 exception。

  12. 令 outArity 为 大小为 ret 的值。

  13. 若 outArity 为 0，返回 undefined。

  14. 否则，若 outArity 为 1，返回 ToJSValue(ret[0])。

  15. 否则，

    1. 令 values 为 « »。

    2. 对于每个 r 属于 ret，

      1. 将 ToJSValue(r) 追加到 values。

    3. 返回 CreateArrayFromList(values)。

注： 调用导出函数 按 内建函数对象 的定义，在被调用的导出函数的 [[Realm]] 中执行。

注： 导出函数没有 [[Construct]] 方法，因此无法使用 new 运算符调用。

要从 JavaScript 对象 func、类型 functype 和 列表的 WebAssembly 值 arguments 运行宿主函数，执行以下步骤：

  1. 令 [parameters] → [results] 为 functype。

  2. 若 parameters 或 results 中任一类型匹配 v128 或 exnref，抛出 TypeError。

  3. 令 jsArguments 为 « »。

  4. 对于每个 arg 属于 arguments，

    1. 将 ! ToJSValue(arg) 追加到 jsArguments。

  5. 令 ret 为 ? Call(func, undefined, jsArguments)。

  6. 令 resultsSize 为 results 的大小。

  7. 若 resultsSize 为 0，返回 « »。

  8. 否则，若 resultsSize 为 1，返回 « ? ToWebAssemblyValue(ret, results[0]) »。

  9. 否则，

    1. 令 method 为 ? GetMethod(ret, %Symbol.iterator%)。

    2. 若 method 为 undefined，抛出 TypeError。

    3. 令 values 为 ? IteratorToList(? GetIteratorFromMethod(ret, method))。

    4. 令 wasmValues 为一个新的空列表。

    5. 若 values 的大小 不等于 resultsSize，抛出 TypeError 异常。

    6. 对 values 与 results 中的每个 value 与 resultType，按顺序配对，

      1. 将 ToWebAssemblyValue(value, resultType) 追加到 wasmValues。

    7. 返回 wasmValues。

要从 JavaScript 对象 func 与类型 functype 创建一个宿主函数，执行以下步骤：

  1. 断言：IsCallable(func)。

  2. 令 stored settings 为在任设置对象（incumbent settings object）。

  3. 令 hostfunc 为一个宿主函数，当以参数 arguments 被调用时，执行以下步骤：

    1. 令 realm 为 func 的关联 Realm。

    2. 令 relevant settings 为 realm 的设置对象。

    3. 准备运行脚本，使用 relevant settings。

    4. 准备运行回调，使用 stored settings。

    5. 令 result 为从 func、functype 与 arguments 运行宿主函数 的结果。

    6. 回调运行后的清理，使用 stored settings。

    7. 脚本运行后的清理，使用 relevant settings。

    8. 断言：result.[[Type]] 为 throw 或 normal。

    9. 令 store 为 外围 agent 的 关联 store。

    10. 若 result.[[Type]] 为 throw，则：

      1. 令 v 为 result.[[Value]]。

      2. 若 v 实现了 Exception，

        1. 令 address 为 v.[[Address]]。

      3. 否则，

        1. 令 type 为 获取 JavaScript 异常标签 的结果。

        2. 令 payload 为 ! ToWebAssemblyValue(v, externref)。

        3. 令 (store, address) 为 exn_alloc(store, type, « payload »)。

        4. 将 外围 agent 的 关联 store 设为 store。

      4. 执行 WebAssembly 指令（ref.exn address）（throw_ref）。

    11. 否则，返回 result.[[Value]]。

  4. 令 (store, funcaddr) 为 func_alloc(store, functype, hostfunc)。

  5. 将 外围 agent 的 关联 store 设为 store。

  6. 返回 funcaddr。

算法 ToJSValue(w) 通过以下步骤将WebAssembly 值强制转换为 JavaScript 值：

  1. 断言：w 的形式不是 v128.const v128。

  2. 断言：w 的形式不是 ref.exn exnaddr。

  3. 若 w 形式为 i64.const u64，

    1. 令 i64 为 signed_64(u64)。

    2. 返回 ℤ(作为数学值解读的 i64)。

  4. 若 w 形式为 i32.const u32，

    1. 令 i32 为 signed_32(u32)。

    2. 返回 𝔽(作为数学值解读的 i32)。

  5. 若 w 形式为 f32.const f32，

    1. 若 f32 为 +∞ 或 −∞，分别返回 +∞𝔽 或 -∞𝔽。

    2. 若 f32 为 nan，返回 NaN。

    3. 返回 𝔽(作为数学值解读的 f32)。

  6. 若 w 形式为 f64.const f64，

    1. 若 f64 为 +∞ 或 −∞，分别返回 +∞𝔽 或 -∞𝔽。

    2. 若 f64 为 nan，返回 NaN。

    3. 返回 𝔽(作为数学值解读的 f64)。

  7. 若 w 形式为 ref.null t，返回 null。

  8. 若 w 形式为 ref.i31 u31，

    1. 令 i31 为 signed_31(u31)。

    2. 返回 𝔽(i31)。

  9. 若 w 形式为 ref.struct structaddr，返回从 structaddr 与 "struct" 创建新的导出 GC 对象 的结果。

  10. 若 w 形式为 ref.array arrayaddr，返回从 arrayaddr 与 "array" 创建新的导出 GC 对象 的结果。

  11. 若 w 形式为 ref.func funcaddr，返回从 funcaddr 创建新的导出函数 的结果。

  12. 若 w 形式为 ref.host hostaddr，返回从 hostaddr 检索宿主值 的结果。

  13. 若 w 形式为 ref.extern ref，返回 ToJSValue(ref)。

注： 值为 NaN 的 Number 可能具有不同的可观察 NaN 载荷；详见 NumericToRawBytes。

对于从宿主地址 hostaddr 检索宿主值，执行以下步骤：

  1. 令 map 为 外围 agent 关联的 宿主值缓存。

  2. 断言：map[hostaddr] 存在。

  3. 返回 map[hostaddr]。

算法 ToWebAssemblyValue(v, type) 通过以下步骤将 JavaScript 值转换为WebAssembly 值：

  1. 断言：type 不是 v128。

  2. 断言：type 不匹配 exnref。

  3. 若 type 为 i64，

    1. 令 i64 为 ? ToBigInt64(v)。

    2. 令 u64 为满足 i64 是 signed_64(u64) 的无符号整数。

    3. 返回 i64.const u64。

  4. 若 type 为 i32，

    1. 令 i32 为 ? ToInt32(v)。

    2. 令 u32 为满足 i32 是 signed_32(u32) 的无符号整数。

    3. 返回 i32.const u32。

  5. 若 type 为 f32，

    1. 令 number 为 ? ToNumber(v)。

    2. 若 number 为 NaN，

      1. 令 n 为实现定义的整数，满足 canon32 ≤ n < 2signif(32)。

      2. 令 f32 为 nan(n)。

    3. 否则，

      1. 令 f32 为将 number 按 IEEE 754-2019 最近偶数舍入规则四舍五入至可表示值所得的结果。[IEEE-754]

    4. 返回 f32.const f32。

  6. 若 type 为 f64，

    1. 令 number 为 ? ToNumber(v)。

    2. 若 number 为 NaN，

      1. 令 n 为实现定义的整数，满足 canon64 ≤ n < 2signif(64)。

      2. 令 f64 为 nan(n)。

    3. 否则，

      1. 令 f64 为 number。

    4. 返回 f64.const f64。

  7. 若 type 形式为 ref null heaptype，

    1. 若 v 为 null，

      1. 令 r 为 ref.null heaptype。

    2. 否则若 match_valtype(type, ref null extern)，

      1. 令 ref 为 ToWebAssemblyValue(v, ref any)。

      2. 令 r 为 ref.extern ref。

    3. 否则若 v 是一个 导出函数（Exported Function） 且 match_valtype(type, ref null func)，

      1. 令 funcaddr 为 v 的 [[FunctionAddress]] 内部槽的值。

      2. 令 r 为 ref.func funcaddr。

    4. 否则若 v 是 Number 且 v 等于 ? ToInt32(v) 且 ℝ(v) < 230 且 ℝ(v) ⩾ -230，

      1. 令 i31 为 ? ToInt32(v)。

      2. 令 u31 为满足 i31 是 signed_31(i31) 的无符号整数。

      3. 令 r 为 ref.i31 u31。

    5. 否则若 v 是 导出 GC 对象（Exported GC Object），

      1. 令 objectaddr 为 v 的 [[ObjectAddress]] 内部槽。

      2. 令 objectkind 为 v 的 [[ObjectKind]] 内部槽。

      3. 若 objectkind 为 "array"，

        1. 令 r 为 ref.array objectaddr。

      4. 若 objectkind 为 "struct"，

        1. 令 r 为 ref.struct objectaddr。

    6. 否则，

      1. 令 map 为 外围 agent 关联的 宿主值缓存。

      2. 若存在宿主地址 hostaddr，使得 map[hostaddr] 与 v 相同，

        1. 返回 ref.host hostaddr。

      3. 令宿主地址 hostaddr 为最小的地址，使得 map[hostaddr] 不存在。

      4. 设置 map[hostaddr] 为 v。

      5. 令 r 为 ref.host hostaddr。

    7. 令 store 为当前 agent 的 关联 store。

    8. 令 actualtype 为 ref_type(store, r)。

    9. 若 match_valtype(actualtype, type) 为 false，

      1. 抛出 TypeError。

    10. 返回 r。

  8. 断言：不会到达此步骤。

算法 AddressValueToU64(v, addrtype) 将 JavaScript 值转换为用于嵌入操作的 WebAssembly u64。它设计为对 AddressType "i32" 的行为与 [EnforceRange] 的 unsigned long 相同，并将这些语义扩展到 AddressType "i64"，执行以下步骤：

  1. 若 addrtype 为 "i32"，

    1. 令 n 为 ? ConvertToInt(v, 32, "unsigned") 的结果，其中目标类型关联 [EnforceRange]。

      注： 这等价于 JS 转换规则 中的 [EnforceRange] unsigned long。

    2. 返回 ℝ(n) 作为 WebAssembly u64。

  2. 若 addrtype 为 "i64"，

    1. 令 n 为 ? ToBigInt(v)。

    2. 若 n < 0 或 n > 264 − 1，抛出 TypeError。

      注： 此操作旨在模拟 [EnforceRange]。

    3. 返回 ℝ(n) 作为 WebAssembly u64。

  3. 断言：不会到达此步骤。

算法 U64ToAddressValue(v, addrtype) 将嵌入操作中的 u64 值转换为对应的 AddressValue 变体（由 AddressType 指定），执行以下步骤：

  1. 若 addrtype 为 "i32"，返回 𝔽(作为数学值解读的 v)。

  2. 否则若 addrtype 为 "i64"，返回 ℤ(作为数学值解读的 v)。

  3. 断言：不会到达此步骤。

5.8. 垃圾回收对象

WebAssembly 的 struct 或 array 在 JavaScript 中以 导出 GC 对象（Exported GC Object） 的形式提供。 导出 GC 对象 是一种封装了垃圾回收的 WebAssembly 引用值的异质对象。 对 导出 GC 对象 的大多数 JavaScript 操作将抛出异常或返回 undefined。

注： 这些操作未来可能进一步细化，以允许在 JavaScript 中与 WebAssembly 的 struct 与 array 进行更丰富的交互。

导出 GC 对象 包含 [[ObjectAddress]] 内部槽，其中保存相对于外围 agent的关联 store的object address， 以及 [[ObjectKind]] 内部槽，保存字符串 "struct" 或 "array"。

导出 GC 对象 的内部方法如下实现。

导出 GC 对象的 [[GetPrototypeOf]] 内部方法 O 不接受参数并返回 null。被调用时执行以下步骤：

  1. 返回 null。

导出 GC 对象的 [[SetPrototypeOf]] 内部方法 O 接受参数 V（一个对象或 null）并返回布尔值。被调用时执行以下步骤：

  1. 返回 false。

导出 GC 对象的 [[IsExtensible]] 内部方法 O 不接受参数并返回布尔值。被调用时执行以下步骤：

  1. 返回 false。

导出 GC 对象的 [[PreventExtensions]] 内部方法 O 不接受参数并返回布尔值。被调用时执行以下步骤：

  1. 返回 false。

导出 GC 对象的 [[GetOwnProperty]] 内部方法 O 接受参数 P（属性键）并返回 undefined。被调用时执行以下步骤：

  1. 返回 undefined。

导出 GC 对象的 [[DefineOwnProperty]] 内部方法 O 接受参数 P（属性键）与 Desc（属性描述符），返回布尔值。被调用时执行以下步骤：

  1. 返回 false。

导出 GC 对象的 [[HasProperty]] 内部方法 O 接受参数 P（属性键）并返回布尔值。被调用时执行以下步骤：

  1. 返回 false。

导出 GC 对象的 [[Get]] 内部方法 O 接受参数 P（属性键）与 Receiver（ECMAScript 语言值）并返回 undefined。被调用时执行以下步骤：

  1. 返回 undefined。

导出 GC 对象的 [[Set]] 内部方法 O 接受参数 P（属性键）、V（ECMAScript 语言值）与 Receiver（ECMAScript 语言值）并抛出异常。被调用时执行以下步骤：

  1. 抛出 TypeError。

导出 GC 对象的 [[Delete]] 内部方法 O 接受参数 P（属性键）并抛出异常。被调用时执行以下步骤：

  1. 抛出 TypeError。

导出 GC 对象的 [[OwnPropertyKeys]] 内部方法 O 不接受参数并返回列表。被调用时执行以下步骤：

  1. 令 keys 为一个新的空列表。

  2. 返回 keys。

要从 WebAssembly object address objectaddr 与字符串 objectkind 创建新的导出 GC 对象，执行以下步骤：

  1. 断言：objectkind 为 "array" 或 "struct"。

  2. 令 map 为 外围 agent 关联的 导出 GC 对象缓存。

  3. 若 map[objectaddr] 存在，

    1. 返回 map[objectaddr]。

  4. 令 object 为 MakeBasicObject(« [[ObjectAddress]] »)。

  5. 将 object.[[ObjectAddress]] 设为 objectaddr。

  6. 将 object.[[ObjectKind]] 设为 objectkind。

  7. 按 导出 GC 对象的 [[GetPrototypeOf]] 内部方法 指定设置 object.[[GetPrototypeOf]]。

  8. 按 导出 GC 对象的 [[SetPrototypeOf]] 内部方法 指定设置 object.[[SetPrototypeOf]]。

  9. 按 导出 GC 对象的 [[IsExtensible]] 内部方法 指定设置 object.[[IsExtensible]]。

  10. 按 导出 GC 对象的 [[PreventExtensions]] 内部方法 指定设置 object.[[PreventExtensions]]。

  11. 按 导出 GC 对象的 [[GetOwnProperty]] 内部方法 指定设置 object.[[GetOwnProperty]]。

  12. 按 导出 GC 对象的 [[DefineOwnProperty]] 内部方法 指定设置 object.[[DefineOwnProperty]]。

  13. 按 导出 GC 对象的 [[HasProperty]] 内部方法 指定设置 object.[[HasProperty]]。

  14. 按 导出 GC 对象的 [[Get]] 内部方法 指定设置 object.[[Get]]。

  15. 按 导出 GC 对象的 [[Set]] 内部方法 指定设置 object.[[Set]]。

  16. 按 导出 GC 对象的 [[Delete]] 内部方法 指定设置 object.[[Delete]]。

  17. 按 导出 GC 对象的 [[OwnPropertyKeys]] 内部方法 指定设置 object.[[OwnPropertyKeys]]。

  18. 设置 map[objectaddr] 为 object。

  19. 返回 object。

dictionary ExceptionOptions {
  boolean traceStack = false;
};

[LegacyNamespace=WebAssembly, Exposed=(Window,Worker,Worklet)]
interface Exception {
  constructor(Tag exceptionTag, sequence<any> payload, optional ExceptionOptions options = {});
  any getArg([EnforceRange] unsigned long index);
  boolean is(Tag exceptionTag);
  readonly attribute (DOMString or undefined) stack;
};

Exception 值表示一个异常。

要从 Exception address exnAddress 初始化一个 Exception 对象 exn，执行以下步骤：

  1. 令 map 为 外围 agent 关联的 Exception 对象缓存。

  2. 断言：map[exnAddress] 不存在。

  3. 将 exn.[[Address]] 设为 exnAddress。

  4. 设置 map[exnAddress] 为 exn。

  5. 令 store 为 外围 agent 的 关联 store。

  6. 令 tagaddr 为 exn_tag(store, exnAddress)。

  7. 令 payload 为 exn_read(store, exnAddress)。

  8. 将 exn.[[Type]] 设为 tagaddr。

  9. 将 exn.[[Payload]] 设为 payload。

  10. 将 exn.[[Stack]] 设为 undefined。

要从 exception address exnAddress 创建一个 Exception 对象，执行以下步骤：

  1. 令 map 为 外围 agent 关联的 Exception 对象缓存。

  2. 若 map[exnAddress] 存在，

    1. 返回 map[exnAddress]。

  3. 令 exn 为一个 new 的 Exception。

  4. 初始化 exn，来自 exnAddress。

  5. 返回 exn。

new Exception(exceptionTag, payload, options) 构造函数步骤为：

  1. 令 JSTagAddr 为 获取 JavaScript 异常标签 的结果。

  2. 若 exceptionTag.[[Address]] 等于 JSTagAddr，

    1. 抛出 TypeError。

  3. 令 store 为 外围 agent 的 关联 store。

  4. 令 [types] → [] 为 tag_type(store, exceptionTag.[[Address]])。

  5. 若 types 的大小 不等于 payload 的大小，

    1. 抛出 TypeError。

  6. 令 wasmPayload 为 « »。

  7. 对于每个 value 与 resultType（来自 payload 与 types，按顺序配对），

    1. 若 resultType 匹配 v128 或 exnref，

      1. 抛出 TypeError。

    2. 将 ? ToWebAssemblyValue(value, resultType) 追加到 wasmPayload。

  8. 令 (store, exceptionAddr) 为 exn_alloc(store, exceptionTag.[[Address]], wasmPayload)

  9. 将 外围 agent 的 关联 store 设为 store。

  10. 初始化 this，来自 exceptionAddr。

  11. 若 options["traceStack"] 为 true，

    1. 将 this.[[Stack]] 设为当前调用栈的 DOMString 表示，或设为 undefined。

getArg(index) 方法的步骤为：

  1. 令 store 为 外围 agent 的 关联 store。

  2. 令 tagaddr 为 exn_tag(store, this.[[Address]])。

  3. 令 payload 为 exn_read(store, this.[[Address]])。

  4. 断言：tagaddr 等于 this.[[Type]]。

  5. 若 index ≥ payload 的大小，

    1. 抛出 RangeError。

  6. 令 [types] → [] 为 tag_type(store, tagaddr)。

  7. 若 types[index] 匹配 v128 或 exnref，

    1. 抛出 TypeError。

  8. 返回 ToJSValue(payload[index])。

is(exceptionTag) 方法的步骤为：

  1. 若 this.[[Type]] 不等于 exceptionTag.[[Address]]，

    1. 返回 false。

  2. 返回 true。

stack 的 getter 步骤为：

  1. 返回 this.[[Stack]]。

  2. 令 store 为 外围 agent 的 关联 store。

  3. 令 (store, tagAddress) 为 tag_alloc(store, « externref » → « »)。

  5. 将当前 agent 关联的 JavaScript 异常标签 设为 tagAddress。

6. 内建函数

JS-API 定义了一组可在编译模块时通过 options 导入的内建函数集合。WebAssembly 的内建函数反映现有的 JavaScript 内建函数，但将其适配为可以直接作为 WebAssembly 函数使用，且开销最小。

所有内建函数被分组为若干集合。每个内建集合都有一个在 WebAssemblyCompileOptions 中使用的 name，以及一个带有 wasm: 前缀、在导入查找期间使用的 qualified name。

要获取某个内建集合的内建函数，给定 builtinSetName，执行以下步骤：

  1. 返回该名称为 builtinSetName 的集合（在本节中定义）的 (name, funcType, steps) 列表。

要用 import 与已启用的内建集合 builtinSetNames 查找内建函数，执行以下步骤：

  1. 断言：验证内建集合名称 builtinSetNames 的结果为 true。

  2. 令 importModuleName 为 import[0]。

  3. 令 importName 为 import[1]。

  4. 对于每个 builtinSetName 属于 builtinSetNames，

    1. 若 builtinSetName 不指向一个内建集合，则继续。

    2. 令 builtinSetQualifiedName 为在 builtinSetName 前加前缀 "wasm:" 的结果。

    3. 若 importModuleName 等于 builtinSetQualifiedName，

      1. 令 builtins 为 获取某个内建集合的内建函数 builtinSetName 的结果。

      2. 对于每个 builtin 属于 builtins，

        1. 令 builtinName 为 builtin[0]。

        2. 若 importName 等于 builtinName，返回 (builtinSetName, builtin)。

  5. 返回 null。

要对 builtinSetNames 验证内建集合名称，执行以下步骤：

  1. 若 builtinSetNames 包含任何重复项，返回 false。

  2. 返回 true。

要从类型 funcType 和执行步骤 steps 创建一个内建函数，执行以下步骤：

  1. 令 hostfunc 为一个 宿主函数，当被调用时执行 steps。

  2. 令 (store, funcaddr) 为 func_alloc(store, functype, hostfunc)。

  3. 将 外围 agent 的 关联 store 设为 store。

  4. 返回 funcaddr。

要用名称 builtinSetName 实例化一个内建集合，执行以下步骤：

  1. 令 builtins 为 获取某个内建集合的内建函数 builtinSetName 的结果。

  2. 令 exportsObject 为 ! OrdinaryObjectCreate(null)。

  3. 对于每个 (name, funcType, steps) 属于 builtins，

    1. 令 funcaddr 为用 funcType 与 steps 创建一个内建函数 的结果。

    2. 令 func 为从 funcaddr 创建 新的导出函数（Exported Function） 的结果。

    3. 令 value 为 func。

    4. 令 status 为 ! CreateDataProperty(exportsObject, name, value)。

    5. 断言：status 为 true。

  4. 返回 exportsObject。

要用 import 与已启用的内建集合 builtinSetNames 验证内建导入，执行以下步骤：

  1. 断言：验证内建集合名称 builtinSetNames 的结果为 true。

  2. 令 maybeBuiltin 为针对 import 与 builtinSetNames 查找内建函数 的结果。

  3. 若 maybeBuiltin 为 null，返回 true。

  4. 令 importExternType 为 import[2]。

  5. 令 builtinFuncType 为 maybeBuiltin[0][1]。

  6. 令 builtinExternType 为 func |builtinFuncType|。

  7. 返回 match_externtype(builtinExternType, importExternType)。

7. 错误条件到 JavaScript 的映射

运行 WebAssembly 程序时会遇到某些事件，导致 WebAssembly 代码停止执行。WebAssembly 代码（当前） 无法捕获这些条件，因此异常将必然传播到外层的非 WebAssembly 调用方（无论是浏览器、 JavaScript 还是其他运行时系统），并在其中像普通的 JavaScript 异常一样被处理。

如果 WebAssembly 通过 import 调用 JavaScript，而 JavaScript 抛出了异常，则该异常会通过 WebAssembly 的激活帧传播到 外层调用方。

由于 JavaScript 异常是可处理的，且在 trap 被处理后 JavaScript 仍可继续调用 WebAssembly 的导出函数，一般而言， trap 并不会阻止后续执行。

7.1. 栈溢出

每当 WebAssembly 代码发生栈溢出时，将抛出与 JavaScript 栈溢出相同类别的异常。具体抛出的异常在两种情况下均为实现定义。

注： ECMAScript 并未规定栈溢出的具体行为；已有实现会抛出 RangeError、 InternalError 或 Error。以上任一行为在此均视为有效。

7.2. 内存不足

当验证、编译或实例化出现内存不足时，将抛出与 JavaScript 中内存不足情况相同类别的异常。 具体抛出的异常在两种情况下均为实现定义。

注： ECMAScript 并未规定内存不足的具体行为；已有实现会抛出 OOMError 或直接崩溃。两者在此均视为有效。

大型表或内存的分配失败可能导致

  • RangeError， 如在 Memory 的 grow() 和 Table 的 grow() 操作中所规定的那样

  • 返回 -1，正如 memory.grow 指令所规定的那样

  • 如本节所述的特定 UA 的 OOM 行为。

在未来的修订中，我们可能会重新考虑针对大内存分配的更可靠、可恢复的错误处理。

更多讨论参见 Issue 879。

8. 实现定义的限制

WebAssembly 核心规范允许实现对模块的句法结构设定限制。 虽然每个 WebAssembly 的嵌入环境可以选择定义自身的限制，但为增强可预测性，本文档所述的标准 WebAssembly JavaScript 接口 定义了以下精确限制。实现必须在模块超过以下任一限制时，抛出 CompileError 来拒绝该模块。 实际上，实现也可能在低于这些限制的有效模块上耗尽资源。

  • 模块的最大大小为 1,073,741,824 字节（1 GiB）。
  • types 段中定义的类型数量上限为 1,000,000。
  • types 段中定义的递归组数量上限为 1,000,000。
  • 一个递归组中定义的类型数量上限为 1,000,000。
  • 已定义子类型层级的最大深度为 63（无父类型定义的类型深度为 0）。
  • 一个模块中定义的函数数量上限为 1,000,000。
  • 一个模块中声明的导入数量上限为 1,000,000。
  • 一个模块中声明的导出数量上限为 1,000,000。
  • 一个模块中定义的全局变量数量上限为 1,000,000。
  • 一个模块中定义的标签数量上限为 1,000,000。
  • 定义的数据段数量上限为 100,000。
  • 表的最大数量（含声明或导入的表）为 100,000。
  • 表的最大大小为 10,000,000。
  • 任意表初始化中的表项数量上限为 10,000,000。
  • 内存的最大数量（含定义与导入的内存）为 100。
  • 32 位内存的 min 或 max 字段上限为 65,536 页（4 GiB）。
  • 64 位内存的 min 或 max 字段上限为 2^37-1 页（2^53 - 2^16 字节）。
  • 任一函数或块的参数数量上限为 1,000。
  • 任一函数或块的返回值数量上限为 1,000。
  • 函数体（含局部变量声明）的最大大小为 7,654,321 字节。
  • 函数中声明的局部变量数量上限（含作为参数隐式声明者）为 50,000。
  • 一个 struct 的字段数量上限为 10,000。
  • array.new_fixed 的操作数数量上限为 10,000。

在运行时若超过以下任一限制，实现必须抛出 RuntimeError： 实际上，实现也可能在低于这些限制的有效模块上耗尽资源。

  • 表的最大大小为 10,000,000。
  • 32 位内存的最大大小为 65,536 页（4 GiB）。
  • 64 位内存的最大大小为 262,144 页（16 GiB）。

一致性

文档约定

一致性要求通过描述性断言与 RFC 2119 术语的组合来表达。 规范性部分中的关键词 “MUST”、“MUST NOT”、“REQUIRED”、“SHALL”、“SHALL NOT”、“SHOULD”、“SHOULD NOT”、“RECOMMENDED”、“MAY” 与 “OPTIONAL” 的解释参见 RFC 2119。 然而，为了可读性，本规范中并未将这些词全部使用大写形式。

除明确标注为非规范性的章节、示例与注释外，本规范的所有文本均为规范性内容。[RFC2119]

一致性算法

在算法中以祈使语气表述的要求（例如 “strip any leading space characters” 或 “return false and abort these steps”）应结合引介该算法时所用关键词（“must”、“should”、“may” 等）的含义进行理解。

以算法或具体步骤表述的一致性要求可以以任何方式实现， 只要最终结果等效即可。 特别地，本规范中定义的算法旨在易于理解，并非旨在追求性能。 鼓励实现者进行优化。
