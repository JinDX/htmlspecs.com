摘要

Web内容可访问性指南（WCAG）2.2涵盖了一系列使Web内容更易于访问的建议。遵循这些指南将使内容对更广泛的残疾人群体更具可访问性，包括对失明和低视力、失聪和听力丧失、运动受限、语言障碍、光敏感性以及这些组合的适应，以及对学习障碍和认知限制的某些适应；但不会解决所有这些残疾用户的需求。这些指南涉及桌面、笔记本电脑、平板电脑和移动设备上的Web内容可访问性。遵循这些指南通常还会使Web内容对普通用户更易于使用。

WCAG 2.2成功标准被写成可测试的声明，不特定于某种技术。关于满足特定技术中成功标准的指导，以及关于解释成功标准的一般信息，提供在单独的文档中。请参见Web内容可访问性指南（WCAG）概述以获取介绍和指向WCAG技术和教育材料的链接。

WCAG 2.2扩展了Web内容可访问性指南2.1 [WCAG21]，该指南于2018年6月发布为 W3C推荐。符合WCAG 2.2的内容也符合WCAG 2.0和WCAG 2.1。工作组意图为要求符合WCAG 2.0或WCAG 2.1的政策提供一个替代符合方式。发布WCAG 2.2并不意味着废止或取代WCAG 2.0或WCAG 2.1。虽然WCAG 2.0和WCAG 2.1仍然是W3C推荐，但W3C建议在可访问性工作中最大化未来适用性时使用WCAG 2.2。 W3C还鼓励在制定或更新Web可访问性政策时使用最新版本的WCAG。

简介

本节为非规范性内容。

WCAG 2 背景

《Web内容无障碍指南》（WCAG）2.2 定义了如何使Web内容对残障人士更为无障碍。无障碍涉及广泛的残疾类型，包括视觉、听觉、肢体、言语、认知、语言、学习和神经方面的残疾。虽然这些指南涵盖了广泛的问题，但它们无法满足所有类型、程度和组合的残疾人士的需求。这些指南还使得Web内容对由于年龄增长而能力改变的老年人更易用，且通常提高了普通用户的可用性。

WCAG 2.2 是通过W3C流程，与世界各地的个人和组织合作开发的，其目标是为国际上的个人、组织和政府提供一个共享的Web内容无障碍标准。WCAG 2.2 基于WCAG 2.0 [WCAG20] 和WCAG 2.1 [WCAG21]，这些版本又基于WCAG 1.0 [WAI-WEBCONTENT]，旨在广泛适用于现今和未来的不同Web技术，且能够通过自动化测试和人工评估相结合的方式进行测试。有关WCAG的简介，请参阅Web内容无障碍指南（WCAG）概述。

在定义更多针对认知、语言和学习残疾的标准时，遇到了显著的挑战，包括开发时间较短以及在可测试性、可实施性和国际考量方面达成共识的困难。未来的WCAG版本将在这一领域继续努力。我们鼓励作者参阅我们关于改善残疾人包容性的补充指导，包括学习和认知残疾、低视力人士等。

Web的无障碍性不仅取决于无障碍的内容，还依赖于无障碍的Web浏览器和其他用户代理工具。创作工具在Web无障碍中也扮演着重要角色。有关这些Web开发和交互组件如何协同工作的概述，请参阅：

  • Web无障碍的基本组件
  • 用户代理无障碍指南（UAAG）概述
  • 创作工具无障碍指南（ATAG）概述

本文件提及的WCAG 2旨在指任何以2开头的WCAG版本。

WCAG 2 指导层次

使用WCAG的个人和组织各不相同，包括Web设计师、开发者、政策制定者、采购代理、教师和学生。为了满足这一受众的不同需求，提供了几个层次的指导，包括整体的原则、一般的指南、可测试的成功标准以及丰富的充分技术、建议技术和常见失败案例，配有示例、资源链接和代码。

  • 原则 - 顶层是四个为Web无障碍性提供基础的原则：可感知、可操作、可理解和健壮。另见理解无障碍性的四大原则。

  • 指南 - 在原则之下是指南。这13条指南提供了作者应努力实现的基本目标，以便让不同残障用户能够更容易地访问内容。这些指南不可测试，但它们提供了框架和总体目标，帮助作者理解成功标准并更好地实施技术。

  • 成功标准 - 每条指南都提供了可测试的成功标准，使WCAG 2.2能够用于需要需求和符合性测试的场合，如设计规范、采购、法规和合同协议。为了满足不同群体和不同情况的需要，定义了三个符合性级别：A（最低）、AA和AAA（最高）。更多关于WCAG级别的信息，请参阅理解符合性级别。

  • 充分和建议技术 - 在WCAG 2.2 文档中的每一条指南和成功标准中，工作组还记录了各种各样的技术。这些技术是说明性的，分为两类：那些足够满足成功标准的技术和那些建议技术。建议技术超出了各自成功标准的要求，允许作者更好地实现指南。一些建议技术解决了不可通过测试的成功标准所涵盖的无障碍障碍。已知的常见失败也有记录。另见理解WCAG 2.2中的充分和建议技术。

所有这些层次的指导（原则、指南、成功标准、充分和建议技术）共同作用，为如何使内容更加无障碍提供指导。鼓励作者查看并尽可能应用所有层次的内容，包括建议技术，以便尽可能广泛地满足用户的需求。

请注意，即使是最高级别（AAA）符合性内容，也无法为所有类型、程度或组合的残疾人士提供无障碍，特别是在认知、语言和学习领域。鼓励作者考虑所有技术，包括建议技术，使内容对认知和学习残疾人士更可用，并寻求当前最佳实践的相关建议，以确保Web内容尽可能对该群体无障碍。元数据可能会帮助用户找到最适合其需求的内容。

WCAG 2.2 支持文档

WCAG 2.2 文档旨在满足那些需要稳定的、可引用技术标准的需求。基于WCAG 2.2文档的其他文件，称为支持文档，针对其他重要目的，包括描述如何应用WCAG于新技术的能力，这些文件可以进行更新。支持文档包括：

  1. 如何符合WCAG 2.2 - 一份可定制的WCAG 2.2快速参考，包含所有指南、成功标准和供作者在开发和评估Web内容时使用的技术。这包括WCAG 2.0、2.1、2.2的内容，并可以通过多种方式过滤，帮助作者专注于相关内容。

  2. 理解WCAG 2.2 - 一份理解和实施WCAG 2.2的指南。每条WCAG 2.2中的指南和成功标准以及关键主题都有一个简短的“理解”文档。

  3. WCAG 2.2 技术 - 技术和常见失败案例的集合，每个案例包含描述、示例、代码和测试。

  4. WCAG 2 文件 - 对WCAG 2支持文档和补充指南的简要介绍。

  5. WCAG 2.2 中的新增内容 介绍了新的成功标准，并通过人物引用展示了无障碍问题。

参见Web内容无障碍指南 (WCAG) 概述，了解WCAG 2.2支持材料的描述，包括与WCAG相关的教育资源。还列出了关于Web无障碍性商业案例、规划实施以提高网站无障碍性及无障碍政策的附加资源，详见 WAI资源。

WCAG 2.2 的要求

WCAG 2.2 符合一套 WCAG 2.2 的要求，这些要求继承自之前的WCAG 2版本。要求结构化了整体的指南框架，并确保了向后兼容性。工作组还使用了一组较为非正式的验收标准，帮助确保成功标准在风格和质量上与WCAG 2.0中的标准类似。这些要求限制了WCAG 2.2中可以包含的内容，这一限制对于保持WCAG 2的小版本特性非常重要。

与WCAG 2.1的比较

WCAG 2.2 的启动目标是继续 WCAG 2.1 的工作：改善针对三大群体的无障碍指南：具有认知或学习障碍的用户、低视力用户和移动设备上的残疾用户。许多满足这些需求的方法被提议并进行了评估，工作组精炼了一些方案。继承自 WCAG 2.0 的结构要求、提案的清晰度与影响以及时间表共同促成了本版本中包含的最终成功标准。工作组认为，WCAG 2.2 逐步推进了针对所有这些领域的网页内容无障碍指导，但强调这些指南不能满足所有用户的需求。

WCAG 2.2 基于并向后兼容 WCAG 2.1，这意味着符合 WCAG 2.2 的网页至少与符合 WCAG 2.1 的页面一样无障碍。新增的要求基于 2.1 和 2.0。WCAG 2.2 移除了一个成功标准，4.1.1 解析。那些根据政策需要符合 WCAG 2.0 或 2.1 的作者可以更新内容以符合 WCAG 2.2，但可能仍需继续测试和报告 4.1.1。遵循多个版本指南的作者应注意以下新增项。

WCAG 2.2 中的新功能

WCAG 2.2 通过添加新的成功标准、支持这些标准的定义和组织新增项的指南扩展了 WCAG 2.1。这种增量方式帮助明确符合 WCAG 2.2 的站点也符合 WCAG 2.1。无障碍指南工作组建议站点采用 WCAG 2.2 作为新的符合目标，即使正式义务提到以前的版本，以提供更好的无障碍性并应对未来政策的变化。

以下成功标准是 WCAG 2.2 中新增的：

  • 2.4.11 焦点不被遮挡（最低） (AA)
  • 2.4.12 焦点不被遮挡（增强） (AAA)
  • 2.4.13 焦点外观 (AAA)
  • 2.5.7 拖动操作 (AA)
  • 2.5.8 目标大小（最小） (AA)
  • 3.2.6 一致性帮助 (A)
  • 3.3.7 冗余输入 (A)
  • 3.3.8 无障碍认证（最低） (AA)
  • 3.3.9 无障碍认证（增强） (AAA)

新的成功标准可能引用了新增的术语，这些术语也已被添加到术语表中，并构成成功标准的规范性要求的一部分。

WCAG 2.2 还引入了新部分，详细说明了可能影响 隐私 和 安全 的规范方面。

WCAG 2.2 中的编号

为了避免对那些对 WCAG 2 版本向后兼容性重要的实施者造成混淆，WCAG 2.2 中的新成功标准已被附加在其指南中的成功标准集的末尾。这避免了因为在现有成功标准之间插入新的成功标准而导致的章节编号变化，但这意味着每个指南中的成功标准不再按合规级别分组。每个指南中的成功标准的顺序并不意味着合规级别的信息；只有成功标准本身的合规级别指示符（A / AA / AAA）才表示这一点。WCAG 2.2 快速参考将提供一种按合规级别查看成功标准的方法，以及许多其他过滤和排序选项。

WCAG 2.2 的合规性

WCAG 2.2 使用与 WCAG 2.0 相同的合规模型。其目的是确保符合 WCAG 2.2 的站点也符合 WCAG 2.0 和 WCAG 2.1，这意味着它们符合引用 WCAG 2.0 或 WCAG 2.1 的政策的要求，同时更好地满足当前网络用户的需求。

后续的无障碍指南版本

与 WCAG 2.2 并行，无障碍指南工作组正在开发另一个主要版本的无障碍指南。这项工作的结果预计将是比 WCAG 2 的小版本更为实质性的网页无障碍指导重构。该工作采用以用户为中心的设计方法，进行研究，以产生最有效和灵活的结果，包括内容创作、用户代理支持和创作工具支持的角色。这是一个多年的工作，因此在完成主要版本之前，WCAG 2.2 被视为一项过渡措施，以提供反映自 WCAG 2.0 发布以来网络变化的更新版无障碍指导。工作组还可能开发其他过渡版本，继续 WCAG 2.2 的类似短时间框架，以在完成主要版本时提供额外支持。

理解悬停或聚焦时的内容 |

(AA级)

注2

5. 符合性

本节列出了符合WCAG 2.2的要求，同时提供了如何做出符合性声明（可选）的信息。最后，它描述了什么是辅助功能支持，因为只有通过辅助功能支持的技术使用方式才能被依赖来符合标准。理解符合性中包含了关于辅助功能支持概念的进一步解释。

5.1 解释规范性要求

WCAG 2.2的主要内容是规范性的，定义了影响符合性声明的要求。介绍性材料、附录、标记为“非规范性”的部分、图表、示例和注释是信息性（非规范性）的。非规范性材料提供了帮助解释指南的建议性信息，但不构成影响符合性声明的要求。

关键字MAY、MUST、MUST NOT、NOT RECOMMENDED、RECOMMENDED、SHOULD和SHOULD NOT应按[RFC2119]中所述进行解释。

5.2 符合性要求

为了使网页符合WCAG 2.2，必须满足以下所有符合性要求：

5.2.1 符合性级别

以下符合性级别之一完全达到。

  • 对于A级符合性（最低符合性级别），网页满足所有A级成功标准，或提供符合性替代版本。
  • 对于AA级符合性，网页满足所有A级和AA级成功标准，或提供AA级符合性替代版本。
  • 对于AAA级符合性，网页满足所有A级、AA级和AAA级成功标准，或提供AAA级符合性替代版本。
注释 1

尽管只能达到规定的符合性级别，但鼓励作者在声明中报告（所达到的级别之外）所有级别的成功标准进展。

注释 2

不建议将AAA级符合性作为整个网站的普遍政策要求，因为某些内容不可能满足所有AAA级成功标准。

5.2.2 完整页面

符合性（以及符合性级别）仅适用于完整的网页，不能将网页的一部分排除在外以达到符合性。

注释 1

为了确定符合性，页面内容的替代选项（例如长描述或视频的替代展示）被视为页面的一部分。

注释 2

网页的作者若因超出控制范围的内容无法实现符合性，可能会考虑发布部分符合性声明。

注释 3

完整页面包括页面根据不同屏幕尺寸自动展示的每个版本（例如响应式网页的变化）。这些版本都需要符合性（或具有符合性替代版本），整个页面才能符合标准。

5.2.3 完整流程

当一个网页属于一系列页面呈现的流程时（即完成活动所需的步骤序列），该流程中的所有页面都需要达到指定的符合性级别或更高。（如果流程中的任何页面未达到该级别或更高，则无法在该级别上实现符合性。）

示例

一个在线商店有一系列用于选择和购买产品的页面。从开始到完成（结账）的所有页面都需符合性要求，流程中的任何页面才能符合。

5.2.4 仅使用支持辅助功能的技术方式

仅使用支持辅助功能的技术方式来满足成功标准。通过不支持辅助功能的方式提供的信息或功能，也需要通过支持辅助功能的方式提供。（参见理解辅助功能支持。）

5.2.5 不干扰

如果使用了不受支持辅助功能支持的技术，或如果它们的使用方式不符合标准，则它们不能阻止用户访问页面的其他部分。此外，整个网页在以下条件下仍需符合性要求：

  1. 当用户代理中启用了不受依赖的技术时，
  2. 当用户代理中关闭了不受依赖的技术时，
  3. 当用户代理不支持不受依赖的技术时。

此外，以下成功标准适用于页面上的所有内容，包括不依赖于符合性的内容，因为不符合这些标准可能会干扰页面的使用：

  • 1.4.2 - 音频控制
  • 2.1.2 - 无键盘陷阱
  • 2.3.1 - 三次闪烁或以下阈值
  • 2.2.2 - 暂停、停止、隐藏
注释

如果页面无法符合性（例如符合性测试页面或示例页面），则不能将其包括在符合性声明的范围内。

有关更多信息及示例，请参见理解符合性要求。

5.3 符合性声明（可选）

符合性仅针对网页定义。然而，符合性声明可以涵盖一个页面、一系列页面或多个相关的网页。

5.3.1 符合性声明的必需组件

符合性声明不是必须的。作者可以在不声明的情况下符合WCAG 2.2。但如果作出符合性声明，则声明必须包括以下信息：

  1. 声明日期
  2. 指南标题、版本和URI "Web内容可访问性指南2.2" 位于 https://www.w3.org/TR/WCAG22/
  3. 符合的级别（A级、AA级或AAA级）

  4. 网页的简要描述，如列出符合性声明所涵盖的URI列表，包括是否涵盖子域名。

    注释 1

    网页可以通过列表或描述所有URIs的表达式来进行描述。

    注释 2

    在安装到客户网站之前没有URI的基于Web的产品可以声明该产品在安装后将符合性。

  5. Web内容技术依赖的列表。
注释 3

如果使用符合性标识，则这将构成声明，并且必须附有上述符合性声明的必需组件。

5.3.2 符合性声明的可选组件

除了上述符合性声明的必需组件外，还可以考虑提供其他信息以帮助用户。推荐的附加信息包括：

  • 超出声明符合级别的成功标准列表。该信息应以用户可使用的形式提供，最好是机器可读的元数据。
  • 使用但不依赖的特定技术列表。
  • 用于测试内容的用户代理和辅助技术列表。
  • 内容的特定可访问性特征列表，提供为机器可读的元数据。
  • 关于超越成功标准以增强可访问性的其他步骤的信息。
  • 依赖技术列表的机器可读元数据版本。
  • 符合性声明的机器可读元数据版本。
注释 1

有关更多信息和符合性声明示例，请参见理解符合性声明。

注释 2

有关在符合性声明中使用元数据的更多信息，请参见理解元数据。

5.4 部分符合性声明 - 第三方内容

将来会添加更多内容的网页可以使用“部分符合性声明”。例如，电子邮件程序、博客、允许用户添加评论的文章，或支持用户贡献内容的应用程序。另一个例子是由多个贡献者汇总内容组成的页面，如门户网站或新闻网站，或者随着时间推移自动插入其他来源内容的网站，例如动态插入广告的情况。

在这些情况下，最初发布时无法知道页面的非控制内容是什么。需要注意的是，非控制内容也可能影响控制内容的可访问性。提供了两种选择：

  1. 可以根据最佳认知来做出符合性的判定。如果这种类型的页面在两天内得到监控和修复（非符合性内容被移除或使其符合），则可以做出符合性判定或声明，因为除了外部贡献内容中的错误会在遇到时得到纠正或移除外，该页面是符合的。如果无法监控或修正非符合性内容，则无法做出符合性声明；

    或者

  2. 可以做出“部分符合性声明”，即该页面不符合，但如果某些部分被移除，则可能符合。该声明的形式为：“此页面不符合，但如果移除以下不受控制的来源部分，则符合WCAG 2.2的X级别。”此外，关于部分符合性声明中描述的非控制内容，还应符合以下条件：

    1. 它不属于作者控制的内容。
    2. 它以用户可以识别的方式描述（例如，不能简单描述为“我们无法控制的所有部分”，除非这些部分明确标记出来）。

5.5 部分符合性声明 - 语言

当页面不符合规定，但如果对页面使用的语言（或所有语言）有可访问性支持，则页面可以符合时，可做出“由于语言部分符合性声明”。声明的形式为：“此页面不符合，但如果存在对以下语言的可访问性支持，则会符合WCAG 2.2的X级别：”

可访问性支持

由用户的 辅助技术 以及浏览器和其他 用户代理 支持

为了使 Web 内容技术的使用（或技术特性的使用）符合可访问性支持，必须满足以下两点：

  1. 所使用的 Web 内容技术 必须被用户的辅助技术（AT）支持。 这意味着所使用的技术已在内容的 人类语言 中进行过与用户辅助技术的互操作性测试。

    同时

  2. 必须有用户可用的支持可访问性的用户代理。 这意味着以下四种情况中的至少一种为真：

    1. 该技术在广泛分发且支持可访问性的用户代理中被原生支持（例如 HTML 和 CSS）；

      或者

    2. 该技术在广泛分发且支持可访问性的插件中得到支持；

      或者

    3. 该内容在封闭的环境中提供，例如大学或公司网络，所需的用户代理也支持可访问性；

      或者

    4. 支持该技术的用户代理是可访问性支持的，并且可供下载或购买，且：

      • 残障人士不需要比没有残障的人支付更多费用 并且
      • 残障人士可以像没有残障的人一样轻松找到并获取。
注 1

相对亮度

注1

对于sRGB色彩空间，颜色的相对亮度定义为L = 0.2126 * R + 0.7152 * G + 0.0722 * B，其中R、G和B定义为：

  • 如果 RsRGB <= 0.04045，则R = RsRGB/12.92，否则R = ((RsRGB+0.055)/1.055) ^ 2.4
  • 如果 GsRGB <= 0.04045，则G = GsRGB/12.92，否则G = ((GsRGB+0.055)/1.055) ^ 2.4
  • 如果 BsRGB <= 0.04045，则B = BsRGB/12.92，否则B = ((BsRGB+0.055)/1.055) ^ 2.4

RsRGB、GsRGB 和 BsRGB 定义为：

  • RsRGB = R8bit/255
  • GsRGB = G8bit/255
  • BsRGB = B8bit/255

"^"字符是指数运算符。(公式摘自 [SRGB].)

注2

在 2021 年 5 月之前，定义中的值为 0.04045（之前为 0.03928）。这是从早期版本的规范中取得的，并已更新。它对这些指南中的计算没有实际影响。

注3

目前几乎所有用于查看网页内容的系统都假设使用 sRGB 编码。除非已知将使用其他色彩空间处理和显示内容，否则作者应使用 sRGB 色彩空间进行评估。如果使用其他色彩空间，请参阅理解成功标准 1.4.3。

注4

如果传送后发生抖动，则使用源颜色值。对于在源头发生抖动的颜色，应使用抖动的颜色的平均值（平均 R，平均 G 和平均 B）。

7. 用户界面组件的输入用途

本节列出了常见的用户界面组件的输入用途。以下术语不是必须使用的关键词，而是必须由网页采用的分类法中捕捉的用途。适用时，作者使用所选分类法标记控件以指示语义用途。这为用户代理和辅助技术提供了应用个性化呈现的潜力，使更多人能够理解和使用内容。

注

输入类型用途列表基于HTML规范的自动填充部分中定义的控件用途，但重要的是要理解，其他技术可能在其规范中定义了相同或部分相同的概念，只有映射到以下含义的概念是必需的。

以下输入控件用途旨在与内容用户相关，并仅限于与该个人相关的信息。

  • name - 全名
  • honorific-prefix - 前缀或称谓（例如，“先生”，“女士”，“博士”，“小姐”）
  • given-name - 名字（在某些西方文化中，也称为first name）
  • additional-name - 额外的名字（在某些西方文化中，也称为middle names，即除名字外的其他名字）
  • family-name - 姓氏（在某些西方文化中，也称为last name或surname）
  • honorific-suffix - 后缀（例如，“小”，“理学学士”，“MBA”）
  • nickname - 昵称、用户名、或别名：通常是用作全名替代的简短名字
  • organization-title - 职称（例如，“软件工程师”，“高级副总裁”，“副总经理”）
  • username - 用户名
  • new-password - 新密码（例如，在创建账户或更改密码时）
  • current-password - 与username字段标识的账户相关的当前密码（例如，登录时）
  • organization - 与该人、地址或联系信息相关的公司名称
  • street-address - 街道地址（多行，保留换行符）
  • address-line1 - 街道地址（每字段一行，第一行）
  • address-line2 - 街道地址（每字段一行，第二行）
  • address-line3 - 街道地址（每字段一行，第三行）
  • address-level4 - 四级行政区划中的最细一级
  • address-level3 - 三级或以上行政区划中的第三级
  • address-level2 - 两级或以上行政区划中的第二级；在拥有两级行政区划的国家中，这通常是找到相关街道地址的城市、镇、村或其他地方
  • address-level1 - 地址中最广泛的行政级别，即找到所在地区的省份；例如，在美国是州，在瑞士是州，在英国是邮镇
  • country - 国家代码
  • country-name - 国家名称
  • postal-code - 邮政编码、邮编、ZIP码、CEDEX码（如为CEDEX，请在address-level2字段后附加“CEDEX”和相关分区）
  • cc-name - 付款工具上给出的全名
  • cc-given-name - 付款工具上给出的名字（在某些西方文化中，也称为first name）
  • cc-additional-name - 付款工具上给出的额外名字（在某些西方文化中，也称为middle names，即除名字外的其他名字）
  • cc-family-name - 付款工具上给出的姓氏（在某些西方文化中，也称为last name或surname）
  • cc-number - 标识付款工具的代码（例如，信用卡号）
  • cc-exp - 付款工具的有效期
  • cc-exp-month - 付款工具的有效期月份
  • cc-exp-year - 付款工具的有效期年份
  • cc-csc - 付款工具的安全码（也称为卡安全码（CSC），卡验证代码（CVC），卡验证值（CVV），签名面板代码（SPC），信用卡ID（CCID）等）
  • cc-type - 付款工具类型
  • transaction-currency - 用户希望交易使用的货币
  • transaction-amount - 用户希望交易的金额（例如，输入竞标或销售价格时）
  • language - 首选语言
  • bday - 生日
  • bday-day - 生日的日期部分
  • bday-month - 生日的月份部分
  • bday-year - 生日的年份部分
  • sex - 性别身份（例如，女性，Fa’afafine）
  • url - 对应于公司、个人、地址或其他联系信息的主页或其他网页
  • photo - 对应于公司、个人、地址或其他联系信息的照片、图标或其他图像
  • tel - 完整的电话号码，包括国家代码
  • tel-country-code - 电话号码的国家代码部分
  • tel-national - 不含国家代码部分的电话号码，并根据适用情况应用国内前缀
  • tel-area-code - 电话号码的区号部分，并根据适用情况应用国内前缀
  • tel-local - 不含国家代码和区号部分的电话号码
  • tel-local-prefix - 电话号码在区号后面的部分的第一部分，当该部分分为两部分时
  • tel-local-suffix - 电话号码在区号后面的部分的第二部分，当该部分分为两部分时
  • tel-extension - 电话号码的内部分机代码
  • email - 电子邮件地址
  • impp - 表示即时消息协议端点的URL（例如，"aim:goim?screenname=example"或"xmpp:fred@example.net"）
