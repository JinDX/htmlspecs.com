摘要

《Web 内容可访问性指南 (WCAG) 2.2》涵盖了广泛的建议，以使 Web 内容更具可访问性。遵循这些指南将使内容对更多残障人士更具可访问性，包括针对盲人和低视力、聋人和听力损失、行动受限、语言障碍、光敏性及其组合的适应性，以及对学习障碍和认知限制的一些适应性；但不能满足所有残障人士的用户需求。这些指南针对任何类型设备上的 Web 内容可访问性（包括台式机、笔记本电脑、自助终端和移动设备）。遵循这些指南通常也会使 Web 内容对普通用户更易用。

WCAG 2.2 的成功标准被写成可测试的声明，与具体技术无关。关于在特定技术中满足成功标准的指导，以及关于解释成功标准的一般信息，均在单独的文档中提供。请参阅Web 内容可访问性指南 (WCAG) 概述以获取介绍和 WCAG 技术及教育材料的链接。

WCAG 2.2 扩展了Web 内容可访问性指南 2.1 [WCAG21]，后者于 2018 年 6 月作为 W3C 推荐标准发布。符合 WCAG 2.2 的内容也符合 WCAG 2.0 和 WCAG 2.1。工作组的意图是，对于要求符合 WCAG 2.0 或 WCAG 2.1 的政策，WCAG 2.2 可以提供一种替代的符合方式。WCAG 2.2 的发布不会废弃或取代 WCAG 2.0 或 WCAG 2.1。虽然 WCAG 2.0 和 WCAG 2.1 仍然是 W3C 推荐标准，但 W3C 建议使用 WCAG 2.2，以最大化未来可访问性工作的适用性。W3C 还鼓励在制定或更新 Web 可访问性政策时使用 WCAG 的最新版本。

WCAG 2 背景

《Web 内容可访问性指南 (WCAG) 2.2》定义了如何使 Web 内容对残障人士更具可访问性。可访问性涉及广泛的残障类型，包括视觉、听觉、身体、语言、认知、语言、学习和神经系统障碍。虽然这些指南涵盖了广泛的问题，但无法满足所有类型、程度和组合的残障人士的需求。这些指南还使 Web 内容对因年龄变化而能力改变的老年人更易用，并且通常会提高普通用户的可用性。

WCAG 2.2 通过 W3C 流程与全球个人和组织合作开发，旨在提供一个共享的 Web 内容可访问性标准，以满足个人、组织和政府的国际需求。WCAG 2.2 基于 WCAG 2.0 [WCAG20] 和 WCAG 2.1 [WCAG21]，而它们又基于 WCAG 1.0 [WAI-WebCONTENT]，旨在广泛适用于当前和未来的不同 Web 技术，并通过自动化测试和人工评估的结合进行测试。有关 WCAG 的介绍，请参阅 Web 内容可访问性指南 (WCAG) 概述。

在定义额外标准以解决认知、语言和学习障碍时遇到了重大挑战，包括开发时间短以及在测试性、可实施性和提案的国际考虑方面达成共识的困难。未来版本的 WCAG 将继续在这一领域开展工作。我们鼓励作者参考我们关于改善对残障人士（包括学习和认知障碍人士、低视力人士等）的包容性的补充指导。

Web 可访问性不仅依赖于可访问的内容，还依赖于可访问的 Web 浏览器和其他用户代理。创作工具在 Web 可访问性中也起着重要作用。有关这些 Web 开发和交互组件如何协同工作的概述，请参阅：

  • Web 可访问性的基本组件
  • 用户代理可访问性指南 (UAAG) 概述
  • 创作工具可访问性指南 (ATAG) 概述

本文件中提到的 WCAG 2 意指以 2 开头的任何和所有版本的 WCAG。

WCAG 2 指导层次

使用 WCAG 的个人和组织范围广泛，包括 Web 设计师和开发人员、政策制定者、采购代理、教师和学生。为了满足这一受众的不同需求，提供了多个层次的指导，包括总体原则、一般指南、可测试的成功标准以及丰富的充分技术、建议技术和记录的常见失败，并附有示例、资源链接和代码。

  • 原则 - 最顶层是四个原则，为 Web 可访问性提供基础：可感知、可操作、可理解和健壮。另请参阅 理解可访问性的四个原则。

  • 指南 - 在原则之下是指南。这 13 条指南提供了作者应努力实现的基本目标，以使内容对不同残障用户更具可访问性。这些指南不可测试，但提供了框架和总体目标，帮助作者理解成功标准并更好地实施技术。

  • 成功标准 - 每条指南都提供了可测试的成功标准，使 WCAG 2.2 可用于需要要求和符合性测试的场景，例如设计规范、采购、法规和合同协议。为了满足不同群体和不同情况的需求，定义了三个符合性级别：A（最低）、AA 和 AAA（最高）。有关 WCAG 级别的更多信息，请参阅 理解符合性级别。

  • 充分技术和建议技术 - 对于 WCAG 2.2 文档中的每个指南和成功标准，工作组还记录了各种技术。这些技术是信息性的，分为两类：满足成功标准的充分技术和建议技术。建议技术超出了单个成功标准的要求，使作者能够更好地解决指南。一些建议技术解决了测试成功标准未涵盖的可访问性障碍。对于已知的常见失败，也进行了记录。另请参阅 理解 WCAG 2.2 中的充分技术和建议技术。

所有这些指导层次（原则、指南、成功标准以及充分技术和建议技术）共同作用，为如何使内容更具可访问性提供指导。鼓励作者查看并应用所有能够应用的层次，包括建议技术，以最好地满足尽可能广泛的用户需求。

请注意，即使是符合最高级别（AAA）的内容也无法对所有类型、程度或组合的残障人士（尤其是在认知、语言和学习领域）完全可访问。鼓励作者考虑所有技术，包括建议技术，参考 使内容对认知和学习障碍人士更易用，并寻求有关当前最佳实践的相关建议，以尽可能确保 Web 内容对这一群体可访问。元数据可能有助于用户找到最适合其需求的内容。

WCAG 2.2 支持文档

WCAG 2.2 文档旨在满足需要稳定、可引用技术标准的用户需求。其他称为支持文档的文件基于 WCAG 2.2 文档，并解决其他重要目的，包括能够更新以描述如何将 WCAG 应用于新技术。支持文档包括：

  1. 如何满足 WCAG 2.2 - 一个可定制的 WCAG 2.2 快速参考，包括所有指南、成功标准和技术，供作者在开发和评估 Web 内容时使用。这包括 WCAG 2.0、2.1 和 2.2 的内容，并可以通过多种方式过滤，以帮助作者专注于相关内容。

  2. 理解 WCAG 2.2 - 一个理解和实施 WCAG 2.2 的指南。每个 WCAG 2.2 的指南和成功标准以及关键主题都有一个简短的“理解”文档。

  3. WCAG 2.2 技术 - 一个技术和常见失败的集合，每个都在单独的文档中，包括描述、示例、代码和测试。

  4. WCAG 2 文档 - 一个简要介绍 WCAG 2 支持文档和补充指导的文档。

  5. WCAG 2.2 的新内容 - 介绍新的成功标准，并通过角色引用说明可访问性问题。

请参阅 Web 内容可访问性指南 (WCAG) 概述，了解 WCAG 2.2 支持材料的描述，包括与 WCAG 2 相关的教育资源。其他资源涵盖主题，例如 Web 可访问性的商业案例、规划实施以改善网站的可访问性以及可访问性政策，这些资源列在 WAI 资源中。

WCAG 2.2 的要求

WCAG 2.2 满足一组WCAG 2.2 的要求，这些要求继承自之前的 WCAG 2 版本的要求。要求构建了指南的整体框架，并确保向后兼容。工作组还使用了一组较为非正式的成功标准验收标准，以帮助确保成功标准在风格和质量上与 WCAG 2.0 中的标准类似。这些要求限制了 WCAG 2.2 中可以包含的内容。这种限制对于保持其作为 WCAG 2 的点版本的性质至关重要。

与WCAG 2.1的比较

WCAG 2.2 的启动目标是延续 WCAG 2.1 的工作：为三个主要群体改进可访问性指导：认知或学习障碍用户、低视力用户以及使用移动设备的残障用户。提出并评估了许多满足这些需求的方法，其中一些被工作组进一步完善。继承自 WCAG 2.0 的结构要求、提案的清晰度和影响力以及时间表共同促成了本版本中包含的最终成功标准集。工作组认为 WCAG 2.2 在所有这些领域中逐步推进了 Web 内容可访问性指导，但强调这些指南并未满足所有用户需求。

WCAG 2.2 基于 WCAG 2.1 并向后兼容，这意味着符合 WCAG 2.2 的网页至少与符合 WCAG 2.1 的网页一样可访问。新增的要求基于 2.1 和 2.0 构建。WCAG 2.2 移除了一个成功标准，4.1.1 解析。根据政策要求必须符合 WCAG 2.0 或 2.1 的作者可以将内容更新为 WCAG 2.2，但可能需要继续测试和报告 4.1.1。遵循多个版本指南的作者应注意以下新增内容。

WCAG 2.2 中的新功能

WCAG 2.2 通过添加新的成功标准、支持这些标准的定义以及组织新增内容的指南扩展了 WCAG 2.1。这种增量方法有助于明确符合 WCAG 2.2 的网站也符合 WCAG 2.1。可访问性指南工作组建议网站采用 WCAG 2.2 作为新的符合目标，即使正式义务提到的是以前的版本，以提供更好的可访问性并预期未来的政策变化。

以下成功标准是 WCAG 2.2 中的新内容：

  • 2.4.11 焦点未被遮挡（最低） (AA)
  • 2.4.12 焦点未被遮挡（增强） (AAA)
  • 2.4.13 焦点外观 (AAA)
  • 2.5.7 拖动动作 (AA)
  • 2.5.8 目标大小（最低） (AA)
  • 3.2.6 一致的帮助 (A)
  • 3.3.7 冗余输入 (A)
  • 3.3.8 可访问认证（最低） (AA)
  • 3.3.9 可访问认证（增强） (AAA)

新的成功标准可能引用已添加到术语表中的新术语，这些术语也是成功标准规范要求的一部分。

WCAG 2.2 还引入了新的章节，详细说明可能影响隐私和安全的规范方面。

WCAG 2.2 中的编号

为了避免对需要向后兼容 WCAG 2 版本的实施者造成混淆，WCAG 2.2 中的新成功标准已被附加到其指南内成功标准集的末尾。这避免了因在指南中的现有成功标准之间插入新成功标准而导致 WCAG 2 的成功标准章节编号发生变化，但这意味着每个指南中的成功标准不再按符合级别分组。每个指南内成功标准的顺序并不暗示有关符合级别的信息；只有成功标准本身的符合级别指示符（A / AA / AAA）表明这一点。WCAG 2.2 快速参考将提供一种按符合级别分组查看成功标准的方法，以及许多其他过滤和排序选项。

WCAG 2.2 的合规性

WCAG 2.2 使用与 WCAG 2.0 相同的符合性模型。其目的是使符合 WCAG 2.2 的网站也符合 WCAG 2.0 和 WCAG 2.1，这意味着它们满足引用 WCAG 2.0 或 WCAG 2.1 的任何政策的要求，同时更好地满足当前 Web 用户的需求。

后续的无障碍指南版本

与 WCAG 2.2 同时，可访问性指南工作组正在开发另一个主要版本的可访问性指南。预计这项工作的结果将是对 Web 可访问性指导进行比 WCAG 2 的点版本更为实质性的重组。该工作遵循以研究为重点、以用户为中心的设计方法，以产生最有效和灵活的成果，包括内容创作、用户代理支持和创作工具支持的角色。这是一项多年努力，因此需要 WCAG 2.2 作为临时措施，提供更新的 Web 可访问性指导，以反映自 WCAG 2.0 发布以来 Web 的变化。工作组可能还会开发额外的临时版本，继续以 WCAG 2.2 为基础，在类似的短时间内提供额外支持，同时完成主要版本。

注2

理解时间可调 |
如何满足时间可调

关闭

调整

注2

5. 符合性

本节列出了符合WCAG 2.2的要求，并提供了有关如何声明符合性的可选信息。最后，它描述了什么是可访问性支持，因为只有技术的可访问性支持使用方式才能依赖于符合性。理解符合性提供了关于可访问性支持概念的进一步解释。

5.1 解释规范性要求

WCAG 2.2 的主要内容是规范性的，定义了影响符合性声明的要求。介绍性材料、附录、标记为“非规范性”的部分、图表、示例和注释是信息性（非规范性）的。非规范性材料提供建议性信息以帮助解释指南，但不会创建影响符合性声明的要求。

关键字 MAY、MUST、MUST NOT、NOT RECOMMENDED、RECOMMENDED、SHOULD 和 SHOULD NOT 应按照 [RFC2119] 中的描述进行解释。

5.2 符合性要求

为了使网页符合WCAG 2.2，必须满足以下所有符合性要求：

5.2.1 符合性级别

以下符合性级别之一完全达到。

  • 对于 Level A 符合性（最低符合性级别），网页 满足 所有 Level A 成功标准，或者提供符合性替代版本。
  • 对于 Level AA 符合性，网页满足所有 Level A 和 Level AA 成功标准，或者提供 Level AA 符合性替代版本。
  • 对于 Level AAA 符合性，网页满足所有 Level A、Level AA 和 Level AAA 成功标准，或者提供 Level AAA 符合性替代版本。
注释 1

虽然符合性只能在规定的级别上实现，但鼓励作者在其声明中报告任何超出已实现符合性级别的所有级别成功标准的进展。

注释 2

不建议将 Level AAA 符合性作为整个网站的一般政策要求，因为某些内容无法满足所有 Level AAA 成功标准。

5.2.2 完整页面

符合性（及符合性级别）仅适用于完整的网页，如果网页的一部分被排除，则无法实现符合性。

注释 1

为了确定符合性，如果页面内容的一部分有替代方案，并且可以直接从页面获取，例如长描述或视频的替代呈现，则这些替代方案被视为页面的一部分。

注释 2

由于作者无法控制的内容导致网页无法符合要求的作者可以考虑部分符合性声明。

注释 3

完整页面包括页面为各种屏幕尺寸自动呈现的每种变体（例如响应式网页中的变体）。为了使整个页面符合要求，每种变体都需要符合要求（或需要具有符合要求的替代版本）。

5.2.3 完整流程

当网页是呈现流程（即完成某项活动所需的一系列步骤）的一部分时，流程中的所有网页都必须符合指定级别或更高。（如果流程中的任何页面未达到该级别或更高，则无法实现该级别的符合性。）

示例

一个在线商店有一系列页面用于选择和购买产品。从开始到完成（结账）的系列中的所有页面都必须符合要求，以使流程中的任何页面符合要求。

5.2.4 仅使用支持辅助功能的技术方式

只有可访问性支持的技术使用方式才能依赖以满足成功标准。任何以不可访问性支持的方式提供的信息或功能，也必须以可访问性支持的方式提供。（请参阅理解可访问性支持。）

5.2.5 不干扰

如果技术以不可访问性支持的方式使用，或者以不符合要求的方式使用，则它们不得阻碍用户访问页面的其他部分。此外，整个网页在以下每种情况下仍需满足符合性要求：

  1. 当用户代理中启用了任何不依赖的技术时，
  2. 当用户代理中关闭了任何不依赖的技术时，
  3. 当用户代理不支持任何不依赖的技术时。

此外，以下成功标准适用于页面上的所有内容，包括不依赖于符合性要求的内容，因为未满足这些标准可能会干扰页面的任何使用：

  • 1.4.2 - 音频控制,
  • 2.1.2 - 无键盘陷阱,
  • 2.3.1 - 三次闪烁或低于阈值,
  • 2.2.2 - 暂停、停止、隐藏.
注释

如果页面无法符合要求（例如，符合性测试页面或示例页面），则不能将其包含在符合性范围或符合性声明中。

有关更多信息，包括示例，请参阅理解符合性要求。

5.3 符合性声明（可选）

符合性仅针对网页定义。然而，符合性声明可以涵盖一个页面、一系列页面或多个相关网页。

5.3.1 符合性声明的必需组件

符合性声明不是必需的。作者可以在不声明的情况下符合 WCAG 2.2。然而，如果做出符合性声明，则符合性声明必须包括以下信息：

  1. 声明日期
  2. 指南标题、版本和 URI："Web 内容可访问性指南 2.2，网址为 https://www.w3.org/TR/WCAG22/"
  3. 满足的符合性级别：（Level A、AA 或 AAA）

  4. 网页的简要描述，例如声明所涵盖的 URI 列表，包括是否包含子域。

    注释 1

    网页可以通过列表或描述声明中包含的所有 URI 的表达式进行描述。

    注释 2

    在安装到客户网站之前没有 URI 的基于 Web 的产品可以声明在安装后符合要求。

  5. Web 内容技术 依赖的技术 的列表。
注释 3

如果使用符合性标志，则它将构成声明，并且必须附有上述符合性声明的必需组件。

5.3.2 符合性声明的可选组件

除了上述符合性声明的必需组件外，请考虑提供其他信息以帮助用户。推荐的附加信息包括：

  • 超出声明符合性级别的成功标准列表。这些信息应以用户可使用的形式提供，最好是机器可读的元数据。
  • “使用但不依赖”的具体技术列表。
  • 用于测试内容的用户代理列表，包括辅助技术。
  • 内容的具体可访问性特性列表，以机器可读的元数据形式提供。
  • 超出成功标准的任何额外步骤的信息，以增强可访问性。
  • “依赖”的具体技术列表的机器可读元数据版本。
  • 符合性声明的机器可读元数据版本。
注释 1

有关更多信息和符合性声明示例，请参阅理解符合性声明。

注释 2

有关在符合性声明中使用元数据的更多信息，请参阅理解元数据。

5.4 部分符合性声明 - 第三方内容

将来会添加更多内容的网页可以使用“部分符合性声明”。例如，电子邮件程序、博客、允许用户添加评论的文章，或支持用户贡献内容的应用程序。另一个例子是由多个贡献者汇总内容组成的页面，如门户网站或新闻网站，或者随着时间推移自动插入其他来源内容的网站，例如动态插入广告的情况。

在这些情况下，最初发布时无法知道页面的非控制内容是什么。需要注意的是，非控制内容也可能影响控制内容的可访问性。提供了两种选择：

  1. 可以根据最佳认知来做出符合性的判定。如果这种类型的页面在两天内得到监控和修复（非符合性内容被移除或使其符合），则可以做出符合性判定或声明，因为除了外部贡献内容中的错误会在遇到时得到纠正或移除外，该页面是符合的。如果无法监控或修正非符合性内容，则无法做出符合性声明；

    或者

  2. 可以做出“部分符合性声明”，即该页面不符合，但如果某些部分被移除，则可能符合。该声明的形式为：“此页面不符合，但如果移除以下不受控制的来源部分，则符合WCAG 2.2的X级别。”此外，关于部分符合性声明中描述的非控制内容，还应符合以下条件：

    1. 它不属于作者控制的内容。
    2. 它以用户可以识别的方式描述（例如，不能简单描述为“我们无法控制的所有部分”，除非这些部分明确标记出来）。

5.5 部分符合性声明 - 语言

当页面不符合规定，但如果对页面使用的语言（或所有语言）有可访问性支持，则页面可以符合时，可做出“由于语言部分符合性声明”。声明的形式为：“此页面不符合，但如果存在对以下语言的可访问性支持，则会符合WCAG 2.2的X级别：”

由用户的 辅助技术 以及浏览器和其他 用户代理 支持

为了使 Web 内容技术的使用（或技术特性的使用）符合可访问性支持，必须满足以下两点：

  1. 所使用的 Web 内容技术 必须被用户的辅助技术（AT）支持。 这意味着所使用的技术已在内容的 人类语言 中进行过与用户辅助技术的互操作性测试。

    同时

  2. 必须有用户可用的支持可访问性的用户代理。 这意味着以下四种情况中的至少一种为真：

    1. 该技术在广泛分发的用户代理中原生支持，并且也支持可访问性（例如 HTML 和 CSS）；

      或者

    2. 该技术在广泛分发且支持可访问性的插件中得到支持；

      或者

    3. 该内容在封闭的环境中提供，例如大学或公司网络，所需的用户代理也支持可访问性；

      或者

    4. 支持该技术的用户代理是可访问性支持的，并且可供下载或购买，且：

      • 残障人士不需要比没有残障的人支付更多费用 并且
      • 残障人士可以像没有残障的人一样轻松找到并获取。
注 1

可访问性指南工作组和 W3C 未规定对 Web 技术的某一特定用途进行分类为可访问性支持所需的辅助技术支持量。 (参见 可访问性支持的辅助技术支持水平)。

注 2

Web 技术可以以不支持可访问性的方式使用，只要这些方式没有 被依赖，并且页面整体符合要求，包括 一致性要求 4 和 一致性要求 5。

注 3

当Web 技术以“可访问性支持”的方式使用时，并不意味着整个技术或该技术的所有使用方式都受到支持。大多数技术，包括 HTML，至少有一个功能或使用方式缺乏支持。只有在可访问性支持的技术使用方式可以依赖于满足 WCAG 要求时，页面才符合 WCAG。

注 4

引用具有多个版本的 Web 内容技术时，应指定受支持的版本。

注 5

作者可以通过查阅已记录的可访问性支持用法列表来找到支持可访问性技术的用法。 (参见 了解可访问性支持的 Web 技术用法)。 作者、公司、技术供应商或其他人可以记录支持可访问性的 Web 技术用法。然而，文档中的所有技术使用方式都需要符合上述的可访问性支持定义。

符合性替代版本

版本，它

  1. 在指定级别上符合要求，并且
  2. 以相同的人类语言提供所有相同的信息和功能，并且
  3. 与不符合要求的内容一样是最新的，并且

    1. 可以通过可访问性支持的机制从不符合要求的页面访问符合要求的版本，或者

注释 2

替代版本不必与原始版本逐页匹配（例如，符合要求的替代版本可以包含多个页面）。

注释 3

注释 4

可以提供替代版本以适应不同的技术环境或用户组。每个版本都应尽可能符合要求。为了满足符合性要求 1，一个版本需要完全符合要求。

注释 5

符合要求的替代版本不必位于符合性范围内，甚至不必位于同一网站上，只要它与不符合要求的版本一样可以免费获得即可。

在内容中设置用户偏好以生成符合要求的版本是访问另一个版本的可接受机制，只要用于设置偏好的方法是可访问性支持的即可。

对比度

(L1 + 0.05) / (L2 + 0.05)，其中

  • L1 是较亮颜色的相对亮度，并且
  • L2 是较暗颜色的相对亮度。
注释 1

对比度范围可以从 1 到 21（通常写为 1:1 到 21:1）。

注释 2

由于作者无法控制用户如何呈现文本的设置（例如字体平滑或抗锯齿），因此可以在关闭抗锯齿的情况下评估文本的对比度。

注释 3

就成功标准 1.4.3 和 1.4.6 而言，对比度是相对于在正常使用中呈现文本的指定背景来测量的。如果未指定背景颜色，则假定为白色。

注释 4

背景颜色是在正常使用中呈现文本的内容的指定颜色。如果在指定文本颜色时未指定背景颜色，则这是一个失败，因为用户的默认背景颜色未知，无法评估其足够的对比度。出于同样的原因，如果在指定背景颜色时未指定文本颜色，则这也是一个失败。

注释 5

当字母周围有边框时，边框可以增加对比度，并将用于计算字母与其背景之间的对比度。字母周围的窄边框将用作字母。字母周围填充字母内部细节的宽边框充当光晕，将被视为背景。

注释 6

WCAG 符合性应针对内容中指定的颜色对进行评估，作者希望这些颜色对在典型呈现中相邻出现。作者无需考虑不寻常的呈现，例如用户代理所做的颜色更改，除非是由作者的代码引起的。

对于 sRGB 颜色空间，颜色的相对亮度定义为 L = 0.2126 * R + 0.7152 * G + 0.0722 * B，其中 R、G 和 B 定义如下：

  • 如果 RsRGB <= 0.04045 则 R = RsRGB/12.92 否则 R = ((RsRGB+0.055)/1.055) ^ 2.4
  • 如果 GsRGB <= 0.04045 则 G = GsRGB/12.92 否则 G = ((GsRGB+0.055)/1.055) ^ 2.4
  • 如果 BsRGB <= 0.04045 则 B = BsRGB/12.92 否则 B = ((BsRGB+0.055)/1.055) ^ 2.4

并且 RsRGB、GsRGB 和 BsRGB 定义如下：

  • RsRGB = R8bit/255
  • GsRGB = G8bit/255
  • BsRGB = B8bit/255

“^”字符是幂运算符。（公式取自 [SRGB]。）

注释 2

2021 年 5 月之前，定义中的 0.04045 值不同 (0.03928)。它取自规范的旧版本，并已更新。在这些指南的背景下，它对计算没有实际影响。

注释 3

如今用于查看 Web 内容的几乎所有系统都假定采用 sRGB 编码。除非已知将使用其他颜色空间来处理和显示内容，否则作者应使用 sRGB 颜色空间进行评估。如果使用其他颜色空间，请参阅理解成功标准 1.4.3。

注释 4

如果在交付后发生抖动，则使用源颜色值。对于在源处抖动的颜色，应使用抖动颜色的平均值（平均 R、平均 G 和平均 B）。

注释 5

注释 6

用户代理通过一个或多个视口呈现内容。视口包括窗口、框架、扬声器和虚拟放大镜。视口可以包含另一个视口（例如，嵌套框架）。用户代理创建的界面组件（例如提示、菜单和警报）不是视口。

注释 2

此定义基于用户代理可访问性指南 1.0 词汇表 [UAAG10]。

视觉定制

可以设置字体、大小、颜色和背景

网页

通过单一 URI 使用 HTTP 获取的非嵌入资源，以及用户代理在渲染或预期与其一起渲染时使用的任何其他资源

注释 1

尽管任何“其他资源”都将与主要资源一起呈现，但它们不一定同时呈现。

注释 2

就符合这些指南而言，资源必须在符合性范围内“非嵌入”，才能被视为网页。

示例 1

包含所有嵌入图像和媒体的 Web 资源。

示例 2

使用异步 JavaScript 和 XML (AJAX) 构建的 Web 邮件程序。该程序完全位于 http://example.com/mail，但包括收件箱、联系人区域和日历。提供了导致显示收件箱、联系人或日历的链接或按钮，但不会更改整个页面的 URI。

示例 3

一个可自定义的门户网站，用户可以从一组不同的内容模块中选择要显示的内容。

示例 4

当您在浏览器中输入“http://shopping.example.com/”时，您会进入一个类似电影的交互式购物环境，您可以在商店中直观地移动，将产品从周围的货架上拖到您面前的虚拟购物车中。单击产品会导致其与浮动在旁边的规格表一起演示。这可能是一个单页网站，也可能只是网站中的一个页面。

7. 用户界面组件的输入用途

本节包含常见用户界面组件输入目的的列表。以下术语并非必须使用的关键字，而是代表网页采用的分类法中必须捕获的目的。在适用的情况下，作者使用选定的分类法标记控件以指示语义目的。这为用户代理和辅助技术提供了应用个性化呈现的潜力，从而使更多人能够理解和使用内容。

注意

输入类型目的列表基于 HTML 规范自动填充部分中定义的控件目的，但重要的是要理解，不同的技术可能在其规范中定义了部分或全部相同的概念，并且只需要映射到以下含义的概念。

以下输入控件目的旨在与内容用户相关，并且仅与该个人相关的信息有关。

  • name - 全名
  • honorific-prefix - 前缀或头衔（例如，“Mr.”、“Ms.”、“Dr.”、“Mlle”）
  • given-name - 名字（在某些西方文化中，也称为教名）
  • additional-name - 中间名（在某些西方文化中，也称为中间名，除名字外的其他教名）
  • family-name - 姓氏（在某些西方文化中，也称为姓氏或姓）
  • honorific-suffix - 后缀（例如，“Jr.”、“B.Sc.”、“MBASW”、“II”）
  • nickname - 昵称、网名、用户名：通常用于代替全名的简称
  • organization-title - 职位（例如，“软件工程师”、“高级副总裁”、“副总经理”）
  • username - 用户名
  • new-password - 新密码（例如，在创建帐户或更改密码时）
  • current-password - 由 username 字段标识的帐户的当前密码（例如，在登录时）
  • organization - 与此字段关联的其他字段中的个人、地址或联系信息对应的公司名称
  • street-address - 街道地址（多行，保留换行符）
  • address-line1 - 街道地址（每字段一行，第 1 行）
  • address-line2 - 街道地址（每字段一行，第 2 行）
  • address-line3 - 街道地址（每字段一行，第 3 行）
  • address-level4 - 在具有四个行政级别的地址中，最细粒度的行政级别
  • address-level3 - 在具有三个或更多行政级别的地址中，第三级行政级别
  • address-level2 - 在具有两个或更多行政级别的地址中，第二级行政级别；在具有两个行政级别的国家/地区，这通常是相关街道地址所在的城市、城镇、村庄或其他地区
  • address-level1 - 地址中最广泛的行政级别，即地区所在的省份；例如，在美国，这是州；在瑞士，这是州；在英国，这是邮政城镇
  • country - 国家代码
  • country-name - 国家名称
  • postal-code - 邮政编码、邮编、ZIP 码、CEDEX 码（如果是 CEDEX，则将“CEDEX”和相关的区（如果适用）附加到 address-level2 字段）
  • cc-name - 支付工具上提供的全名
  • cc-given-name - 支付工具上提供的名字（在某些西方文化中，也称为教名）
  • cc-additional-name - 支付工具上提供的中间名（在某些西方文化中，也称为中间名，除名字外的其他教名）
  • cc-family-name - 支付工具上提供的姓氏（在某些西方文化中，也称为姓氏或姓）
  • cc-number - 标识支付工具的代码（例如，信用卡号）
  • cc-exp - 支付工具的到期日期
  • cc-exp-month - 支付工具到期日期的月份部分
  • cc-exp-year - 支付工具到期日期的年份部分
  • cc-csc - 支付工具的安全码（也称为银行卡安全码 (CSC)、银行卡验证码 (CVC)、银行卡验证值 (CVV)、签名栏代码 (SPC)、信用卡 ID (CCID) 等）
  • cc-type - 支付工具类型
  • transaction-currency - 用户希望交易使用的货币
  • transaction-amount - 用户希望交易的金额（例如，在输入出价或售价时）
  • language - 首选语言
  • bday - 生日
  • bday-day - 生日的日期部分
  • bday-month - 生日的月份部分
  • bday-year - 生日的年份部分
  • sex - 性别认同（例如，女性、Fa’afafine）
  • url - 与此字段关联的其他字段中的公司、个人、地址或联系信息对应的主页或其他网页
  • photo - 与此字段关联的其他字段中的公司、个人、地址或联系信息对应的照片、图标或其他图像
  • tel - 完整电话号码，包括国家代码
  • tel-country-code - 电话号码的国家代码部分
  • tel-national - 不带国家代码部分的电话号码，如果适用，则应用国家内部前缀
  • tel-area-code - 电话号码的区号部分，如果适用，则应用国家内部前缀
  • tel-local - 不带国家代码和区号部分的电话号码
  • tel-local-prefix - 当区号后的电话号码部分分为两部分时，该部分的第一部分
  • tel-local-suffix - 当区号后的电话号码部分分为两部分时，该部分的第二部分
  • tel-extension - 电话号码内部分机号
  • email - 电子邮件地址
  • impp - 表示即时消息协议端点的 URL（例如，“aim:goim?screenname=example”或“xmpp:fred@example.net”）
