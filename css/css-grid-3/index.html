1. 引言

本节不具备规范性。

网格布局是一种 CSS 的布局模型，具有强大的控制盒子及其内容的尺寸和定位的能力。 网格布局专为二维布局优化：即在两个方向都需要内容对齐的布局场景。

虽然许多布局可以通过常规网格布局表达， 但在两个轴上将项目限制在网格中，也会导致无法表达 Web 上一些常见布局。

本模块定义了一种网格布局变体，移除了该限制， 使得项目可在一个轴上放入类似网格的轨道， 而在另一个轴上顺序堆叠。 未被明确放入特定轨道的项目， 会根据当前已放置项目的布局尺寸， 被放到剩余空间最多的列（或行）中。

1.1. 背景与动机

1.1.1. 瀑布流布局与自动放置项目

网格车道布局， 有时也称为“瀑布流布局”或“masonry 布局”， 是一种常见的 Web 设计模式，其中多个项目（通常是图片或短文摘要） 被逐个放入列中， 排列方式类似于砖石结构建造。 不同于多列布局和多行列弹性布局， 在这些布局中，内容会在第一列从上向下排布 直到溢出到第二列， 自动放置在网格车道布局里为每个新项目选择列， 保证新项目一般位于比后放置项更靠上的位置。

传统的 Pinterest 搜索结果页面就是这种布局的典型代表：

这里，每个项目高度都不同 （取决于内容和列宽）， 检查 DOM 可看到 （视觉内容本身无法体现顺序） 每个项目都被放入当前高度最小的那一列。

这种布局表面上与多列布局相似； 但它的优点在于，向下滚动页面时， 布局中的“后面”项目 （如在搜索结果中相关度较低的项目） 会更自然地出现在下方。

无法用早期的 CSS 布局模型实现此布局， 除非事先已知每个项目的高度， 或用 JavaScript 进行内容测量和放置。

使用display: grid-lanes配合grid-template-columns 就可以实现这种网格车道布局

1.1.2. 一维网格布局

网格布局 支持强大的轨道尺寸与双轴显式放置， 但有时布局仅需在一个维度上对齐项目。

将网格车道布局 与显式定位项目结合使用， 可实现这种一维网格布局。

这个例子由 Douglas Graham 提供， 使用显式定位将每个项目放入对应列； 但没有行。 每列的项目沿列方向依次堆叠。 这种布局无法用网格布局复现， 因为相邻列项目间的“跨越”关系不是固定的：依赖其相对高度 以及可选横幅或广告项目是否包含。 也无法在弹性布局中实现， 因为项目的源顺序（用于阅读、顺序导航、一列手机布局时） 在两列之间来回切换。

1.2. 属性值定义

本规范遵循CSS 属性定义约定，引用自[CSS2]， 并采用属性值定义语法（见[CSS-VALUES-3]）。 本规范未定义的值类型在 CSS 属性值与单位[CSS-VALUES-3]中定义。 与其他 CSS 模块组合可扩展这些值类型的定义。

2. 网格车道布局模型

网格车道布局 会将项目按预设轨道排布，类似于网格布局的单一轴线 （称为网格轴）， 而在另一轴上则如弹性布局般自由流动 （称为堆叠轴）。 类似于网格布局且不同于弹性布局， 网格车道布局的自动放置 会在多个轨道间分配项目， 以尽量保持各轨道长度接近。

网格项目的形成与块化方式 与常规网格容器完全一致。

所有 CSS 属性的行为与常规网格容器一致， 除非本规范另有规定。 例如，可以用order 指定项目布局顺序。

注：支持子网格项目， 但子网格仅在网格轴上生效； 详见§ 3.2 子网格。

网格车道容器 指网格容器， 其内容参与网格车道布局。 网格车道容器会创建列轨道 如果其堆叠轴是块轴， 或创建行轨道 如果其堆叠轴是行内轴。

网格车道容器比较
列式车道 
grid-template-columns: 1fr 2fr 3fr;
行式车道 
grid-template-rows: 1fr 2fr 3fr;

2.1. 重排序与无障碍性

虽然网格车道布局通常是“向前推进”的模式 （即在至少一个轴上将下一个项目放在当前项目的末端，符合自然“阅读顺序”）， 但它可以在内联轴或块轴之间以看似任意的方式切换“向后”或“向前”。 在简单场景下，item-tolerance 属性有助于减少因自动放置项目时 堆叠轴上的微小尺寸差异所产生的回溯感。 但当自动放置与显式放置或跨区项目混合时， 还是可能发生一定程度的“回溯”现象。

例如，在下方的标记示例中， 第四个项目是个跨区项，无法放入第一行剩余的空列。 它最终被定位到第一列， 即可容纳它的最高的空白区域。 接下来的几个跨度为 1 的项目， 最终被布置在它“上方”的空列， 违背了自然阅读顺序。 
<section class=masonry>
  <div class=item>1</div>
  <div class=item>2</div>
  <div class="item tall">3</div>
  <div class="item wide">4</div>
  <div class=item>5</div>
  <div class=item>6</div>
  <div class=item>7</div>
</section>
<style>
.masonry {
  display: grid-lanes;
  grid-template-columns: repeat(5, auto);
}
.item { height: 50px; }
.item.wide { grid-column: span 3; }
.item.tall { height: 90px; }
</style>
同理，显式放入特定轨道的项目也可能在其后面留下空白， 后续自动放置的项目可能被放到视觉顺序之外。

作者应了解这种情况 并尽量设计能减少“回溯”的布局， 便于聚焦和正确阅读顺序的把握。 或者，如若这些项目之间没有本质的顺序， 可用reading-flow属性让UA根据阅读和线性导航重排项目。

或者，是否应该默认自动放置项目即重排序？

减少回溯的技术包括：

  • 为item-tolerance选择合适的值， 即数值应足够大，避免对相近尺寸轨道的无谓区分， 但又不能大到忽略了真实的尺寸差异。

  • 用显式放置方式将相关项目分组， 而不是打乱原有项目序。

  • 避免在网格轴上混合跨度、 同时又在堆叠轴上项目尺寸差异很大， 这会导致项目顺序被打乱（见上述示例）。

与网格布局 和弹性布局一致， 作者也可以使用order 属性重排序； 但注意事项也同样适用。 参见CSS Grid Layout 2 § 4 重排序与无障碍性 以及CSS Display 4 § 3.1 重排序与无障碍性。

3. 网格车道轨道规范

在网格轴上， 可以充分利用网格布局的能力来定义轨道：

  • 轨道大小、线名和区域可在网格车道容器的网格轴上定义， 和网格布局一致。

  • 显式网格和隐式网格的形成与常规 网格容器 相同。

  • 项目也可以如下文直接放在这些网格模板上，就像网格布局一样。

但对于自动放置项目， 它们会对所有轨道都产生尺寸贡献， 而不是只贡献最终被放置的那个轨道； 参见§ 3.4 网格轴轨道尺寸。

注：这是因为自动放置项目必须在放置的同时进行布局， 这样每个轨道才能知道自己有多“满” （也就是下一个自动放置项目该放哪个轨道）； 因此，轨道本身必须已确定尺寸， 这样项目才能在布局时获得可用空间。

3.1. 声明网格车道轨道模板：grid-template-* 属性

grid-template-* 以及grid-auto-rows/grid-auto-columns 属性 （及其速记）都适用于网格轴上的 网格车道容器， 并像常规网格容器一样建立轨道。 （在堆叠轴则忽略。）

3.1.1. 内在轨道与 repeat()

Level 3 扩展了 repeat() 表示法，允许重复时 最小轨道尺寸函数和 最大轨道尺寸函数都不是 确定值； 换言之，<auto-repeat> 的语法放宽如下： 

<auto-repeat> = repeat(  [ auto-fill | auto-fit ] , [ <line-names>? <track-size> ]+ <line-names>? )

为了解决重复次数， 需为此类轨道计算一个假设尺寸， 按CSS Grid Layout 2 § 12.4 初始化轨道尺寸和 CSS Grid Layout 2 § 12.5 解析内在轨道尺寸初始化轨道尺寸和解析内在轨道尺寸， 并以§ 3.4 网格轴轨道尺寸的假定作为依据：

  • 忽略显式项目放置。 （即假设所有项目都为 自动位置。）

  • 不要合并任何轨道。

  • 将重复轨道列表扩展到可承载所有自动放置项目的程度， 即轨道列表应重复 2 + (最大跨度 - 2)/(repeat()中的轨道数) 次, 并向下取整。

在repeat()列表中每个轨道的假设尺寸， 由该条目（按索引）对应的所有轨道中的最大值给出。

例如， 模板为 repeat(auto-fill, 50px auto auto)， 最大跨度为 2 时， 就要将轨道列表重复两次， 得到 50px auto auto 50px auto auto。 做完假设布局后， 第2、第5个轨道取最大值给第一个 auto， 第3、第6个轨道取最大值给第二个 auto。 用这些具体尺寸决定能填充进容器的重复次数。

这种做法要在常规网格布局中也支持吗，还是回退到单次重复？如需支持，方法如何？[Issue #10915]

注： 这种简化的布局启发式定义是为“足够好”， 且保持快速和一致性。 忽略放置是为了让概念可执行； 你在不知道所需重复次数前， 无法确定某项目实际会落入哪个轨道 （即便其声明有确定放置）。

技术上，如果你的显式放置并不跨越或覆盖重复片段， 我们实际上是能知道放置的。 是否希望上述算法调整支持这种情景？ 即对于 auto repeat(auto-fill, ...) auto， 如果显式放置在 line 1 或 -1， 可以只影响对应轨道 （而不假定会出现在每个轨道）。

3.2. 子网格

子网格允许嵌套的网格车道容器（和网格容器） 共享轨道尺寸。 如果父元素的对应轴为网格轴， 则子网格的轴按网格容器的规范从父容器继承； 如果父元素的对应轴为堆叠轴， 子网格的轴也作为堆叠轴处理。

若该行为与车道方向冲突（或导致两个轴都堆叠），应如何处理？

在网格车道布局中， 自动放置的子网格 不会从父网格继承任何线名， 因为那样会使项目放置依赖于布局结果； 但子网格的轨道仍照常与父网格轨道对齐。

这里有一个子网格示例： 
<style>
.grid {
  display: inline-grid-lanes;
  grid-template-rows: auto auto 100px;
  align-content: center;
  height: 300px;
  border: 1px solid;
}

.grid > * {
  margin: 5px;
  background: silver;
}
.grid > :nth-child(2n) {
  background: pink;
}

.grid subgrid {
  display: grid;
  grid: subgrid / subgrid;
  grid-row: 2 / span 2;
  grid-gap: 30px;
}
.grid subgrid > * { background: cyan; }
</style>

<div class="grid">
  <item>1</item>
  <item>2</item>
  <item>3</item>
  <subgrid>
    <item style="height:100px">subgrid.1</item>
    <item>sub.2</item>
    <item>s.3</item>
  </subgrid>
  <item>4</item>
  <item>5</item>
  <item style="width: 80px">6</item>
  <item>7</item>
</div>

注意子网格的第一个项目（"subgrid.1"）会贡献到父网格第二行的内在尺寸。 这是因为子网格指定了明确位置，因此可知其会占用哪些轨道。 另请注意，若对子网格的堆叠轴做子网格处理， 会导致子网格变为网格车道容器， 并且其行内轴成为堆叠轴。

作为子网格的 网格车道容器 可以称为网格车道子网格。

3.3. 轨道重复：repeat() 表示法

本规范为repeat()表示法引入了新关键字以及特有的 grid-lanes 行为。

3.3.1. repeat(auto-fit)

在网格车道容器（和常规网格容器）中， auto-fit的行为类似auto-fill， 但空轨道会被折叠。 不过，由于放置发生在轨道尺寸之后， 网格车道容器用启发式方法判断轨道是否被占用：

  • 所有被显式放置的轨道都视为已占用。

  • 所有自动放置项目的跨度和为N，则前N个未占用轨道都视为已占用。

所有通过auto-fit生成且被此启发判定为未占用的轨道， 均视为“空的”并被折叠。 折叠网格轨道无法用于放置自动项目。

注：在使用auto-fill时，自动放置项有可能被放入某个轨道，但如果使用auto-fit则该轨道会被折叠，特别是在存在跨度大于1的自动放置项，且与显式放置项目混用时，显式放置后留下的空隙太小，无法容纳自动放置项的情况下。

3.4. 网格轴轨道尺寸

轨道尺寸算法与CSS Grid相同， 不同之处在于确定哪些项目贡献内在尺寸时：

  • 所有被显式放入该轨道的项目都贡献，且

  • 所有自动网格位置的项目都贡献 （无论最终是否放入该轨道）。

例如，假设网格轴有两列：

  • A、B、C项目都没有显式位置。

  • D项目被显式放入第一列。

这时，A、B、C、D 都会参与第一列尺寸计算， 第二列则只有 A、B、C（不含 D）。

对于跨度项目且拥有自动网格位置的情况， 视为在每个可起始位置都放了一份， 逐个计算贡献。

例如，假设网格轴有5列， 中间列宽固定为100px， 其余为auto定宽。 对轨道尺寸而言， 一个跨度为2、内在尺寸为220px的项目， 理论上等价于复制如下：

  • 在网格线1，从前两列各贡献110px。

  • 在网格线2，为第二列贡献120px。

  • 在网格线3，为第四列贡献120px。

  • 在网格线4，为第4、5列各贡献110px。

注： 此算法确保每个轨道至少足够容纳最终被放置其内的所有项目， 不会在放置与轨道尺寸间产生依赖循环。 但如果项目尺寸相差大， 轨道可能会比“实际需要”更大： 只有所有项目都显式放在网格轴， 或所有项目等尺寸 （或多组等尺寸跨度项）时，轨道尺寸才一定精确贴合。

3.4.1. 子网格项目尺寸贡献

为常规网格容器或网格车道容器的轨道定尺寸时， 网格车道子网格 对拥有自动网格位置项目的处理特别如下：

  • 此类项目会放置进子网格可跨越的每个网格轨道。 （若子网格有确定位置，则只放跨越轨道； 若有自动位置，则在父网格所有轨道。）

  • 此类项目获得它可被虚放的每个边缘的最大 margin/border/padding 贡献。 若跨越整个子网格，四边都取。 （见CSS Grid Layout §9）

3.4.2. 优化轨道尺寸

可将具有相同跨度和放置的项目聚合为单个虚拟项，优化轨道尺寸，方法如下：

  1. 按下列属性将所有网格项目分为项目组：

    • 项目跨度

    • 项目放置位置 （即允许放在哪些轨道）

    • 项目的基线分组

    注： 如跨度为2且定位于第二轨的项目，与跨度为2且自动位置的项目是不同组。

  2. 为每个项目组合成一个虚拟网格项， 其内在尺寸贡献是组内所有项的最大值。

    若这些项用基线对齐， 则把所有项都放入假想轨道， 得到共享基线及 shim， 相应调整该组内在尺寸贡献。

  3. 将每个虚拟网格项的假想副本放入网格轴轨道每个可用位置， 按轨道尺寸算法处理。 结果就是网格车道容器的轨道尺寸。

注： 如与上文轨道尺寸算法结果不符， 则是错误， 请向 CSSWG 报告。

4. 网格车道轨道放置

在网格轴上， 可通过熟悉的网格放置属性语法将项目显式放置到轨道并进行跨越。 自动放置则采用§ 4.4 网格车道布局与放置算法， 将每一个自动网格位置的项目放入当前“最短”的轨道。

下面的网格车道布局示例展示了显式放置与跨越的项目： 
.container {
  grid-template-columns: repeat(3, auto);
}
.container > :nth-child(2) {
  /* auto-placed, but spanning. */
  grid-column: span 2;
}
.container > :nth-child(3) {
  /* manually placed */
  grid-column: 3;
}
/* all other children are auto-placed */

4.1. 指定网格轴项目放置：grid-column-* 和 grid-row-* 属性

grid-column-* 和 grid-row-* 属性 （包括速记） 适用于项目的网格轴， 并如常规网格布局一般完成放置。

4.2. 放置精度：item-tolerance 属性

名称: item-tolerance
值: normal | <length-percentage> | infinite
初始值: normal
适用元素: 网格车道容器
继承性: no
百分比: 相对网格轴 内容盒尺寸的网格车道容器
计算值: 一个计算后的 <length-percentage> 值
规范顺序: 按语法
动画类型: 按长度

网格车道容器 通过将每个网格项目放入当前最少填充的网格轨道而填满。 若有多个轨道并列为最短， 顺序依次放置会更美观。 但若轨道高度仅有极小差别时， 非顺序填充可能会显得怪异， 因为这些高度差异并不会被认为是有意义的不同。

item-tolerance 属性指定了判断轨道为“等高”时的阈值， 让它们按顺序依次填充。

<length-percentage>

为网格车道容器指定并列阈值。 如果可选位置距离最短位置在指定范围内， 则视为“并列”等价选择。

注： 初始值是“较小”距离（1em）， 通常用来表达“足够接近”。

normal

在网格车道布局 里，其实际值为1em。 在其他布局模式下，则实际值为0。

infinite

指定一个无穷大的并列阈值。 这样项目始终严格按顺序分布， 完全无视轨道长度。

注： 此值会导致连续项目在堆叠轴上的位置差异非常大， 这会令读者困惑。 若1em初值太小， 建议用更大值（如10em或50vh）， 不建议用infinite。

4.3. 密集放置：dense 关键字

在网格布局中，grid-auto-flow: dense允许在网格项目放置算法里回溯。 dense关键字同样允许网格车道放置算法里回溯。 但由于网格车道布局里项目放置和尺寸计算交织， 仅当空位轨道总占用尺寸与项目正常放置轨道占用尺寸相等时， 项目才能回溯占用兼容空位。

注： 此限制避免多次布局同一项目。

4.4. 网格车道布局与放置算法

对于网格轴中的每个轨道， 持有一个初始为零的累计位置。 还要维护一个自动放置光标，初始指向第一条线。

对顺序修改文档顺序的每个项目：

  1. 如果该项目在网格轴上有确定放置位置， 就采用这个位置进行放置。

    本步骤是否也需更新放置光标？

    否则，按如下子步骤解析其网格轴放置：

    1. 从网格轴隐式网格中的第一条线开始， 计算若把该项目从此线放置会跨越的轨道的最大累计位置，记为max_pos。
    2. 对每个后续线号重复上述步骤， 直到该项目无法继续放入网格。
    3. 令possible lines为使max_pos最小的线， 以及所有在并列阈值范围内的其他线。
    4. 选择possible lines中第一个大于等于自动放置光标的作为该项目在网格轴上的位置； 若无则选第一个。
    5. 将自动放置光标更新为指向该项目的末尾线。
  2. 将该项目放置在其网格轴对应的轨道中， 位置为其跨越轨道的最大累计位置。
  3. 计算该项目的包含块尺寸并布局该项目。 将所跨越网格轴轨道的累计位置 设为 max_pos + 外部尺寸 + grid-gap。

    在这里，盒子的外部尺寸会下限为零。

    注：即，如果盒子有足够大的负外边距， 这里不会计算为负的尺寸。 任一轨道的累计位置均不会减少， 所以负尺寸盒不会导致后续正常尺寸项目在该轨道上与前项重叠。

  4. 如果网格车道容器用的是dense密集分布， 且布局中存在被跳过的空间（比如因跨越项所致）， 当前尺寸的该项目如果放在更早时可被放进这些空间， 且所跨轨道的总占用尺寸等于当前所放轨道， 那么要放进最高的可用空间。 若在最高空间内有多个并列有效位置， 则放在最靠起始端的那个。 将自动放置光标和累计位置 回溯到该项目放置前的值。

    注：视觉上“插入”前方空白的项目 （通过负外边距、position: relative、变换等实现）， 并不影响空白空间的尺寸。 后续项目依旧可以放进这些空间中，并在视觉上与前项目重叠。

    注：密集分布在回填时会忽略自动放置光标， 且放置后不会更新光标。 若没有可回填的位置，则行为同未指定dense时一样。

注：该算法总会选择能让项目放得最高的轨道。 若有多个等价位置，则选择最靠前的轨道， 并尽量选在上一个项目后面 （即使有并列，也能保证总是“向前推进”）。

4.4.1. 包含块

参与网格车道布局的 网格项目的包含块 由其网格区域在网格轴上的范围 以及网格容器内容盒在堆叠轴上的范围组成。

4.4.2. 放置与书写模式

注： 与网格布局一致， 网格车道布局与放置同样受书写模式影响。 例如对direction: rtl， 无论内联轴是网格轴还是堆叠轴， 项目都按从右到左放置，而非左到右。

下面是一个示例，direction: rtl用于网格轴： 
<style>
  .grid {
    display: inline-grid-lanes;
    direction: rtl;
    grid-template-columns: repeat(4, 2ch);
    border: 1px solid;
  }

  item { background: silver }
  item:nth-child(2n+1) {
    background: pink;
    height: 4em;
  }
  </style>

<div class="grid">
  <item>1</item>
  <item style="grid-column:span 2">2</item>
  <item>3</item>
  <item>4</item>
</div>
下面是一个示例， direction: rtl用于堆叠轴： 
<style>
.grid {
  display: inline-grid-lanes;
  direction: rtl;
  width: 10ch;
  column-gap: 1ch;
  grid-template-rows: repeat(4, 2em);
  border: 1px solid;
}

item { background: silver }
item:nth-child(2n+1) {
  background: pink;
  width: 4ch;
}
</style>

<div class="grid">
  <item>1</item>
  <item style="grid-row:span 2">2</item>
  <item>3</item>
  <item>4</item>
</div>

5. 网格容器定尺寸

网格容器定尺寸与常规网格容器一致， 但对堆叠轴如下替换： 网格容器在堆叠轴的max-content 尺寸（min-content 尺寸）为 网格车道容器 在该轴 max-content 约束（min-content 约束）下的 堆叠范围的尺寸。 堆叠范围 是指每个轨道首项的最起始外边缘和末项的最末端外边缘之间的距离。

这里有个简单示例： 
<style>
.grid {
  display: inline-grid-lanes;
  grid-template-columns: 50px 100px auto;
  grid-gap: 10px;
  border: 1px solid;
}
item { background: silver; margin: 5px; }
</style>

<div class="grid">
  <item style="border:10px solid">1</item>
  <item>2</item>
  <item>3</item>
  <item style="height:50px">4</item>
  <item>5</item>
  <item>6</item>
</div>

6. 对齐与间距

沟槽在两个轴上都支持。 在堆叠轴， 间隙应用于每对相邻项目的外边距盒之间。 两轴上的外边距均不会折叠。

在网格轴上， 对齐 与常规网格容器一致。

在堆叠轴， 内容分布 作用于整体内容， 类似块容器中的表现。 更具体地说，对齐对象为堆叠范围。

注： 在堆叠轴上这些属性始终只有一个对齐对象， 所以align-content / justify-content的唯一有效值只剩 start、 center、 end以及基线对齐。 （normal 和 stretch行为与start一致， 分布式对齐的行为与其回退对齐一致。） 若网格项目在堆叠轴上溢出网格容器的内容盒， 那么堆叠范围可能会大于 网格容器的 内容盒。

6.1. 基线对齐

项目在基线对齐下， 若处于网格轴轨道内， 行为与常规网格容器一致， 且网格容器基线的确定方法与该轴上的常规网格容器一样。

基线对齐在堆叠轴不支持。 该轴上网格容器的首基线集 取自各轨道首项的最高对齐基线， 尾基线集取自每个轨道最后放项目的最低对齐基线。

名称: item-direction, item-track

该属性控制项目是按行还是按列放置， 以及轨道内是起始到末端还是末端到起始排列。

有两点争议： a) 应该叫什么； b) 是描述主放置方向还是轨道方位——即主轴由该属性定义，指主放置轴还是主轨道轴。 这对flex 布局和网格布局来说无异， 但对网格车道布局而言主放置方向与其轨道方向正交。[Issue #11480]

auto

根据布局模式计算为row或column：

  • 对flex 容器和网格容器，计算为row。

  • 对网格车道容器， 若grid-template-rows不是none 且grid-template-columns为none，计算为行轨道方向； 否则计算为列轨道方向。

row

面向轨道方案 表示放入行，即平行于行内轴的轨道或线。 项目在这些行内按起始到末端顺序填充。

面向流方案 表示行主项目放置， 即在行内轴上起始到末端依次放置， 在flex 布局中为 flex 行， 在网格车道布局中为列轨道。

column

面向轨道方案 表示放入列，即平行于块轴的轨道或线。 项目在这些列内按起始到末端顺序填充。

面向流方案 表示列主项目放置， 即在块轴上起始到末端依次放置， 在flex 布局中为 flex 列， 在网格车道布局中为行轨道。

row-reverse

同row， 但用末端到起始顺序。

column-reverse

同column， 但用末端到起始顺序。

项目交叉轴放置模式：item-cross/item-wrap

名称: item-wrap, item-cross
值: [ auto | nowrap | wrap ] || [ normal | reverse ] | wrap-reverse
初始值: auto
适用元素: flex 容器，网格容器，网格车道容器
继承性: no
百分比: N/A
计算值: 与声明值一致
规范顺序: 按语法
动画类型: 离散

用于控制在主轴正交的轴上的项目放置。

auto

在nowrap（flex 容器）和wrap（其他）之间自动切换。

nowrap

所有项目始终在主放置轴上排， 即便溢出空间。 在flex 布局时表现为单行 flex 容器； 网格布局中则必要时在主放置轴上生成隐式轨道。

wrap

主放置轴空间用尽时项目换行。 flex 布局会变成多行 flex 容器； 网格布局的自动放置算法则在主放置轴上无显式轨道时换到下一行/列。

normal

项目在主轨道轴正交轴上起始到末端排布。

在flex 布局和网格布局， 此项决定新轨道（flex 行/网格轨道）的生成方向。

在网格车道布局中， 面向轨道语法对应于相等高度轨道时选中哪条轨道； 面向流语法对应于项目在轨道中填充的方向

reverse

项目在主轨道轴正交轴上以末端到起始顺序排布。

wrap-reverse

等价于 wrap reverse。

注： 该值为与flex-wrap一致而存在。

此属性的解释和命名取决于item-direction/item-track的轴解释。[Issue #11480]

项目放置打包模式：item-pack 属性

名称: item-pack
值: normal | dense || balance
初始值: normal
适用元素: flex 容器，网格容器，网格车道容器
继承性: no
百分比: N/A
计算值: 与声明值一致
规范顺序: 按语法
动画类型: 离散

该属性以布局相关方式控制项目在轨道间如何分布。

normal

采用布局模式的默认打包策略。

dense

允许回溯以把项目放入前面被跳过的位置。 （由于前项太大而空着的空间会这样填补）

例如， 在flex 布局中允许把项目放进前面仍有剩余空隙的行。

balance

在flex 布局 中，表示各行内容平均分配 （包括最后一行）， 类似text-wrap-style: balance。

适用元素: 见各个长属性

变更

Level 2 以来的新增内容

自 Level 2 起新增了以下功能：

  • 新增网格车道布局。
  • 扩展了 repeat(auto-fill) 和 repeat(auto-fit)，允许其接受不定轨道尺寸函数。 (Issue 9321, Issue 12899)
  • 增加了item-flow属性及其长属性， 用于通用项目排序与放置控制。 参见附录A：通用布局项流控制：item-* 属性。 (Issue 11480)

最近变更

自 2025年9月17日工作草案以来进行了如下更改：

  • 定义了一个新的内部 display 类型 grid-lanes， 用于建立网格车道布局， 并更新了规范用词以保持一致。 (Issue 12022)

  • 从repeat()中移除了不必要的 auto-areas 值。 (Issue 10854)

  • 调整了 repeat(auto-fill) 和 repeat(auto-fit) 在不定轨道尺寸函数下的启发式处理方式。 (Issue 12899)

  • 将项目对轨道累计位置的贡献下限设为零。 (Issue 12918)

另见更早的变更。

一致性

文档约定

一致性要求由描述性断言和 RFC 2119 术语共同表达。规范性部分中的关键字 “MUST”、“MUST NOT”、“REQUIRED”、“SHALL”、“SHALL NOT”、“SHOULD”、“SHOULD NOT”、“RECOMMENDED”、“MAY”、“OPTIONAL” 按 RFC 2119 解释。 为便于阅读，规范中这些词不会全部大写。

本规范所有内容均为规范性，除非明确标注为非规范性、示例或注释。[RFC2119]

本规范中的示例以 “例如” 开头或用 class="example" 单独标示，如下：

这是一个说明性示例。

说明性注释以 “注意” 开头并用 class="note" 单独标示，如下：

注意，这是一个说明性注释。

通知性建议为规范性内容，采用 <strong class="advisement"> 特别样式标示，如下： UA 必须提供可访问替代方案。

    一致性类别

    本规范定义三类兼容对象：

    样式表
    CSS 样式表。
    渲染器
    解释样式表语义并渲染使用样式表的文档的UA。
    编辑工具
    编写样式表的UA。

    样式表符合本规范， 需其使用本模块定义语法的所有语句均有效， 且符合通用 CSS 语法和本模块中每项特性的语法。

    渲染器符合本规范， 需其除按相关规范解释样式表外，还应正确解析并渲染本规范定义的所有特性。 但因设备限制无法正确渲染文档不视为不兼容 UA。（如 UA 不要求在单色显示器上渲染颜色。）

    编辑工具符合本规范， 需其编写的样式表在语法上符合通用 CSS 语法和本模块所有特性语法， 并满足本模块样式表的所有其他一致性要求。

    部分实现

    为便于作者利用向前兼容的解析规则指定回退值， CSS 渲染器必须将所有不支持的 at-规则、属性、属性值、关键字及其他语法结构视为无效（并按需忽略）。 特别是，UA不得在对多值属性声明进行解析时，仅忽略不支持的组件值而保留支持值：任何值被视为无效时，CSS 要求整个声明被忽略。

    不稳定与专有特性的实现

    为避免与未来稳定 CSS 特性冲突， CSSWG 推荐遵循最佳实践实现不稳定和专有扩展特性。

    非实验性实现

    当规范进入候选推荐阶段（CR）， 可进行非实验性实现，且实现者应发布任何 CR 级特性的无前缀实现， 前提是可证明其按规范正确实现。

    为确保 CSS 在各实现间的互操作性， CSS 工作组要求非实验性 CSS 渲染器在发布无前缀实现前向 W3C 提交实现报告（如有需要还应提交用于该报告的测试用例）。 向 W3C 提交的测试用例需经 CSS 工作组审核与修正。

    有关提交测试用例和实现报告的更多信息，请参见 CSS 工作组网站 https://www.w3.org/Style/CSS/Test/。 如有问题请联系 public-css-testsuite@w3.org 邮件列表。
