1. 介绍

本节不是规范性内容。

网格布局是一个强大的 CSS 布局模型，能够控制盒子及其内容的大小和定位。 网格布局针对二维布局进行了优化： 这些布局在两个维度上都希望内容对齐。

尽管许多布局可以通过常规的网格布局表示， 但在两个轴上限制项目到网格中也使得 一些常见的网页布局无法实现。

本模块定义了一种布局系统，消除了这一限制， 使得项目可以仅在一个轴上放置到类似网格的轨道中， 同时在另一个轴上一个接一个地堆叠它们。 项目根据已放置项目的布局大小， 被放置到具有最多剩余空间的列（或行）中。 本模块还扩展了 CSS 网格，引入了这种新的网格项目 放置策略 和 CSS 盒子对齐，提供了新的对齐特性。

1.1. 背景和动机

砖石布局，有时也称为“瀑布布局”， 是一种常见的网页设计模式，多个项目—— 通常是图像或短文章摘要—— 以一种松散地类似于石砌的方式， 一个接一个地放置到列中。 与 多列布局不同， 在该布局中，内容垂直放置在第一列， 直到必须溢出到第二列，砖石布局 为每个新项目选择一列， 使得它通常更靠近布局的顶部，而不是稍后放置的项目。

Pinterest 搜索结果页面就是这一布局的示例：

在这里，每个项目的高度不同 （取决于内容和列的宽度）， 检查 DOM 可以揭示 （因为视觉内容本身没有排序的指示） 每个项目已被放置到当前高度最小的列中。

这种布局表面上看起来类似于多列布局； 但它的优势在于向下滚动 自然会导致布局中的“后面”项目 （也就是在搜索结果中相关性较低的项目）出现。

除非你预先知道每个项目的高度， 否则无法使用早期的 CSS 布局模型实现这种布局， 或使用 JavaScript 进行内容测量或放置。

在此语法模型中，砖石布局 集成到 网格布局 模型中， 重新使用所有 grid-* 属性。
独立网格选项

2. 砌体布局模型

砌体布局 将项目布局到预定义的轨道中，类似于 网格布局 在一个轴上 （称为 网格轴）， 但在另一个轴上像 弹性布局 一样自由流动 （称为 堆叠轴）。 类似于 网格布局，与 弹性布局 不同，砌体布局 的自动放置 将项目分布在轨道上，以保持这些轨道的长度尽可能相似。

网格 项目 的形成和 块化 与常规的 网格容器 完全相同。

(为清晰起见，网格项目 和 网格轨道 的 砌体容器 可以 被称为 砌体项目 和 砌体轨道.)

除非本规范另有说明，否则所有 CSS 属性的工作方式与常规的 网格容器 相同。 例如，顺序 可用于指定 项目的不同布局顺序。

注意： 支持子网格项目， 但子网格仅在 网格 轴 中发生； 有关详细信息，请参见 § 3.3 子网格。

一个 砌体容器 是一个其内容参与 砌体布局 的框。 如果其 堆叠轴 是 块轴，则 砌体容器 是一个 列砌体容器， 或者如果其 堆叠轴 是 行内轴，则是一个 行砌体容器。

2.1. 重新排序与可访问性

虽然 砖石布局 通常是向前发展的 （将下一个项目放在当前项目的下方或末尾， 匹配自然的“阅读顺序”）， 它可以在这两者之间以看似任意的方式切换。 在简单情况下，masonry-slack 属性可以帮助减少 由于在 块轴 布局自动放置项目时的小尺寸差异造成的回溯感。 但是当 自动放置 与 显式放置 或跨越项目混合时， 可能会发生一定程度的回溯。

例如，在以下标记示例中， 第四个项目是一个跨越器，它不适合 第一行剩余的空列。 它最终被放置在第一列， 这是它能适应的最高可用空间。 接下来的几个项目，跨度为1， 最终在空列中“在它上面”布局， 违反了自然的阅读顺序。 
<section class=masonry>
  <div class=item>1</div>
  <div class=item>2</div>
  <div class="item tall">3</div>
  <div class="item wide">4</div>
  <div class=item>5</div>
  <div class=item>6/div>
  <div class=item>7</div>
</section>
<style>
.masonry {
  display: masonry;
  masonry-tracks: repeat(5, auto);
}
.item { height: 50px; }
.item.wide { masonry-track: span 3; }
.item.tall { height: 90px; }
</style>
同样，显式放置在特定轨道中的项目可能会在它们后面留下空隙， 随后自动放置的项目可以在视觉上不按顺序放置。

作者应意识到这些可能性 并设计布局，以便尽量减少此类回溯， 以便更容易跟随焦点和阅读顺序。 或者，如果项目没有固有顺序， 使用 reading-flow 属性，允许用户代理重新排序项目 以便于阅读和线性导航。

或者，是否应该将重新排序 作为自动放置项目的默认行为？

减少回溯的技术包括：

  • 使用适当的值来设置 masonry-slack， 即值应足够大，以避免在大小相似的轨道之间产生多余的区别， 但又不能大到忽视有意义的差异。

  • 通过选择位置以使自动放置不会 违反自然的阅读顺序，来显式放置项目。

与 网格布局 和 弹性布局 作者可以使用 顺序 属性 来重新排序项目； 同样的警告适用。 请参见 CSS 网格布局 2 § 4 重新排序与可访问性 和 CSS 弹性布局 1 § 5.4.1 重新排序与可访问性。

这样是否合理，或者我们应该始终遵循 grid-auto-flow 轴 （这样会像 flexbox 一样填充，但具有明确大小的轨道）？

dense 打包是否适用于砌体布局？ 在砌体中，这要昂贵得多， 因为你必须在每个可能的空隙位置布局元素 以查看它是否足够短以适应； 网格只需处理一些整数。 [问题 #9326]

3. 砌体轨道规范

在网格轴中， 网格布局的全部功能可用于轨道规范：

  • 轨道大小、线名称和区域可以在砌体容器的网格轴上进行指定， 就像在网格布局中一样。

  • 显式网格和隐式 网格的形成方式与常规网格 容器相同。

  • 项目可以根据这些网格模板进行放置，就像在网格布局中一样。

然而， 自动放置的项目对所有轨道都贡献大小， 而不仅仅是它们最终放置的轨道； 见§ 3.5 网格轴轨道大小。

注意：这是因为自动放置的项目必须在 放置时进行布局， 这样每个轨道才能知道它是多么“满”， （因此哪条轨道应接收下一个自动放置的项目）； 因此，轨道本身必须已经有一个确定的大小， 以便项目在布局过程中知道它们的可用空间。

在网格轴中，砌体 容器的轨道是通过masonry-template-tracks、masonry-template-areas， 和masonry-auto-tracks建立的。

值： <'grid-template-columns'>

适用对象： 砌体 容器
继承： 否
百分比： 参考内容区域的相应维度
计算值： 按计算轨道列表中的每个项目的计算值类型

3.4. 轨道重复：repeat() 符号

本规范引入了新的关键字和特定于砖石的行为 用于repeat() 符号。

3.4.1. repeat(auto-areas)

用于repeat() 符号的新auto-areas值 代表了所需的重复次数，以使显式轨道的总数与 grid-template-areas / masonry-template-areas 值 在相应轴上生效。 如果列出了多个轨道用于重复， 最终的重复会根据需要截断 以产生正确数量的轨道。

注意：与auto-fill不同——它总是至少重复一次并始终 完全重复轨道列表——auto-areas的重复次数可以为零 （如果已经有足够的显式轨道）， 最终的重复可以是部分的。

如果grid-template-areas / masonry-template-areas为none， 此值的行为类似于auto-fill。

注意：此值同时适用于常规网格容器和砖石容器。

目前尚不清楚我们 是否真的需要这个值。 [问题 #10854]

3.4.2. repeat(auto-fit)

在砖石容器 （如常规网格容器）中，auto-fit的行为类似于auto-fill， 但空轨道被折叠。 然而，由于放置发生在轨道尺寸之后，砖石容器使用启发式 来确定轨道是否会被占用：

  • 所有被显式放置项目占用的轨道都被视为已占用。

  • 以所有自动放置项目的跨度之和为N， 所有未占用的轨道直到第N个轨道 被视为已占用。

所有通过auto-fit重复产生的轨道，并被该启发式认为未占用 都被假定为“空的”，并且被折叠。 折叠轨道不能接受自动放置项目的放置。

注意：可能会出现一个自动放置的项目 被放置在一个轨道中，当auto-fill被使用时 如果auto-fit被使用时会被折叠， 如果存在跨度大于1的自动放置项目与显式放置项目混合，导致间隙太小，无法容纳自动放置项目。

3.5. 网格轴轨道大小

轨道大小的工作方式与 CSS 网格 相同， 只是考虑到哪些项目贡献于内在大小时：

  • 所有明确放置在该轨道中的项目都会贡献，并且

  • 所有具有 自动网格位置 的项目都会贡献 （无论它们是否最终放置在该轨道中）。

例如，假设在 网格轴 中有两列，并且

  • 项目 A、B 和 C 没有明确的位置。

  • 项目 D 明确放置在第一列中。

在这种情况下，项目 A、B、C 和 D 都会贡献于第一列的大小， 而只有 A、B 和 C（而不是 D）会对第二列贡献。

对于具有 自动网格位置 的跨越项目， 假设它们放置在每个可能的起始位置，并相应地贡献。

例如，假设在 网格轴 中有 5 列， 中间的列固定大小为 100px，其他两列为 auto 尺寸。 在轨道大小的目的下， 一个跨越 2 个轨道且内在贡献为 220px 的项目 本质上被复制并假设存在：

  • 在网格线 1， 对前两条轨道贡献 110px。

  • 在网格线 2， 对第二条轨道贡献 120px。

  • 在网格线 3， 对第四条轨道贡献 120px。

  • 在网格线 4， 对第四和第五条轨道贡献 110px。

注意： 此算法确保每个轨道至少足够大 以容纳每个最终放置在其中的项目， 并且不会在放置和轨道大小之间创建依赖循环。 然而，根据大小的变化， 轨道可能会比必要的更大： 只有在所有项目明确放置在 网格轴 中或所有项目相同大小时 （或在跨越项目的情况下匹配该大小的倍数）才能保证精确适配。

3.5.1. 子网格项目贡献

在调整常规 网格容器 或 砌体 容器 的轨道时， 子砌体 对 具有 自动网格位置 的项目有特殊处理：

  • 任何此类项目都被放置到 子砌体 可以跨越的每个可能网格轨道中。 （如果子砌体有 确定网格位置，则只在跨越的轨道中； 如果它具有 自动网格位置，则在父网格中的所有轨道中。）

  • 任何此类项目都会在其可以假设放置的每个边缘处获得最大的边距/边框/内边距贡献。 如果该项目跨越整个子网格，则同时获得两者。 （见 CSS 网格布局 §9。）

4.4. 砌体布局和放置算法

对于 网格轴 中的每个轨道， 保持一个 运行位置， 初始化为零。 同时维护一个 自动放置光标， 初始指向第一条线。

对于 顺序修改文档顺序 中的每个项目：

  1. 如果该项目在 确定的网格位置 中， 则使用该放置。

    这是否也应更新放置光标？

    否则，使用以下子步骤解析其 网格轴 放置：

    1. 从 网格轴 中的第一条线开始， 查找如果项目被放置在此线，所跨的 网格轴 轨道的最大 运行位置， 并称该位置为 max_pos。
    2. 增加线号并重复步骤 2 直到项目不再适合网格。
    3. 让 possible lines 为导致最小 max_pos 的线，以及所有导致此 max_pos 在该 绑定阈值 内的线。
    4. 选择在 possible lines 中第一个大于或等于 自动放置光标 的线作为项目在 网格轴 中的位置； 如果没有这样的线，则选择第一个线。
    5. 更新 自动放置光标 指向项目的最后一条线。
  2. 在其 网格轴 轨道中放置项目 在它所跨轨道的最大 运行位置 处。
  3. 计算项目的 包含块 的大小，然后布局该项目。 将所跨 网格轴 轨道的 运行位置 设置为 max_pos + 外部大小 + 网格间隙。

注意： 此算法选择将项目尽可能高放置的轨道。 如果有绑定，它选择最早的轨道，在 最近放置的项目之后，如果可能的话， 确保它在有绑定的情况下“向前移动”。

如果以反向顺序放置项目（见 grid-auto-flow/masonry-flow）， 算法是类似的，但顺序相反。

9. 性能说明

通常，堆叠布局的性能应显著优于等效的常规（2轴）网格布局，尤其是当 堆叠轴 为 块轴 时，因为网格行的内在大小通常非常昂贵。在 网格轴 中的任何内在轨道大小也应更便宜，因为通常只有一部分项目对堆叠布局中的内在大小有贡献，而与所有跨越内在大小轨道的项目相反。拉伸项目在新大小下进行第二次布局（当实际发生变化时），因此如果有大量每个包含很多内容的拉伸项目，这可能是昂贵的。尤其是，除非它们很小或简单，否则应避免嵌套的拉伸堆叠布局。

作者可以通过选择大多数项目不进行拉伸来改善这种情况，例如指定 align-items:start，然后仅为少数项目选择 align-self:stretch，以让这些项目填充 堆叠轴。（这种性能分析来自于 Gecko 视角，但我怀疑对于其他布局引擎也是如此。）
