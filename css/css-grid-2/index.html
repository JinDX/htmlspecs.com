摘要

本 CSS 模块定义了一个二维的基于网格的布局系统，优化了用户界面设计。在网格布局模型中，网格容器的子元素可以定位在预定义的灵活或固定大小的布局网格中的任意插槽中。第 2 级通过添加“子网格”功能扩展了网格，以使嵌套网格参与其父网格的尺寸调整。

CSS 是一种用于描述结构化文档（例如 HTML 和 XML）在屏幕、纸张等上的渲染语言。

1. 引言

本节不具有规范性。

网格布局是 CSS 的一种布局模型，具有强大的能力来控制盒子及其内容的大小和位置。与 弹性盒布局不同，后者是单轴取向的，网格布局为二维布局进行了优化：即在两个维度上都希望对内容进行对齐的布局。

此外，由于能够在网格中明确定位项目，网格布局允许在视觉布局结构上进行显著的变化，而不需要相应的标记更改。通过结合 媒体查询 与控制网格容器及其子元素布局的 CSS 属性，作者可以根据设备形态、方向和可用空间的变化调整布局，同时在多个展示中保留内容更理想的语义结构。

虽然许多布局可以用网格或弹性盒表达，但它们各有其专长。网格强制二维对齐，采用自上而下的布局方式，允许项目显式重叠，并具有更强大的跨越功能。弹性盒侧重于单轴内的空间分配，采用更简单的自下而上的布局方式，可以使用基于内容尺寸的换行系统控制其次轴，并依赖底层的标记层次构建更复杂的布局。预计两者都将是 CSS 作者的宝贵且互补的工具。

网格第 2 级添加了 子网格 功能：子网格轴是将其网格线与元素父网格的线对齐，并通过与父网格的集成来派生其轨道大小的轴。

1.1. 背景与动机

随着网站从简单的文档演变为复杂的交互式应用程序，文档布局技术（如浮动）不一定适合应用程序布局。通过使用表格、JavaScript 或对浮动元素的精确测量，作者找到了实现所需布局的变通方法。适应可用空间的布局通常较为脆弱，并且在空间受限时会导致不合逻辑的行为。作为替代方案，许多 Web 应用程序的作者选择了固定布局，无法利用屏幕上可用渲染空间的变化。

网格布局的能力解决了这些问题。它为作者提供了一种机制，可以使用一组可预测的尺寸行为将可用的布局空间划分为列和行。然后，作者可以将其应用程序的构建块元素精确定位和调整大小到这些列和行的交叉点所定义的 网格区域 中。以下示例展示了网格布局的自适应能力，以及它如何实现内容与样式的更清晰分离。

1.1.1. 适应可用空间的布局

网格布局可以用于智能地调整网页内元素的大小。相邻的图例展示了一个包含五个主要组件的游戏布局：游戏标题、统计区域、游戏板、得分区域和控制区域。作者的意图是将游戏的空间划分为：

  • 统计区域始终位于游戏标题下方。
  • 游戏板位于统计信息和标题的右侧。
  • 游戏标题和游戏板的顶部始终对齐。
  • 当游戏达到其最小高度时，游戏板底部与统计区域底部对齐。在其他情况下，游戏板将扩展以利用所有可用空间。
  • 控制位于游戏板下方居中。
  • 得分区域的顶部与控制区域的顶部对齐。
  • 得分区域位于统计区域下方。
  • 得分区域与统计区域下方的控制对齐。

以下网格布局示例展示了作者如何声明性地实现所有的尺寸调整、放置和对齐规则。

/**
 * 通过声明 网格项目 的网格，定义每个 网格容器 的空间。
 */
#grid {
  /**
   * 两列：
   *  1. 第一列适应内容大小，
   *  2. 第二列接收剩余空间
   *     （但永远不小于棋盘或游戏控制的最小尺寸，它们占据了该列 [图4]）
   *
   * 三行：
   *  3. 第一行适应内容大小，
   *  4. 中间行接收剩余空间
   *     （但永远不小于棋盘或统计区域的最小高度）
   *  5. 最后一行适应内容大小。
   */
  display: grid;
  grid-template-columns:
    /* 1 */ auto
    /* 2 */ 1fr;
  grid-template-rows:
    /* 3 */ auto
    /* 4 */ 1fr
    /* 5 */ auto;
}

/* 使用 'grid-row' 和 'grid-column' 属性的坐标指定每个 网格项目 的位置和尺寸。
 */
#title    { grid-column: 1; grid-row: 1; }
#score    { grid-column: 1; grid-row: 3; }
#stats    { grid-column: 1; grid-row: 2; align-self: start; }
#board    { grid-column: 2; grid-row: 1 / span 2; }
#controls { grid-column: 2; grid-row: 3; justify-self: center; }

<div id="grid">
  <div id="title">游戏标题</div>
  <div id="score">得分</div>
  <div id="stats">统计信息</div>
  <div id="board">棋盘</div>
  <div id="controls">控制</div>
</div>

注意：有多种方法可以指定网格的结构并定位和调整网格项目的尺寸，每种方法都针对不同的场景进行了优化。

1.1.2. 源顺序独立性

延续前面的示例，作者还希望游戏能够适应不同的设备。同时，当以纵向或横向查看时，游戏应优化组件的布局（图 6 和图 7）。通过将网格布局与媒体查询结合，作者能够使用相同的语义标记，但可以独立于源顺序重新排列元素的布局，以在两种方向上实现所需的布局。

以下示例使用网格布局命名将被 网格项目 占据的空间的能力。这允许作者在网格定义发生变化时避免重写 网格项目 的规则。

@media (orientation: portrait) {
  #grid {
    display: grid;

    /* 网格的行、列和区域使用 grid-template-areas 属性视觉上定义。每个字符串代表一行，每个单词代表一个区域。字符串中的单词数量决定列数。请注意，每个字符串中的单词数量必须相同。 */
    grid-template-areas: "title stats"
                         "score stats"
                         "board board"
                         "ctrls ctrls";

    /* 可以使用 grid-template-columns 和 grid-template-rows 属性分配列和行的大小。 */
    grid-template-columns: auto 1fr; 
    grid-template-rows: auto auto 1fr auto; 
  }
}

@media (orientation: landscape) {
  #grid {
    display: grid;

    /* 同样，模板属性定义相同名称的区域，但这次以不同方式定位，以更好地适应横向方向。 */
    grid-template-areas: "title board"
                         "stats board"
                         "score ctrls";

    grid-template-columns: auto 1fr; 
    grid-template-rows: auto 1fr auto; 
  }
}

/* grid-area 属性将网格项目放置到网格的命名区域。 */
#title    { grid-area: title } 
#score    { grid-area: score } 
#stats    { grid-area: stats } 
#board    { grid-area: board } 
#controls { grid-area: ctrls } 

<div id="grid">
  <div id="title">游戏标题</div> 
  <div id="score">得分</div> 
  <div id="stats">统计信息</div> 
  <div id="board">棋盘</div> 
  <div id="controls">控制</div> 
</div>

注意：网格布局的重新排序功能有意仅影响视觉呈现，保留基于源顺序的语音顺序和导航。这允许作者在操作视觉展示时保留源顺序，以优化非 CSS 用户代理和线性模型（如语音和顺序导航）。

网格项目的布局和重新排序不应替代正确的源顺序，否则可能会破坏文档的可访问性。

2. 概述

本节不具有规范性。

网格布局通过使用 网格 控制其内容的布局：一组交叉的水平和垂直线为 网格容器 的内容创建一个尺寸和定位的坐标系统。网格布局功能包括：

  • 固定的、灵活的和基于内容的 轨道尺寸函数
  • 通过正（前向）和负（后向）数字网格坐标、命名的网格线和命名的网格区域进行 显式项目放置； 自动将项目放置到空区域，包括使用 重新排序的 order
  • 空间敏感的轨道重复以及为了容纳更多内容自动添加行或列
  • 使用 边距、槽 和 对齐属性 控制对齐和间距
  • 能够重叠内容并通过 使用 z-index 控制层次

网格容器 可以根据需要嵌套或与 弹性容器 混合使用以创建更复杂的布局。

2.1. 声明网格

网格的轨道（行 和 列）可以通过 显式网格 属性明确声明并调整大小，或者当项目被放置在 显式网格 之外时隐式创建。网格 简写及其子属性定义网格的参数。§ 7 定义网格

以下是一些网格声明示例：
  • 以下声明了一个具有四个命名区域的网格：H、A、B 和 F。第一列的大小适应其内容（auto），第二列占据剩余空间（1fr）。行默认为 auto（基于内容的）尺寸；最后一行的固定大小为 30px。 
    main {
  grid: "H    H "
        "A    B "
        "F    F " 30px
  /     auto 1fr;
}
  • 以下声明了一个网格，其高度为 100vh，并包含尽可能多的至少为 5em 的行。网格没有显式的列，而是随着内容的添加添加列，生成的列宽是均等的（1fr）。由于溢出到右边的内容不会打印，因此为打印添加的备用布局是添加行。 
    main {
  grid: repeat(auto-fill, 5em) / auto-flow 1fr;
  height: 100vh;
}
@media print {
  main {
    grid: auto-flow 1fr / repeat(auto-fill, 5em);
  }
}
  • 以下声明了一个网格，包含 5 个大小均等的列和三行，中间行占据所有剩余空间（并且至少足够容纳其内容）。 
    main {
  grid: auto 1fr auto / repeat(5, 1fr);
  min-height: 100vh;
}

2.2. 放置项目

网格容器的内容被组织为单个 网格项目（类似于 弹性项目），然后将它们分配到 网格 中的预定义 区域。它们可以通过 网格放置属性 使用坐标显式放置，或使用 自动放置 自动放置到空区域中。§ 8 放置网格项目

下面是一些使用 grid-area 简写的网格放置声明示例： 
grid-area: a;          /* 放置到命名网格区域“a” */
grid-area: auto;       /* 自动放置到下一个空区域 */
grid-area: 2 / 4;      /* 放置到第 2 行，第 4 列 */
grid-area: 1 / 3 / -1; /* 放置到第 3 列，跨越所有行 */
grid-area: header-start / sidebar-start / footer-end / sidebar-end;
                       /* 使用命名线进行放置 */

这些等价于以下 grid-row 和 grid-column 声明：

grid-row: a;                         grid-column: a;
grid-row: auto;                      grid-column: auto;
grid-row: 2;                         grid-column: 4;
grid-row: 1 / -1;                    grid-column: 3;
grid-row: header-start / footer-end; grid-column: sidebar-start / sidebar-end;

它们可以进一步分解为 grid-row-start / grid-row-end / grid-column-start / grid-column-end 长手声明，例如：

grid-area: a;
/* 等价于 grid-row-start: a; grid-column-start: a; grid-row-end: a; grid-column-end: a; */

grid-area: 1 / 3 / -1;
/* 等价于 grid-row-start: 1; grid-column-start: 3; grid-row-end: -1; grid-column-end: auto; */

2.3. 调整网格大小

一旦 网格项目 被 放置，网格轨道（行和列）的大小会根据内容的大小和网格定义中指定的可用空间进行计算。

生成的网格大小根据 align-content 和 justify-content 属性在 网格容器 内对齐。§ 11 对齐与间距

以下示例通过在所有列之间分配多余空间来对齐所有列，并在网格小于 100vh 时将其居中于 网格容器 中。 
main {
  grid: auto-flow 1fr / repeat(auto-fill, 5em);
  min-height: 100vh;
  justify-content: space-between;
  align-content: safe center;
}

最后，每个 网格项目 会根据其自身的 尺寸 [CSS2] 和 对齐属性 在其分配的 网格区域 内进行调整和对齐。[CSS-ALIGN-3]

3. 网格布局概念与术语

在 网格布局 中，网格容器 的内容通过定位和对齐放置到 网格 中。网格 是一组交叉的水平和垂直 网格线，将 网格容器 的空间划分为 网格区域，网格项目（表示 网格容器 的内容）可以放置到其中。有两组 网格线：一组定义了沿着 块轴 运行的 列，另一组定义了沿 内联轴 的 行。[CSS3-WRITING-MODES]

3.1. 网格线

网格线 是 网格 的水平和垂直分割线。 网格线 存在于列或行的两侧。 它们可以通过数字索引或作者指定的名称来引用。 网格项目 使用 网格线 来确定其在 网格 中的位置，使用 网格放置属性。

以下两个示例都创建了三列 网格线 和四行 网格线。

第一个示例演示了作者如何使用 网格线 编号来定位 网格项目：

#grid { 
  display: grid; 
  grid-template-columns: 150px 1fr; 
  grid-template-rows: 50px 1fr 50px; 
} 

#item1 { grid-column: 2; 
         grid-row-start: 1; grid-row-end: 4; }

第二个示例使用显式命名的 网格线：

/* 与上一个示例等价，但使用命名线 */ 
#grid { 
  display: grid; 
  grid-template-columns: 150px [item1-start] 1fr [item1-end]; 
  grid-template-rows: [item1-start] 50px 1fr 50px [item1-end]; 
} 

#item1 { 
  grid-column: item1-start / item1-end; 
  grid-row: item1-start / item1-end; 
}

3.2. 网格轨道与单元格

网格轨道 是 网格列 或 网格行 的通用术语，也就是说，它是两条相邻 网格线 之间的空间。每个 网格轨道 被分配一个大小调整函数，控制列或行的宽度或高度，因此确定其边界 网格线 之间的距离。相邻的 网格轨道 可以通过 槽 分隔，否则会紧密排列。

网格单元 是网格行和网格列的交叉点。它是定位 网格项目 时可以引用的最小单位。

在以下示例中有两列三行。 第一列固定为 150px。 第二列使用灵活的大小调整，这是网格中未分配空间的函数，因此随着 网格容器 宽度的变化而变化。 如果 网格容器 的使用宽度为 200px，则第二列宽 50px。 如果 网格容器 的使用宽度为 100px，则第二列为 0px，放置在该列中的任何内容将溢出 网格容器。 
#grid {
  display: grid;
  grid-template-columns: 150px 1fr;  /* 两列 */
  grid-template-rows: 50px 1fr 50px; /* 三行 */
}

3.3. 网格区域

网格区域 是用于布局一个或多个 网格项目 的逻辑空间。 网格区域 由一个或多个相邻的 网格单元 组成。 它由四条 网格线 限定，每条 网格区域 的每一侧都有一条网格线，并且参与它所交叉的 网格轨道 的大小调整。 可以通过 grid-template-areas 属性显式命名 网格区域，也可以通过其边界 网格线 隐式引用。 使用 网格放置属性 将 网格项目 分配到 网格区域。 

/* 使用模板语法 */
#grid  {
  display: grid;
  grid-template-areas: ". a"
                       "b a"
                       ". a";
  grid-template-columns: 150px 1fr;
  grid-template-rows: 50px 1fr 50px;
}

#item1 { grid-area: a }
#item2 { grid-area: b }
#item3 { grid-area: b }

/* 将项目 2 和 3 对齐在网格区域 "b" 中的不同位置。*/
/* 默认情况下，网格项目会拉伸以适应其网格区域，这些项目会相互重叠。 */
#item2 { align-self: start; }
#item3 { justify-self: end; align-self: end; }

网格项目 的 网格区域 形成了放置它的包含块。分配到同一 网格区域 的 网格项目 之间不会直接影响彼此的布局。 然而，间接地，占据 网格轨道 的网格项目 与 内在大小调整函数 相关，可以影响该轨道的大小（从而影响其边界 网格线 的位置），进而影响另一个 网格项目 的位置或大小。

3.4. 嵌套与子网格项目

网格项目 可以通过赋予 display: grid 来本身成为 网格容器。 通常情况下，嵌套网格 的内容布局将独立于它所在的 父网格 的布局。

然而，在某些情况下，可能需要多个 网格项目 的内容相互对齐。 嵌套网格 可以将其行和/或列的定义推迟到其父 网格容器，从而成为 子网格。 在这种情况下，子网格 的网格项目会参与父网格的大小调整，从而使两个网格的内容能够对齐。 详见 § 9 子网格。

子网格 通过 subgrid 关键字在 grid-template-rows 或 grid-template-columns 中建立，并且可以在任一轴或两者上 子网格化。没有 子网格化 轴的网格是 独立网格。

例如，假设我们有一个由一系列带有标签的输入框组成的表单： 
<ul> 
  <li><label>姓名:</label> <input name=fn> 
  <li><label>地址:</label> <input name=address> 
  <li><label>电话:</label> <input name=phone> 
</ul>

我们希望标签和输入框对齐，并且希望为每个列表项添加边框。 这可以通过子网格布局来实现：

ul {
  display: grid;
  grid: auto-flow / auto 1fr;
}
li {
  grid-column: span 2;
  display: grid;
  grid-template-columns: subgrid;
  border: solid;
}
label {
  grid-column: 1;
}
input {
  grid-column: 2;
}

4. 排序和可访问性

网格布局为作者提供了强大的文档重排功能。 然而，这并不能替代文档源的正确排序。 order 属性和 网格放置 不会 影响非视觉媒体中的排序（例如 语音）。 同样，视觉上重新排列网格项目不会影响顺序导航模式（如循环浏览链接，见 tabindex [HTML]）的默认遍历顺序。

作者必须仅将 order 和 网格放置属性 用于视觉内容的重排，而不是逻辑内容的重排。 使用这些功能进行逻辑重排的样式表是不符合规范的。

注意： 这是为了让非视觉媒体和非CSS的用户代理（通常线性展示内容）能够依赖逻辑的源顺序，而网格布局的放置和排序功能则用于调整视觉排列。（由于视觉感知是二维和非线性的，视觉顺序和阅读顺序并不总是相同的。）

许多网页在标记中有相似的结构，顶部是标题，底部是页脚，中间是内容区及一到两个附加列。 通常，理想的做法是内容首先出现在页面的源代码中，附加列则在其后。然而，这使得许多常见设计（如将附加列放在左侧，内容区域在右侧）难以实现。 多年来，这个问题通过多种方式得到了处理，通常在有两列附加列的情况下被称为“圣杯布局”。 网格布局使这个示例变得简单。以下是页面代码和预期布局的草图： 
<!DOCTYPE html> 
<header>...</header> 
<article>...</article> 
<nav>...</nav> 
<aside>...</aside> 
<footer>...</footer>

使用网格布局可以轻松实现此布局：

body { display: grid; 
       grid: "h h h" 
             "a b c" 
             "f f f"; 
       grid-template-columns: auto 1fr 20%; } 
article { grid-area: b; min-width: 12em;     } 
nav     { grid-area: a; /* 自动最小宽度 */ } 
aside   { grid-area: c; min-width: 12em;     }

一个额外的好处是，所有列默认情况下都将是等高的，主要内容将尽可能宽以填满屏幕。 此外，还可以结合媒体查询在窄屏上切换到全垂直布局：

@media all and (max-width: 60em) { 
  /* 屏幕太窄，无法支持三列 */ 
  main { display: block; } 
}

为了在所有展示模式下保留作者的预期顺序，创作工具——包括所见即所得编辑器以及基于网页的创作辅助工具——必须重新排序底层文档源，而不是使用 order 或 网格放置属性 来进行重排，除非作者明确表示底层文档顺序（决定语音和导航顺序）应与视觉顺序不同步。

例如，工具可能会提供拖放网格项目的功能以及根据屏幕尺寸范围处理媒体查询以生成备用布局的功能。

由于大多数时候，重排应影响所有屏幕范围以及导航和语音顺序，工具将通过同时重新排序DOM层来匹配拖放视觉排列的结果。 然而，在某些情况下，作者可能希望在不同的屏幕尺寸下呈现不同的视觉排列。 工具可以通过使用 网格放置属性和媒体查询提供此功能，但也应将最小屏幕尺寸的排列与底层DOM顺序绑定（因为这很可能是逻辑线性呈现顺序），同时在其他尺寸范围内使用网格放置属性重新排列视觉呈现。

此类工具将符合规范，而仅通过使用网格放置属性处理拖放网格重排的工具（无论其实现多么方便）将不符合规范。

5. 网格容器

5.1. 创建网格容器: grid 和 inline-grid display 值

名称： display
新值： grid | inline-grid
grid
该值会使元素生成一个 网格容器 框， 在 流布局 中，该框是 块级的。
inline-grid
该值会使元素生成一个 网格容器 框， 在 流布局 中，该框是 行内级的。

不是 子网格 的 网格容器 为其内容建立了一个 独立 的 网格格式化上下文。 这与为内容建立一个独立的 块级格式化上下文 相同， 只是使用的是网格布局而非块布局： 浮动不会侵入网格容器， 网格容器的边距不会与其内容的边距合并。 网格容器 的内容被布局为 网格， 网格线 形成每个 网格项目 包含块的边界。

与普通嵌套网格不同， 子网格 的内容参与其父 网格格式化上下文； 因此子网格不会建立 独立的格式化上下文。

网格容器不是块级容器， 因此在假定块布局的情况下设计的一些属性在网格布局中不适用。 特别是：

  • float 和 clear 对 网格项目 无效。 然而，float 属性仍然影响 display 在网格容器子元素上的计算值， 因为这发生在 确定 网格项目 之前。
  • vertical-align 对网格项目无效。
  • ::first-line 和 ::first-letter 伪元素不适用于 网格容器， 且 网格容器 不会为其祖先贡献首个格式化行或首个字母。

如果元素的指定 display 值为 inline-grid，且该元素被浮动或绝对定位， 则 display 的计算值为 grid。 因此，CSS 2.1 第9.7章中的表格会增加一行， "指定值" 列中的 inline-grid 和 "计算值" 列中的 grid。

5.2. 网格容器的大小

请参阅 [CSS-SIZING-3] 了解本节中的术语定义。

网格容器 的大小通过其参与的格式化上下文的规则来确定：

  • 作为 块级 框在 块级格式化上下文 中， 它的大小像一个 块级框，用于建立格式化上下文， 并且 auto 的 行内尺寸 按照非替换块级框计算。
  • 作为行内级框在 行内格式化上下文 中， 它的大小像一个原子行内级框（如行内块）。

在行内和块级格式化上下文中， 网格容器 的 auto 块大小 是其最大内容大小。

块布局规范可能需要定义这一点， 但尚未编写。

最大内容大小（最小内容大小） 网格容器 是 其轨道大小（包括排水沟）的总和， 当网格在 最大内容限制（最小内容限制）下进行调整时。

5.3. 可滚动网格的溢出

overflow 属性适用于 网格容器。

如同其包含在固有大小中一样（见上文）， 网格 也包含在 网格容器 的 可滚动溢出区域 中。

注意： 当 网格容器 是 滚动容器 时要注意其与内边距的交互： 额外的内边距被定义为添加到 可滚动溢出矩形， 以便启用可滚动内容的 place-content: end 对齐。 参见 CSS 溢出 3 §2.2 可滚动溢出

5.4. 限制大型网格

由于内存有限， UA 可能会将 隐式网格 的可能大小限制在 UA 定义的范围内 （应该可以容纳 [-10000, 10000] 范围内的线）， 放弃超出该限制的所有线条。 如果网格项放置在此限制之外， 其网格区域必须 被限制 在此限定网格内。

要 限制网格区域：

  • 如果 网格区域 会 跨越 限制网格的边界， 则其跨度被限制到限定网格的最后一条线。

  • 如果 网格区域 会被 完全放置在限定网格之外， 则其跨度必须被截断为 1， 并且该区域被重新放置在该侧网格的最后一个 网格轨道 中。

例如，如果 UA 仅支持每个维度最多 1000 个轨道的网格， 则以下放置属性： 
.grid-item {
  grid-row:500 / 1500;
  grid-column:2000 / 3000;
}

最终将等效于：

.grid-item {
  grid-row:500 / 1001;
  grid-column:1000 / 1001;
}

6. 网格项

宽泛地讲，网格项是 网格容器 的流内内容所代表的盒子。

网格容器 的每个流内子元素都会成为一个 网格项， 并且每个连续的子 文本运行 会被包装在 一个 匿名 块容器 中 成为网格项。 但是，如果整个子 文本运行 序列仅包含 空白字符（即可以被 white-space 属性影响的字符） 则它不会被渲染（就像其 文本节点 是 display:none 一样）。

网格项示例：

<div style="display: grid">

  <!-- 网格项：块级子元素 -->
  <div id="item1">块级元素</div>

  <!-- 网格项：浮动元素；浮动被忽略 -->
  <div id="item2" style="float: left;">浮动元素</div>

  <!-- 网格项：匿名块级盒子包裹行内内容 -->
  匿名项 3

  <!-- 网格项：行内子元素 -->
  <span>
    项目 4
    <!-- 网格项不会 拆分 块级元素 -->
    <q style="display: block" id="not-an-item">项目 4</q>
    项目 4
  </span>
</div>

注意： 元素之间的空白不会生成自己的网格项， 但元素之间的文本 确实 会被包装在匿名网格项中。

注意： 匿名项的盒子不可样式化， 因为没有元素可以分配样式规则。 然而，其内容会继承来自网格容器的样式（如字体设置）。

6.1. 网格项显示

除非是 子网格， 否则 网格项 会为其内容 建立独立的格式化上下文。 但是，网格项是 网格级别 盒子，而不是块级盒子： 它们参与其容器的 网格格式化上下文， 而不是块格式化上下文。

如果元素最近的祖先元素的 计算 display 值是 grid 或 inline-grid， 那么该元素的 display 值会被 块化。 （请参阅 CSS2.1§9.7 [CSS2] 和 CSS Display 3 §2.7 自动盒子类型转换 了解此类 display 值转换的详细信息。）

注意： 块化仍然会发生，即使 grid 或 inline-grid 元素没有生成 网格容器 盒子， 例如当它是 替换元素 或处于 display:none 子树中时。

注意： 一些 display 值通常会触发围绕原始盒子创建匿名盒子。 如果这样的盒子是一个 网格项， 它会先被块化，因此不会创建匿名盒子。 例如，两个连续的 网格项 使用 display: table-cell 会变成两个独立的 display: block 的 网格项， 而不会被包装成一个匿名表格。

6.2. 网格项尺寸调整

网格项 的尺寸由其 网格区域 定义的包含块来决定。

网格项的 尺寸计算 在特定维度上根据其 自对齐值 而有所不同：

normal

如果网格项是具有相关维度 替换元素，且具有 自然尺寸 （或具有 首选纵横比，且在其他维度上有自然尺寸）， 则网格项的尺寸将按照 align-self: start 进行尺寸调整（与 CSS 2 §10.3.4 块级替换元素的宽度计算规则 一致）。

否则， 如果网格项具有 首选纵横比， 则网格项将按照 块级盒子 的规则进行尺寸调整。

否则， 网格项将按照 stretch 的规则进行尺寸调整。

stretch

使用非替换盒子的 内联尺寸 计算规则 （定义于 CSS 2 §10.3.3 块级非替换元素的宽度计算规则）。

注意： 如果其尺寸在另一轴上也受到限制，这可能会扭曲具有 首选纵横比 的项目的纵横比。

所有其他值

将项目的尺寸设定为 fit-content。

以下信息表总结了网格项的自动尺寸调整行为：
网格项自动尺寸调整行为摘要
对齐方式 非替换元素尺寸 替换元素尺寸
normal 填充网格区域 使用 自然尺寸
stretch 填充网格区域 填充网格区域
start/center/etc. fit-content 尺寸调整（类似浮动） 使用 自然尺寸

注意： auto 值的 min-width 和 min-height 会影响相关轴上的轨道尺寸调整， 类似于它如何影响 flex 项目 的主尺寸。 参见 § 6.6 网格项的自动最小尺寸。

6.3. 重新排序的网格项：order 属性

order 属性同样适用于 网格项。 它影响它们的 自动布局 和 绘制顺序。

与重新排序弹性项一样， order 属性应仅在视觉顺序需要与语音和导航顺序不同步时使用； 否则，应重新排列底层文档源。 参见 CSS Flexbox 1 §5.4.1 重新排序和可访问性，在 [CSS-FLEXBOX-1] 中。

6.4. 网格项的边距和内边距

由于相邻的网格项独立包含在由其 网格区域 形成的包含块内， 相邻的 网格项 的边距不会 折叠。

网格项 上的百分比边距和内边距， 如同 块盒子 上的百分比边距和内边距一样， 是根据其 内联尺寸 解析的， 例如，左/右/上/下百分比都根据其 包含块 的宽度 在水平 书写模式 中解析。

自动边距扩展以吸收相应维度中的额外空间， 因此可用于对齐。 参见 § 11.2 使用自动边距对齐

6.5. Z轴顺序：z-index 属性

网格项 可以在它们被放置到相交的 网格区域 中时重叠， 甚至在由于负边距或定位而放置在非相交区域时也会重叠。 网格项 的绘制顺序与内联块 [CSS2] 完全相同， 只是使用 order 修改的文档顺序 代替了原始文档顺序， 并且 z-index 值除了 auto 之外， 即使 position 是 static（行为与 position 为 relative 的情况完全相同），也会创建堆叠上下文。 因此，z-index 属性可以轻松地用于控制网格项的Z轴顺序。

注意：定位在网格项外的后代仍参与该网格项建立的任何堆叠上下文。

下图显示了几个重叠的网格项， 使用隐式的源顺序和显式的 z-index 来控制它们的堆叠顺序。 
<style type="text/css">
#grid {
  display: grid;
  grid-template-columns: 1fr 1fr;
  grid-template-rows: 1fr 1fr
}
#A { grid-column: 1 / span 2; grid-row: 2; align-self: end; }
#B { grid-column: 1; grid-row: 1; z-index: 10; }
#C { grid-column: 2; grid-row: 1; align-self: start; margin-left: -20px; }
#D { grid-column: 2; grid-row: 2; justify-self: end; align-self: start; }
#E { grid-column: 1 / span 2; grid-row: 1 / span 2;
     z-index: 5; justify-self: center; align-self: center; }
</style>

  <div id="A">A</div>
  <div id="B">B</div>
  <div id="C">C</div>
  <div id="D">D</div>
  <div id="E">E</div>
</div>

6.6. 网格项的自动最小尺寸

注意：本节（以及规范的其余部分）中使用的许多尺寸术语在 CSS 内在和外在尺寸 [CSS-SIZING-3] 中有定义。

为了为 网格项 提供更合理的默认 最小尺寸， 如果满足以下所有条件，则其在给定轴上的 自动最小尺寸 的使用值为 基于内容的最小尺寸：

  • 它不是 滚动容器

  • 它至少跨越该轴上一个 轨道， 且该轨道的 最小轨道尺寸函数 是 auto

  • 如果它在该轴上跨越多个轨道，则这些轨道中没有 弹性轨道

否则，自动最小尺寸 通常为零。

在给定维度上的 网格项 的 基于内容的最小尺寸 是其 指定尺寸建议（如果存在）， 否则是其 转移尺寸建议（如果存在）， 否则为其 内容尺寸建议，见下文。 但是， 如果在给定维度上 网格项 仅跨越具有 固定 最大轨道尺寸函数 的 网格轨道， 那么其 指定尺寸建议 和 内容尺寸建议 （以及其从此维度输入到 转移尺寸建议 中的输入） 进一步被钳制为小于或等于其 网格区域 在该维度上的最大尺寸， 如这些 网格轨道 的 最大轨道尺寸函数 的总和再加上任何中间的 固定 槽隙。

在所有情况下，尺寸建议还受该轴上的 最大尺寸 钳制， 如果它是确定的。

注意： fit-content() 的参数并不会像 固定 最大轨道尺寸函数 那样钳制 基于内容的最小尺寸。

内容尺寸建议、指定尺寸建议 和 转移尺寸建议 在此计算中考虑了相关的最小/最大/首选尺寸属性， 以防止 基于内容的最小尺寸 干扰任何作者提供的限制， 并且这些定义如下：

指定尺寸建议
如果该项的相关轴上的 首选尺寸是 确定的， 则 指定尺寸建议 为该尺寸。 否则未定义。
转移尺寸建议
如果该项具有 首选纵横比，并且其在相对轴上的 首选尺寸是 确定的， 则 转移尺寸建议 为该尺寸 （如果它们是确定的，则通过纵横比钳制相对轴的 最小 和 最大尺寸）。 否则未定义。
内容尺寸建议
内容尺寸建议 为 相关轴上的 min-content 尺寸， 如果它具有 首选纵横比， 则通过任何确定的相对轴 最小 和 最大尺寸进行钳制。

为了计算盒子的内在尺寸 （例如，盒子的 min-content 尺寸）， 基于内容的最小尺寸 使得该轴上的盒子尺寸变得不确定 （即使例如其 宽度 属性指定了 确定的尺寸）。 请注意，这意味着根据此尺寸计算的百分比 将 表现为 auto。

尽管如此，尽管在某些情况下这可能需要额外的布局过程来重新解析百分比， 但这种值 （例如min-content、max-content 和 fit-content 值，在 [CSS-SIZING-3] 中定义） 并不会阻止项内的百分比尺寸的解析。

注意，虽然基于内容的最小尺寸通常是合适的， 并有助于防止内容重叠或溢出容器， 但在某些情况下它并不合适：

特别是，如果网格布局用于文档的主要内容区域， 最好设置一个明确的与字体相关的最小宽度，例如 min-width: 12em。 基于内容的最小宽度可能会导致大表格或大图像拉伸整个内容区域的尺寸，可能进入溢出区域， 从而使文本行不必要地变长，难以阅读。

还请注意，当内容项具有大量内容时使用基于内容的尺寸， 布局引擎必须遍历所有这些内容才能找到其最小尺寸， 而如果作者设置了明确的最小值，则无需这样做。 （然而，对于内容较少的项， 这种遍历是微不足道的，因此不是性能问题。）

7. 定义网格

7.1. 显式网格

三个属性 grid-template-rows、grid-template-columns 和 grid-template-areas 共同定义了 显式网格，即 网格容器的显式网格轨道。 最终网格可能会由于放置在 显式网格外的 网格项而变大； 在这种情况下将创建隐式轨道， 这些隐式轨道的大小由 grid-auto-rows 和 grid-auto-columns 属性来控制。

显式网格的大小由以下两者中较大的值决定： 由 grid-template-areas 定义的行/列数 和由 grid-template-rows/grid-template-columns 定义的行/列数。 任何由 grid-template-areas 定义但未由 grid-template-rows/grid-template-columns 定义大小的行/列将从 grid-auto-rows/grid-auto-columns 属性中获取其大小。 如果这些属性没有定义任何 显式轨道， 那么 显式网格仍然在每个轴上包含一个 网格线。

在 网格放置属性中的数字索引是从 显式网格的边缘开始计数的。 正索引从 开始边开始计数 （从最开始的 开始线开始，从1开始）， 负索引从 结束边开始计数 （从最末尾的 结束线开始，从-1开始）。

grid 和 grid-template 属性是 简写，可同时设置所有三个 显式网格属性（grid-template-rows、grid-template-columns 和 grid-template-areas）。 grid 简写还会重置控制 隐式网格的属性， 而 grid-template 属性则不影响这些属性。

7.2. 显式轨道大小：grid-template-rows 和 grid-template-columns 属性

名称: grid-template-columns, grid-template-rows
值: none | <track-list> | <auto-track-list> | subgrid <line-name-list>?
初始值: none
适用对象: 网格容器
是否继承: 否
百分比: 参照内容区域的对应尺寸
计算值: 关键词 none 或 计算轨道列表
规范顺序: 按语法
动画类型: 如果列表长度匹配，则根据 计算轨道列表 中的每项值类型；否则为离散

这些属性指定了网格的线名称和轨道大小函数，并以空格分隔的形式作为一个 轨道列表。 grid-template-columns 属性用于指定网格列的 轨道列表， 而 grid-template-rows 属性用于指定网格行的 轨道列表。

这些值的含义如下：

none
表示该属性不创建任何 显式网格轨道 （虽然仍可通过 grid-template-areas 创建显式网格轨道）。

注意：在没有显式网格的情况下，任何行/列将被隐式生成， 它们的大小将由 grid-auto-rows 和 grid-auto-columns 属性决定。

<track-list> | <auto-track-list>
指定轨道列表， 由多个 轨道大小函数 和 线名称 组成。 每个 轨道大小函数 可以是长度、网格容器大小的百分比、占据该列或行内容的测量值或网格中可用空间的分数。 还可以使用 minmax() 语法指定范围，该语法可结合前述任何机制指定列或行的最小和最大轨道大小函数。
subgrid <line-name-list>?
subgrid 值表示网格将采用其父网格在该轴上的跨越部分。 列或行的大小将从 父网格 的定义中继承，而 子网格的项将参与与父网格共享的任何轨道的内在大小计算。 简言之，子网格 提供了通过嵌套元素传递网格参数的能力，并将基于内容的大小信息传递回它们的 父网格。

<line-name-list> 参数允许为与 父网格 共享的网格线进行本地命名： 如果给定了 <line-name-list>， 则为子网格的显式网格的每条线分配指定的 <line-names>， 从第1条线开始。超出的线名称将被忽略。

如果没有 父网格， 则该值等同于初始值 none， 此时网格容器不是 子网格。

未子网格化的轴称为 独立轴。

轨道列表的语法如下：

<track-list>          = [ <line-names>? [ <track-size> | <track-repeat> ] ]+ <line-names>?
<auto-track-list>     = [ <line-names>? [ <fixed-size> | <fixed-repeat> ] ]* <line-names>? <auto-repeat>
                        [ <line-names>? [ <fixed-size> | <fixed-repeat> ] ]* <line-names>?
<explicit-track-list> = [ <line-names>? <track-size> ]+ <line-names>?

<line-name-list>      = [ <line-names> | <name-repeat> ]+
<track-size>          = <track-breadth> | minmax( <inflexible-breadth> , <track-breadth> ) | fit-content( <length-percentage> )
<fixed-size>          = <fixed-breadth> | minmax( <fixed-breadth> , <track-breadth> ) | minmax( <inflexible-breadth> , <fixed-breadth> )
<track-breadth>       = <length-percentage> | <flex> | min-content | max-content | auto
<inflexible-breadth>  = <length-percentage> | min-content | max-content | auto
<fixed-breadth>       = <length-percentage>
<line-names>          = '[' <custom-ident>* ']'

这些组件值定义如下……

7.2.1. 轨道尺寸

<length-percentage>
一个非负长度或百分比，由CSS3 Values定义。[CSS-VALUES-3]

<percentage>值相对于 内部行内 尺寸的网格容器中的列网格轨道， 和块 尺寸的网格容器在行网格轨道中。 如果网格容器的大小取决于其轨道的大小， 则<percentage>必须被视为auto， 以计算网格容器的固有尺寸，然后根据该结果的网格容器尺寸进行解析 以布局网格及其项目。

<flex>
一个具有单位fr的非负维度，指定轨道的flex系数。 每个<flex>大小的轨道根据其flex系数占据剩余空间的比例。 例如，给定一个轨道列表1fr 2fr， 这些轨道将分别占据剩余空间的1/3和2/3。 参见§ 7.2.4 灵活长度：fr单位了解更多详细信息。

注意：如果flex系数的总和小于1， 它们将只占据相应比例的剩余空间， 而不是扩展到填满整个空间。

当出现在minmax()符号之外时， 意味着一个自动的最小值（即"minmax(auto, <flex>)"）。

minmax(min, max)
定义一个大于或等于min且小于或等于max的尺寸范围。 如果max小于min， 则max将被min取下限（本质上得到minmax(min, min)）。 作为最大值，<flex>值设置了轨道的flex系数； 它作为最小值是无效的。

注意：本规范的未来版本可能允许<flex>最小值， 并将更新轨道尺寸算法以正确处理这一点。

auto
作为最大值：表示占据网格轨道的网格项的最大最大内容贡献； 然而，与max-content不同， 允许通过align-content和justify-content属性扩展轨道。

作为最小值：表示占据网格轨道的网格项的最大最小尺寸 （由min-width/min-height指定）。 （这通常最初与min-content最小值相等，但不总是如此——参见§ 6.6 网格项的自动最小尺寸。）

当出现在minmax()符号之外时： 等效于minmax(auto, auto)， 表示上述的最小值和最大值之间的范围。 （在最基本的情况下，这类似于minmax(min-content, max-content)， 但具有额外的能力。）

max-content
表示占据网格轨道的网格项的最大最大内容贡献。
min-content
表示占据网格轨道的网格项的最大最小内容贡献。
fit-content( <length-percentage> )
表示公式 max(minimum, min(limit, max-content))， 其中minimum表示一个auto最小值 （通常，但不总是等于min-content最小值）， 而limit是传递给轨道尺寸函数的 fit-content()的参数。 这本质上是将minmax(auto, max-content)与minmax(auto, limit)中的较小者进行计算。
给定以下grid-template-columns声明： 
grid-template-columns: 100px 1fr max-content minmax(min-content, 1fr);

创建了五条网格线：

  1. 位于网格容器的起始边缘。
  2. 距离网格容器起始边缘100px。
  3. 距离上一条线的距离等于自由空间（网格容器宽度减去非弹性 网格轨道的宽度）的二分之一。
  4. 距离上一条线的距离等于位于这两条线之间的任意网格项的最大尺寸。
  5. 距离上一条线的距离至少与位于这些线之间的任意网格项的最大最小尺寸相同， 但不大于自由空间的另一半。

如果非弹性尺寸 (100px、max-content和min-content) 的总和大于网格容器的宽度， 最后一条网格线将距离网格容器起始边缘的距离等于它们的总和（1fr尺寸都解析为0）。 如果总和小于网格容器的宽度， 最后一条网格线将正好位于网格容器的终止边缘。 只要<flex>值存在于网格轨道尺寸中时，这通常是正确的。

其他有效的网格轨道定义示例： 
/* 有效轨道定义示例 */
grid-template-rows: 1fr minmax(min-content, 1fr);
grid-template-rows: 10px repeat(2, 1fr auto minmax(30%, 1fr));
grid-template-rows: calc(4em - 5px);

注意：网格的尺寸不仅仅是轨道尺寸的总和， 因为行间距、列间距以及justify-content和align-content可以在轨道之间添加额外的空间。

7.2.2. 命名网格线：[<custom-ident>*] 语法

虽然网格线总是可以通过它们的数字索引引用，线名称可以使网格放置属性 更容易理解和维护。线名称可以通过grid-template-rows和grid-template-columns属性进行显式分配， 或者通过命名的网格区域，使用grid-template-areas属性隐式分配。

例如，以下代码为网格中的所有线赋予了有意义的名称。 注意，有些线有多个名称。 
#grid {
  display: grid;
  grid-template-columns: [first nav-start] 150px [main-start] 1fr [last];
  grid-template-rows: [first header-start] 50px [main-start] 1fr [footer-start] 50px [last];
}

线名称不能是span或auto， 即<custom-ident>在<line-names> 生产中排除关键词span和auto。

7.2.3. 重复行和列：repeat() 语法

repeat() 表示轨道列表的重复片段， 允许将显示重复模式的大量列或行以更紧凑的形式编写。

此示例显示了编写相同网格定义的两种等效方式。 两种声明都生成了四个“主”列，每个宽度为250px， 被10px的“间隔”列包围。 
grid-template-columns: 10px [col-start] 250px [col-end]
                       10px [col-start] 250px [col-end]
                       10px [col-start] 250px [col-end]
                       10px [col-start] 250px [col-end] 10px;
/* 与上面相同，但更容易编写 */
grid-template-columns: repeat(4, 10px [col-start] 250px [col-end]) 10px;
7.2.3.1. repeat() 的语法

repeat() 语法的大致通用形式为：

repeat( [ <integer [1,∞]> | auto-fill | auto-fit ] , <track-list> )

第一个参数指定重复的次数。 第二个参数是一个轨道列表， 该列表将被重复指定次数。 但是，有一些限制：

  • repeat() 语法不能嵌套。

  • 自动重复（auto-fill 或 auto-fit） 不能与固有或灵活尺寸结合使用。

因此，repeat() 语法的精确定义有多种形式：

<track-repeat> = repeat( [ <integer [1,∞]> ] , [ <line-names>? <track-size> ]+ <line-names>? )
<auto-repeat>  = repeat( [ auto-fill | auto-fit ] , [ <line-names>? <fixed-size> ]+ <line-names>? )
<fixed-repeat> = repeat( [ <integer [1,∞]> ] , [ <line-names>? <fixed-size> ]+ <line-names>? )
<name-repeat> = repeat( [ <integer [1,∞]> | auto-fill ], <line-names>+)

  • <track-repeat> 变体可以表示任何<track-size>的重复， 但仅限于固定次数的重复。

  • <auto-repeat> 变体可以自动重复以填充 一个空间， 但需要确定的轨道尺寸，以便计算重复的次数。 它在轨道列表中只能出现一次， 但相同的轨道列表中还可以包含<fixed-repeat>。

  • <name-repeat> 变体用于向子网格添加线名称。 它只能与subgrid关键字一起使用， 并且不能指定轨道尺寸，只能指定线名称。

如果repeat() 函数不是<name-repeat>的情况下在两个<line-names> 之间相邻出现， 则名称列表会合并。 例如，repeat(2, [a] 1fr [b]) 相当于[a] 1fr [b a] 1fr [b]。

7.2.3.2. 填充重复：auto-fill 和 auto-fit 重复

在子网格化的轴上， auto-fill关键字在每个<line-name-list>中只能出现一次， 并重复足够多的次数，以便名称列表与子网格的指定网格跨度匹配（如果跨度已满足，则默认为0）。

否则，在独立轴上， 当auto-fill作为重复次数时， 如果网格容器在相关轴上具有确定的尺寸或最大尺寸， 则重复的次数是不会导致网格溢出其内容框的最大正整数 网格容器（将每个轨道视为其最大轨道尺寸函数，如果该函数是确定的，否则将其视为其最小轨道尺寸函数， 如果两者都是确定的，则将最大轨道尺寸函数取下限为最小轨道尺寸函数， 并考虑间隙）； 如果任何重复次数会导致溢出， 则重复一次。 否则，如果网格容器在相关轴上具有确定的最小尺寸， 则重复的次数是满足该最小要求的最小正整数。 否则，指定的轨道列表仅重复一次。

例如，以下代码将根据窗口宽度创建尽可能多的25字符列。 如果有任何剩余空间， 它将分布在25字符列之间。 
body {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(25ch, 1fr));
}

auto-fit关键字的行为与auto-fill相同， 除了在网格项放置后，任何空的重复轨道将被折叠。 空轨道是指没有放入或跨越它的流内网格项的轨道。 （这可能会导致所有轨道被折叠， 如果它们都是空的。）

被折叠的轨道被视为具有固定的轨道尺寸函数为0px， 并且它两侧的沟槽——包括通过分布对齐分配的任何空间——折叠。

为了找到独立轴上的自动重复轨道的数量， UA必须将轨道尺寸下限为UA指定的值， 以避免被零除。 建议此下限为1px。

7.2.3.3. repeat()的插值/组合

如果两个repeat() 语法的第一个参数（重复次数）相同 且第二个参数（轨道列表）的轨道数量相同， 它们通过 通过其计算轨道列表的每个组件的计算值组合（就像组合顶层轨道列表一样）。 否则，它们会以离散方式组合。

7.2.4. 灵活长度：fr 单位

灵活长度或<flex>是具有fr单位的维度， 它表示剩余空间在网格容器中的一部分。 使用fr单位的轨道称为灵活轨道，因为它们会响应剩余空间进行调整，类似于弹性项目如何在弹性容器中填充空间。

剩余空间的分配发生在所有非灵活的轨道尺寸函数达到其最大值之后。 这些行或列的总尺寸从可用空间中减去，得出剩余空间， 然后按其灵活系数的比例在灵活尺寸的行和列之间分配。

每个列或行在剩余空间中的份额可以通过以下公式计算： <flex> * <leftover space> / <sum of all flex factors>。

<flex> 值在 0fr 和 1fr 之间有一些特殊行为： 当灵活系数总和小于1时， 它们将占用少于100%的剩余空间。

轨道的<flex>值 实际上是对某些剩余空间比例的请求， 其中1fr意味着“100%的剩余空间”； 然后如果该轴上的轨道总共请求超过100%， 请求会重新平衡以保持相同的比例但刚好使用100%的空间。 然而，如果轨道请求的少于全部空间 （例如三个轨道每个为.25fr）， 那么它们将获得各自请求的确切数量 （每个获得剩余空间的25%， 最后的25%保持未填充）。 请参见§ 12.7 扩展灵活轨道了解剩余空间如何分配的具体细节。

此模式对于当fr值接近零时的连续行为是必要的 （这意味着轨道希望占用无剩余空间）。 否则，1fr轨道将占据所有剩余空间； 但0.1fr轨道也会占据所有空间， 0.01fr轨道也一样， 直到值小到足以下溢为零， 轨道突然不占用任何剩余空间。 有了这种行为， 轨道随着其灵活系数缩小到1fr以下会逐渐占用更少的剩余空间， 平滑过渡到零时不占用任何剩余空间。

除非特别需要这种“部分填充”行为， 否则作者应坚持使用值 ≥ 1； 例如，使用1fr和2fr通常比使用.33fr和.67fr更好， 因为它们在添加或移除轨道时更可能表现为预期的效果。

当可用空间是无限的 （当网格容器的宽度或高度为不确定的时）， 灵活尺寸的网格轨道会根据其内容进行调整，同时保持其各自的比例。 每个灵活尺寸的网格轨道的使用尺寸通过 确定每个灵活尺寸的网格轨道的最大内容尺寸并将该尺寸除以相应的灵活系数来确定“假设的1fr尺寸”。 这些值中的最大值被用作解析的1fr长度（灵活分数）， 然后将其乘以每个网格轨道的灵活系数以确定其最终尺寸。

注意：<flex>值不是<长度> （它们也不与<长度>兼容， 如某些<百分比>值）， 因此它们不能在calc()表达式中表示或与其他单位类型组合使用。

7.2.5. 轨道列表的计算值

计算的轨道列表是独立轴的一个列表，在线名称集和轨道部分之间交替， 第一个和最后一个项目是线名称集。

线名称集是表示线名称的标识符的（可能为空的）有序集。

轨道部分可以是：

  • minmax()函数表示单个轨道的大小， 其中每个<length-percentage>已计算

  • repeat()函数表示重复的轨道列表部分， 其中其<整数>已计算 其<轨道列表>表示为计算的轨道列表

子网格化轴的计算轨道列表是 subgrid 关键字后跟表示该轴中每条线的列表的线名称集。

7.2.6. 轨道列表的解析值

grid-template-rows和grid-template-columns属性是解析值的特殊情况属性。[CSSOM]

7.2.6.1. 独立轨道列表的解析值

当一个元素生成一个网格容器框时， 其解析值为grid-template-rows或grid-template-columns属性 在独立轴上的解析值为 使用值， 序列化方式如下：

  • 每个轨道单独列出， 无论是隐式还是显式创建， 不使用repeat()语法。
  • 每个轨道尺寸都以像素长度给出， 无论使用何种尺寸函数。
  • 相邻的线名称合并为一个括号集。
上述列表的第一个要点 意味着隐式轨道也被序列化为 grid-template-rows/等的一部分， 尽管作者实际上不能在这些属性中指定隐式轨道尺寸！ 因此，grid-template-rows和grid-template-columns的值 可能无法正确地循环往复： 
const s = getComputedStyle(gridEl);

// 这将把它们转换为显式行，
// 可能会改变网格项的定位，
// 并改变网格的整体大小！

这是一个早期实现的意外特性， 它未经深思熟虑就泄露到了后来的实现中。 我们打算从规范中删除它， 但要等到我们定义了一个CSSOM API来获取隐式轨道的信息之后， 因为目前这是获取该信息的唯一方法， 而且有许多页面依赖于此。

否则，（例如当元素具有display: none或不是一个网格容器时） 解析值只是计算值。

</div>

7.2.6.2. 子网格轨道列表的解析值

当一个元素生成一个网格容器框，且它是一个子网格时， 其解析值在grid-template-rows和 grid-template-columns属性中 表示使用的列数， 序列化为subgrid关键字， 后跟表示其每条线的列表， 这些线作为所有线名称的线名称集 显式定义在子网格上（不包括从父网格继承的名称）， 且不使用repeat()语法。

例如， 当应用于一个具有子网格，且使用grid-column: span 4时， 以下每个grid-template-columns 指定值将成为相应的解析值： 
specified: subgrid [a] repeat(auto-fill, [b]) [c]
resolved:  subgrid [a] [b] [b] [b] [c]

specified: subgrid [a] [a] [a] [a] repeat(auto-fill, [b]) [c] [c]
resolved:  subgrid [a] [a] [a] [a] [c]

specified: subgrid [] [a]
resolved:  subgrid [] [a] [] [] []

specified: subgrid [a] [b] [c] [d] [e] [f]
resolved:  subgrid [a] [b] [c] [d] [e]

注意： 这种做法违反了通用的“最短等效序列化”原则， 因为序列化了空的尾部线名称集， 因为这些尾部的线名称集提供了关于子网格跨越了多少轨道的潜在有用信息。

名称: grid-template-areas
值: none | <string>+
初始值: none
适用于: 网格容器
是否继承: 否
百分比: n/a
计算值: 关键字none或字符串值列表

此属性指定了命名网格区域， 这些区域不与任何特定的网格项关联， 但可以从网格放置属性中引用。 grid-template-areas属性的语法还提供了网格结构的可视化， 使网格容器的整体布局更易于理解。

值的含义如下：

none
表示该属性未定义任何命名网格区域， 同样也未定义任何显式网格轨道， （尽管仍然可以通过grid-template-columns或grid-template-rows创建显式网格轨道）。

注意： 在没有显式网格的情况下，任何行/列将被隐式生成， 其大小将由grid-auto-rows和grid-auto-columns属性决定。

<string>+
为grid-template-areas属性中列出的每个单独的字符串创建一行， 并为字符串中的每个单元创建一列， 解析如下：

将字符串标记为以下令牌列表， 使用最长匹配语义：

  • 一系列名称代码点， 表示一个命名单元令牌，其名称由这些代码点组成。
  • 一系列一个或多个"." (U+002E FULL STOP)， 表示一个空单元令牌。
  • 一系列空白， 表示什么都不表示 （不要生成令牌）。
  • 一系列其他字符， 表示一个垃圾令牌。

注意： 这些规则可以生成与<ident>语法不匹配的单元名称， 例如 "1st 2nd 3rd"， 在其他属性中通过名称引用这些区域时需要转义， 例如grid-row: \31st;以引用名为1st的区域。

  • 空单元令牌表示网格容器中的未命名区域。
  • 命名单元令牌创建一个具有相同名称的命名网格区域。 在行内和行间的多个命名单元令牌 创建一个跨越相应网格单元的单个命名网格区域。
  • 垃圾令牌是语法错误， 并使声明无效。

所有字符串必须定义相同数量的单元令牌 (命名单元令牌和/或空单元令牌)， 并且至少有一个单元令牌， 否则声明无效。 如果一个命名网格区域跨越多个网格单元， 但这些单元没有形成一个单一的填充矩形， 则声明无效。

注意： 非矩形或不连续的区域可能在此模块的未来版本中被允许。

7.3.1. 模板字符串的序列化

当序列化指定值 或计算值时，<string>值 的grid-template-areas属性中， 每个空单元令牌都序列化为一个 "."（U+002E FULL STOP）， 并且连续的单元令牌之间用一个空格（U+0020 SPACE）分隔， 其他所有空白将被省略。

7.3.2. 隐式分配的线名称

grid-template-areas属性从模板中的命名网格区域生成隐式分配的线名称。 对于每个命名网格区域foo，生成四个隐式分配的线名称： 两个名为foo-start，用于命名行起始和列起始线， 以及两个名为foo-end，用于命名行结束和列结束线。

这些隐式分配的线名称与任何其他线名称行为相同， 除了它们不会出现在grid-template-rows/grid-template-columns的值中。 即使定义了具有相同名称的显式分配的线名称， 隐式分配的线名称也只是具有相同名称的更多线。

7.3.3. 隐式命名区域

由于命名网格区域由其生成的隐式分配的线名称引用， 显式添加同一形式的命名线（foo-start/foo-end） 实际上会创建一个命名网格区域。 这样的隐式命名区域不会出现在grid-template-areas的值中， 但仍然可以由网格放置属性引用。

7.5. 隐式网格

grid-template-rows、grid-template-columns 和 grid-template-areas 属性定义了构成 显式网格 的固定数量的轨道。当 网格项目 位于这些边界之外时，网格容器 会通过向 网格 添加 隐式网格轨道生成 隐式网格线。这些网格线与 显式网格 一起构成 隐式网格。grid-auto-rows 和 grid-auto-columns 属性定义了这些 隐式网格轨道 的大小，以及由 grid-template-areas 创建但未通过 grid-template-rows 或 grid-template-columns 显式定义大小的任何 显式网格轨道。

grid-auto-flow 属性控制了没有显式位置的 网格项目 的自动布局。一旦 显式网格 填满（或没有 显式网格），自动布局还会导致生成 隐式网格轨道。

grid 简写 属性可以同时声明 隐式网格属性（grid-auto-flow、grid-auto-rows 和 grid-auto-columns），以及 显式网格属性。

7.7. 自动布局：grid-auto-flow属性

名称： grid-auto-flow
值： [ row | column ] || dense
初始值： row
适用于： 网格容器
是否继承： 否
百分比： 不适用
计算值： 指定的关键字
规范顺序： 按语法规则
动画类型： 离散

网格项如果未显式放置，将通过自动布局算法自动放置在网格容器中的空白处。 grid-auto-flow属性控制自动布局算法的工作方式， 具体规定了自动放置的项目如何在网格中流动。 有关自动布局算法如何工作的详细信息，请参阅§ 8.5 网格项布局算法。

row
自动布局算法逐行放置项目， 按需添加新行。 如果未提供row或column，则默认为row。
column
自动布局算法逐列放置项目， 按需添加新列。
dense
如果指定了该值，自动布局算法将使用“密集”排列算法， 该算法会尝试在后续较小项目出现时填补网格中先前留下的空隙。 这可能会导致项目的顺序混乱， 以便填补较大项目留下的空隙。

如果省略此值，则会使用“稀疏”算法， 即布局算法在放置项目时只会在网格中向“前”移动， 从不回退填补空隙。 这确保了所有自动放置的项目按顺序排列， 即使这会留下可由后续项目填补的空隙。

注意：预计此模块的未来版本将添加一个值，该值将自动定位的项目流入一个单一的“默认”单元格。

自动布局按网格项在修改的文档顺序进行。

名称： grid
值： <'grid-template'> | <'grid-template-rows'> / [ auto-flow && dense? ] <'grid-auto-columns'>? | [ auto-flow && dense? ] <'grid-auto-rows'>? / <'grid-template-columns'>
初始值： none
适用于： 网格容器
是否继承： 见个别属性
百分比： 见个别属性
计算值： 见个别属性
规范顺序： 按语法规则

grid 属性是一个简写，可同时设置显式网格属性（grid-template-rows、grid-template-columns 和 grid-template-areas）和隐式网格属性（grid-auto-rows、grid-auto-columns 和 grid-auto-flow）， 一次性声明。 （它不会重置槽属性。） 其语法与grid-template的语法匹配， 加上用于定义自动流网格的附加语法：

<'grid-template-rows'> / [ auto-flow && dense? ] <'grid-auto-columns'>?
[ auto-flow && dense? ] <'grid-auto-rows'>? / <'grid-template-columns'>
设置自动流， 通过在一个轴上显式设置轨道 （分别设置grid-template-rows或grid-template-columns，并将另一轴设置为none）， 并指定如何在另一轴上自动重复轨道 （分别设置grid-auto-rows或grid-auto-columns，并将另一轴设置为auto）。grid-auto-flow 也相应设置为 row或column， 若指定了 dense，则包括它。

所有其他grid的子属性都将重置为其初始值。

注意： 注意，你只能在一个 grid 声明中指定显式或隐式网格属性。 没有指定的子属性将按照正常的简写行为设置为其初始值。

除了接受 grid-template 简写语法以设置显式网格， grid 简写还可以轻松设置自动格式化网格的参数。 例如，grid: auto-flow 1fr / 100px; 相当于 
grid-template: none / 100px;
grid-auto-flow: row;
grid-auto-rows: 1fr;
grid-auto-columns: auto;

类似地，grid: none / auto-flow 1fr 相当于

grid-template: none;
grid-auto-flow: column;
grid-auto-rows: auto;
grid-auto-columns: 1fr;

8. 放置网格项目

每个网格项目都与一个网格区域相关联， 该区域是由相邻网格单元组成的矩形集， 网格项目占据这些单元。 此网格区域定义了网格项目的包含块， 其中对齐属性（justify-self和align-self）决定其实际位置。 网格项目占据的单元也影响网格行和列的大小， 详见§ 12 网格大小调整。

网格项目的网格区域在网格中的位置由其放置决定， 包括网格位置和网格跨度：

网格位置
网格项目在每个轴上的网格中的位置。 网格位置可以是确定的（明确指定的） 或自动的（由自动放置确定的）。
网格跨度
网格轨道中网格项目在每个轴上占据的数量。 如果子网格轴的*-start或*-end值 是auto， 则网格项目有一个不确定的跨度， 并且由其<line-name-list>决定。 否则， 网格项目的网格跨度是确定的， 如果在该轴上无法确定，则默认为1。

网格放置属性——包括长属性 grid-row-start、grid-row-end、grid-column-start、grid-column-end， 以及它们的简写属性 grid-row、grid-column 和 grid-area—— 允许作者通过提供以下六项信息中的任何一项（或不提供）来指定网格项目的放置：

行 列
起点 行起点线 列起点线
终点 行终点线 列终点线
跨度 行跨度 列跨度

在给定的维度中，任意两个确定值起点、终点和跨度将隐含确定第三个值。

下表总结了网格位置或跨度何时为确定或自动的条件：

位置 跨度
确定 至少指定了一条线 显式、隐式或默认的跨度。
自动 未明确指定任何线 不适用

8.1. 网格放置的常见模式

本节为说明性内容。

网格放置属性的长属性被组织为三个简写：

grid-area
grid-column grid-row
grid-column-start grid-column-end grid-row-start grid-row-end

8.1.1. 命名区域

可以通过在grid-area中指定区域名称， 将项目放置在命名网格区域中（如通过grid-template-areas模板生成的命名区域）：

article {
  grid-area: main;
  /* 将项目放置在命名为“main”的区域中。 */
}

项目也可以部分与命名网格区域对齐， 而其他边缘则与某些其他线对齐：

.one {
  grid-row-start: main;
  /* 将行起点与命名区域“main”的起始边对齐。 */
}

8.1.2. 数字索引和跨度

网格项目可以通过数字进行定位和调整大小， 这对于脚本驱动的布局特别有帮助：

.two {
  grid-row: 2;    /* 将项目放置在第二行。 */
  grid-column: 3; /* 将项目放置在第三列。 */
  /* 等价于 grid-area: 2 / 3; */
}

默认情况下，网格项目的跨度为1。 不同的跨度可以显式指定：

.three {
  grid-row: 2 / span 5;
  /* 从第2行开始，
     跨度为5行（在第7行结束）。 */
}

.four {
  grid-row: span 5 / 7;
  /* 在第7行结束，
     向上跨度为5行（从第2行开始）。 */
}

注意： 请注意，网格索引是相对书写模式的。 例如，在像阿拉伯语这样的从右到左的语言中， 第一列是最右边的列。

8.1.3. 命名线和跨度

线可以通过其名称引用， 而不是按数字计数：

.five {
  grid-column: first / middle;
  /* 从线“first”到线“middle”的跨度。 */
}

注意： 请注意，如果命名网格区域与线名同名， 放置算法将优先使用命名网格区域的线。

如果有多个相同名称的线， 它们将有效地建立一组命名的网格线， 可以通过名称过滤放置进行独占索引：

.six {
  grid-row: text 5 / text 7;
  /* 跨度为从第5到第7条名为“text”的线。 */
  grid-row: text 5 / span text 2;
  /* 同上——从第5条名为“text”的线开始，
     然后跨过另外两条“text”线，到达第7条。 */
}

8.1.4. 自动放置

网格项目可以自动放置到下一个可用的空网格单元中， 如果没有剩余空间，网格会自动扩展。

.eight {
  grid-area: auto; /* 默认值 */
}

例如，这可以用于在目录网站上以网格形式列出多个促销商品。

自动放置可以与显式跨度结合使用， 如果项目需要占用多个单元格：

.nine {
  grid-area: span 2 / span 3;
  /* 自动放置的项目，占据两行和三列。 */
}

自动放置算法是通过添加行或列来控制网格的扩展， 这一行为由grid-auto-flow属性控制。

注意： 默认情况下，自动放置算法线性地遍历网格而不回溯； 如果必须跳过一些空格以放置更大的项目， 它不会返回填补这些空隙。 要更改此行为， 可以在grid-auto-flow中指定dense关键字。

8.2. 网格项目放置与源代码顺序

“能力越大，责任越大。”

网格放置属性的功能允许内容在网格中自由安排和重新排序， 从而视觉呈现可以与基础文档的源代码顺序大相径庭。 这些功能让作者在根据不同设备和展示模式定制渲染时拥有极大的自由度， 例如使用媒体查询。 然而它们不能替代正确的源代码顺序。

正确的源代码顺序对于语音浏览器、顺序导航（如键盘导航）以及非CSS用户代理（例如搜索引擎、触觉浏览器等）非常重要。 网格放置仅影响视觉呈现！ 这使得作者能够优化文档源代码以适应非CSS/非视觉交互模式， 并使用网格放置技术进一步操控视觉呈现， 同时保持源代码顺序不变。

8.3. 基于线条的放置：grid-row-start、grid-column-start、grid-row-end 和 grid-column-end 属性

名称: grid-row-start, grid-column-start, grid-row-end, grid-column-end
值: <grid-line>
初始值: auto
适用于: 网格项目 和 绝对定位的框，其包含块为 网格容器
是否继承: 否
百分比: n/a
计算值: 指定的关键词、标识符和/或整数
规范顺序: 按语法
动画类型: 离散 
<grid-line> =
  auto |
  <custom-ident> |
  [ <integer> && <custom-ident>? ] |
  [ span && [ <integer> || <custom-ident> ] ]

grid-row-start、grid-column-start、grid-row-end 和 grid-column-end 属性 确定了网格项目在网格中的大小和位置，通过提供一条线、一个跨度或没有（自动） 来指定其网格放置， 从而指定其内联起始、块起始、 内联结束 和 块结束 边界。

值的含义如下：

<custom-ident>
首先尝试将网格区域的边缘与命名网格区域对齐： 如果有网格线的线名称为 <custom-ident>-start（对于 grid-*-start） / <custom-ident>-end（对于 grid-*-end）， 则该线贡献给网格项目的放置。

注意：命名网格区域会自动生成此类形式的隐式分配的线名， 因此指定 grid-row-start: foo 将选择该 命名网格区域的起始边缘（除非之前明确指定了另一个名为 foo-start 的线）。

否则， 视为已与 <custom-ident> 一起指定了整数 1。

<integer> && <custom-ident>?
提供网格项目网格线的第 N 条线以进行放置。 如果给出了负整数， 则反向计数， 从显式网格的结束边缘开始。

如果给定名称为 <custom-ident>， 则仅计算具有该名称的线。 如果没有足够多具有该名称的线， 则所有隐式网格线假定为具有该名称，以便找到此位置。

整数值为零使声明无效。

span && [ <integer> || <custom-ident> ]
提供网格项目网格跨度，使得其网格区域的对应边缘距离其相对边缘的 N 条线。 例如，grid-column-end: span 2 表示从 grid-column-start 线开始的第二条线。

如果给定名称为 <custom-ident>， 则仅计算具有该名称的线。 如果没有足够多具有该名称的线， 则所有隐式网格线在对应搜索方向的显式网格一侧假定为具有该名称，以便计算此跨度。

例如，给定以下声明： 
.grid { grid-template-columns: 100px; }
.griditem { grid-column: span foo / 4; }

该网格容器具有一个显式网格，包含两条网格线， 编号为 1 和 2。 该网格项目 的列结束边缘指定在第4条线上， 因此在隐式网格的结束方向生成了两条线。

其列起始边缘必须是从该方向上第一个 "foo" 线。 但网格中没有 "foo" 线， 因此唯一的可能是隐式网格中的线。 第3条线不是候选者，因为它位于显式网格的结束方向， 而 grid-column-start 的跨度要求它在起始方向上搜索。 因此，唯一的选择是让 隐式网格 在显式网格的起始方向上生成一条线。

如果省略了整数值，则默认为 1。 负整数或零是无效的。

auto
此属性不对网格项目的放置作任何贡献，表示自动放置或默认跨度为1。 （参见上面的放置网格项目第8节。）

在以上所有生产中， <custom-ident> 还排除了关键词 span 和 auto。

给定一个单行的8列网格和以下9个命名线： 
1  2  3  4  5  6  7  8  9
+--+--+--+--+--+--+--+--+
|  |  |  |  |  |  |  |  |
A  B  C  A  B  C  A  B  C
|  |  |  |  |  |  |  |  |
+--+--+--+--+--+--+--+--+

以下声明将网格项目放置在所指示的线之间：

grid-column-start: 4; grid-column-end: auto;
/* 线4到线5 */

grid-column-start: auto; grid-column-end: 6;
/* 线5到线6 */

grid-column-start: C; grid-column-end: C -1;
/* 线3到线9 */

grid-column-start: C; grid-column-end: span C;
/* 线3到线6 */

grid-column-start: span C; grid-column-end: C -1;
/* 线6到线9 */

8.3.1. 网格放置冲突处理

如果放置的网格项目包含两条线， 且起始线比结束线更接近终点， 则交换两条线。 如果起始线与结束线相等， 则移除结束线。

如果放置包含两个跨度， 则移除由结束 网格放置属性贡献的跨度。

如果放置仅包含命名线的跨度， 则将其替换为跨度1。

grid-row 和 grid-column 属性是 grid-row-start/grid-row-end 和 grid-column-start/grid-column-end的简写。

如果指定了两个<grid-line>值， 则 grid-row-start/grid-column-start 设置为斜杠前的值， grid-row-end/grid-column-end 设置为斜杠后的值。

当省略第二个值时， 如果第一个值是<custom-ident>， 则 grid-row-end/grid-column-end 也设置为该<custom-ident>; 否则，它设置为 auto。

名称: grid-area
值: <grid-line> [ / <grid-line> ]{0,3}
初始值: auto
适用对象: 网格项目 和 其包含块为网格容器的绝对定位框

百分比: N/A
计算值: 参见各个属性

grid-area 属性是 简写属性，用于同时设置 grid-row-start、grid-column-start、grid-row-end 和 grid-column-end。

如果指定了四个<grid-line>值，grid-row-start 设置为第一个值，grid-column-start 设置为第二个值，grid-row-end 设置为第三个值，grid-column-end 设置为第四个值。

当省略 grid-column-end 时，若 grid-column-start 是 <custom-ident>，则 grid-column-end 也会设置为该 <custom-ident>；否则，它会被设置为 auto。

8.5. 网格项目放置算法

以下 网格项目放置算法 解析 自动位置 为 确定的位置，确保每个 网格项目都有明确的 网格区域 进行布局。（网格跨度无需特殊解析；如果未显式指定，默认为 1。）

注意：此算法可能会在 显式网格中没有空间放置自动定位的 网格项目时，生成新的行或列至 隐式网格中。

每个 网格单元（无论在 显式网格还是 隐式网格中）都可以是被占用或未被占用。当网格单元被一个具有 确定的网格位置的 网格项目的 网格区域覆盖时，该单元格是 被占用；否则为 未被占用。单元格的 占用/未占用 状态可在此算法执行过程中改变。

为清晰起见，假设此算法使用 grid-auto-flow 设置为 row。如果设置为 column，则算法中的行与列、内联与块方向等概念需交换。

注意：自动放置算法按 修改过的文档顺序处理 网格项目，而不是它们的原始文档顺序。

  1. 生成匿名网格项目，如 § 6 网格项目中所述。（匿名的 网格项目总是自动放置的，因为它们的框不能指定任何 网格放置属性。）

  2. 定位所有非自动定位的项目。

  3. 处理固定在特定行上的项目。

    对于每个具有 确定的行位置 的 网格项目，按修改过的文档顺序处理：

    “稀疏”排列（默认行为）

    设置项目的 放置 的列开始线为最早（最小的正索引）行，以确保此项目的 网格区域 不会与任何 已占用的网格单元重叠，且在本步骤中先前放置的其他项目之后。

    “密集”排列（指定 dense）

    设置项目的 放置 的列开始线为最早（最小的正索引）行，以确保此项目的 网格区域 不会与任何 已占用的网格单元重叠。

  4. 确定隐式网格中的列。

    在 隐式网格中创建列：

    1. 从 显式网格 中的列开始。

    2. 在所有具有 确定的列位置 的项目（显式定位的项目、前一步中定位的项目，以及尚未定位但具有确定列的项目）中，必要时在 隐式网格 的开始和结束处添加列以容纳这些项目。

    3. 如果所有没有 确定列位置 的项目中的最大 列跨度 大于 隐式网格 的宽度，向隐式网格末尾添加列以容纳该 列跨度。

    例如，以下样式片段： 
    #grid {
  display: grid;
  grid-template-columns: repeat(5, 100px);
  grid-auto-flow: row;
}
#grid-item {
  grid-column: 4 / span 3;
}

    所需列数为 6。显式网格 提供了 5 列（来自 grid-template-columns），线编号为 1 到 6，但 #grid-item 的列位置意味着它结束于第 7 条线，因此需要在 隐式网格 的末尾添加一列。

  5. 定位剩余的网格项目。

    自动放置游标 定义了网格中的当前“插入点”，由一对行和列的 网格线指定。最初，自动放置游标 设置为 隐式网格中的最开始的行和列线。

    grid-auto-flow 值决定如何放置这些项目：

    “稀疏”排列（默认行为）

    对于所有未被先前步骤放置的 网格项目，按修改过的文档顺序处理：

    如果项目具有 确定的列位置：

    1. 将 游标的列位置设置为 网格项目 的列开始线。如果该位置小于游标之前的列位置，则将行位置加 1。

    2. 增加 游标的行位置，直到找到一个位置，在该位置上 网格项目 不与任何 已占用的网格单元重叠（必要时在 隐式网格中创建新行）。

    3. 将项目的行开始线设置为 游标的行位置，并根据该位置的跨度设置项目的行结束线。

    如果项目在两个轴上均有 自动网格位置：

    1. 增加 自动放置游标的列位置，直到该项目的 网格区域不与任何 已占用的网格单元重叠，或者游标的列位置加上项目的列跨度溢出隐式网格中的列数（如本算法前面确定的）。

    2. 如果在前一步找到一个不重叠的位置，则将项目的行开始线和列开始线设置为 游标的位置。否则，将游标的行位置加 1（必要时在 隐式网格中创建新行），并将列位置重置为 隐式网格中的最开始列线，然后返回前一步。

    “密集”排列（指定 dense）

    对于所有未被先前步骤放置的 网格项目，按修改过的文档顺序处理：

    如果项目具有 确定的列位置：

    1. 将游标的行位置设置为 隐式网格中最开始的行线。将游标的列位置设置为项目的列开始线。

    2. 增加 自动放置游标的行位置，直到找到一个位置，在该位置上 网格项目 不与任何 已占用的网格单元重叠（必要时在 隐式网格中创建新行）。

    3. 将项目的行开始线索引设置为游标的行位置。（隐式设置项目的行结束线根据其跨度。）

    如果项目在两个轴上均有 自动网格位置：

    1. 将游标的行位置和列位置设置为 隐式网格中的最开始行和列线。

    2. 增加游标的列位置，直到该项目的网格区域不与任何已占用的网格单元重叠，或游标的列位置加上项目的列跨度溢出隐式网格中的列数。

    3. 如果在前一步找到不重叠的位置，则将项目的行开始线和列开始线设置为游标的位置。否则，将游标的行位置加 1，并将列位置重置为隐式网格中的最开始列线，然后返回前一步。

9. 子网格

一个子网格的行为与普通的网格容器相同，除了以下几点：

  • 放置子网格会在它的子网格轨道与其在父网格中的跨度之间建立关联。 这些在子网格与其父网格之间共享的网格线形成子网格的显式网格， 且其轨道大小由父网格控制。
  • 在子网格的子网格维度中的显式轨道数量 始终与其父网格中的网格轨道的跨度相对应：
    • 如果子网格在子网格维度中的 网格跨度是确定的， 那么每个子网格维度中的显式轨道数量将根据其该维度中的网格跨度 决定（不管其grid-template-*属性）。
    • 如果它有不确定的网格跨度，那么其网格跨度将根据该轴上由grid-template-*属性指定的显式轨道数量确定， 最低值为1。
  • 网格放置属性 的 子网格中的网格项目及其使用的行号 都限定在 子网格所覆盖的行， 与 子网格之外的行一致地被排除在其显式网格之外。 例如，数字索引从 子网格的第一行开始计数，而不是从父网格的第一行开始。

    行编号和放置规则 遵循 子网格自己的 书写模式， 就像它们对嵌套的独立网格一样。

  • 由于 子网格 可以在内容放置之前放置， 子网格的线 自动接收父网格相应线上的显式指定的线名称。

    这些名称是对 线名称的补充， 它们本地指定于 子网格。

  • 当 子网格与其父网格中的命名网格区域重叠时， 该命名网格区域是由grid-template-areas属性声明创建的，隐式指定的线名称将被分配 以表示父网格的命名网格区域在子网格中的位置。

    注意： 如果命名网格区域仅部分重叠子网格， 则其隐式指定的线名称将被分配 给 子网格的第一和/或最后一行，以便一个 命名网格区域存在， 表示该子网格部分重叠的区域； 因此 线名称分配在子网格中的位置可能并不总是与父网格中的线名称完全一致。

    这些名称也是对 本地指定于子网格的线名称的补充。

    在以下示例中， 祖父网格拥有4列，既有显式的线名称， 也有由grid-template-areas生成的隐式线名称： 
    <style type="css">
  .outer {
    display: grid;
    grid-template-columns:
      [outer-edge] 20px [main-start] 1fr [center] 1fr max-content [main-end];
    grid-template-areas:
      "gutter info info photos";
  }
  .middle {
    grid-column: main-start / main-end;
    display: grid;
    grid: subgrid / subgrid;
  }
  .inner {
    grid-column: center / -1;
    display: grid;
    grid: subgrid / subgrid;
  }
</style>

<div class="outer">
  <div class="middle">
    <div class="inner">&hellip;</div>
  </div>
</div>

    经过所有类型的名称解析后， 每个网格的名称将是：

    .outer = [outer-edge gutter-start] [gutter-end info-start main-start] [center] [info-end photos-start] [main-end photos-end]
.middle = [info-start main-start] [center] [info-end photos-start] [main-end photos-end]
.inner = [center info-start] [info-end photos-start] [main-end photos-end]

    请注意，所有显式指定的线名称一直继承到 .inner， 但隐式指定的线名称是 基于每个子网格与原始命名网格区域的重叠计算出来的。

  • 子网格在子网格轴维度中没有任何隐式网格轨道。 当显式网格没有足够的网格线时， 通常会使用假设的隐式网格线来解决放置问题； 然而，每个网格项目的网格区域会根据 子网格的显式网格进行限制 （使用与在过大的网格中限制放置相同的过程）。
    例如，如果一个span 1子网格有一个网格项目，其属性为grid-column: 2 / span 3;， 那么该项目将被限制在子网格的第一个（也是唯一的）轨道中。
  • 子网格本身作为其父网格中的普通网格项目布局， 但在子网格轴维度中，行为则完全为空以进行轨道尺寸调整。
  • 子网格自己的网格项目参与其在子网格轴维度的尺寸调整， 并且在这些维度上对其进行对齐。

    在此过程中，子网格的每个边缘的边距、内边距和边框的总和 被应用为对这些边缘的项目的额外层（可能是负的）边距。 这种额外的“边距”会通过多个层次的子网格累积。

    例如，如果我们有一个3×3的网格，轨道如下： 
    #parent-grid { grid-template-columns: 300px auto 300px; }

    如果一个子网格覆盖了最后两个轨道， 那么它的前两列对应于父网格的最后两列， 并且定位到这些轨道中的任何项目都会参与父网格的尺寸调整。 具体而言，定位在子网格第一个轨道中的项目 会影响父网格中间轨道的自动调整。

    #subgrid { grid-column: 2 / span 2; } /* 覆盖父网格的第2和第3轨道 */
#subgrid > :first-child { grid-column: 1; } /* 子网格的第1轨道，父网格的第2轨道 */

    如果子网格有边距/边框/内边距， 那么这些边距/边框/内边距的大小也会影响尺寸。 例如，如果子网格有100px的内边距：

    #subgrid { padding: 100px; }

    那么在子网格的第一个轨道中的网格项目 会像它有额外的100px的顶部、左侧和底部边距一样行动， 影响父网格轨道的尺寸 以及网格项目自己的位置。

    与此同时， 子网格的槽距（行间距/列间距）与其父网格的 槽距之间差值的一半 被应用为边缘物体的额外层（可能是负的）边距。 这种额外的“边距”也通过多个层次的子网格累积。 如果normal值被使用，表示子网格的槽距与其父网格相同， 即差异为零。

    注意： 最终结果是父网格的轨道将按指定的尺寸进行调整， 而子网格的槽距将在视觉上与父网格的 槽距对齐。

    例如， 假设我们有一个300px宽的外部网格， 间距为50px， 列的规格为100px 1fr。 一个跨越两个轨道的子网格将会……

    ……如果其列间距是normal（或50px）：
    • 其左列中的网格项目 以没有任何特殊调整的方式进行尺寸调整和布局 （并贡献其尺寸用于父网格的尺寸计算）， 因此拉伸至100px宽 并保持与子网格的左边缘对齐。
    • 其右列中的网格项目 以没有任何特殊调整的方式进行尺寸调整和布局 （并贡献其尺寸用于父网格的尺寸计算）， 因此拉伸至150px宽， 并保持与子网格的右边缘对齐。
    • 有效的视觉槽距 在物体之间为50px，与父网格完全匹配。

    ……如果其列间距是0：
    • 其左列中的网格项目 以仿佛它有-25px右边距的方式进行尺寸调整和布局， 因此拉伸至125px宽 并保持与子网格的左边缘对齐。
    • 其右列中的网格项目 以仿佛它有-25px左边距的方式进行尺寸调整和布局， 因此拉伸至175px宽， 并保持与子网格的右边缘对齐。
    • 有效的视觉槽距在物体之间为零，正如其列间距所指定的那样。

    ……如果其列间距是25px：
    • 其左列中的网格项目 以仿佛它有-12.5px右边距的方式进行尺寸调整和布局， 因此拉伸至112.5px宽 并保持与子网格的左边缘对齐。
    • 其右列中的网格项目 以仿佛它有-12.5px左边距的方式进行尺寸调整和布局， 因此拉伸至162.5px宽， 并保持与子网格的右边缘对齐。
    • 有效的视觉槽距在物体之间为25px，正如其列间距所指定的那样。

    ……如果其列间距是75px：
    • 其左列中的网格项目 以仿佛它有12.5px右边距的方式进行尺寸调整和布局， 因此拉伸至87.5px宽 并保持与子网格的左边缘对齐。
    • 其右列中的网格项目 以仿佛它有12.5px左边距的方式进行尺寸调整和布局， 因此拉伸至137.5px宽， 并保持与子网格的右边缘对齐。
    • 有效的视觉槽距在物体之间为75px，正如其列间距所指定的那样。

  • 对于非空的子网格的每个边缘， 为了考虑子网格在该边缘的边距/边框/内边距， 为每个跨距大小贡献了一个假设项到轨道尺寸算法中， 这些项目跨越了靠近子网格边缘的占用轨道。 这些项目的尺寸取自每个跨距大小的最大项目， 并额外增加了子网格在该边缘（或两边，如果它恰好是两侧最极端的项目且也是跨距最小的项目）的边距/边框/内边距。 同样，假设项的跨距取自相同的真实项目的跨距， 并通过它与相关的子网格边缘之间的空轨道数量进行膨胀。

    注意： 如果靠近子网格边缘的轨道包含真实项目，这一步骤可以被跳过， 因为这些项目已经考虑了子网格的边距/边框/内边距，如上所述。

    例如，在以下子网格布局中： 
    5px auto auto 5px
.   aaaaaaaaa .
.   bbbb cccc .

    假设子网格项目a、b和c占据了它们相应的网格区域， 子网格的内边距为25px， 两个假设网格项目将被贡献到轨道尺寸算法中， 以处理子网格的行内起始内边距： 一个项目的尺寸为b加25px， 跨越前两列； 另一个项目的尺寸为a加25px， 跨越前三列。

    如果只有项目a存在， 它将同时 跨越最接近的占用列的两侧， 并且是每一侧跨距最小的项目， 那么它贡献的假设项的尺寸将被增加50px， 并跨越所有四列。

  • 子网格在其 子网格轴维度中总是被拉伸： 它的align-self和justify-self属性将被忽略， 任何指定的宽度/高度约束也将被忽略。
  • 在布局上，子网格的网格总是与其父网格的相应部分对齐； 它的align-content和justify-content属性在子网格轴维度中也将被忽略。
  • overflow属性确实适用于子网格， 因此子网格溢出的内容可以通过滚动来查看。 （注意：滚动行为不会影响布局。）

10. 绝对定位

10.1. 以网格容器为包含块

如果一个绝对定位的元素的包含块是由网格容器生成的， 那么包含块对应于其网格区域，该区域由其网格放置属性决定。 偏移属性（上/右/下/左） 然后表示从这个包含块的相应边缘向内的偏移，按照正常方式。

注意： 虽然将一个元素绝对定位到网格容器确实允许 它与该容器的网格线对齐， 但这些元素不会占用空间或以其他方式参与网格的布局。

.grid {
  grid: 1fr 1fr 1fr 1fr / 10rem 10rem 10rem 10rem;
  /* 4 个等高的行填充网格容器，
     4 列，每列 10rem */
  justify-content: center;
  /* 水平居中网格在网格容器中 */
  position: relative;
  /* 建立绝对定位的包含块 */
}

.abspos {
  grid-row-start: 1;     /* 第1网格行线 = 网格容器顶部 */
  grid-row-end: span 2;  /* 第3网格行线 */
  grid-column-start: 3;  /* 第3网格列线 */
  grid-column-end: auto; /* 右内边距边缘 */
  /* 包含块覆盖网格容器的右上象限 */

  position: absolute;
  top: 70px;
  bottom: 40px;
  left: 100px;
  right: 30px;
}

注意： 网格和网格放置属性 是流相对的， 而偏移属性（左，右，上和下） 是物理的， 所以如果direction或writing-mode属性发生改变， 网格会相应地转换， 但偏移量不会。

取代自动放置，auto值对于网格放置属性贡献了一个特殊的线， 其位置是对应的网格容器内边距边缘（如果网格容器 溢出，则为可滚动区域的内边距边缘）。 这些线成为增强网格的第0和倒数第0条线，用于定位绝对定位的项目。

注意： 因此，默认情况下，绝对定位框的包含块将对应于网格容器的内边距边缘，正如它对块级容器所做的一样。

绝对定位发生在网格及其流内内容布局之后， 并且不会对任何网格轨道的尺寸做出贡献， 也不会以任何方式影响网格的大小或配置。 如果网格放置属性引用了不存在的线， 无论是通过显式指定这样的线，还是通过跨越现有隐式网格外部， 它将被视为指定auto（而不是创建新的隐式网格线）。

注意： 请记住，隐式线条假定具有所有线条名称， 因此，引用的线条可能存在，尽管它没有被显式命名。

如果放置只包含一个网格跨距， 用该轴的两个auto线代替。 （当该轴的两个网格放置属性最初都贡献了一个跨距时， § 8.3.1 网格放置冲突处理导致第二个跨距被忽略。）

10.2. 以网格容器为父级

一个网格容器的绝对定位子元素是脱离流的，并且不是网格项， 因此不会影响其他项的放置或网格的尺寸。

一个网格容器的绝对定位子元素的静态位置 [CSS2] 是假设它是唯一的网格项 位于网格区域中， 该区域的边缘与 网格容器的内容边缘重合。 但是，如果网格容器父级也是绝对定位元素的包含块的生成者， 则使用在§ 10.1 以网格容器为包含块中确定的网格区域。

注意： 请注意，这个位置受到子元素上的justify-self和align-self值的影响， 并且与大多数其他布局模型一样， 绝对定位的子元素不会影响包含块的大小或其内容的布局。

11. 对齐和间距

当网格容器的网格轨道已确定大小， 并且所有网格项的尺寸已最终确定后，网格项可以在其网格区域内进行对齐。

边距属性可以像在块布局中那样用于对齐项。网格项也遵循盒对齐属性，这些属性来自CSS 盒对齐模块 [CSS-ALIGN-3]， 它们允许在行和列中轻松地基于关键词对项目进行对齐。

默认情况下，网格项会拉伸以 填满其网格区域。 然而，如果justify-self或align-self的计算值不是stretch，或者边距是auto，网格项将自动调整大小以适应其内容。

11.1. 槽：行间距 (row-gap)、 列间距 (column-gap) 和 间距 (gap) 属性

行间距和列间距属性 （及其间距简写）， 在网格容器上指定时， 定义了槽在网格行和网格列之间的间距。 它们的语法在CSS盒对齐模块第3版 §8 盒间的间隙中定义。

这些属性的效果 类似于受影响的网格线 获得厚度： 两条网格线之间的网格轨道是表示它们的槽之间的空间。 为了轨道尺寸计算的目的， 每个槽被视为 一个额外的、空的、固定大小的轨道， 任何跨过其对应网格线的网格项都将跨越此轨道。

注意： 额外的间距可能会因 justify-content / align-content而增加。 详见§ 12.1 网格尺寸算法。 此空间会有效地增加槽的大小。

如果网格在轨道之间发生分段， 这些轨道之间的槽间距必须被抑制。注意，槽即使在强制换行后也会被抑制， 不同于边距。

槽 只会出现在隐式网格的轨道之间； 第一个轨道之前或最后一个轨道之后都没有槽。 （特别地，槽不会出现在 隐式网格的第一个/最后一个轨道和 增强网格中的“auto”线之间。）

当一个折叠轨道的 槽折叠时， 它们精确重叠——两个槽重叠，起点和终点边缘重合。 如果折叠轨道的一侧没有槽 （例如，如果它是隐式网格的第一个或最后一个轨道）， 那么折叠其槽将导致轨道两侧都没有槽。

11.2. 使用auto边距对齐

本节为非规范性内容。 关于边距如何影响网格项的规范性定义在§ 12 网格尺寸。

在网格项上的自动边距的效果 与块流中的自动边距非常相似：

  • 在计算网格轨道尺寸时，自动边距被视为0。
  • auto边距吸收正的空闲空间 然后通过盒对齐属性进行对齐。
  • 溢出的元素会忽略其auto边距， 并根据其盒对齐属性进行溢出处理。

11.3. 行内轴对齐：justify-self 和 justify-items 属性

网格项可以通过 在网格项上使用justify-self属性，或者在网格容器上使用justify-items属性 来在行内轴维度上进行对齐，详见[CSS-ALIGN-3]。

例如， 在英文文档中， 行内轴是水平的， 因此justify-*属性可以水平对齐 网格项。

如果基线对齐在某个网格项上被指定， 且该轴上的尺寸依赖于内在大小的轨道 （其尺寸因此既依赖于该项的大小又依赖于基线对齐，从而形成循环依赖）， 该项不会参与基线对齐， 而是使用其回退对齐，仿佛最初指定了该对齐方式。 对于此目的，当网格容器 在相关轴上有不确定的大小时， <flex>轨道尺寸将视为“内在大小”。

注意： 使用回退对齐与否 不会在布局过程中发生变化： 如果存在循环依赖，那么它始终存在。

11.4. 块轴对齐：align-self 和 align-items 属性

网格项也可以在 块维度上对齐 （垂直于行内维度）， 通过在网格项上使用align-self属性，或者在网格容器上使用align-items属性进行对齐，详见 [CSS-ALIGN-3]。

如果基线对齐在某个网格项上被指定， 且该轴上的尺寸依赖于内在大小的轨道 （其尺寸因此既依赖于该项的大小又依赖于基线对齐，从而形成循环依赖）， 该项不会参与基线对齐， 而是使用其回退对齐，仿佛最初指定了该对齐方式。 对于此目的，当网格容器 在相关轴上有不确定的大小时， <flex>轨道尺寸将视为“内在大小”。

11.5. 对齐网格：justify-content 和 align-content 属性

如果网格的外边缘 与网格容器的 内容边缘不一致 （例如，没有列是弹性大小）， 网格轨道将在内容框内根据justify-content和align-content属性对齐在网格容器中。

例如，以下网格在垂直方向居中， 并对齐到其网格容器的右边缘： 
.grid {
  display: grid;
  grid: 12rem 12rem 12rem 12rem / 10rem 10rem 10rem 10rem;
  justify-content: end;
  align-content: center;
}

如果没有网格轨道 （显式网格为空，且没有在隐式网格中创建轨道）， 则每个轴中的唯一网格线将 与网格容器的起始边对齐。

请注意，某些justify-content和align-content值可能会导致轨道间隔开 （space-around，space-between，space-evenly） 或被调整大小（stretch）。 如果网格在轨道之间 被分割， 则必须抑制这些轨道之间的任何附加间距。

例如，在以下网格中， 跨越的项的网格区域增加以适应 由于对齐分配给槽的额外空间： 
.wrapper {
  display: grid;
  /* 3行/4列的网格容器 */
  grid: repeat(3, auto) / repeat(4, auto);
  gap: 10px;
  align-content: space-around;
  justify-content: space-between;
}

.item1 { grid-column: 1 / 5; }
.item2 { grid-column: 1 / 3; grid-row: 2 / 4; }
.item3 { grid-column: 3 / 5; }
/* 最后两个项目自动放置在最后两个网格单元中 */

请注意，对齐（与间隙间距不同） 发生在网格轨道被调整大小之后， 因此如果轨道尺寸由跨越项的内容决定， 它将在对齐阶段获得多余的空间 以适应对齐间距。

11.6. 网格容器的基线

网格容器的第一个（或最后一个）基线的确定规则如下：

  1. 找到包含至少一个网格容器的第一行（或最后一行）。

    如果任何与该行相交的网格项参与了该行中的基线对齐， 那么网格容器的基线集 将从这些网格项的共享对齐基线生成。

    否则， 网格容器的第一个（或最后一个）基线集将从第一个（或最后一个）按行主顺序的网格项的对齐基线生成 （依据网格容器的书写模式）。 如果该网格项在网格的内联轴上没有对齐基线， 则首先从其边框边缘合成一个基线。

  2. 如果网格容器中没有包含任何网格项， 则网格容器没有第一个（或最后一个）基线集， 并且如果需要，将根据其对齐上下文的规则合成一个基线。 从该算法中退出。

网格修改后的文档顺序（网格顺序）是遍历网格的网格单元时遇到的网格项的顺序。 如果同时遇到两个项目， 它们将按文档顺序处理。

按照上述规则计算基线时， 如果贡献基线的盒子有允许滚动的overflow值， 则在确定其基线时必须将其视为处于初始滚动位置。

在确定表格单元的基线时， 网格容器就像行框或表格行一样提供基线。[CSS2]

有关基线的更多信息，请参见CSS Writing Modes 3 §4.1 基线简介和 CSS Box Alignment 3 §9 基线对齐细节。

12. 网格尺寸计算

本节定义了网格尺寸计算算法， 用于确定所有网格轨道的尺寸，并由此扩展到整个网格的尺寸。

每个轨道都有指定的最小和最大尺寸函数（两者可能相同）。 每个尺寸函数可以是以下之一：

  • 固定尺寸函数（<长度> 或可解析的<百分比>）。
  • 固有尺寸函数（min-content，max-content，auto，fit-content()）。
  • 灵活尺寸函数（<flex>）。

网格尺寸计算算法定义了如何将这些尺寸约束解析为实际使用的轨道尺寸。

12.1. 网格大小调整算法

注意: 所有网格项的放置， 包括子网格及其子项， 都发生在调整大小之前。

  1. 首先，轨道大小调整算法用于确定 网格列的大小。

    在此过程中， 任何在网格容器 的行内轴上 被子网格化的网格项将被视为空， 而它的网格项（即孙项） 将被视为网格容器的直接子项（即祖父项）。 此检查是递归的。

    仅在块轴 上被子网格化，且其在网格容器中 的行内轴大小取决于其内容大小的网格项也会被检查： 由于该维度的大小可能取决于其在另一个维度中的子网格轨道的大小贡献， 因此此类项对网格列大小调整的贡献 （见解析内在轨道大小） 是基于它在网格大小调整算法中达到的相同步骤的基础上作出的。 例如，在此步骤的第一次通过时， 该项的轨道只会在第一步完成调整； 如果触发了该步骤的第二次通过， 则该项将在返回其大小以供考虑网格列大小调整之前完成第一到三步的第一次通过以及该步骤的第二次通过。 再次，此检查是递归的。

    如果在此步骤中计算网格项的布局 取决于可用空间 在块轴上的可用空间， 假设可用空间是 所有具有明确的 最大轨道大小调整函数 的行具有该大小而其他行是无限的。 如果网格容器和所有轨道都具有明确的大小， 还需要应用align-content属性，以确定由这些项跨越的间隙的最终有效大小； 否则在估算中忽略轨道对齐的效果。

  2. 接着，轨道大小调整算法 用于确定网格行的大小。

    在此过程中， 任何在网格容器 的块轴 上被子网格化的网格项将被视为空， 而其网格项（即孙项） 将被视为网格容器的直接子项（即祖父项）。 此检查是递归的。

    如同调整列大小一样， 仅在行内轴上 被子网格化且其网格容器 的大小取决于其内容的项也会被检查。 （如同调整列大小一样， 该项对网格行大小调整的贡献是在相应点的轨道大小调整完成后作出的； 再次，此检查是递归的。）

    要找到行内轴 的可用空间 ， 对于其块轴 大小贡献需要使用先前步骤中计算的网格列的大小。 如果网格容器的 行内大小是 明确的， 同时应用justify-content来考虑列间隙的有效大小。

  3. 然后，如果任何网格项的最小内容贡献根据第二步中计算的行大小和对齐发生了变化， 则需要使用新的最大内容贡献（仅一次）重新确定 网格列的大小。

    要找到任何块轴 上的可用空间 ， 对于其行内轴 大小贡献需要使用先前步骤中计算的网格行的大小。 如果网格容器的 块大小是 明确的， 同时应用align-content来考虑行间隙的有效大小。

    此重复操作对于某些情况下是必要的，例如 行内大小 取决于其块大小的 网格区域。 示例包括包装列的弹性容器 （flex-flow: column wrap）， 正交流（writing-mode）， 和多列容器。
  4. 接着，如果任何网格项的最小内容贡献根据第三步中计算的列大小和对齐发生了变化， 则使用新的最小内容和最大内容贡献 （仅一次）重新确定网格行的大小。

    要找到任何行内轴上的可用空间， 对于其块轴大小贡献，需要使用先前步骤中计算的网格列的大小。 如果网格容器的行内大小 是明确的， 还需应用justify-content来考虑列间隙的有效大小。

  5. 最后，网格容器的大小 通过将网格的最终大小作为其内容大小确定， 轨道根据align-content 和justify-content属性在 网格容器中对齐。

    注意: 这可能会在轨道之间引入额外的空间， 从而可能增加跨越间隙的任何网格项的网格区域大小， 超出轨道大小调整过程中分配的空间。

注意: 在子网格化维度中， 每个给定轨道的项目都被视为父网格的成员。 这种交错方式要求网格大小调整在每个轴上钻取子网格， 而不是在递归过程中完成两个轴的调整。 注意，这意味着如果一个子网格建立了正交流， 它的轨道大小调整顺序会与嵌套网格相反。

一旦每个网格区域的大小确定， 网格项将被放置在各自的包含块中。 网格区域的宽度和高度在此过程中被视为明确的。

注意: 由于仅使用明确大小计算的公式，例如拉伸适配公式， 也是明确的，因此被拉伸的网格项的大小也被视为明确的。

以下示例说明了如何根据每个轴调整子网格的大小：

假设我们有一个父网格容器A，它包含一个具有子网格化列的项B， B包含一个子网格化行的孙项C，以及一个简单嵌套网格的孙项D。 

<grid-A>
  <grid-B subgrid=columns>
    <grid-C subgrid=rows/>
    <grid-D>
  </grid-B>
<grid-A>

当A调整其列的大小时， 它将B的项插入到A的相应列中， 但当A调整其行的大小时， 它将B视为一个单一项（即一个拥有自己行和子项C和D的网格容器）。 同样，当B调整其行的大小时， 它将C的项插入到B的行中， 但当B调整其列的大小时， 它将C视为一个单一项，就像对待D一样。 C的行与A的行没有关系， 因为B中的行是嵌套的，而不是子网格化的。

从总体上看，网格算法的顺序是：

  1. 调整列的大小
  2. 调整行的大小
  3. 根据最终的行大小调整列（如果需要）

在这个示例中，网格大小调整算法如下：

  1. 解析A网格列的大小， 使用A的网格项的大小， 将B视为空， 但将其子项（包括C和D）视为网格A的项。

    网格算法简单地递归到D。 对于C，情况更复杂：

    1. 调整C的列大小。
    2. 通过调整B的行来调整C的行大小。
    3. 调整C的列。
    4. 返回C的最终列大小。

    对于B的行的正确大小需要C的最终列大小， 因为行大小取决于列大小， 因此B的行很可能取决于C的最终列大小。 为了打破这种循环依赖， 我们需要将算法分割，依赖于C列最终大小的初步估计， 并在后期进行调整。 因此，对于C，我们只需递归调整列的大小， 并将该初步大小传递给A以进行其初步列大小调整。

    当我们稍后调整B的行大小时， 我们将调整C的行（它是子网格化的）， 并通过最终确定C的列来完成C的大小调整。 如果这导致了变化，我们有机会在A的调整过程中触发A列的调整。

  2. 接着，解析A的行的大小， 使用A的网格项的大小， 将B视为一个单一项。

    由于B作为子网格， 其大小调整被分成多个步骤， 网格算法只递归调整B的行： 调整B的行大小，将D视为一个单一项，获取其最终大小， 并将C视为空项， 将其子项提升为B网格中的项。

    B返回其最终行大小， 该大小纳入A的行大小调整过程。

  3. 最后，确定A的列的最终大小。 如果C的最终大小由于通过B调整行大小而发生了变化， 则应触发B的列大小调整， 进而触发A列的调整。

12.2. 轨道大小调整术语

最小轨道大小调整函数
如果轨道使用了minmax()函数调整大小， 这是该函数的第一个参数。 如果轨道使用了<flex>值或fit-content()函数调整大小，则为auto。 否则，则为该轨道的大小调整函数。
最大轨道大小调整函数
如果轨道使用了minmax()函数调整大小， 这是该函数的第二个参数。 否则，则为该轨道的大小调整函数。 在所有情况下，将auto和fit-content()视为max-content， 除非在特定情况下对fit-content()另有说明。
可用网格空间
在每个维度中独立计算，可用网格空间是：

  • 如果网格容器的大小是明确的， 则使用其内容框的大小。

  • 如果网格容器 正在根据最小内容约束或最大内容约束进行大小调整， 则可用网格空间就是该约束 （且是不确定的）。

注意: 表示基于内容大小调整的auto大小， 例如在水平书写模式下块级盒子的高度，等同于max-content。

在所有情况下，根据网格容器 的最小/最大宽度/高度属性限制可用网格空间， 如果这些属性是明确的。

空余空间
等于可用网格空间减去 所有网格轨道的基本大小总和（包括沟槽）， 结果取0为下限。 如果可用网格空间是不确定的， 则空余空间也是不确定的。
跨度计数
在适用的维度中，网格项跨越的 网格轨道的数量。

注意: 记住沟槽在网格大小调整算法中被视为固定大小的轨道——其最小和最大大小调整函数都设置为沟槽的实际大小——因此它们的宽度需要相应地包含在轨道大小调整算法的计算中。

12.3. 轨道大小调整算法

本节的其余部分是轨道大小调整算法， 它根据最小和最大轨道大小调整函数 来计算轨道的使用大小。 每个轨道都有一个基础大小， 一个<length>， 它在整个算法中会逐步增长，最终成为轨道的最终大小， 还有一个增长限制， 一个<length>， 它为基础大小提供了一个期望的最大值。 共5个步骤：

  1. 初始化轨道大小
  2. 解析内在轨道大小
  3. 最大化轨道
  4. 扩展弹性轨道
  5. [[#algo-stretch|扩展拉伸的auto轨道]]

12.4. 初始化轨道大小

初始化每个轨道的基础大小和增长限制。 对于每个轨道， 如果轨道的最小轨道大小调整函数是：

固定大小调整函数
解析为一个绝对长度，并将该大小用作轨道的初始基础大小。

注意: 不确定 长度不会出现， 因为它们被视为auto。

内在大小调整函数
使用初始基础大小为零。

对于每个轨道， 如果轨道的最大轨道大小调整函数是：

一个固定尺寸函数
解析为绝对长度，并使用该尺寸作为轨道的初始增长限制。
一个内在尺寸函数
一个灵活尺寸函数
使用无限的初始增长限制。

在所有情况下，如果增长限制 小于基础大小， 则增加增长限制以匹配基础大小。

注意: 沟槽在轨道大小调整算法中被视为空的固定大小轨道。

12.5. 解析内在轨道大小

此步骤将内在轨道大小调整函数解析为绝对长度。 首先，它根据完全包含在单个轨道中的项目解析这些大小。 然后逐渐加入跨越多个轨道的项目的空间需求， 尽可能均匀地将额外空间分配到这些轨道上。

注意: 当此步骤完成时， 所有内在的基础大小和增长限制都将解析为绝对长度。

注意: 请记住 fit-content() 和 auto 最大轨道大小调整函数 与 max-content 的处理方式相同，除非明确指定了其他情况。

  1. 对齐基线对齐的项目 使它们的内在大小贡献反映其基线对齐。 对于每个基线共享组中的项目， 在每个项目的起始/结束侧（用于首/末基线对齐）添加“垫片”（有效的额外边距）， 这样当它们起始/结束对齐时， 它们的基线按照指定对齐。

    在轨道大小调整时，将这些“垫片”视为项目内在大小贡献的一部分。 如果一个项目使用多个内在大小贡献，它可以为每个贡献拥有不同的垫片。

    例如，当 网格容器的大小为 不确定时， 它首先在最小/最大内容约束下进行布局以找到大小， 然后在此大小下进行“实际”布局 （这会影响诸如百分比轨道之类的项目）。 每个阶段添加的“垫片”是独立的， 仅在该阶段影响布局。

    注意: 请注意，基线自对齐和基线内容对齐的项目 都在此步骤中被考虑。

    注意: 由于网格项目的大小 取决于内在大小调整轨道的大小不参与基线对齐， 它们不会被添加“垫片”。

  2. 调整轨道以适应非跨越的项目： 对于每个具有内在 轨道大小调整函数而不是 弹性大小调整函数的轨道， 考虑其范围为1的项目：
    对于最小内容最小值：
    如果轨道具有min-content 最小轨道大小调整函数， 将其基础大小 设置为项目的最大最小内容贡献，并且取0为下限。
    对于最大内容最小值：
    如果轨道具有max-content 最小轨道大小调整函数， 将其基础大小 设置为项目的最大最大内容贡献，并且取0为下限。
    对于自动最小值：
    如果轨道具有auto 最小轨道大小调整函数 且网格容器正在进行最小/最大内容约束下的大小调整， 则将该轨道的基础大小 设置为项目的受限的最小/最大内容贡献的最大值，取0为下限。 受限的最小/最大内容贡献是项目的最小/最大内容贡献，受限于最大轨道大小调整函数（该函数可以是 fit-content()轨道大小调整函数的参数）， 如果它是固定的，最终以其最小贡献为下限。

    否则， 将轨道的基础大小设置为项目的 最小贡献 的最大值，取0为下限。 最小贡献是项目可以拥有的最小 外部大小。 具体来说， 如果项目的计算首选大小 表现为auto或取决于其在相关轴上的包含块的大小， 则其最小贡献是 假设项目使用的最小大小作为其首选大小时的 外部大小； 否则项目的最小贡献是其 最小内容贡献。 由于最小贡献通常取决于 项目的内容大小， 因此它被视为内在大小贡献的一种。

    注意: 对于指定最小大小为 auto（初始值）的项目， 最小贡献通常等同于 最小内容贡献——但在某些情况下可能有所不同，请参阅 § 6.6 网格项目的自动最小大小。 此外，最小贡献 ≤ 最小内容贡献 ≤ 最大内容贡献。

    对于最小内容最大值：
    如果轨道具有min-content 最大轨道大小调整函数， 将其增长限制 设置为项目的最大最小内容贡献。
    对于最大内容最大值：
    如果轨道具有max-content 最大轨道大小调整函数， 将其增长限制 设置为项目的最大最大内容贡献。 对于fit-content()最大值， 进一步根据增长限制进行限制， 由fit-content()参数确定。

    在所有情况下，如果轨道的增长限制现在小于其基础大小， 则将增长限制增加至匹配基础大小。

    注意: 此步骤是以下处理跨越项目步骤的简化， 应产生与对范围为1的项目运行这些指令相同的行为。

  3. 增加尺寸以容纳跨越内容大小轨道的项： 接下来，考虑跨度为 2 且未跨越具有 灵活大小功能的轨道的项。
    1. 对于固有最小值：首先通过 分配额外空间 增加具有 固有 最小轨道尺寸功能的轨道的 基础大小， 以容纳这些项的 最小贡献。

      如果网格容器在 最小内容限制 或 最大内容限制 下调整大小， 请将这些项的 有限最小内容贡献 替代其 最小贡献。 （对于跨越多个轨道的项，用于计算其 有限最小/最大内容贡献 的上限是其所跨越轨道中 所有 固定 最大轨道尺寸功能 的总和，并且仅当它仅跨越这些轨道时应用。）

    2. 对于基于内容的最小值：接下来继续通过 分配额外空间 增加具有 最小轨道尺寸功能 的轨道的 基础大小，以适应这些项的 最小内容贡献。
    3. 对于最大内容的最小值：接下来，如果网格容器在 最大内容限制下调整大小， 继续通过 分配额外空间 增加具有 最小轨道尺寸功能的轨道的 基础大小，以适应这些项的 有限最大内容贡献。

      在所有情况下，继续通过 分配额外空间 增加具有 最小轨道尺寸功能 的轨道的 基础大小，以适应这些项的 最大内容贡献。

    4. 如果此时任何轨道的 增长限制 低于其 基础大小，请将其 增长限制 增加到与其 基础大小 一致。
    5. 对于固有最大值：接下来，通过 分配额外空间 增加具有 固有 最大轨道尺寸功能 的轨道的 增长限制，以适应这些项的 最小内容贡献。 标记在此步骤中增长限制从无限更改为有限的任何轨道为 无限可增长，以便在下一步中使用。
      为什么存在无限可增长标记？

      Peter Salas 解释：

      考虑以下情况：

两个“自动”轨道（即 minmax(min-content, max-content) minmax(min-content, max-content)）。
项目 1 在轨道 1 中，min-content = max-content = 10。
项目 2 跨越轨道 1 和 2，min-content = 30，max-content = 100。

在为第一个项目解决 min-content/max-content 后，我们有如下情况：

轨道 1：基础大小 = 10，增长限制 = 10

轨道 2：基础大小 = 0，增长限制 = 无限

接下来我们为第二个项目解决 min-content/max-content。

第一阶段将轨道 2 的基础大小设置为 20，以便两个轨道的基础大小之和为 30。
第二阶段没有操作，因为没有相关轨道。
第三阶段将轨道 2 的增长限制设置为 20，以便两个轨道的增长限制之和为 30。
在第四阶段，我们需要通过 70 来增加增长限制的总和，以适应项目 2。
两种选择是：

1. 等量增加每个轨道的增长限制，
  得到的增长限制 = [45, 55]。
2. 仅增加第二个轨道的增长限制，
  得到的增长限制 = [10, 90]。

通过在空间分配期间不将刚刚设置的增长限制视为约束（即将其视为无限），
我们得到了第二个结果，
我们认为这是更好的结果，因为第一个轨道正好按第一个项目大小调整。
    6. 对于最大内容的最大值：最后，继续通过 分配额外空间 增加具有 最大轨道尺寸功能 的轨道的 增长限制，以适应这些项的 最大内容贡献。 但是，限制任何 fit-content() 轨道的增长， 以其 fit-content() 参数为限。

    逐步重复此操作以考虑跨度更大的项，直到所有项都已考虑。

  4. 增加尺寸以容纳跨越灵活轨道的项： 接下来，重复上一步，考虑 所有确实跨越具有灵活大小功能的轨道的项，同时
    • 仅将空间分配给灵活轨道（即将所有其他轨道视为具有 固定大小功能）
    • 如果项跨越的所有灵活轨道的 灵活大小功能之和大于零， 则根据其灵活大小功能的比例而不是平均分配空间。
  5. 如果仍然有轨道的增长限制是无限的（例如，因为它没有放置任何项或它是一个 灵活轨道），将其 增长限制设置为其 基础大小。

注意：在跨越多个轨道时，没有单一的方法可以满足固有尺寸约束。该算法结合了已在现实世界用例中表现出良好效果的多种启发式方法， 例如本文档前面提到的“游戏”示例。未来可能会更新此算法，以便考虑到识别出的更高级的启发式方法。

12.5.1. 分配跨越轨道的额外空间

为了通过增加一组轨道的受影响尺寸来分配额外空间，满足一组固有尺寸贡献的要求，

  1. 为每个受影响的基础大小或 增长限制分别维护一个 计划增加，初始值为0。 （这可以防止尺寸增加成为依赖顺序的问题。）
  2. 对于每个被考虑的项，
    1. 找到要分配的空间： 从项的尺寸贡献中减去每个跨越轨道的对应尺寸（ 基础大小或 增长限制）。 （对于无限的增长限制，替代使用轨道的 基础大小。）这就是要分配的空间，将其下限设置为0。 
      extra-space = max(0, size-contribution - ∑track-sizes)
    2. 在限制内分配空间： 通过将空间在受影响的轨道之间平均分配，找到每个跨越轨道的 项引起的增加，当受影响的尺寸和项引起的增加达到其限制时，冻结该轨道的 项引起的增加（并根据需要继续增加未冻结的轨道）。

      对于基础大小，其 限制是其增长限制。 对于增长限制，其限制为无限， 如果其标记为无限可增长，否则为其 增长限制。

      注意：如果受影响的尺寸是 增长限制且轨道未标记为 无限可增长， 则每个项引起的增加将为零。

    3. 超出限制时分配空间： 如果在所有轨道被冻结后仍有空间，则解冻并继续将空间分配给 项引起的增加…
      • 在适应最小贡献或 适应最小内容贡献时： 任何受影响的轨道，也恰好具有固有最大轨道大小功能； 如果没有这样的轨道，则所有受影响的轨道。
      • 在适应最大内容贡献时： 任何受影响的轨道，也恰好具有 max-content 最大轨道大小功能； 如果没有这样的轨道，则所有受影响的轨道。
      • 处理任何固有的增长限制时： 所有受影响的轨道。

      为此目的， 最大轨道大小功能的 fit-content()轨道， 被视为max-content，直到其达到 fit-content()参数指定的限制， 之后它将被视为具有该参数的固定大小功能。

      注意：此步骤优先分配空间，以适应轨道 最小轨道大小功能当前增长限制之外的空间要求， 基于它们的最大轨道大小功能类型。

    4. 对于每个受影响的轨道， 如果该轨道的项引起的增加大于轨道的计划增加，则将该轨道的 计划增加设置为该值。
  3. 通过添加计划增加更新轨道的受影响尺寸，以便下一轮空间分配将考虑到这一增加。 （如果受影响的尺寸是无限的增长限制， 则将其设置为轨道的基础大小 加上计划增加。）

12.6. 最大化轨道

如果可用空间为正数， 则将其平均分配给所有轨道的基础大小， 当轨道达到其增长限制时将其冻结（并根据需要继续增加未冻结的轨道）。

在此步骤中： 如果在网格容器上应用 最大内容约束， 则可用空间为无限大； 如果应用最小内容约束， 则可用空间为零。

如果这会导致网格大于 网格容器的内尺寸，受到其 最大宽度/高度的限制， 则重新执行此步骤， 并将可用网格空间视为等于 网格容器的内尺寸，当其尺寸为 最大宽度/高度时。

12.7. 扩展灵活轨道

此步骤通过使用可分配给灵活轨道的最大值来调整其尺寸， 不超过可用空间。

首先，找到网格使用的灵活比例：

如果可用空间为零，或在 网格容器上应用 最小内容约束：
使用的灵活比例为零。
否则，如果可用空间是 确定的长度：
使用的灵活比例是通过使用所有 网格轨道和 填充空间的 可用网格空间来 计算出的结果。
否则，如果可用空间是 不确定的长度：
使用的灵活比例是以下两者中的最大值：
  • 对于每个灵活轨道，如果灵活轨道的 灵活系数大于1， 则为该轨道的基础大小除以其灵活系数的结果； 否则，取轨道的基础大小。
  • 对于每个跨越灵活轨道的网格项， 使用该项跨越的所有网格轨道，并将其 填充空间 设置为该项的最大内容贡献，通过计算得出的结果。

如果使用此灵活比例 会导致网格小于 网格容器的 最小宽度/高度（或大于 网格容器的 最大宽度/高度）， 则重新执行此步骤， 将可用空间视为确定的， 并将可用网格空间视为等于 网格容器的 内尺寸， 当其尺寸为最小宽度/高度（最大宽度/高度）时。

对于每个灵活轨道， 如果使用的灵活比例 乘以该轨道的灵活系数的结果大于该轨道的 基础大小， 则将其基础大小设置为该乘积值。

12.7.1. 找到 fr 的大小

此算法找到 fr 单位可以达到的最大值，且不超过目标大小。 必须传入一组网格轨道和某个数量的 可填充空间。

  1. 令剩余空间等于 可填充空间减去 非灵活网格轨道的 基础大小。
  2. 令灵活系数总和 等于灵活系数的总和， 其中包括灵活轨道。 如果该值小于1， 则将其设置为1。
  3. 令假设的 fr 大小等于 剩余空间除以 灵活系数总和。
  4. 如果假设的 fr 大小与某个 灵活轨道的 灵活系数的乘积小于该轨道的基础大小， 则重新开始此算法，将所有此类轨道视为非灵活的。
  5. 返回假设的 fr 大小。

12.8. 扩展 auto 轨道

此步骤通过将剩余的正数确定的 可用空间均分到具有 auto 最大轨道大小函数的轨道上来扩展它们。 如果可用空间是不确定的， 但网格容器具有 确定的 最小宽度/高度， 则使用该大小来计算此步骤的可用空间。

13. 分割网格布局

网格容器可以在页面之间跨行或跨列进行分割， 也可以在项目内部进行分割。 break-*属性对网格容器的应用与它们参与的格式化上下文中的应用方式相同。 本节定义了它们如何应用于网格项目及网格项目的内容。

以下分割规则将分割容器称为“页面”。 同样的规则适用于任何其他分割上下文。 （根据需要将“页面”替换为适当的分割容器类型。） 参见CSS分割模块 [CSS3-BREAK]。

在本级别的网格布局中，未定义分割网格容器的确切布局。 然而，网格容器内部的分割需遵守以下规则：

  • break-before和break-after属性在网格项目上应用时，会传播到它们所在的网格行。 第一个行的break-before属性和最后一个行的 break-after属性会传播到网格容器。
  • 网格项目中的强制分割实际上会增加其内容的大小； 它不会在兄弟项目中触发强制分割。
  • A类断点机会发生在行或列之间（取决于合适的轴）， 而C类断点机会发生在第一行/最后一行 （列）与网格容器的内容边缘之间。 [CSS3-BREAK]
  • 当网格容器在分割后继续时，其网格项目 在分割上下文中的块流方向上的可用空间会被之前页面上网格容器片段所占据的空间减少。 网格容器片段所占的空间是其在该页面上的内容框的大小。 如果由于此调整导致可用空间变为负值，则将其设置为零。
  • 除了前一点强加的项目重排之外，用户代理应尽量减少相对于未分割流的网格容器变形。

本规范中的示例用“例如”一词引入，或像这样与规范文本分开： class="example"，例如：

这是一个信息示例。

信息性注释以“注释”一词开头，并使用class="note"与规范文本分开，例如：

注释，这是一个信息性注释。

建议性内容是规范部分，应引起特别注意，并使用<strong class="advisement">标记开头，例如： UA 必须提供可访问的替代方案。

一致性类别

本规范定义了三个一致性类别：

样式表
CSS 样式表。
渲染器
UA，解释样式表的语义并渲染使用它们的文档。
创作工具
UA，编写样式表。

如果样式表中使用的所有语法都符合本模块中定义的通用 CSS 语法以及本模块中每个功能的语法，那么该样式表就符合本规范。

如果渲染器除了解析样式表并根据相关规范定义进行渲染外，还支持本规范中定义的所有功能，解析它们正确并相应地渲染文档，那么该渲染器符合本规范。 然而，UA 因设备限制而无法正确渲染文档的情况并不使其不符合规范。（例如，UA 不要求在单色显示器上渲染颜色。）

如果创作工具编写的样式表符合通用 CSS 语法以及本模块中每个功能的语法，并且符合本模块中描述的样式表的所有其他一致性要求，则该创作工具符合本规范。

部分实现

为了使作者能够利用向前兼容的解析规则分配后备值，CSS 渲染器必须将任何尚未得到支持的 at 规则、属性、属性值、关键字和其他语法构造视为无效，并按需忽略。 特别是，用户代理不得选择性地忽略不受支持的组件值，而只保留多值属性声明中的支持值： 如果任何值被视为无效（如不支持的值必须那样），CSS 要求忽略整个声明。

不稳定和专有功能的实现

为了避免与未来稳定的 CSS 功能发生冲突，CSSWG 建议遵循最佳实践，以实现不稳定功能和 CSS 的专有扩展。

非实验性实现

一旦规范达到候选推荐标准（CR）阶段，即可进行非实验性实现，实施者应发布任何根据规范证明正确实现的 CR 级别功能的无前缀实现。

为了建立并维护跨实现的 CSS 互操作性，CSS 工作组要求非实验性的 CSS 渲染器在发布无前缀实现之前向 W3C 提交实现报告（如果有必要，还需提交用于该实现报告的测试用例）。提交给 W3C 的测试用例将由 CSS 工作组进行审核和修正。

有关提交测试用例和实现报告的更多信息，请访问 CSS 工作组网站：https://www.w3.org/Style/CSS/Test/。问题应发送至邮件列表 public-css-testsuite@w3.org。

CR 退出标准

要将本规范推进到提议推荐阶段（PR），每个功能必须至少有两个独立的、互操作的实现。每个功能可以由不同的产品集实现，不要求所有功能都由单个产品实现。为此标准的目的，我们定义以下术语：

独立
每个实现必须由不同的组织开发，且不能共享、重用或基于其他符合条件的实现代码。与本规范实现无关的代码部分不受此要求限制。
互操作
通过官方 CSS 测试套件中的相应测试用例，或如果实现不是 Web 浏览器，则通过等效测试。每个相关测试都应创建等效测试，如果某个用户代理（UA）要用于声称互操作性，则必须有一个或多个其他 UA 能够以相同方式通过这些等效测试，目的是实现互操作性。等效测试必须公开可用以便进行同行评审。
实现
一个用户代理，其：
  1. 实现了该规范。
  2. 对公众可用。实现可以是发布的产品或其他公开可用的版本（如测试版、预览版或“夜间构建”）。非发布产品版本必须至少实现该功能一个月，以证明其稳定性。
  3. 不是实验性的（即，该版本并非专为通过测试套件而设计，也不打算在未来用于正常使用）。

本规范将至少保留在候选推荐状态六个月。

问题索引

上述列表的第一个要点意味着隐式轨道被序列化为 grid-template-rows/等的一部分， 尽管作者实际上无法在这些属性中指定隐式轨道大小！ 所以grid-template-rows和grid-template-columns的值 可能无法正确地进行回合操作： 
const s = getComputedStyle(gridEl);
gridEl.style.gridTemplateRows = s.gridTemplateRows;
// 这样的代码 应该是无操作，
// 但如果存在任何隐式行，
// 这将会把它们转变为显式行，
// 可能会改变网格项的位置，
// 并改变网格的整体大小！

这是早期实现的意外特性， 后来的实现也泄露了这一点， 没有过多考虑。 我们打算将其从规范中移除， 但要等到我们定义了一个CSSOM API用于获取隐式轨道信息之后， 因为目前这是获取该信息的唯一方法， 而许多页面依赖于此。
