摘要

CSS 是一种描述结构化文档（如 HTML 和 XML）在屏幕、纸张等上呈现的语言。

1. 介绍

本节为非规范性内容。

视图过渡（View Transitions），如在 [css-view-transitions-1] 中所规定，是一种允许 开发者 在 文档 的视觉状态之间创建动画过渡的特性。

第二级扩展了该规范，增加了必要的 API 和生命周期，以支持同源跨文档导航中的过渡，并增加了一些功能，使得编写具有更丰富视图过渡的页面变得更加容易。

第二级定义了以下内容：

  • 跨文档视图过渡，包括 @view-transition 规则及启用跨文档视图过渡生命周期的算法。

  • 选择性视图过渡，一种根据 活动视图过渡 的存在来匹配样式的方法， 更具体地说是根据活动视图过渡的某种类型。

  • 视图过渡伪元素之间共享样式，一种一次声明样式的方法， 并在多个视图过渡伪元素中使用。包括 view-transition-class 属性，以及 对命名伪元素的附加

2.跨文档视图过渡

2.1. 概述

本节为非规范性内容。

2.1.1. 激活

在同文档视图过渡中，作者通过调用 startViewTransition 来启动视图过渡。在跨文档视图过渡中，触发视图过渡的条件是两个文档之间的导航，只要满足以下条件：

  • 两个文档属于 同源;

  • 页面在整个导航过程中保持可见;

  • 导航由页面发起，例如通过点击链接或提交表单，或为 遍历 导航（后退/前进）。这排除了例如由地址栏发起的导航;

  • 导航不包括跨源重定向;

  • 两个文档均选择使用 @view-transition 规则进行跨文档视图过渡。

有关更多详细信息，请参见 生命周期 部分。

2.1.2. 等待新状态稳定

在同一文档的视图过渡中，作者可以使用传递给 startViewTransition 的回调函数来表明新状态何时处于稳定状态。 由于跨文档视图过渡是声明性的，因此没有这样的明确承诺。相反，用户代理依赖 渲染阻塞机制 来决定文档何时达到稳定状态。 通过这种方式，作者可以使用 blocking 属性来延迟过渡，直到以下情况发生：

  • 通过在所需的脚本上使用脚本的 blocking 属性，所有预期的脚本都已执行。

  • 通过在所需的样式上使用样式或链接的 blocking 属性，所有预期的样式都已执行。

  • 使用 expect HTMLLinkElement 元素，解析器已经看到所有预期的 HTML 元素。

注意： 过度使用渲染阻塞机制可能会导致旧状态冻结过久，进而导致糟糕的用户体验。 为避免这种情况，建议确保渲染阻塞元素能及时获取。

在此示例中，旧文档的最后一帧将会被展示，并且动画会延迟，直到满足以下条件：

  • style.css 已应用，以确保新状态具有正确的样式。

  • fixup.js 已运行，以确保展示内容通过基于脚本的修正保持最新。

  • 解析并看到 main-article 部分，以确保加载了足够的内容后再允许过渡继续。

<!DOCTYPE html>
<html>
  <head>
    < !-- 这将默认是渲染阻塞的 -->
    <link rel="stylesheet" href="style.css">

    < !-- 因为这个脚本修正了布局，标记为渲染阻塞将确保它在视图过渡激活前运行 -->
    <script async href="fixup.js" blocking="render"></script>

    < !-- 在看到并完全解析 main-article 元素之前等待，之后再激活过渡 -->
    <link rel="expect" href="#main-article" blocking="render">
  </head>
  <body>
    <header>...</header>
    <main>
      <article id="main-article">...</article>
    </main>
    <article id="secondary-article">...</article>
  </body>
</html>

2.1.3. 自定义

ViewTransition 对象允许通过脚本自定义过渡效果。 同一文档的视图过渡使用从 ViewTransition 调用返回的单个对象，贯穿整个生命周期。 跨文档视图过渡有两个 ViewTransition 对象，一个在旧文档中，一个在新文档中。
2.1.3.1. 处理旧文档中的视图过渡

pageswap 事件在文档即将被卸载并由另一个文档替换的最后时刻触发。可以用它来判断是否即将发生视图过渡，使用 types 自定义过渡，对捕获的元素进行最后的更改，或者在必要时跳过过渡。 PageSwapEvent 接口提供了一个 viewTransition 对象，当导航符合视图过渡条件时，该对象不为空，还有一个 activation 对象，提供有关导航的实用信息，比如重定向后的 URL。 过渡的 finished Promise 可用于在过渡后进行清理，以防文档稍后从 BFCache 恢复。

2.1.3.2. 处理新文档中的视图过渡

pagereveal 事件在呈现新文档的第一帧之前触发。 可以通过查询 viewTransition 属性，判断视图过渡是否仍然有效。 类似于同一文档的视图过渡，作者现在可以选择不同的 types， 对捕获的元素进行最后的更改，等待过渡 ready 后进行动画，或者完全跳过过渡。

2.1.4. 生命周期

此部分为非规范性内容。

成功的跨文档视图过渡经历以下阶段：

  1. 在旧的 文档 中：

    1. 用户通过点击链接、提交表单、按下浏览器的返回按钮等方式发起导航。

      注意：某些导航不会触发视图过渡，例如在地址栏中输入新地址。

    2. 当新 文档 准备好激活时，触发 pageswap 事件。

    3. 如果导航是 同源 的，且没有跨域重定向，并且旧 文档 已选择 加入跨文档视图过渡，则事件的 viewTransition 属性会是一个 ViewTransition 对象。

    4. 作者现在可以自定义过渡，例如通过修改其 types， 或者完全 跳过 它。

    5. 如果没有跳过 ViewTransition， 则旧文档的状态会被 捕获。

    6. 导航继续：旧 文档 被卸载，新 文档 现在处于激活状态。

  2. 然后，在新的 文档 中：

    1. 当新的 文档 准备好进行第一次 渲染机会 时，名为 pagereveal 的事件会在新 文档 上触发， 并带有 viewTransition 属性。

    2. 该 ViewTransition 的 updateCallbackDone promise 已经解析， 它的 捕获的元素 来自旧 文档。

    3. 这是另一个机会，作者可以自定义过渡，例如通过修改其 types， 或者完全 跳过 它。

    4. 新文档的状态会作为过渡的 "新" 状态被 捕获。

    5. 从此时开始，过渡会像同一文档的过渡一样继续进行，按照 激活视图过渡 的方式。

2.2. 示例

要生成与第一个示例中的跨淡入淡出效果相同的效果 CSS 视图过渡 1 § 1.6 示例， 但跨文档，我们不需要 JavaScript。

相反，我们在页面 1 和页面 2 中选择在导航时触发视图过渡：

// 在两个文档中：
@view-transition {
  navigation: auto;
}

从页面 1 到页面 2 的链接将为示例 1 生成一个跨淡入淡出过渡。 要实现示例 2、3 和 4 的效果，只需将伪元素的 CSS 放入两个文档中。

注意，@view-transition 规则可以 与媒体查询一起使用。 例如，这将在屏幕宽度大于以下值时执行过渡： 
@view-transition {
  navigation: auto;
}

@media (max-width: 600px) {
  navigation: none;
}
要实现示例 5，我们必须做几件事：

  • 在两个页面中选择启用导航触发的视图过渡。

  • 通过 sessionStorage 将点击位置传递到新文档中。

  • 在新文档中拦截 ViewTransition 对象，使用 pagereveal 事件。

在两个页面中：

@view-transition {
  navigation: auto;
}

在旧页面中：

addEventListener('click', event => {
  sessionStorage.setItem("lastClickX", event.clientX);
  sessionStorage.setItem("lastClickY", event.clientY);
});

在新页面中：

// 这将在初次加载和从 BFCache 恢复时运行。
addEventListener("pagereveal", async event => {
  if (!event.viewTransition)
    return;

  const x = sessionStorage.getItem("lastClickX") ?? innerWidth / 2;
  const y = sessionStorage.getItem("lastClickY") ?? innerHeight / 2;

  const endRadius = Math.hypot(
    Math.max(x, innerWidth - x),
    Math.max(y, innerHeight - y)
  );

  await event.viewTransition.ready;

  // 为新文档的视图执行动画
  document.documentElement.animate(
    {
      clipPath: [
        `circle(0 at ${x}px ${y}px)`,
        `circle(${endRadius}px at ${x}px ${y}px)`,
      ],
    },
    {
      duration: 500,
      easing: 'ease-in',
      pseudoElement: '::view-transition-new(root)'
    }
  );
})
根据导航的属性选择捕获哪些元素， 以及某些图像是否已加载：

在旧页面中：

window.addEventListener("pageswap", event => {
  // 例如，页面被隐藏，或者导航是跨文档的。
  if (!event.viewTransition)
    return;

    // 如果你不想为前进/后退导航执行视图过渡...
  if (event.activation.navigationType === "traverse") {
    event.viewTransition.skipTransition();
  }

  const newURL = new URL(event.activation.entry.url);
  if (newURL.pathname === "/details" && thumbnail.complete) {
    thumbnail.classList.add("transition-to-hero");

    // 如果页面从 BFCache 恢复，这将清理状态。
    event.viewTransition.finished.then(() => {
      thumbnail.classList.remove("transition-to-hero");
    });
  }

});

在新页面中：

window.addEventListener("pagereveal", event => {
  // 例如，页面被隐藏，导航是跨文档的，或者在旧文档中跳过了过渡。
  if (!event.viewTransition)
    return;

  const oldURL = new URL(navigation.activation.from.url);
  if (newURL.pathname === "/list") {
    event.viewTransition.types.add("coming-from-list");

    // 这将在视图过渡完成之前显示缩略图。
    if (!hero.complete) {
      setToThumbnail(hero);
      event.viewTransition.finished.then(() => {
        setToFullResolution(hero);
      })
    }
  }
});

2.3. 选择加入跨文档视图过渡

2.3.1. 规则 @view-transition

@view-transition 规则用于指示文档应在跨文档导航时设置和激活 视图过渡。

@view-transition 规则的语法如下：

@view-transition {
  <声明列表>
}

@view-transition 规则接受 导航 和 类型 描述符。

注意： 根据默认行为，@view-transition 规则可以嵌套在 条件组规则 中，例如 @media 或 @supports。

当 @view-transition 规则对 文档 document 发生变化时，用户代理必须 更新给定文档的出站过渡的选择状态。

注意： 这需要在布尔值中缓存，因为结果需要在 导航时并行读取。

2.3.2. 描述符 navigation

名称： navigation
适用于： @view-transition
值： auto | none
初始值： none

‘navigation’ 描述符选择自动开始视图过渡，当执行某种类型的导航时。它必须在旧文档和新文档中均存在。

none

将没有过渡。

auto

如果导航是同源的，没有跨源重定向，并且其 NavigationType 为：

  • traverse， 或

  • push 或 replace， 其 用户导航参与 不等于 "browser UI"。

注意： 被排除在 auto 的导航例如，通过 URL 地址栏导航或点击书签，以及任何形式的用户或脚本发起的 reload。

2.3.3. 通过 CSSOM 访问 @view-transition 规则

CSSRule 接口扩展如下：

partial interface CSSRule {
  const unsigned short VIEW_TRANSITION_RULE = 15;
};

CSSViewTransitionRule 表示 @view-transition 规则。

enum ViewTransitionNavigation { "auto", "none" };

[Exposed=Window]
interface CSSViewTransitionRule : CSSRule {
  readonly attribute ViewTransitionNavigation navigation;
  [SameObject] readonly attribute FrozenArray<CSSOMString> types;
};

3. 选择性视图过渡

3.1. 概述

本节为非规范性。

对于简单页面，仅有一个视图过渡，在参与元素上设置 view-transition-name 属性应该就足够了。 然而，在更复杂的场景中，作者可能希望声明各种视图过渡，并且只同时运行其中一个。 例如，在点击导航栏时滑动整个页面，以及在拖动列表中的一项时对列表进行排序。

为了确保每个视图过渡仅捕获它所需的内容，并且不同的过渡不会相互干扰， 本规范引入了 活动类型 的概念，配合 :active-view-transition 和 :active-view-transition-type() 伪类。

:active-view-transition 匹配具有 文档元素 的 活动视图过渡，而 :active-view-transition-type() 匹配 文档元素，如果选择器中的类型与 活动视图过渡 的 活动类型 匹配。

ViewTransition 的 活动类型 通过以下方式之一填充：

  1. 作为参数传递给 startViewTransition(callbackOptions)

  2. 随时突变，使用过渡的 types

  3. 使用 types 描述符声明跨文档视图过渡。

3.2. 示例

例如，开发者可以以以下方式开始一个过渡： 
document.startViewTransition({update: updateTheDOMSomehow, types: ["slide-in", "reverse"]});

这将激活以下选择器中的任何一个：

:root:active-view-transition-type(slide-in) {}
:root:active-view-transition-type(reverse) {}
:root:active-view-transition-type(slide-in, reverse) {}
:root:active-view-transition-type(slide-in, something-else) {}
:root:active-view-transition {}

而不提供过渡类型启动过渡，仅会激活 ':active-view-transition'：

document.startViewTransition(updateTheDOMSomehow);
// 或
document.startViewTransition({update: updateTheDOMSomehow});

/* 这将处于活动状态 */
:root { }
:root:active-view-transition {}

/* 这将不会处于活动状态 */
:root:active-view-transition-type(slide-in) {}
:root:active-view-transition-type(any-type-at-all-except-star) {}

3.3. 根据活动视图过渡进行选择

3.3.1. 活动视图过渡 :active-view-transition 伪类

:active-view-transition 伪类适用于文档的根元素，如果它具有 活动视图过渡。

特异性 的 :active-view-transition 是一个伪类选择器。

:active-view-transition 伪类在 文档元素 匹配，当其 节点文档 有一个非空的 活动视图过渡。

3.3.2. 活动视图过渡类型 :active-view-transition-type() 伪类

:active-view-transition-type() 伪类适用于文档的根元素，如果它具有匹配的 活动视图过渡。它的语法定义如下：

:active-view-transition-type(<custom-ident>#)

特异性 的 :active-view-transition-type() 是一个伪类选择器。

:active-view-transition-type() 伪类在 文档元素 匹配，当其 节点文档 有一个非空的 活动视图过渡，其 活动类型 包含 至少一个 <custom-ident> 参数。

3.4. 更改正在进行的视图过渡的类型

ViewTransition 接口扩展如下：

[Exposed=Window]

4. 在视图过渡伪元素之间共享样式

4.1. 概述

本节为非规范性。

在DOM中以相似方式样式化多个元素时，通常使用 class 属性： 设置一个在多个元素之间共享的名称，然后使用 类选择器 来声明共享样式。

视图过渡伪元素（例如 view-transition-group()）不是在DOM中定义的，而是通过使用 view-transition-name 属性定义。 为此，view-transition-class CSS属性为视图过渡提供了HTML等效的 类。 当一个具有 view-transition-name 的元素同时具有 view-transition-class 值时，该类可以被伪元素选择，如 示例 所示。

4.2. 示例

此示例创建一个过渡，每个方块在其自己的名称下参与，同时为所有方块的动画应用 1秒的持续时间： 
<div class="box" id="red-box"></div>
<div class="box" id="green-box"></div>
<div class="box" id="yellow-box"></div>

div.box {
  view-transition-class: any-box;
  width: 100px;
  height: 100px;
}
#red-box {
  view-transition-name: red-box;
  background: red;
}
#green-box {
  view-transition-name: green-box;
  background: green;
}
#yellow-box {
  view-transition-name: yellow-box;
  background: yellow;
}

/* 以下样式将适用于所有方块，得益于 'view-transition-class' */
::view-transition-group(
  *.any-box)
{
  animation-duration: 1s;
}

4.3. view-transition-class 属性

名称： view-transition-class
值： none | <custom-ident>+
初始： none
适用： 所有元素
继承： 否
百分比： 不适用
计算值： 如指定
规范顺序： 按语法
动画类型： 离散

view-transition-class 可用于将相同的 样式规则应用于多个 命名的视图过渡伪元素，这些伪元素可能 有不同的 view-transition-name。 虽然 view-transition-name 用于 匹配旧状态中的元素与新状态中的相应元素之间的关系，view-transition-class 仅用于 使用视图过渡伪元素应用样式 （::view-transition-group()， ::view-transition-image-pair()， ::view-transition-old()， ::view-transition-new()）。

请注意，view-transition-class 本身并不会标记一个元素 以供捕获，它仅作为样式应用于已经具有 view-transition-name 的额外方式。

none

没有类会应用于该元素生成的命名视图过渡伪元素。

<custom-ident>+

所有指定的 <custom-ident> 值（除了 none）在使用 命名视图转换伪元素 选择器时应用。none 是无效的 <custom-ident>，用于 view-transition-class，即使与另一个 <custom-ident> 结合使用。

注意： 如果在转换的旧状态和新状态中都为一个元素指定了相同的 view-transition-name， 则只有来自新状态的 view-transition-class 值适用。这同样适用于跨文档视图转换： 旧文档中的类仅在新文档中未指定其对应的 view-transition-name 时适用。

4.4. 命名视图过渡伪元素的扩展

命名视图过渡伪元素（view-transition-group()、view-transition-image-pair()、view-transition-old()、view-transition-new()）的语法扩展如下：

::view-transition-group(<pt-name-and-class-selector>)
::view-transition-image-pair(<pt-name-and-class-selector>)
::view-transition-old(<pt-name-and-class-selector>)
::view-transition-new(<pt-name-and-class-selector>)

其中<pt-name-selector>的作用与之前定义的一样，而<pt-name-and-class-selector>具有以下语法定义：

<pt-name-and-class-selector> = <pt-name-selector> <pt-class-selector>? | <pt-class-selector>
<pt-class-selector> = ['.' <custom-ident>]+

在解释上述语法时，禁止出现空格：

  • 在<pt-name-selector>和<pt-class-selector>之间

  • 在<pt-class-selector>的任何组件之间。

具有一个或多个<custom-ident>值的命名视图过渡伪元素的选择器，只有在伪元素的view-transition-name的命名元素中，类列表值包含所有这些值时，才能匹配一个元素。

特异性具有以下任一条件的命名视图过渡伪元素的选择器：

  • 一个带有<pt-name-selector>和一个<custom-ident>；或

  • 一个带有至少一个<pt-class-selector>的<custom-ident>，

相当于一个类型选择器。

一个带有特异性的命名视图过渡伪元素的选择器，带有一个*参数，并且带有一个空的<pt-class-selector>的特异性为零。

5. 扩展 document.startViewTransition()

dictionary StartViewTransitionOptions {
  UpdateCallback? update = null;
  sequence<DOMString>? types = null;
};

partial interface Document {

  ViewTransition startViewTransition(optional (UpdateCallback or StartViewTransitionOptions) callbackOptions = {});
};

方法步骤 for startViewTransition(callbackOptions) 如下：

  1. 让 updateCallback 为 null。

  2. 如果 callbackOptions 是一个 UpdateCallback， 将 updateCallback 设置为 callbackOptions。

  3. 否则，如果 callbackOptions 是一个 StartViewTransitionOptions， 则将 updateCallback 设置为 callbackOptions 的 update。

  4. 如果 this 的 活动视图过渡 不为 null 且其 出站后捕获步骤 不为 null， 则：

    1. 让 preSkippedTransition 为一个新的 ViewTransition 在 this 的 相关领域，其 更新回调 为 updateCallback。

      注意： preSkippedTransition 的 types 在这里被忽略，因为过渡从未被激活。

    2. 跳过 preSkippedTransition，并抛出一个 "InvalidStateError" DOMException。

    3. 返回 preSkippedTransition。

    注意： 这确保在触发 pageswap 后启动的同一文档过渡会被跳过。

  5. 让 viewTransition 为运行 方法步骤 的结果，针对 startViewTransition(updateCallback) ，给定 updateCallback。

  6. 如果 callbackOptions 是一个 StartViewTransitionOptions， 则将 viewTransition 的 活动类型 设置为 clone 的 types 作为一个 集合。

  7. 返回 viewTransition。

6.2. 解析 @view-transition 规则

要为一个 Document document 解析 @view-transition 规则：

注意： 这在旧文档和新文档中都会调用。

  1. 如果 document 的 可见性状态 为 "hidden"， 则返回 "skip transition"。

  2. 让 matchingRule 为 document 中的最后一个 @view-transition 规则。

  7. 如果 typesDescriptor 的 计算值 为 none，则返回一个 集合 « »。

  8. 返回一个 集合 的字符串，对应于 typesDescriptor 的 计算值。

注意： 这在导航期间被调用，可能 并行进行。

  1. 如果用户代理决定显示一个 实现定义的 导航体验，例如基于手势的后退导航过渡， 则用户代理可能会忽略作者定义的视图过渡。如果是这种情况，则返回 false。

为了 设置 跨文档视图过渡，给定一个 Document oldDocument，一个 Document newDocument，以及一个接受无参数的算法 proceedWithNavigation：

注意： 这来自HTML规范的调用。

  1. 断言：这些步骤作为 任务 的一部分在 oldDocument 上排队运行。

    注意： 我们还不知道 newDocument 是否选择加入，因为它可能尚未被解析。 我们在触发 pagereveal 事件时检查 newDocument 的选择加入状态。

安全考虑

为了防止跨源问题，此时只能为同源导航启用跨文档视图过渡。如在 WICG/view-transitions#200 中讨论的， 这仍然存在两个潜在威胁：

  1. 两个文档中的 跨源隔离能力 可能不同。 这可能导致一个 文档 在 跨源隔离 的情况下能够读取来自非跨源隔离文档的图像数据。这已经在 [[css-view-transitions-1#sec] 中缓解，因为对捕获的跨源 iframe 也适用相同的限制。

  2. 同源导航仍可能通过跨源重定向发生，例如， https://example.com 链接到 https://auth-provider.com/，然后重定向回 https://example.com/loggedin。

    这可能导致一个（小）情况，即跨源方会将用户重定向到意外的第一方 URL，从而导致意外的过渡并掩盖存在重定向的事实。 为了缓解这一点，目前如果 文档 是通过跨源重定向创建的，则禁用视图过渡。 请注意，当 文档 被 重新激活 时，此检查不适用，因为在这种情况下跨源重定向已经发生。

    注意：这仅适用于服务器端重定向。客户端重定向，例如使用 [^meta/http-equiv/refresh^]，等同于新的导航。

  3. 此功能向 CSS 暴露了更多信息，因为到目前为止 CSS 并不知道任何与导航相关的内容。 这可能引发关于安全的第三方 CSS 的担忧。然而，作为一般规则，第三方样式表应该来自可信来源，因为 CSS 可以以多种方式了解文档或改变文档。

请参见 问题 #8684 和 WICG/view-transitions#200 以获取详细讨论。

部分实现

为了使作者能够利用向前兼容的解析规则来分配后备值，CSS 渲染器 必须 将任何无可用支持级别的 at-rules、属性、属性值、关键字和其他语法构造视为无效（并且 适当地忽略）。特别是，用户代理 必须不 有选择性地忽略不受支持的组件值，而在单个多值属性声明中尊重受支持的值：如果任何值被视为无效（由于不受支持的值必须如此），CSS 要求整个声明被忽略。

不稳定和专有特性的实现

为了避免与未来稳定 CSS 特性发生冲突，CSS 工作组建议在实现不稳定特性和专有扩展时，遵循最佳实践。

非实验性实现

一旦规范达到候选推荐阶段，非实验性实现就可能出现，实施者应发布任何能够证明根据规范正确实现的 CR 级特性的无前缀实现。

为了建立和维护 CSS 在各个实现之间的互操作性，CSS 工作组请求非实验性 CSS 渲染器在发布任何 CSS 特性的无前缀实现之前，向 W3C 提交实施报告（如果必要，还需提交用于该实施报告的测试用例）。提交给 W3C 的测试用例将接受 CSS 工作组的审查和修正。

有关提交测试用例和实施报告的更多信息，可以访问 CSS 工作组网站 https://www.w3.org/Style/CSS/Test/。如有问题，请联系public-css-testsuite@w3.org 邮件列表。
